## 概要

`docs/adr/` 配下 21 ファイル（`retired/` 除外・README.md + ADR-0101〜ADR-0128 の現行 20 件）に対する japanese-tech-writing（ADR 適用サブセット: 整形・LLM 表現禁止・冗長排除・論証の厳密さ・用語政策）および document-type-responsibilities.md（配置基準・用語政策）の逸脱を抽出したカタログ。個別修正は後続 Issue で扱う（本カタログは見える化が目的）。遡及準拠の範囲は REQ-0140-026 に基づく。

## 検出事項

### F-001（high）: 関連 ADR 箇条書きでの em-dash（—）使用（パターン）

- **対象**: `README.md` L36-53, L67-97 / `ADR-0101.md` L47-48 / `ADR-0102.md` L35-37 / `ADR-0103.md` L56-58 / `ADR-0104.md` L42-46 / `ADR-0105.md` L46-49 / `ADR-0106.md` L34-36 / `ADR-0107.md` L41-42 / `ADR-0108.md` L40-41 / `ADR-0109.md` L48-49 / `ADR-0110.md` L49-51 / `ADR-0111.md` L64-66 / `ADR-0112.md` L86-89 / `ADR-0114.md` L70-71 / `ADR-0123.md` L83-84 / `ADR-0125.md` L89-96
- **該当規範**: 整形（ダッシュを日本語の地の文・見出しで使わない。同格・補足の挿入は括弧（）に）
- **現状**: 「関連 ADR」「References」節の箇条書きで em-dash による同格・補足挿入が広く浸透している。
  - 例（`README.md` L36-37）: `- [ADR-0101](ADR-0101.md) — AgentDevFlow プラグイン名前空間の統一`
  - 例（`ADR-0101.md` L47）: `- ADR-0005（廃止 — 本判断の前身）`
  - 例（`ADR-0114.md` L70）: `- ADR-0112（サブエージェント委譲の一般化）— 本判断は ADR-0112の一般原則の特定適用である。`
- **改善方向**: 同格・補足の挿入は括弧（全角）に置き換える。パターン単位で機械的修正可能。
  - 例: `- [ADR-0101](ADR-0101.md)（AgentDevFlow プラグイン名前空間の統一）`
  - 例: `- ADR-0005（廃止、本判断の前身）`
- **補足**: 16 ファイルにわたり同一パターン。一括変換スクリプト等で対処可能。リポジトリ全体の表記揺れ是正 Issue として扱う価値がある。

### F-002（med）: 空虚な形容「包括的」の使用（パターン）

- **対象**: `ADR-0103.md` L13 / `ADR-0104.md` L15, L31 / `ADR-0125.md` L13（前後）/ `ADR-0126.md` L44（前後）
- **該当規範**: LLM 表現禁止（空虚な形容「包括的」）
- **現状**:
  - `ADR-0103.md` L13: 「文書種別全体の責務境界を**包括的**に宣言していなかった」
  - `ADR-0104.md` L15: 「実行時独立性の**包括的**な判断として統合する必要があった」
  - `ADR-0104.md` L31: 「**包括的**な判断として不十分」
- **改善方向**: 「包括的」は何をどう網羅したかを書かない LLM 口調。具体的な対象範囲に置き換える。
  - 「文書種別全体の責務境界を一括して宣言していなかった」
  - 「実行時独立性について、複数の個別宣言を統合した判断が必要だった」
  - 「個別宣言のままでは判断として不十分」

### F-003（med）: 中黒（・）による日本語並列（パターン）

- **対象**: `README.md`、`ADR-0102.md`、`ADR-0103.md`、`ADR-0104.md`、`ADR-0107.md`、`ADR-0112.md`、`ADR-0123.md`、`ADR-0125.md`、`ADR-0126.md` 等（広範）
- **該当規範**: 整形（中黒を日本語の並列で使わない）
- **現状例**:
  - `ADR-0102.md` L35: 「実行時・編集時の関心分離」
  - `ADR-0107.md` L41: 「コマンド・スキル・テンプレート・スクリプト責任分界」
  - `ADR-0125.md` L13: 「子Issue単位オーケストレーションモデル（SC-007）において子Issue を1件ずつ順次選択し case-run に委譲する設計だった」
- **改善方向**: 日本語並列には読点（、）またはスラッシュ／箇条書きを使用する。固定された複合名詞内部（「実行時・編集時」等の確定 tech term）は許容範囲もあるが、3 つ以上の並列や流動的な並列は読点に統一する方針が望ましい。リポジトリ全体方針として横断 Issue で扱うことを推奨。

### F-004（med）: ADR-0126 の長文複文

- **対象**: `ADR-0126.md` L44
- **該当規範**: 読み手の負荷の管理（抽象的な言い回しの指す内容が文脈から一意に決まらないときは同格挿入で特定する）・整形（一文一行）
- **現状**: 「本 ADR は `agentdev-adr-guidelines` の False Negative 防止基準（SKILL.md lines 49-59）に基づく。「ADRを作成してはならない条件」（ディレクトリ規約・アーティファクト契約・入出力形式等）に表面的には該当するが、`src/opencode-local/` の3層ソースモデル導入と `generated_by`/ジャンクション検出の安全性制約は「将来の設計・運用・文書システムを制約する決定」を含むため、例外として ADR を認める。」
- **改善方向**: 長文複文を条件・判断・結論に分割する。
  - 「本 ADR は `agentdev-adr-guidelines` の False Negative 防止基準（SKILL.md lines 49-59）に基づく。」
  - 「ディレクトリ規約・アーティファクト契約・入出力形式といった『ADR を作成してはならない条件』に表面的には該当する。」
  - 「ただし `src/opencode-local/` の 3 層ソースモデル導入と `generated_by` /ジャンクション検出の安全性制約は、将来の設計・運用・文書システムを制約する決定を含む。」
  - 「そのため例外として ADR を認める。」

### F-005（low）: ADR-0125 の長文（順次モデル導入背景）

- **対象**: `ADR-0125.md` L13
- **該当規範**: 整形（一文一行）・読み手の負荷の管理
- **現状**: 「AgentDevFlow の case-auto は、子Issue単位オーケストレーションモデル（SC-007）において子Issue を1件ずつ順次選択し case-run に委譲する設計だった（REQ-0114-045, G15）。この順次モデルは、かつて存在した Epic Orchestrator Contract（親エージェントによる Wave 一括実行）を置き換えるものとして導入された。」
- **改善方向**: 1 文目が長く、修飾構造が読みづらい。背景と現状を 2 文に分ける。

### F-006（info）: ADR 本文の規範準拠度（参考記録）

- **対象**: ADR-0101〜ADR-0128 の本文（Decision / Consequences / Rationale 節）

- **改善方向**: （対応不要。検出事項の見える化が目的なため準拠状況も記録する）
