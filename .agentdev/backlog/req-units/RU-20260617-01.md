# REQ/SPEC責務境界とREQ粒度基準の明確化・運用維持

## 背景

本セッションでは、REQ-0108 を中心に、active REQ の要件行が過剰に細かい、schema / enum / path / filename / template 名 / fixture 条件 / checker 詳細 / 過去の作業履歴を要件化している、という問題を確認した。

議論の結果、問題は REQ-0108 単体ではなく、REQ と SPEC の関心境界、および REQ 要件行の粒度基準が active REQ 全体に十分適用されていないことにあると整理した。

ユーザーは、REQ を「満たしたい状態・主な関心」、SPEC を「その状態を実現するための現在仕様・schema・判定表・パラメータ等」と理解しており、この理解は大枠で合意済みである。ただし、境界は画一的ではなく、特定の値・分類・パラメータが外部契約または安定した境界条件そのものである場合は、REQ に要約として残し、詳細定義を SPEC に置く方針とした。

## 問題

active REQ には、以下の問題が混在している。

1. 要件行が過剰に小さく、複数行が本来 1 つの状態要件または外部契約に統合できる。
2. 自明な再説明、既存要件の言い換え、責務境界を増やさない補足が要件行として残っている。
3. schema field、enum値、route/category/status の詳細、draft frontmatter、template variant、file pattern 等のパラメータ的内容が REQ 要件行として定義されている。
4. docs-check / inspect-docs / inspect-skills 等の個別検査ルール、fixture detail、false positive 抑制方式、検出語彙、checker 実装構造が REQ に混入している。
5. Step番号、Phase番号、retry回数、token目安、行数上限、具体的なコマンド内部手順が REQ に混入している。
6. Case / RU / Issue / PR / OU 由来の作業履歴、移行完了記録、改名・削除・移管の経緯が active REQ 本文または Update Notes に過剰に残っている。
7. 「パラメータだからSPECへ」と機械的に判断すると、work_type、domain state、公開command名、出力先、side effect境界など、REQに残すべき外部契約まで失うリスクがある。

## 統合理由

本RUは、特定REQの単純な圧縮ではなく、REQ体系全体の品質維持基準を整えるためのRUである。

REQ-0108 の肥大化は代表例だが、同種の問題は REQ-0102、REQ-0103、REQ-0104、REQ-0107、REQ-0109、REQ-0112、REQ-0114、REQ-0119、REQ-0123、REQ-0124、REQ-0125〜0133 にも見られる。したがって、個別REQごとの場当たり修正ではなく、以下を一体で扱う必要がある。

- REQ/SPEC責務境界の明文化
- REQ要件行の粒度基準の明文化
- 例外判定としての「安定契約ならREQに残す」基準
- req-define / req-save / inspect-docs / docs-check による継続的な運用
- 既存 active REQ の再分類・再構成に向けた判断基準

## 要件化の方向

### 1. REQ/SPEC境界基準を明文化する

REQ は、満たすべき状態・振る舞い・制約・外部契約を記述する文書とする。

SPEC は、その要件を現在構成として成立させる schema、lifecycle、command topology、rule catalog、fixture detail、判定表、enum、format、内部パラメータを記述する文書とする。

ただし、特定の値・分類・パラメータが外部契約または安定した境界条件そのものである場合は、REQ に要約として保持し、詳細定義を SPEC に配置する。

### 2. REQ要件行の粒度基準を定義する

REQ要件行は、以下を満たすものに限定する。

- 変更後または現行で満たすべき状態・振る舞い・制約である。
- 外部から見える責務、command間契約、domain state、公開入口、安全境界、恒久的な文書体系制約を表している。
- その行が存在しないと、対象が何を満たすべきか不明になる。
- 検証可能であり、単なる作業手順、履歴、移行記録、自明な再説明ではない。

### 3. SPEC等へ移す基準を定義する

以下は原則として REQ 要件行から外し、適切な詳細層へ配置する。

- schema field
- enum値一覧
- route / category / status の詳細判定表
- file pattern
- template variant
- report format
- fixture detail
- regression test 条件
- checker個別ルール
- false positive 抑制方式
- retry回数、token目安、行数上限等の内部パラメータ
- Step番号、Phase番号、内部アルゴリズム
- 具体的な migration / rename / delete / move / update 作業履歴
- Case / RU / Issue / PR / OU 由来の作業記録

### 4. 例外判定を明示する

パラメータ的に見える内容でも、以下に該当する場合は REQ に要約として残せる。

- 公開command名や公開入口である。
- domain state の位置づけを定義している。
- 他commandとの接続契約である。
- 利用者に見える分類体系である。
- 安全境界または停止条件の大枠である。
- 後続工程が依存する安定した外部契約である。

ただし、詳細な値一覧、判定表、内部処理条件は SPEC 等へ配置する。

### 5. docs-check / inspect-docs による継続運用を追加する

docs-check は、active REQ 要件行に schema / field / enum / fixture / checker detail / Step番号 / Phase番号 / 作業履歴 / 自明な再説明が混入していないかを検出できること。

inspect-docs は、REQ体系の SPLIT / MERGE / MOVE / DUPLICATE / RETIRE / DRIFT 観点に加え、REQ/SPEC境界違反、REQ粒度過小、SPEC detail混入、外部契約と内部パラメータの誤分類を診断できること。

req-define / req-save は、新規または更新される要件行について、REQに残すべき外部契約か、SPEC等へ配置すべき詳細かを保存前に判定できること。

## 対象外

本RUでは以下を対象外とする。

- 各REQ要件行の削除・移管・統合をこのRU内で確定すること。
- 前回抽出した問題候補をそのまま全削除または全SPEC移管として扱うこと。
- REQ-0108以外を含む active REQ 全件の具体的な差分作成。
- 新規SPECファイル名の確定。
- 個別checkerの実装。
- integrity rule の具体的な正規表現。
- command / skill / reference の実装変更。
- Issue / PR 作成。
- 過去履歴の完全削除方針の確定。
- retired REQ / retired ADR の大規模再編集。

## 実際の対処例

### 例1: finding schema

不適切なREQ粒度:

- finding は `level`, `category`, `route`, `file`, `line`, `evidence`, `expected`, `message` を持つこと。
- finding category は `document-drift`, `broken-reference`, ... のいずれかであること。

改善後のREQ要約:

- docs-check は、後続処理が機械的に扱える構造化 finding を出力すること。

詳細配置:

- field一覧、必須/任意、category enum、route enum、report schema は `docs/specs/integrity-contracts.md` に配置する。

### 例2: docs-check個別ルール

不適切なREQ粒度:

- retired REQ の filename と frontmatter id の一致を検査すること。
- mapping-table の canonical path を検査すること。
- command frontmatter の許可fieldを検査すること。

改善後のREQ要約:

- docs-check は、REQ / ADR / SPEC / command / skill / template の整合性を、rule catalog に基づき検査すること。

詳細配置:

- 個別検査ルール、検出条件、severity、例外条件は `docs/specs/integrity-rule-catalog.md` に配置する。

### 例3: work_type

機械的にSPEC送りにしてはいけない例:

- work_type が `bugfix / feature / maintenance / docs_chore` の4種であること。

理由:

- これは単なるenumではなく、ユーザー、req-define、case-open、case-auto、case-run の間で共有される安定したworkflow契約である。

REQに残す形:

- AgentDevFlow workflow は、work_type に基づいて後続工程を分類すること。標準 work_type は `bugfix / feature / maintenance / docs_chore` の4種であること。

SPECへ移す形:

- work_type + scale から workflow_route を導出する詳細表。
- 各 work_type のtemplate variant、label、draft消費パターン。

### 例4: retry回数

不適切なREQ粒度:

- squash merge 失敗時、5秒間隔で最大5回自動リトライすること。

改善後のREQ要約:

- case-close は、一時的な merge 失敗に対して安全な再試行と、失敗時のユーザー判断へのフォールバックを提供すること。

詳細配置:

- retry間隔、最大回数、対象エラー、fallback手順は command reference または workflow contract に配置する。

### 例5: Update Notes / Case構成 / 受入条件

不適切なREQ状態:

- active REQ 本文に Case構成、AC、移行元行番号、PR/RU/Issue由来の作業履歴が残っている。

改善後:

- active REQ には現在満たすべき状態・制約・外部契約を残す。
- 過去のCase/RU/Issue/PR由来の履歴は、必要な場合のみ retired / mapping / changelog / Issue / PR に保持する。
- Update Notes は最小限にし、現行要件の理解に必要な範囲に限定する。

## 受け入れ条件

1. REQ/SPEC責務境界として、REQが満たすべき状態・振る舞い・制約・外部契約を記述し、SPECがschema・lifecycle・command topology・rule catalog・fixture detail・判定表・enum・format・内部パラメータを記述する方針が明文化されている。
2. パラメータ的内容でも、外部契約または安定した境界条件である場合は REQ に要約を残せる、という例外基準が明文化されている。
3. REQ要件行の粒度基準として、過小要件、自明な再説明、既存要件の言い換え、作業履歴、schema/detail混入を除外する判断基準が定義されている。
4. req-define / req-save が、新規または更新される要件行について、REQに残すべき内容かSPEC等へ配置すべき内容かを保存前に判定できる。
5. docs-check が、REQ要件行への schema / field / enum / fixture / checker detail / Step番号 / Phase番号 / 作業履歴 / 自明な再説明の混入を検出できる。
6. inspect-docs が、REQ/SPEC境界違反、REQ粒度過小、SPEC detail混入、外部契約と内部パラメータの誤分類を診断観点として扱える。
7. REQ-0108 は docs-check の外部契約を中心に再構成され、finding schema、個別checker、fixture、false positive抑制、rule catalog詳細は適切なSPECまたはrule catalogへ分離される。
8. active REQ 全件の再構成は、前回抽出した問題候補をそのまま削除・移管するのではなく、外部契約 / 状態要件 / 安定境界 / 内部仕様 / schema / 実装手順 / 履歴に再分類して判断される。
9. 具体的な要件行の移管先が未確定の内容は、REQ本文に確定事項として混入していない。
