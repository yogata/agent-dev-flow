# project-extensions化とcommand/skillのADR/REQ/SPEC具体参照禁止

## 背景

AgentDevFlow の command / skill は配布対象である。
ADR / REQ / SPEC の具体ID、具体パス、固定URLは、その文書を持つプロジェクト内でしか解決できない。

そのため、配布対象 command / skill 本文に ADR / REQ / SPEC への具体参照を持たせると、AgentDevFlow 適用先プロジェクトで解決不能な参照が混入する。

一方で、各プロジェクトでは、AgentDevFlow が提供する標準 command / skill に対して、プロジェクト固有の文脈、規約、検査、受け入れゲート、禁止事項を追加したい。
この追加・拡張は、標準 command / skill 本文へ直接書き込むのではなく、プロジェクト側の設定層で与える必要がある。

## 問題

現行の `.agentdev/doc-inputs/**` は、command / skill が読む docs の参照リストとしての意味が強い。
しかし必要なのは、単なる参照リストではなく、AgentDevFlow が提供する command / skill をプロジェクト固有に追加・拡張する設定層である。

また、command / skill 本文に ADR / REQ / SPEC の具体参照を禁止するルールを徹底するには、次を分離する必要がある。

- 標準 command / skill 本文は、プロジェクト非依存の共通手順だけを持つ。
- プロジェクト固有の参照、規約、検査は `.agentdev/extensions/**` に置く。
- `.agentdev/extensions/**` は、そのプロジェクトの ADR / REQ / SPEC を参照してよい。
- command / skill 本文では、ADR / REQ / SPEC の具体ID、具体パス、固定URLを禁止する。
- REQ / ADR / SPEC 本文内の相互参照は許容する。ただし今回の主題ではない。

## 統合理由

本件は、以下の複数論点が同一の設計境界に属するため、1つの Requirement Unit として扱う。

- 配布対象 command / skill 本文からプロジェクト固有参照を排除する境界。
- `.agentdev/doc-inputs/**` を参照リストから project extension に再定義する変更。
- AgentDevFlow 適用プロジェクトごとに異なるガードレールを注入する仕組み。
- command / skill 実行時に対応 extension だけを読む契約。
- extension 定義、保守、検査のための標準 skill / command の追加。
- agent-dev-flow 本体における ADR / REQ / SPEC 具体参照禁止の持続的検査。

これらを分離して要件化すると、command / skill の参照境界と project extension の責務が再び分散する。
したがって、配布物参照境界と project extension 機構を一体として要件化する。

## 要件化の方向

### 1. project-extensions の定義

`docs/specs/foundations/project-doc-inputs.md` を `docs/specs/foundations/project-extensions.md` に置き換え、`.agentdev/extensions/**` を次のように定義する。

`.agentdev/extensions/**` は、AgentDevFlow が提供する標準 command / skill に対して、プロジェクト固有の文脈、規約、検査、受け入れゲート、禁止事項を追加・拡張する設定層である。

標準配置は以下とする。

```text
.agentdev/extensions/commands/<command>.yaml
.agentdev/extensions/skills/<skill>.yaml
```

### 2. extension の基本構造

extension は以下の基本構造を持つ。

```yaml
version: 1
kind: command-extension
id: /agentdev/<command>

context: []
rules: []
checks: []
acceptance_gates: []
must_not: []
```

skill extension は `kind: skill-extension` とし、`id` は対象 skill を識別する値とする。

各セクションの意味は以下とする。

| セクション | 意味 |
|---|---|
| `context` | command / skill に追加で与える文脈 |
| `rules` | command / skill に追加で守らせる規約 |
| `checks` | command / skill に追加で実行させる検査 |
| `acceptance_gates` | command / skill extension が追加する実行完了前ゲート |
| `must_not` | command / skill に追加で課す禁止事項 |

`acceptance_gates` は REQ の受け入れ条件ではなく、case-close / QG 本体でもない。
あくまで command / skill extension によって追加される実行完了前ゲートである。

### 3. 実行時読み込み契約

command / skill は、実行時に自分に対応する extension だけを読む。

- command は `.agentdev/extensions/commands/<command>.yaml` を対象とする。
- skill は `.agentdev/extensions/skills/<skill>.yaml` を対象とする。
- 対応 extension が存在しない場合は、標準動作で続行する。
- 対応 extension が破損している場合は、エラーを表示し、当該 extension を無視して標準動作で続行する。
- extension は標準 command / skill の上書きではなく、追加・拡張としてのみ扱う。

### 4. 標準 skill の追加

標準 skill `agentdev-project-extensions` を追加する。

`agentdev-project-extensions` は以下を担う。

- 対応 extension ファイルの探索。
- extension 不在時の空 extension 扱い。
- extension 破損時のエラー表示と無視。
- `context`, `rules`, `checks`, `acceptance_gates`, `must_not` の読み取り。
- extension が標準 command / skill の上書きではなく追加・拡張であることの扱い。
- `rules` / `checks` から project-local skill 委譲対象を抽出すること。

### 5. 保守用 command の追加

保守用 command `/agentdev/inspect-extensions` を追加する。

`inspect-extensions` は以下を担う。

- `.agentdev/extensions/**` の一覧化。
- extension YAML の構造確認。
- `kind` と配置の整合確認。
- `id` と対象 command / skill の対応確認。
- `context.paths` の実在確認。
- `rules.skill` / `checks.skill` に記述された project-local skill の存在確認。
- 旧 `.agentdev/doc-inputs/**` の残存検出。
- extension が標準 command / skill の上書きとして記述されていないことの確認。

### 6. project-local skill 委譲

`rules` / `checks` は、`skill:` に具体的な project-local skill 名を記述し、その skill に実行を委譲する。

初期契約では `action`, `required`, `fail_on` は採用しない。
呼び出された skill は、extension entry の `id`, `when`, `skill` および周辺文脈をもとに判断する。

例:

```yaml
rules:
  - id: no-doc-id-in-command-skill
    when: command または skill を変更する場合
    skill: repo-agentdev-runtime-boundary

checks:
  - id: runtime-reference-guard
    when: command または skill を変更した場合
    skill: repo-agentdev-runtime-boundary
```

### 7. command / skill 本文の参照禁止

command / skill 本文には、ADR / REQ / SPEC の具体ID、具体パス、固定URLを記述しない。

禁止対象は文書種別名としての `ADR`, `REQ`, `SPEC` ではなく、プロジェクト固有文書を直接指す具体参照である。

`.agentdev/extensions/**` は、そのプロジェクトの ADR / REQ / SPEC を参照してよい。
REQ / ADR / SPEC 本文内の参照も許容する。

### 8. agent-dev-flow 本体での持続的検査

agent-dev-flow 本体では、project-local integrity skill により、配布対象 command / skill 本文に ADR / REQ / SPEC の具体ID、具体パス、固定URLが残っていないことを検査する。

この検査は `.agentdev/extensions/**` 内の ADR / REQ / SPEC 参照を許可対象として扱う。

## 主対象REQまたは変更対象候補

主な変更対象候補は以下とする。

- `docs/specs/foundations/project-doc-inputs.md`
  - `docs/specs/foundations/project-extensions.md` へ置き換える。
- `.agentdev/doc-inputs/**`
  - `.agentdev/extensions/**` へ置き換える。
- command / skill の project doc-inputs 読み込み記述
  - project extensions 読み込み記述へ置き換える。
- 標準 skill
  - `agentdev-project-extensions` を追加する。
- 標準 command
  - `/agentdev/inspect-extensions` を追加する。
- agent-dev-flow project-local integrity skill
  - command / skill 本文の ADR / REQ / SPEC 具体参照禁止を検査する責務を持たせる。
- 既存 integrity / docs check 系
  - `.agentdev/extensions/**` を許可領域として扱えるようにする。

## 対象外

以下は本 RU の対象外とする。

- 標準 command / skill の責務を extension から上書きすること。
- 標準ワークフロー状態の追加。
- 標準ガードレールの無効化。
- `action`, `required`, `fail_on` の初期契約への追加。
- REQ / ADR / SPEC 本文内の参照方針の再設計。
- REQ / ADR / SPEC 間参照の推奨可否の詳細設計。
- 具体的な実装手順、コード差分、移行スクリプトの詳細。
- Issue 作成、PR 作成。
- 会話URLの記録。
