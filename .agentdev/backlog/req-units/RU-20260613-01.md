# RU: 複数RU処理における operation unit queue と Issue/Epic 単位分離

## 背景

複数の RU を `req-define` にまとめて渡す運用では、現状でも要件整理自体は実行できている。

しかし、`req-define` の実施単位が、そのまま Issue または Epic Issue の単位に転写されると、意味的に関係しない RU 群まで 1 つの大きな Epic に畳まれる。これにより、Epic Issue が実行制御単位ではなく、単なる `req-define` 実行単位の入れ物になり、Issue / PR / close の粒度が不自然になる。

一方で、関係する RU 群をまとめること自体は許容する。結果として 1 つの大きな作業単位になる場合は、Epic Issue を使ってよい。

## 問題

現在の運用では、複数 RU 入力、複数 REQ 操作、複数 Issue、Epic Issue の境界が曖昧である。

特に以下が問題である。

- `req-define` の入力単位と、Issue / Epic Issue の発行単位が混同される
- 複数RUを渡しただけで Epic Issue 化に寄る
- OU 間の依存関係が、Epic 化の条件として扱われる可能性がある
- `case-auto` が複数操作単位を逐次 close まで完了するための queue 制御を持たない
- 複数RU処理の進行が、draft / queue ではなくセッション中の推論に依存する

## 統合理由

本RUは、複数RU入力、draft構造、operation unit、Issue/Epic化、case-auto逐次処理を一体の要件として扱う。

これらを分割すると、以下の不整合が残る。

- `req-define` が queue 情報を出さないまま、`case-auto` だけが queue 制御を要求される
- `case-open` が従来どおり複数REQ即Epic化し、operation unit queue と衝突する
- OU 間依存の扱いが、queue dependency と Epic hierarchy dependency に分離されない
- `case-auto` が本来持つべきではない REQ操作判定や Issue階層判定を再実装するリスクが残る

そのため、`req-define` / `req-save` / `case-open` / `case-auto` の入出力契約をまとめて整える必要がある。

## 要件化の方向

### draft と operation unit

`req-define` は feature / RU入力系では原則 1 draft を作成する。

draft は Issue 単位ではなく、要件整理結果のコンテナとして扱う。draft は 1 件以上の operation unit を保持できる。

operation unit は、以下を持つ。

- source RU
- 対象REQまたは新規REQ候補
- 操作種別 CREATE / APPEND / UPDATE
- scale
- 依存関係
- 推奨処理順
- case-open への Issue 化方針候補

### Issue / Epic 化

`case-open` は draft 単位ではなく、operation unit 単位で Issue 構造を決定する。

複数RU、複数REQ操作、複数operation unit であることだけを理由に Epic Issue を作成しない。

Epic Issue を作成するのは、複数 operation unit または複数 Issue を同一目的・同一実行制御単位として束ねる必要がある場合に限る。

### OU 間依存

OU 間依存は、まず operation unit queue の依存関係として扱う。

OU 間依存があるだけでは Epic Issue を作成しない。

queue dependency は、`case-auto` が次に処理可能な operation unit を判断するために使う。

Epic Issue の親子構造や Wave は、複数 Issue を同一実行制御単位として束ねる必要がある場合にのみ使う。

### case-auto

`case-auto` は operation unit queue の管理・制御だけを担う。

`case-auto` は、各 operation unit を次の順で処理する。

1. `req-save` 相当処理
2. `case-open` 相当処理
3. `case-run` 相当処理
4. `case-close` 相当処理

前の operation unit が close まで完了した場合のみ、次の operation unit に進む。

`case-auto` は operation unit の統合・分割、REQ操作分類、Issue階層判定を再評価しない。
