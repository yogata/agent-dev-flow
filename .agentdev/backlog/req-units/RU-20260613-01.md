# RU: REQ tags の仕様・実装からの完全除去

## 背景

現行の REQ ファイルには frontmatter `tags` が存在する。

これまでの検討で、REQ tags は既存REQ照合の補助情報として位置づけられていたが、REQ本文、背景、問題、受け入れ条件、対象外、責務境界を確認する照合に比べて有効性が低いことを確認した。

また、namespace 付き tags や repo-local taxonomy による統制も検討したが、namespace セットはリポジトリごとの成果物構造に依存するため、agent-dev-flow plugin の汎用仕様として固定できない。namespace や taxonomy を自由に育てる方式では、現行の自由記述 tags と同様に統制されない分類語彙が乱立する。

## 問題

REQ tags は canonical な要件情報ではなく、検索・照合・棚卸しの補助情報にとどまる。

その一方で、tags を仕様・テンプレート・検査・照合ロジックに残すと、以下の問題が残る。

- REQ作成時にタグ付け判断が追加負荷になる
- タグ語彙の統制、taxonomy、メンテナンスが別責務として発生する
- タグがREQ照合の根拠であるかのように誤用される
- repoごとに異なる分類軸を plugin 共通仕様へ持ち込みやすくなる
- タグ廃止後も「タグ廃止要件」を残すと履歴都合のノイズになる

## Source Summary

チャット内で以下を合意した。

- REQ照合は frontmatter tags ではなく、REQ本文、背景、問題、受け入れ条件、対象外、責務境界を確認して行う
- tags は照合補助としても弱く、主判断材料には適さない
- namespace 付き tags は一見有効だが、namespace 自体が repo-local であり、汎用固定できない
- repo-local taxonomy を継続運用する案は可能だが、補助情報のために運用コストが大きい
- agent-dev-flow plugin の汎用設計として、tags 管理を中核機能に持ち込むのは不適切
- REQ tags は仕様・実装から完全廃止する
- 廃止すること自体を新規要件として残さず、要件・仕様・実装からの完全除去をもって廃止とする

## 統合理由

本RUは、REQ tags の扱いを「統制して活用する」方向ではなく、「仕様・実装から除去する」方向へ統一する。

タグ廃止を独立した新規REQとして定義すると、廃止後もタグに関するREQが active requirements に残り、REQ体系に不要な履歴要素を残すことになる。

そのため、REQ tags の廃止は新規REQ化せず、既存の要件定義、仕様、テンプレート、検査実装、既存REQファイルから tags 参照を消し切る変更として扱う。

## 要件化の方向

REQ tags を agent-dev-flow の active specification から完全に除去する。

具体的には、以下の方針で既存REQまたは関連仕様へ反映する。

- REQ frontmatter の必須項目から `tags` を削除する
- 既存 REQ ファイルの frontmatter から `tags` を削除する
- REQ templates / examples / fixtures から `tags` を削除する
- 既存REQ照合ロジック・説明から `tags` を評価軸として削除する
- docs-check / integrity の frontmatter validation から `tags` 必須確認を削除する
- tag taxonomy、tag 管理コマンド、tag namespace 運用は導入しない
- tags 廃止そのものを新規REQとして active requirements に残さない

## 主対象REQまたは変更対象候補

変更対象候補は以下。

- REQ frontmatter schema を定義している既存REQ
- REQファイル作成・更新を扱う既存REQ
- 既存REQ照合を扱う既存REQまたは skill
- docs-check / integrity の frontmatter validation を扱う既存REQまたは仕様
- REQ template / sample / fixture
- `docs/requirements/REQ-*.md`
- `agentdev-req-file-manager` 相当の既存REQ照合スキル
- `req-define`
- `req-save`
- docs-check / integrity 関連実装

具体的な REQ ID は、後続の既存REQ照合で確定する。

## 対象外

以下は対象外とする。

- tag taxonomy の新設
- tag 管理コマンドの新設
- namespace 付き tags の導入
- repo-local taxonomy 運用の導入
- tags 廃止を説明する新規 active REQ の作成
- 過去の Issue / PR / closed discussion / 履歴ログからの tags 記述削除
- REQ照合方式全体の再設計
- tags 以外の frontmatter 項目の見直し
- 無関係なREQ本文、体裁、表現の改善

## 受け入れ条件

- active REQ frontmatter schema に `tags` 必須項目が残っていない
- 既存の active `docs/requirements/REQ-*.md` frontmatter に `tags` が残っていない
- REQ template、sample、fixture に `tags` が残っていない
- 既存REQ照合の評価軸から `tags` が削除されている
- `req-define` / `req-save` の説明または処理に、REQ tags を参照する現行仕様が残っていない
- docs-check / integrity の frontmatter validation で `tags` 必須確認が行われていない
- tag taxonomy または tag 管理コマンドが新設されていない
- tags 廃止そのものを表す新規 active REQ が作成されていない
- active docs、現行仕様、現行実装、現行テンプレートから REQ tags の有効運用を前提とする記述が消えている
- 変更は REQ tags の完全除去に直結する範囲に限定されている
