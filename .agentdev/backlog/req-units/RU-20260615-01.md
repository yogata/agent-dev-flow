# RU: diagnostics 系 workflow family の inspect への全面改名

## 背景

diagnostics 系コマンドとして、文書・Command/Skill の意味整合性や参照妥当性を検出し、finding を保存・分類・promote する workflow family が検討されている。

ただし、`diagnostics` は語として重く、診断・分析・原因特定・改善提案まで含むように見える。ゼロベースで検討した結果、コマンド接頭辞および workflow family 名としては `inspect` がより妥当であると合意した。

本 RU は、既存導入状態を前提にせず、`diagnostics` / `diagnostic` を `inspect` へ統一するための要件化入力である。

## 問題

`diagnostics` / `diagnostic` を残すと、以下の問題が発生する。

- コマンド責務が「検出」より広く見える
- `diagnostic` と `diagnostics` の単複揺れが発生する
- finding artifact、保存ディレクトリ、promote route、backlog-review 入力型の用語が分裂する
- 後続工程で `diagnostic/diagnostics/inspect` の3系統が混在する
- コマンド名だけ置換すると、ディレクトリ構成・README・REQ/ADR/SPEC/guides・DOC-MAP・スキル参照に不整合が残る

## Source Summary

チャット内で以下を合意した。

- `diagnostics` という語そのものを懸念対象とする
- 既存導入状態ありきではなく、ゼロベースで命名を評価する
- `diagnostics` より `inspect` を採用する
- 変更対象はコマンド名だけではない
- すべてのドキュメント上の言及、ディレクトリ構成、artifact 名、source_type、参照、README、DOC-MAP、スキル名を対象にする
- `diagnostic finding` のような単数形派生も残さず、`inspect finding` に統一する

## 統合理由

この変更は、個別コマンド名の修正ではなく workflow family 全体の改名である。

`diagnostics-*` コマンド、`.agentdev/diagnostics/`、`diagnostics finding`、`source_type: diagnostics`、関連スキル名、README / DOC-MAP / docs 内参照は同じ概念を指すため、分離して扱うと中間状態の不整合が残る。

したがって、1つの Requirement Unit として統合し、後続の req-define で対象ファイル群と canonical name を一括で扱えるようにする。

## 要件化の方向

`diagnostics` 系 workflow family を `inspect` 系 workflow family へ全面改名する。

以下を canonical name とする。

| 種別 | canonical |
|---|---|
| docs inspection command | `/agentdev/inspect-docs` |
| command/skill inspection command | `/agentdev/inspect-skills` |
| finding promotion command | `/agentdev/inspect-promote` |
| domain state directory | `.agentdev/inspect/` |
| inbox directory | `.agentdev/inspect/inbox/` |
| promoted directory | `.agentdev/inspect/promoted/` |
| rejected archive directory | `.agentdev/inspect/archive/rejected/` |
| finding term | `inspect finding` |
| source type | `inspect` |
| skill name | `agentdev-inspect-skills` |

以下の旧称は現行用語として使用しない。

- `diagnostics-*`
- `diagnostic-*`
- `.agentdev/diagnostics/`
- `diagnostics finding`
- `diagnostic finding`
- `source_type: diagnostics`
- `type: diagnostics`
- `agentdev-diagnostics-*`

履歴説明として旧称を記載する必要がある場合は、現行名ではなく旧称であることを明記する。

## 対象外

以下は対象外とする。

- `inspect` workflow の新規機能追加
- finding 分類ルールの再設計
- intake / learning / backlog-review の責務再設計
- review / check / audit / scan など他 family の命名見直し
- 旧 docs-review / skill-review の復活
- Issue 作成
- PR 作成
- 実装手順の指定
- コード差分案の作成

## 受け入れ条件

- `/agentdev/inspect-docs`、`/agentdev/inspect-skills`、`/agentdev/inspect-promote` が canonical command name として定義されている
- `diagnostics-docs.md`、`diagnostics-skills.md`、`diagnostics-promote.md` が現行コマンド定義として残っていない
- `.agentdev/inspect/inbox/`、`.agentdev/inspect/promoted/`、`.agentdev/inspect/archive/rejected/` が inspect finding の保存先として定義されている
- `.agentdev/diagnostics/` が現行 domain state directory として残っていない
- backlog-review の promoted artifact 入力対象が `.agentdev/inspect/promoted/*.md` を含む
- `source_type` および `sources[].type` の inspect 系入力型が `inspect` として定義されている
- `source_type: diagnostics` および `type: diagnostics` が現行仕様として残っていない
- `diagnostics finding` および `diagnostic finding` が現行用語として残っていない
- 必要な場合は `inspect finding` に統一されている
- `agentdev-diagnostics-*` 系スキル名・参照が現行名として残っていない
- `agentdev-inspect-skills` が Command/Skill inspection 用スキル名として使われている
- README、`.agentdev/README.md`、command README、DOC-MAP、REQ/ADR/SPEC/guides 内の導線が inspect 系に更新されている
- 履歴説明として `diagnostics` / `diagnostic` を残す場合は、旧称であることが明記されている
- 隣接する review / intake / learning / backlog の設計変更が混入していない
