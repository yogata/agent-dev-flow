# RU: ADR意思決定境界の全体見直しと再発防止

## 背景

本セッションでは、ADR の役割について「ADR は意思決定であり、どうあるべきかを対象とする」という方針が確認された。

その観点から、`ADR-0113 review 系コマンド完全削除` や `ADR-0110 requirements/views 廃止` は、ADR として不適切であると整理された。削除・廃止・移行・完全削除は変更作業であり、ADR の主題ではない。

廃止した内容は、新たに「廃止するADR」を作成するのではなく、対象ADRを retire / supersede すれば足りる。ADR として残す場合は、削除・廃止ではなく、将来の構造・責務境界・運用原則などの「あるべき状態」を記述する必要がある。

## 問題

現行の ADR 群には、意思決定ではなく変更作業そのものを主題にした ADR が混入している。

具体的には、以下の問題がある。

1. ADR のタイトルに「廃止」「削除」「完全削除」「移行」「導入」など、変更作業を主題化する表現が含まれている。
2. ADR の決定本文に、あるべき状態ではなく、ファイル削除・コマンド削除・移管・移行手段などが含まれている。
3. retired にすべき過去判断について、新規ADRで「廃止する」と記録する構造が発生しうる。
4. ADR本文を修正しても、ADR作成・更新・検査・レビューの入口に基準がなければ再発する。
5. ADR / REQ / SPEC / command / skill / integrity の責務境界に、ADRとして書いてよいもの・いけないものの判定基準が十分に制度化されていない。

## Source Summary

本セッションで合意された内容は以下である。

- ADR は意思決定であり、「どうあるべきか」が対象である。
- 削除・廃止・移行・完全削除そのものは ADR の主題ではない。
- `ADR-0113 review 系コマンド完全削除` は、ADR として不適切である。
- `ADR-0110 requirements/views 廃止` も同様に、ADR として不適切である。
- 廃止した内容は、廃止対象の ADR を retire / supersede すればよい。
- 「廃止するというADR」は許容しない。
- `docs/adr/` 配下の全 ADR ファイル、各ADRのタイトル、各ADR-IDごとの内容を漏れなく再評価する必要がある。
- 本件は「見直し」と「再発防止」の2本立てである。
- 見直しだけでは不足し、ADRで記述する方針を REQ / SPEC / command / skill / integrity/docs-check に反映する必要がある。

## 統合理由

本件は、個別ADRの文言修正だけではなく、ADRという文書種別の境界を再定義し、全ADRの現行基盤としての妥当性を再評価し、さらに再発防止のために生成入口・判断入口・検査入口へ反映する必要がある。

そのため、以下を単一の Requirement Unit として扱う。

- 既存ADR群の全体見直し
- ADRとして不適切な主題の除去
- accepted / retired / superseded の扱い整理
- ADR作成・更新判断の再発防止
- integrity/docs-check による継続検出

## 要件化の方向

ADR は、将来の設計・運用・文書システムを制約する意思決定を記録する文書でなければならない。

ADR は、削除・廃止・移行・統合・再構築・完全削除そのものを主題にしてはならない。

過去の判断を現行基盤から外すだけの場合は、新規ADRを作成せず、対象ADRの status 遷移、retired 配置、または superseded 扱いで処理する。

新規ADRを作成できるのは、旧判断を置き換える新しい「あるべき状態」の意思決定が存在する場合に限る。

accepted ADR は、タイトル・背景・決定・根拠・影響が、変更作業ではなく「あるべき状態」「責務境界」「採否理由」を記述していなければならない。

retired ADR は歴史記録として保持し、現行基盤の根拠として扱ってはならない。

ADRの妥当性基準は、ADR本文だけではなく、REQ、SPEC、command、skill、integrity/docs-check に反映し、継続的に維持できるようにする。

## 主対象REQまたは変更対象候補

主対象候補:

- `REQ-0101`: 文書種別定義として、ADRの責務境界を強化する。
- `REQ-0108`: integrity / docs-check の検査観点に、ADR主題妥当性の検出を追加する。

変更対象候補:

- `docs/adr/README.md`
  - accepted ADR / retired ADR の区別を現行基盤として正しく表現する。
  - ADR index が作業主題ADRを accepted として扱わないようにする。

- `docs/adr/ADR-0101.md` 〜 `docs/adr/ADR-0113.md`
  - 全 accepted ADR を対象に、タイトル・本文・status・現行基盤としての妥当性を再評価する。
  - 不適切なADRは retire / supersede するか、あるべき状態の意思決定へ再構成する。

- `docs/adr/retired/ADR-0001.md` 〜 `docs/adr/retired/ADR-0023.md`
  - retired ADR は本文を書き換える対象ではなく、歴史記録として保持されていること、現行基盤から外れていること、後継ADRとの関係が妥当であることを確認する。

- `docs/specs/*`
  - ADR / REQ / SPEC の文書種別責務境界を反映する。
  - ADR は why / decision、SPEC は current architecture / what is、REQ は満たすべき要件、という役割分担を明確化する。

- `src/opencode/commands/agentdev/req-define.md`
  - ADR作成可否ゲートに、削除・廃止・移行・完全削除そのものをADR化しない判定基準を追加する。

- `src/opencode/skills/agentdev-adr-guidelines/SKILL.md`
  - ADRを作成してよい条件 / 作成してはならない条件を更新する。
  - 「微妙な場合はADR側に寄せる」趣旨の記述がある場合は、今回の基準と衝突しないように修正する。

- `src/opencode/skills/agentdev-adr-file-manager/SKILL.md`
  - ADR lifecycle / status 遷移に、単なる廃止・削除・移行は新規ADRではなく retire / supersede で扱う基準を反映する。

- integrity/docs-check 関連
  - accepted ADR のタイトル・本文・README index に、削除・廃止・移行・完全削除そのものを主題化した記述が混入していないか検出する。
  - retired ADR を現行基盤として参照していないか検出する。
  - accepted ADR と retired ADR の index 整合性を検査する。
