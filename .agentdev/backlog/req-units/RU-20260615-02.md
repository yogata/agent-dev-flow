# RU: ADR意思決定境界の制度化と再発防止

## 背景

本セッションでは、ADR の役割について「ADR は意思決定であり、どうあるべきかを対象とする」という方針が確認された。

また、既存ADRの見直しだけでは不十分であり、ADRとして書いてよいもの・いけないものを生成入口、判断入口、検査入口に反映しなければ再発すると整理された。

ADR本文だけを修正しても、`req-define`、ADR関連skill、文書種別SPEC、integrity/docs-check が同じ境界を持っていなければ、削除・廃止・移行・完全削除そのものを主題にしたADRが再び作成されうる。

## 問題

ADRの文書種別境界が、作成・更新・検査の各入口に十分に制度化されていない。

具体的には、以下の問題がある。

1. ADR本文を修正しても、ADR作成・更新・検査・レビューの入口に基準がなければ再発する。
2. ADR / REQ / SPEC / command / skill / integrity の責務境界に、ADRとして書いてよいもの・いけないものの判定基準が十分に制度化されていない。
3. 「廃止するADR」「削除するADR」「移行するADR」を作成しないための明示的な判定基準が、生成入口に必要である。
4. accepted ADR に作業主題が混入した場合、それを検出する継続的な検査観点が必要である。

## Source Summary

本セッションで合意された内容は以下である。

- ADR は意思決定であり、「どうあるべきか」が対象である。
- 削除・廃止・移行・完全削除そのものは ADR の主題ではない。
- 廃止した内容は、廃止対象の ADR を retire / supersede すればよい。
- 「廃止するというADR」は許容しない。
- 本件は「見直し」と「再発防止」の2本立てであり、本RUはそのうち「再発防止」を扱う。
- 見直しだけでは不足し、ADRで記述する方針を REQ / SPEC / command / skill / integrity/docs-check に反映する必要がある。

## 統合理由

本RUは、ADR意思決定境界を継続的に維持するための制度化を対象とする。

既存ADR群の棚卸し、retire / supersede、タイトル・本文の再構成は別RUで扱うため、本RUでは以下に限定する。

- ADRの文書種別定義の強化
- ADR作成可否ゲートの更新
- ADR関連skillの判断基準更新
- docs/specs の責務境界反映
- integrity/docs-check による継続検出

## 要件化の方向

ADR は、将来の設計・運用・文書システムを制約する意思決定を記録する文書でなければならない。

ADR は、削除・廃止・移行・統合・再構築・完全削除そのものを主題にしてはならない。

過去の判断を現行基盤から外すだけの場合は、新規ADRを作成せず、対象ADRの status 遷移、retired 配置、または superseded 扱いで処理する。

新規ADRを作成できるのは、旧判断を置き換える新しい「あるべき状態」の意思決定が存在する場合に限る。

ADRの妥当性基準は、ADR本文だけではなく、REQ、SPEC、command、skill、integrity/docs-check に反映し、継続的に維持できるようにする。

accepted ADR は、タイトル・背景・決定・根拠・影響が、変更作業ではなく「あるべき状態」「責務境界」「採否理由」を記述しているか検査できなければならない。

## 主対象REQまたは変更対象候補

主対象候補:

- `REQ-0101`
  - 文書種別定義として、ADRの責務境界を強化する。
  - ADRは作業そのものを主題にしないことを明文化する。

- `REQ-0108`
  - integrity / docs-check の検査観点に、ADR主題妥当性の検出を追加する。

変更対象候補:

- `docs/specs/*`
  - ADR / REQ / SPEC の文書種別責務境界を反映する。
  - ADR は why / decision、SPEC は current architecture / what is、REQ は満たすべき要件、という役割分担を明確化する。

- `src/opencode/commands/agentdev/req-define.md`
  - ADR作成可否ゲートに、削除・廃止・移行・完全削除そのものをADR化しない判定基準を追加する。

- `src/opencode/skills/agentdev-adr-guidelines/SKILL.md`
  - ADRを作成してよい条件 / 作成してはならない条件を更新する。
  - 「微妙な場合はADR側に寄せる」趣旨の記述がある場合は、今回の基準と衝突しないように修正する。

- `src/opencode/skills/agentdev-adr-file-manager/SKILL.md`
  - ADR lifecycle / status 遷移に、単なる廃止・削除・移行は新規ADRではなく retire / supersede で扱う基準を反映する。

- integrity/docs-check 関連
  - accepted ADR のタイトル・本文・README index に、削除・廃止・移行・完全削除そのものを主題化した記述が混入していないか検出する。
  - retired ADR を現行基盤として参照していないか検出する。
  - accepted ADR と retired ADR の index 整合性を検査する。

## 対象外

以下は本RUの対象外とする。

- 既存ADR群の棚卸しそのもの
- 個別ADRの retire / supersede / 本文再構成
- `docs/adr/retired/` 配下の歴史記録本文を、現行方針に合わせて書き換えること
- コード実装そのもの
- Issue 作成
- PR 作成
- 会話URLの記録
- 未合意の個別ADR採否判断を確定要件として扱うこと

## 受け入れ条件

- `REQ-0101` または該当REQに、ADRの文書種別定義として「作業そのものを主題にしない」基準が反映されていること。
- `REQ-0108` または該当REQに、ADR主題妥当性を継続検出する観点が反映されていること。
- `req-define` のADR作成可否ゲートに、削除・廃止・移行・完全削除そのものをADR化しない基準が反映されていること。
- `agentdev-adr-guidelines` に、ADRを作成してよい条件 / 作成してはならない条件が反映されていること。
- `agentdev-adr-file-manager` に、単なる廃止・削除・移行は新規ADRではなく retire / supersede で扱う基準が反映されていること。
- docs/specs の文書種別責務境界が、ADRの役割と矛盾していないこと。
- integrity/docs-check が、accepted ADR の作業主題混入、retired ADR の現行基盤参照、ADR index 不整合を検出できること。
- 本RUに、既存ADR群の個別棚卸し・retire・本文再構成が混入していないこと。
- 本RUに、未合意の個別実装手順やコード差分案が含まれていないこと。
