# RU: コマンド・スキル間参照を skill 名単位へ正規化する

## 背景

`case-open.md` において、`agentdev-workflow-lifecycle` skill の内部プロトコル名である「Issue 生成プロトコル」まで参照している記述が確認された。

この記述は、skill を参照するだけでなく、skill 内部の見出し・protocol・Step 構造へ command が依存している状態である。

その後の確認で、同型の問題は `case-open.md` に限定されず、複数の command / skill に存在することが整理された。

## 問題

command から skill の内部見出し、protocol 名、Step 番号、Section 名まで参照すると、以下の問題が発生する。

- command が skill 内部構造に依存する
- skill の責務境界が曖昧になる
- `agentdev-workflow-lifecycle` が lifecycle 判定以外の汎用置き場になる
- command 固有の実行手順が skill 側へ漏れる
- skill 側が command 固有 Step 番号を一次情報として保持する
- 後続の command / skill 分割時に参照切れや責務重複が発生する

特に `agentdev-workflow-lifecycle` は、lifecycle / work_type / scale / upstream handoff などの判定系を扱うべきであり、Issue 本文生成、Epic / child Issue 生成、intake pipeline、docs-review 診断、backlog-review 統合手順などの command 固有処理を受け持つべきではない。

## Source Summary

チャット上で、以下の修正方針が合意された。

- command から skill を参照する場合、原則として skill 名までに抑える
- `skill の <protocol>`、`skill の「Step N: ...」`、`skill の Section N` のような内部構造参照は修正対象とする
- `case-open.md` の `agentdev-workflow-lifecycle skill の Issue 生成プロトコル` 参照は過剰である
- 同様の問題を全 command / skill で確認し、修正候補を洗い出す
- `agentdev-workflow-lifecycle` に Issue 生成ロジックを追加しない
- 共有利用が確認できない段階で新 skill を作るのは避ける
- ただし、複数 command が同一領域の詳細手順を共有している場合は、専用 skill または reference への分離を検討する

## 統合理由

本RUは、単一文言の修正ではなく、command / skill 間の参照境界を正規化する要件として扱う。

`case-open.md` の Issue 生成プロトコル参照、`backlog-review.md` の lifecycle Step 参照、`intake-*` の intake pipeline 相当処理の lifecycle 参照、`docs-review.md` の診断手順の lifecycle 参照、skill 側の command Step 番号保持は、すべて同じ構造問題である。

したがって、個別ファイル修正ではなく、共通ルールを定義したうえで全 command / skill へ適用する。

## 要件化の方向

command / skill 間参照について、以下の規則を導入する。

1. command から skill を参照する場合、原則として skill 名までにする。
2. command から skill 内部の protocol 名、Step 名、Section 名、見出し名を参照してはならない。
3. 例外は、実行時に明示的に読み込む必要がある template / reference / script の repo-root-relative path に限定する。
4. command 固有の実行順序、Step 番号、Phase 名を skill 側の一次情報として保持してはならない。
5. skill の `USE FOR` / `DO NOT USE FOR` と参照元用途が矛盾する場合、参照元または参照先を修正する。
6. `agentdev-workflow-lifecycle` は lifecycle / work_type / scale / upstream handoff などの判定系に限定し、Issue生成、intake処理、docs-review診断、backlog統合処理の詳細を持たせない。
7. 複数 command で共有される詳細手順がある場合のみ、専用 skill または reference への分離を検討する。

## 対象外

以下は本RUの対象外とする。

- command / skill の実装ロジック変更
- GitHub Issue 作成
- PR 作成
- 未確認の既存REQ再編
- `agentdev-workflow-lifecycle` の全面再設計
- 単発用途の新規 skill 作成
- 文言改善のみを目的とした隣接記述の整形
- 配布対象外ファイルの無関係な整理
- `case-auto` の自走制御や queue 制御の変更
- template / reference / script の内容変更。ただし参照境界の修正に直結する場合は対象に含める
