# RU-0004: チャット経由のBacklog登録としての session-sourced RU 正式化

## 背景

現行のRU（Requirement Unit）は、`req-backlog` が intake / learning の promoted artifact を分析・統合して `.agentdev/backlog/req-units/` に生成する前提で定義されている。

一方、実運用では、チャット内の要件壁打ちで `req-define` 向け入力が十分に整理され、ユーザー合意まで完了するケースがある。

この場合、`intake-capture → intake-review → intake-promote → req-backlog` を経由せず、チャット合意内容を直接RUとして保存し、後続の `req-define` に渡したい。

## 問題

現行のRU frontmatterは `source_type: intake | learning | mixed`、`generated_by: req-backlog`、`sources[].type: intake | learning` を前提としており、`chat` または `session` 由来のRUが正式に定義されていない。

そのため、チャット合意からRUを作成すると、実運用上は有効でも、現行フォーマットから見ると非準拠または未定義の扱いになる。

また、チャット合意をRU化できる条件、`session:*` の意味、RU本文に自足させるべき情報が明文化されていない。

## Source Summary

チャット内でユーザーと合意した `req-define` 向け入力を、`session-sourced RU` として `.agentdev/backlog/req-units/RU-*.md` に保存できるようにする。

このルートは intake / learning pipeline の置き換えではない。

promoted artifact 由来RUは、引き続き `req-backlog` が生成する。

session-sourced RU は、チャット内で要件化の方向、対象外、受け入れ条件まで整理済みの場合に限り作成できる。

## 統合理由

1:1。

本要件は、チャット合意内容をRUとして保存し、`req-define` に渡す軽量ルートを正式化する単一論点である。

対象はRU source type / generated_by / sources schema / session path / RU本文の自足性であり、同一の仕様拡張として扱う。

現行の `req-backlog` 由来RU定義を壊さず、session-sourced RU を追加ルートとして許容する。

## 要件化する内容

### 1. session-sourced RU の許容

チャット内でユーザーと合意済みの `req-define` 向け入力は、session-sourced RU として `.agentdev/backlog/req-units/RU-*.md` に保存できる。

### 2. frontmatter 拡張

session-sourced RU では、以下のfrontmatterを許容する。

```yaml
source_type: chat
generated_by: session
generated_at: {ISO 8601 timestamp}
status: draft
sources:
  - type: chat
    path: session:YYYY-MM-DD-{topic}
```

### 3. session path の扱い

`sources[].path` の `session:YYYY-MM-DD-{topic}` は、会話URLではなく、チャット内合意を示す識別子として扱う。

### 4. RU本文の自足性

後続工程で必要な背景、問題、合意内容、対象外、受け入れ条件は、RU本文に自足する形で記録する。

外部URLを根拠にせず、RU本文だけで `req-define` が成立する内容にする。

### 5. RU化対象の制限

session-sourced RU は、`req-define` に渡せる粒度まで整理済みの内容に限る。

以下はRU化しない。

- 未合意の論点
- 採否未確定の気づき
- 単なるメモ
- `req-define` に渡せる粒度まで整理されていない内容
