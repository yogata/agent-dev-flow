# RU: RU間の実施順制約をRU-IDで記述する

## 背景

`backlog-review` は、ユーザー承認済みの内容から複数のRUを作成する場合がある。

複数RUを同時に扱う場合でも、内容によっては先に処理すべきRUと後続RUの順序制約が発生する。

## 問題

現状のRUに実施順制約を表現する明確な項目がない場合、同一バッチ内のRU処理順がエージェント判断に委ねられる。

その結果、前提となるRUより先に後続RUが処理される可能性がある。

## Source Summary

チャット内で、以下を合意した。

- 実施順制約はRUファイルに記述する
- 制約対象は一連のRUファイル群の中に存在するRUである
- RU-IDはRU作成時に判明する
- 依存関係は promoted artifact path ではなく RU-ID で記述する
- `backlog-review` は同時実施時にもRU-ID依存に基づいて順序を保つ

## 統合理由

この要件は、`backlog-review` がユーザー承認済み内容からRUを作成し、同時に複数RUを扱う場合の処理順保証に関するものである。

そのため、`backlog-review` / `backlog-save` 統合後の `backlog-review` 責務に含めるのが妥当である。

## 要件化の方向

RUは、同一バッチ内の他RUへの依存を frontmatter の `depends_on` にRU-IDで記述できること。

`backlog-review` は、同時に処理するRU群に `depends_on` が含まれる場合、依存先RUを先に処理すること。

`depends_on` の依存先はRU-IDに限定すること。

`backlog-review` は、RU保存前または処理前に以下を検証すること。

- `depends_on` に記述されたRU-IDが同一バッチ内または処理対象として存在すること
- 循環依存が存在しないこと
- 依存順に並べ替え可能であること

未解決依存、循環依存、または同時処理対象外の依存がある場合、`backlog-review` は当該RUを処理せず、理由を提示すること。

## 主対象REQまたは変更対象候補

主対象REQ候補:

- REQ-0105 UPDATE

変更対象候補:

- `backlog-review` のRU作成責務
- RU frontmatter仕様
- `backlog-review` の同時処理時の順序保証
- promoted artifact削除前の検証条件

## 対象外

以下は対象外とする。

- Issue実行順の制御
- PR作成順の制御
- `case-run` 内部の実装手順制御
- RU本文への実装手順の記述
- promoted artifact path を依存先として扱うこと
- 新しい状態名の追加

## 受け入れ条件

- RU frontmatter に `depends_on` を記述できる
- `depends_on` の値はRU-IDである
- 同一バッチ内で複数RUを処理する場合、依存先RUが依存元RUより先に処理される
- 存在しないRU-IDを `depends_on` に指定した場合、処理されず理由が提示される
- 循環依存がある場合、処理されず理由が提示される
- promoted artifact path を依存先として扱わない
- 順序制約は実装手順ではなくRU間の前提関係として扱われる
