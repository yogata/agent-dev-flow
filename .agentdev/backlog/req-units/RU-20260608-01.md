# RU: case-auto 複数操作単位の継続実行キュー

## 背景

`/agentdev/req-define` に `RU-0001` と `RU-0002` を同時入力した結果、両RUは同一の要件定義セッションで統合/分離判定され、複数操作単位として整理された。

その後、`case-auto` により操作単位1（RU-0002: ADR再編・01XX baseline化）は `req-save → case-open → case-run → case-close` まで完了した。

一方で、操作単位2（RU-0001: docs-* command suite）は同一 draft 内に `status: analyzed` の状態で残存したにもかかわらず、`case-auto` は後続処理へ進まず、「RU-0001 を入力として /agentdev/req-define を実行」と報告した。

## 問題

`RU-0001` はすでに `/agentdev/req-define` 済みであり、再度 `req-define` に戻すべきではない。

問題は、`case-auto` が入力要件doc内の複数操作単位を処理キューとして保持せず、1つの操作単位の `case-close` 完了後に同一 draft 内の未実行操作単位を継続処理しないことである。

このため、複数RUを同時に `req-define` した場合、先行操作単位だけが完了し、後続操作単位が未実行のまま残存する。

## 統合理由

本RUは、`case-auto` の自走性・工程間状態管理・複数操作単位処理に関する単一の仕様不足を扱う。

既存の `depends_on` や backlog-review のRU順序制約ではなく、`req-define` 後に生成された複数操作単位 draft を `case-auto` が最後まで消費するための要件である。

したがって、主対象は `REQ-0114` への APPEND とし、`REQ-0102` や `REQ-0105` の根本変更は対象外とする。

## 主対象REQまたは変更対象候補

### 変更対象候補

- `src/opencode/commands/agentdev/case-auto.md`

  - 現在の操作単位を選択して `req-save → case-open → case-run → case-close` を実行する。
  - `case-close` 後に未実行操作単位を確認し、次操作単位へ進む。
  - 未実行操作単位が残る場合は全体完了報告を禁止する。

- 必要に応じて completion report template

  - 不可能であれば、操作単位指定の受け渡しを最小限追加する。

## 受け入れ条件

- 複数RUを同時に `/agentdev/req-define` し、複数操作単位として draft に保持した場合、`case-auto` が全操作単位を処理対象として認識する。
- 先行操作単位の `case-close` 完了後、同一 draft 内の未実行操作単位が存在すれば、`case-auto` が次操作単位の `req-save` 相当処理へ進む。
- `status: analyzed` 等の `req-define` 済み操作単位について、`case-auto` が再度 `/agentdev/req-define` を案内しない。
- 未実行操作単位が残っている場合、`case-auto` は全体完了として報告しない。
- 停止時の報告に、完了済み操作単位、実行中操作単位、未実行操作単位、再開すべき次工程が含まれる。
- 全操作単位が `req-save → case-open → case-run → case-close` を完了し、対応する Issue / PR / close / RU削除確認が完了した場合のみ、`case-auto` が全体完了を報告する。
