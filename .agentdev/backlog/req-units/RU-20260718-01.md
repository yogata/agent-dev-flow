# Command／Skill実行契約の単一定義化と横断是正

## 背景

AgentDevFlowの配布CommandおよびSkillは、Commandを薄い公開入口とし、再利用可能な判断や詳細手順をSkill、references、scriptへ分離する構造を採用している。

一方、現行配布物には、同一または類似する実行契約、判断基準、状態遷移、共通手順が、Command、Skill、referencesの複数箇所に記述されている箇所がある。この重複により、定義間の差異、更新漏れ、意味的な矛盾、実行時コンテキストの肥大化が発生し得る。

外部のモデル向けガイダンスを直接要件化するのではなく、そこから得られる一般的な知見を、モデル非依存のCommand／Skill作成方針として既存の責務分界へ統合する。

## 問題

現行のCommand／Skill構造には、次の問題がある。

- 同一の実行契約や判断基準が複数の配布物に再記述され、どの記述が正規な定義であるかが不明確になる
- CommandとSkillの間で、工程、状態、停止条件、責務範囲の意味的な差異が発生する
- 複数Commandに共通する契約が個別Commandへ反復記述される
- 長い説明、例、根拠、変更経緯が実行時に読み込まれる本文へ残り、必要な実行指示が埋没する
- 責務分離後も参照元に元の定義が残り、参照と再定義が併存する
- 現行の診断では、参照の存在や構造は確認できても、意味的な重複や矛盾を横断的に検出しきれない

## Source Summary

本セッションでは、以下を合意した。

- 外部ガイダンスのURL、特定モデル名、特定API機能は要件化しない
- ガイダンスから得られる一般原則を、Command／Skillの恒久的な作成・配置方針として取り込む
- 新規ADRは作成せず、既存のCommand／Skill責務分界REQを精緻化する
- 対象を一部Commandに限定せず、全配布Command／Skillを横断的に評価し、違反箇所を是正する
- `authoritative source` という英語表現は用いず、「責務ごとの正規な定義元」などの日本語で記述する
- 責務ごとの正規な定義元は、Command、Skill、references、script、harness側文書の既存分界に従う
- 意味的な重複、矛盾、正規な定義元からの逸脱は `inspect-skills` の診断対象とする
- 新規のintegrity機械検査ルールは今回の対象に含めない
- 外部的なCommandの目的、入力、出力、停止条件、成果物契約は変更しない

## 統合理由

Command／Skillの配置原則、既存配布物の横断是正、`inspect-skills`による再発検出は、すべて「実行契約が複数箇所で再定義されること」を根因とする。

配置原則だけを追加した場合、既存の重複や矛盾が残る。既存配布物だけを修正した場合、再発を防止する診断基準が残らない。

したがって、次の三点を一つのRequirement Unitとして扱う。

1. 責務ごとの正規な定義元の明文化
2. 全配布Command／Skillの横断的な是正
3. 意味的な重複・矛盾を検出する診断基準の追加

## 要件化の方向

### 責務ごとの正規な定義元

実行契約の正規な定義元は、内容の責務に応じて次のように配置する。

| 内容 | 正規な定義元 |
|---|---|
| 公開入力、出力、成果物、停止条件、高レベル工程、委譲接続点 | Command |
| 再利用可能な判断基準、状態判定、状態遷移、再開判定、共通手順 | Skill |
| 長い判定表、分類表、例、詳細プロトコル | Skillの`references/` |
| 決定的な列挙、照合、変換、生成、検査 | Script |
| 実行エージェント選定、起動方法、並列度、timeout、retry、context管理 | `AGENTS.md`またはharness側reference |
| 背景、要件、設計判断、仕様上の詳細 | REQ、ADR、SPEC |

参照元は、正規な定義元の内容を再記述せず、参照と参照元固有の差分のみを保持する。

同じ内容を要約して再掲する場合も、その要約が独立した契約として解釈されない表現とし、正規な定義元との意味的な差異を生じさせない。

### 横断是正

全配布Command、全配布Skill、および関連するreferences、templates、scriptsを評価対象とする。

評価により次の問題が確認されたファイルだけを変更対象とする。

- 同一契約の複数箇所での再定義
- CommandとSkill間の工程、状態、責務、停止条件の矛盾
- 複数Commandにおける共通契約の反復
- Command固有手順の共通Skillへの混入
- referencesへ分離済みの詳細が本文にも残る状態
- 実行判断に不要な背景、変更経緯、根拠説明の本文残留
- 分離または縮約による意味、条件、成果物契約の欠落

横断是正は既存のCommand／Skill／references／scripts構造の中で行い、新しい標準階層や新しい成果物種別を追加しない。

### 診断

`inspect-skills`の意味的診断観点へ、少なくとも次を追加する。

- 同一実行契約の意味的重複
- 正規な定義元以外での契約再定義
- CommandとSkill間の意味的矛盾
- 共通契約の複数Commandへの反復
- Command固有責務と共通Skill責務の混在
- referencesまたはscriptへの分離後に残る本文重複
- 薄型化または共通化によるセマンティクス欠落
- 実行時に不要な説明が主要な実行指示を埋没させる構造

診断は修正を実行せず、対象、分類、根拠、正規な定義元の候補、推奨経路を報告する。

意味判断を要するため、今回の診断観点を新規の文字列ベースintegrityルールにはしない。将来、安定した構文パターンが確立した項目だけを別途機械検査候補とする。

## 主対象REQまたは変更対象候補

### 主対象REQ

- `REQ-0119`：Command／Skill／サブエージェント責務分界
  - 責務ごとの正規な定義元
  - 同一契約の再定義抑止
  - 参照元は参照と固有差分のみを保持
  - 配置変更前後における外部契約維持

- `REQ-0125`：inspect-skills／Command・Skill参照妥当性検出
  - 意味的重複
  - 意味的矛盾
  - 正規な定義元からの逸脱
  - セマンティクス欠落
  - 診断結果の報告契約

### 関連REQ

- `REQ-0162`：配布物のharness実行制御分離
  - 既存原則を維持する
  - 今回の主変更対象とはしない

### 変更対象SPEC候補

- Command／Skill責務境界を定義する既存SPEC
- `agentdev-inspect-skills` SPEC
- Command authoring standards
- Skill authoring standards
- 必要に応じて、個別Command／Skill SPECの責務記述

### ADR

新規ADRおよび既存ADRの変更は行わない。

今回の変更は、既存の責務分界とharness分離の意思決定を変更せず、その適用条件を精緻化するものである。

## 対象外

- 外部ガイダンスURLの記録
- OpenAI、GLMその他の特定モデルを前提とする規則
- 特定APIのパラメータまたは機能の標準化
- 新しいCommand／Skill階層の追加
- 新しいharness抽象化レイヤの追加
- harness固有の起動方法、推論設定、並列制御の実装
- 新規integrity機械検査ルール
- Commandの公開目的、入力、出力、停止条件、永続成果物契約の変更
- ワークフローの新機能追加
- 実装手順、ファイル別修正手順、作業順序の確定

## 受け入れ条件

1. `REQ-0119`に、責務ごとの正規な定義元と、同一契約を複数箇所で再定義しない原則が記載されている
2. `REQ-0125`に、意味的重複、意味的矛盾、正規な定義元からの逸脱を診断する要件が記載されている
3. 正規な定義元の責務分担が、Command、Skill、references、script、harness側文書について日本語で定義されている
4. 全配布Commandおよび全配布Skillが横断的な評価対象となっている
5. 横断評価で違反が確認された配布物について、重複定義または矛盾が解消されている
6. 参照元は、正規な定義元の再記述ではなく、参照と参照元固有の差分のみを保持している
7. Commandは公開入口、高レベル工程、停止条件、成果物、委譲接続点に限定されている
8. 再利用可能な判断、状態判定、状態遷移、詳細手順は、責務に応じてSkill、references、scriptへ配置されている
9. `inspect-skills`が、意味的重複、矛盾、責務逸脱、セマンティクス欠落を診断し、対象、分類、根拠、推奨経路を報告できる
10. 横断是正前後で、各Commandの公開目的、入力、出力、停止条件、永続成果物契約が維持されている
11. 配布物の規範が特定モデルや特定APIに依存せず、モデル非依存の一般原則として記述されている
12. 新規ADR、新規標準階層、新規integrity機械検査ルールが追加されていない
