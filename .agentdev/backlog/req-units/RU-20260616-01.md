# RU: OMO委譲による実装計画・レビュー外部化とagentdev実行バックエンド境界

## 背景

本セッションでは、agent-dev-flow における `case-run` の実装計画策定とレビューをどこで扱うべきかを議論した。

現状認識として、agentdev は `.agentdev/drafts/`、GitHub Issue、PR、case-open/case-run/case-close の workflow state を管理する。一方、具体的な実装計画、plan review、実装タスクの具体処理は、個別タスクの実行具体であり、agentdev 本体に抱え込むべきではない。

また、oh-my-openagent には Prometheus / ulw-plan による plan 作成、および Momus による plan review の仕組みが存在する。Momus は plan の実行可能性、参照の妥当性、QAシナリオの具体性、blocking issue の有無を確認する review agent として位置づけられる。

本セッションでは、agentdev 側が Issue / PR / `.agentdev/` の状態管理を保持しつつ、Issue 単位の具体的な実装計画・レビュー・実行を OMO 側へ委譲する方針で合意した。

## 問題

現行の `case-run` が具体的な実装計画ファイルの生成・再開・レビュー・実行制御まで直接持つ場合、以下の問題がある。

- agentdev の workflow state 管理と、個別実装計画の詳細責務が混在する。
- メインセッションが OMO / Momus / plan 反復 / 実行ログを直接抱え、コンテキスト消費が増える。
- `.sisyphus/plans/` のような具体的 plan artifact への依存が agentdev 側に残る。
- 将来的に OMO 以外のハーネスへ切り替える余地が狭くなる。
- Epic / 複数Issueの orchestration と、1 Issue の実装具体が混線しやすい。
- Momus review の REJECT / ITERATE を main session が直接扱うと、agentdev 本体に OMO 固有の反復制御が漏れる。

## Source Summary

## 統合理由

本件は複数の小論点を含むが、すべて「agentdev と OMO の実行責務境界」を定める同一テーマである。

特に、以下は分離せず一体で要件化する必要がある。

- `case-run` から具体実装計画を外すこと
- OMO / Momus への委譲境界
- driver subagent の導入
- 外部実行バックエンド skill の新設
- `.sisyphus/plans/` 依存撤去
- driver result の最小契約
- blocked / failed の Issue コメント化
- PR本文による capture handoff 維持
- Epic / 複数Issue orchestration を agentdev 側に残すこと

これらを別々に扱うと、agentdev が plan artifact を読んでしまう、OMO が Issue state を更新してしまう、または main session が OMO 固有の反復制御を抱えるなどの境界崩れが起きる。

## 要件化の方向

`case-run` は、Issue 単位の実装実行を外部実行バックエンドへ委譲する adapter 呼び出しに再整理する。

`case-run` 本体は以下のみを担う。

- 対象 Issue を解決する。
- worktree 前提を維持する。
- 1 Issue 単位で driver subagent を起動する。
- driver の最小結果を受け取る。
- `completed(pr)` の場合は agentdev workflow を継続する。
- `blocked` / `failed` の場合は、Issueコメントを参照する停止状態として報告する。
- 複数Issue / Epic の順序制御は引き続き agentdev 側で行う。

新規の独立 skill は、外部実行バックエンド driver の契約を定義する。名称は後続工程で確定するが、役割は `agentdev-execution-backend` 相当とする。

driver subagent は以下を担う。

- Issue本文を読み取る。
- 必要に応じて ADR / REQ / SPEC / docs / repository context を再確認する。
- OMO に Issue 単位の作業を委譲する。
- OMO / Momus からの REJECT / ITERATE / blocker を受け取る。
- repository context で回答可能な blocker は自律的に補足して OMO に差し戻す。
- repository context で回答不能な blocker は Issue に構造化コメントとして記録する。
- OMO 成功時は PR番号を返す。
- OMO blocked / failed 時は Issueコメントを SSoT として記録し、driver result は最小状態のみ返す。
- OMO 内部 plan artifact の中身を agentdev 制御に持ち込まない。
- Findings / Capture候補が存在する場合は、PR本文へ記述されていることを OMO 側の完了契約として扱う。

OMO は以下を担う。

- Issueから具体的な実装計画を作成する。
- Momus review を実行する。
- Momus が ITERATE を返す場合は OMO 内部で plan を修正し再レビューする。
- Momus が blocker を返す場合は driver に戻す。
- 実装、検証、PR作成を行う。
- Findings / Capture候補があれば PR本文に記述する。

`blocked` / `failed` の詳細は driver result ではなく Issueコメントに記録する。

最小 driver result は以下の3状態とする。

- `completed(pr)`
- `blocked`
- `failed`

`issue` は driver 呼び出し入力で既知のため返さない。  
`backend` は OMO 前提のため返さない。  
`plan_artifact` は agentdev が読まないため返さない。  
`verification` は PR / CI / case-close 側で扱うため返さない。  
`completion_conditions` は Issue本文を case-close が再読込して検証するため返さない。  
`capture_candidates` は PR本文への記述を契約とするため返さない。

## 主対象REQまたは変更対象候補

変更対象候補は以下。

- `case-run` コマンド定義
  - Step 5 の plan 生成・実行・TDD実装を、外部実行バックエンド driver 委譲へ再整理する。
  - `.sisyphus/plans/` 依存を撤去する。
  - OMO plan artifact の中身を読まない制約を追加する。
  - driver result の3状態を扱う制御を追加する。

- 新規独立 skill
  - 外部実行バックエンド adapter protocol を定義する。
  - OMO 委譲、Momus review response、blocked / failed の Issueコメント記録、PR返却契約を定義する。
  - 名称は後続工程で確定する。

- `agentdev-workflow-orchestration`
  - Epic / 複数Issue orchestration と、1 Issue OMO委譲の境界を明確にする。
  - 必要な場合のみ、新規 skill への参照を追加する。

- `case-auto`
  - `case-run` の内部委譲方式変更に伴う記述追従が必要な場合のみ更新する。
  - case-auto が OMO / Momus / plan artifact を直接制御しないことを確認する。

- PRテンプレートまたは workflow template
  - Findings / Capture候補を PR本文に記述する契約が、OMO 委譲後も維持されるように確認する。

## 受け入れ条件

- `case-run` が 1 Issue 単位で外部実行バックエンド driver subagent に委譲すること。
- OMO への直接委譲を main session が行わず、driver subagent 経由に限定されていること。
- 外部実行バックエンド driver の独立 skill が定義されていること。
- `case-run` 本体に OMO / Momus の詳細反復手順が展開されていないこと。
- OMO 前提で動作し、OMO なし fallback が導入されていないこと。
- `.sisyphus/plans/` を agentdev 側の再開判定・plan制御に使わないこと。
- OMO plan artifact は agentdev から不透明な外部成果物として扱われ、agentdev が中身を読んで制御しないこと。
- OMO 委譲単位が 1 Issue であること。
- 複数Issue / Epic Issue の orchestration が agentdev 側に残っていること。
- driver は ADR / REQ / SPEC / docs / Issue本文 / repository context を再確認し、回答可能な blocker を自律的に OMO へ差し戻せること。
- repository context で回答不能な blocker は Issueコメントに構造化して記録されること。
- blocked / failed の詳細本文は driver result ではなく Issueコメントに記録されること。
- blocked / failed を返す前に、driver が Issue コメントの書き込みと確認を行うこと。
- driver result は `completed(pr)` / `blocked` / `failed` の3状態に限定されていること。
- `completed` は PR番号を含むこと。
- `blocked` / `failed` は PR番号を含まないこと。

- case-close は引き続き PR本文を capture 入力源として扱うこと。

- OMO plan TODO の完了状態を agentdev の正式完了判定として扱わないこと。
