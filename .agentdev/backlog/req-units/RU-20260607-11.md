# RU: ADR作成フローにおけるADR禁止ゲートの追加

## 背景

agent-dev-flow のADR確認において、ADR-0023 が「技術的な選択」ではなく、command仕様・artifact仕様・workflow制約の変更を記録した文書であることを確認した。

ADR-0023の内容は、`backlog-save` 廃止、`backlog-review` へのRU生成統合、review draft廃止、RU frontmatter `depends_on` 追加、`learning-promote` 後の直行禁止などであり、ADRではなくREQ/SPECで扱うべき内容である。

同様に、仕様変更・仕様記述に寄っており、技術的な意思決定として成立しにくいADR候補が複数存在することも確認した。

このため、問題はADR-0023単体の誤分類ではなく、ADR作成フローを担うcommand / skill側に、ADRとして成立しない内容を停止・移送する判定ゲートが不足していることにある。

## 問題

現在のADR作成フローでは、仕様変更・command変更・artifact contract変更だけの内容でも、`adr-required: true` が立つとADR作成に進み得る。

特に以下が問題である。

- `req-define` のADR判定で、REQ/SPEC相当の内容をADR候補から除外する必須ゲートが弱い
- `req-save` が `adr-required: true` を前提にADR作成へ進み、保存直前にADRとして成立するかを再判定しない
- `agentdev-adr-guidelines` に、仕様変更だけならADRを作らないという禁止条件が不足している
- `agentdev-adr-file-manager` はADR要否判定ではなく、採番・ファイル操作が主責務であるため、意味的誤分類を止められない
- `repo-agentdev-integrity` は意味的妥当性判定を対象外としているため、作成後の誤分類も検出できない

## Source Summary

チャット内で以下を合意した。

- ADR-0023はADRではない
- ADR-0023は新規ADRとして作成すべきではなかった
- ADR-0023の内容はREQ/SPEC変更として扱うべきである
- 同様に仕様変更・仕様記述だけで技術的決定ではないADRが存在する
- ADRの不適切な運用は、agent-dev-flowのcommand / skill側の不備に起因する可能性が高い
- 再発防止には、ADR-0023単体の整理ではなく、ADR作成フロー側にADR禁止ゲートを追加する必要がある

## 統合理由

ADR誤分類の再発防止は、個別ADRの修正では達成できない。

`req-define`、`req-save`、`agentdev-adr-guidelines`、`agentdev-adr-file-manager`、`repo-agentdev-integrity` は、ADR候補の生成、保存、管理、検査に関係するため、同一Requirement Unitとして扱う。

ただし、ADRファイル群そのものの再編成は別作業とし、本RUではADR作成フローの判定不備を対象とする。

## 要件化の方向

ADR作成フローは、技術判断を含まない仕様変更・command変更・artifact contract変更をADRとして保存してはならない。

仕様変更・command仕様・artifact schema・template・検査条件・workflow制約は、ADR候補ではなくREQ/SPEC更新候補として扱う。

ADR候補として扱うには、少なくとも以下を満たす必要がある。

- 技術的な選択が存在する
- 採用案と却下案が存在する
- 選択理由が存在する
- 変更後の仕様差分だけではない
- 後から単純なREQ/SPEC更新で戻せる作業指示だけではない

ADRとして成立しない場合は、ADR作成を停止し、REQ/SPECへの移送候補として提示する。

## 主対象REQまたは変更対象候補

変更対象候補は以下。

- `.opencode/commands/agentdev/req-define.md`
- `.opencode/commands/agentdev/req-save.md`
- `.opencode/skills/agentdev-adr-guidelines/SKILL.md`
- `.opencode/skills/agentdev-adr-file-manager/SKILL.md`
- `.opencode/skills/repo-agentdev-integrity/SKILL.md`
- 関連するREQ/SPEC文書

想定されるREQ/SPEC反映先は、後続の `req-define` で確定する。

## 要件

### ADR禁止ゲート

`req-define` は、ADR候補を立てる前に、対象内容がREQ/SPEC相当ではないかを判定しなければならない。

以下のみを含む変更はADR候補にしてはならない。

- commandの入出力変更
- command step変更
- guardrail変更
- frontmatter field追加・変更
- artifact lifecycle変更
- template配置・template内容変更
- integrity-check等の検査条件変更
- workflow上の禁止条件・停止条件・承認条件
- 既存commandの廃止・統合
- 既存ディレクトリ・path・namespaceの現在仕様化

### REQ/SPEC移送

ADR候補がREQ/SPEC相当と判定された場合、`req-define` はADR候補として扱わず、REQ/SPEC更新候補として整理しなければならない。

分類は以下を基本とする。

- 振る舞い・制約・状態はREQ
- 現在仕様・schema・path・lifecycle・artifact contractはSPEC
- 人間向け説明・操作導線はGuide
- 採用判断・代替案・却下理由・影響はADR

### req-save保存前再検証

`req-save` は、`draft-meta` に `adr-required: true` がある場合でも、保存直前にADR妥当性を再検証しなければならない。

ADRとして成立しない場合、`req-save` はADR作成を停止し、REQ/SPECへ移送すべき理由を報告しなければならない。

### adr-guidelines強化

`agentdev-adr-guidelines` は、ADRを作る条件だけでなく、ADRを作ってはならない条件を明示しなければならない。

特に、agent-dev-flow自身のcommand / skill / artifact contract / workflow仕様変更を、技術判断なしにADR化してはならない。

### adr-file-manager責務境界

`agentdev-adr-file-manager` はADR要否の意味判定を主責務にしない。

ただし、ADRファイル操作の前提として、ADR妥当性判定が `agentdev-adr-guidelines` によって完了していることを明記する。

### integrity-check補助検出

`repo-agentdev-integrity` は、ADRの意味的妥当性をstrict判定してはならない。

ただし、以下のようなADR誤分類の兆候をwarningまたはobservationとして検出できるようにする。

- Decisionが「追加」「廃止」「更新」「禁止」「テンプレート変更」だけで構成される
- 代替案が存在しない、または実質的な技術選択になっていない
- command step、frontmatter field、template path、artifact lifecycleの記述がDecisionの中心になっている
- REQ/SPECへ移すべき規範記述がADR本文に多い

## 対象外

以下は本RUの対象外とする。

- ADR-0023本文の再編成・削除・retire
- 既存ADR群の一括再分類
- 新規ADR作成
- command / skill の実装変更
- Issue作成
- PR作成
- GitHubへの追加変更

## 受け入れ条件

- `req-define` に、REQ/SPEC相当の内容をADR候補から除外する判定が定義されている
- `req-save` に、ADR保存直前の妥当性再検証が定義されている
- `agentdev-adr-guidelines` に、ADR禁止条件が明示されている
- `agentdev-adr-file-manager` の責務が、ADR要否判定ではなくファイル操作であることが維持されている
- `repo-agentdev-integrity` に、ADR誤分類の兆候を検出するwarningまたはobservation方針が定義されている
- 仕様変更のみの内容がADRとして新規保存されない流れになっている
- ADRとして不成立な内容はREQ/SPEC更新候補として提示される
- 本RU内に未合意のADR再編成方針が混入していない
