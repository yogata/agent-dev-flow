# RU: reporting skill 廃止と skill/SPEC 責務分離

## 背景

`agentdev-workflow-reporting` は現状、completion report variant registry、completion report template 本文、completion report 共通仕様、検査観点を保持している。

しかし、completion report variant 本文は command 固有の出力 template であり、variant 選択条件も command 固有である。  
また、completion report 共通仕様や template 配置契約は SPEC 相当であり、skill reference に置くべき内容ではない。

既存の artifact 責務境界では、command は入口・入出力・guardrail・高レベル手順、skill は runtime 実行支援・再利用可能な判断基準、template は出力構造とプレースホルダー、SPEC は repo-internal な現在仕様を扱う。

## 問題

- `agentdev-workflow-reporting` が skill として過剰である。
- 修正後は実質的に単なる completion report template registry となるため、skill として残す必然性がない。
- completion report variant 本文が skill 配下にあり、template artifact としての責務と合っていない。
- `completion-reports.md` が skill reference ではなく SPEC 相当の内容を持っている。
- skill references に SPEC 相当の状態モデル・契約・分類体系・schema・現在仕様が混在している。
- 現行 REQ / SPEC は completion report template の配置先として skill 配下を前提にしており、整理後構造と矛盾する。
- SPEC へ移管した内容を runtime command / skill が直接参照し始めると、runtime 配布物が `docs/specs/` に固定依存する不整合が発生する。

## Source Summary

本RUは、このチャット内で合意した以下の内容に基づく。

- `agentdev-workflow-reporting` は不要とする。
- completion report variant 本文は command-local template として扱う。
- 配置先は `.opencode/commands/agentdev/templates/{command}/{variant}.md` とする。
- `_templates` は既存仕様に根拠がないため採用しない。
- `completion-reports/` 階層は現時点では用途が単一であり過剰なため採用しない。
- `<command>/` 階層は variant の所有者を明確にするため採用する。
- `completion-reports.md` は SPEC 化した上で廃止する。
- すべての skill / skill references を確認対象とする。
- SPEC へ移管するのは SPEC 相当内容に限る。
- runtime 実行時に必要な短い判断表・チェックリスト・操作注意は skill / skill references に残す。
- SPEC へ移管した内容を runtime command / skill が直接の実行時依存として参照してはならない。
- 今回は既存の文書責務境界に基づく更新であり、新規ADRは原則作成しない。
- 「スキルを見直す」というADRは作成しない。
- 必要に応じて REQ / SPEC / skill / command の定義を更新する。

## 統合理由

`agentdev-workflow-reporting` の廃止、completion report template の command-local 配置、`completion-reports.md` の SPEC 化、skill references 内の SPEC 相当記述分離は、同一の責務境界是正である。

これらを分離したRUにすると、以下の不整合が発生しやすい。

- reporting skill を廃止したのに旧 template 配置が残る。
- template を移動したのに REQ-0107 の配置指定が旧構造のまま残る。
- SPEC 相当内容を移したのに skill reference が runtime 参照として肥大化したまま残る。
- SPEC へ移管した内容を runtime command / skill が直接参照し、runtime 配布物が `docs/specs/` に固定依存する。
- integrity-check が旧配置と新配置を混在して検査する。

そのため、Epic 単位で一括管理し、子Issueに分割して実施する。

## 要件化の方向

### 1. `agentdev-workflow-reporting` を廃止する

`agentdev-workflow-reporting` は廃止する。

廃止理由:

- completion report variant 本文は template artifact である。
- variant 選択条件は command 固有である。
- completion report 共通仕様は SPEC に置くべきである。
- 廃止後に skill として残す再利用可能な判断基準がない。

### 2. completion report variant 本文を command-local template に移す

completion report variant 本文は以下へ配置する。

```text
.opencode/commands/agentdev/templates/{command}/{variant}.md
```

例:

```text
.opencode/commands/agentdev/templates/req-define/feature.md
.opencode/commands/agentdev/templates/req-define/feature-epic.md
.opencode/commands/agentdev/templates/req-define/lightweight.md
.opencode/commands/agentdev/templates/req-save/standard.md
.opencode/commands/agentdev/templates/case-close/agentdev-push-failed.md
```

採用理由:

- command と密結合な template を command 領域に配置できる。
- `templates/` は既存 artifact contract の語彙と整合する。
- `{command}/` 階層により variant の所有者が明確になる。
- `completion-reports/` 階層は現時点では用途が単一であり過剰である。
- `_templates` は既存仕様に根拠がないため採用しない。

### 3. command 定義は template 参照だけを持つ

command 定義ファイルは completion report 本文を直接保持しない。

command 定義に保持してよいもの:

- 使用する template path
- variant 選択条件
- 置換変数

command 定義に保持してはならないもの:

- completion report の最終出力本文
- command 別 completion report の追加ブロック本文
- 次コマンド文面の本文定義

### 4. `completion-reports.md` を廃止する

`agentdev-workflow-reporting/references/completion-reports.md` は廃止する。

含まれている SPEC 相当内容は `docs/specs/` へ移管する。

移管対象:

- completion report の共通必須フィールド
- `結果` / `検証結果` / `git 永続化` / `次のコマンド` の責務境界
- git 操作結果の重複記載禁止
- 汎用締め文禁止
- 完了報告後の追加出力禁止
- completion report template の配置契約
- integrity-check による completion report template 検査契約

### 5. skill 内の SPEC 相当記述を分離する

SPEC へ移すもの:

- 現在仕様
- 状態モデル
- artifact lifecycle
- workflow contract
- command topology
- schema
- frontmatter contract
- finding category / route / report schema
- completion report contract
- 文書種別・artifact 間の責務境界
- repo-internal な設計説明

skill に残すもの:

- runtime 実行時に参照する短い判断表
- USE FOR / DO NOT USE FOR
- command 実行時のチェックリスト
- 操作時の注意
- 失敗回避のための実務知識

補足:

SPEC へ移管した内容を、runtime command / skill が直接の実行時依存として参照してはならない。  
runtime 実行時に必要な最小情報は、command または skill references に自己完結させる。  
`docs/specs/` は repo-internal な現在仕様の記録・検査基準として扱う。

### 6. 移管対象

すべての skill / skill references を確認対象とする。

ただし、SPEC へ移管するのは SPEC 相当内容に限る。  
runtime 実行時に必要な短い判断表・チェックリスト・操作注意は skill / skill references に残す。

Epic で一括管理し、子Issueに分割して実施してよい。

子Issue候補:

1. reporting 廃止と completion report template 移動
2. artifact contract / reporting contract の SPEC 更新
3. workflow-lifecycle 系 references の SPEC 分離
4. spec-compliance / integrity 系 references の SPEC 分離
5. req / adr / file-manager 系 references の SPEC 分離
6. integrity-check 更新
