# RU: アーキテクチャ方針のADR化

## 背景

過去チャットで、AgentDevFlow の今後の開発方針に関わるアーキテクチャ上の考え方を複数議論した。

現行リポジトリには既に以下のADRが存在する。

- ADR-0001: Command/Skill/Template/Script責任分界の正式定義
- ADR-0002: Orchestration skill作成基準の導入
- ADR-0006: Epic Issue 本文を実行順序 SSoT とする設計
- ADR-0008: DOC-MAP導入と requirements/views 廃止
- ADR-0009: REQ体系再基準化

ただし、以下の3点は既存ADRだけでは今後の開発判断基準として十分に固定されていない。

1. HITL boundary を command 分割原則にする
2. manager / orchestrator pattern を限定採用する
3. Requirement Source pipeline を promoted artifact → RU → req-define に一本化する

## 問題

AgentDevFlow の command / skill / workflow を今後追加・修正する際、上記3点がADRとして明文化されていないと、以下の判断が揺れる。

- command をどの単位で分割するか
- orchestration skill や manager pattern をどこまで適用するか
- intake / learning / promoted artifact / RU / req-define の境界をどう扱うか
- ユーザー承認前の情報を後続工程へ流してよいか
- Requirement Source と単なる候補情報をどう分けるか

## Source Summary

チャット内で、過去議論に含まれるアーキテクチャ方針を確認し、現行ADRおよびREQとの重複を整理した。

確認結果として、以下をADR化候補とした。

1. HITL boundary による command 分割原則
2. manager / orchestrator pattern の限定採用
3. Requirement Source pipeline と RU boundary

既存ADR・REQで足りるものはADR化対象外とした。

- Command / Skill / Template / Script 責務分界は ADR-0001 が既存
- Orchestration skill 作成基準は ADR-0002 が既存
- Epic Issue 本文を実行順序 SSoT とする設計は ADR-0006 が既存
- DOC-MAP / guides / SPEC / REQ / ADR の基準境界は ADR-0008、ADR-0009、REQ-0101で既存
- references/ と reference/ の統一は REQ-0103 で足りるためADR不要

## 統合理由

3件はいずれも個別command修正ではなく、今後の開発判断に継続的に影響する方針である。

HITL boundary は command 分割の上位原則であり、manager / orchestrator pattern は複雑workflowへの設計pattern適用境界であり、Requirement Source pipeline は要件化前情報の正本化境界である。

これらは相互に関連するが、決定対象が異なるため、1件のADRに統合せず、3件のADRとして分離する。

## 要件化の方向

Pattern は docs / architecture decision 変更として扱う。

実装対象は以下。

1. `docs/adr/ADR-{NNNN}.md` を3件追加する
2. `docs/adr/README.md` に追加ADRを登録する
3. 必要に応じて `docs/DOC-MAP.md` または関連guideに参照を追加する
4. 既存ADR-0001、ADR-0002、ADR-0006、ADR-0008、ADR-0009との関係をADR本文に明記する
5. ADR status は初期値 `proposed` とする

### ADR候補1: HITL boundary による command 分割原則

AgentDevFlow の workflow command は、原則として HITL boundary、すなわちユーザー判断・承認を伴う境界を基準に分割する。

1 command は、前段のHITL boundaryで確定した入力を受け取り、次段のHITL boundaryで確認・承認可能な成果物を出力する。

### ADR候補2: manager / orchestrator pattern の限定採用

manager / orchestrator pattern は、AgentDevFlow 全体の標準command構造とはしない。

このpatternは、複数Issue、Wave scheduling、依存関係、サブエージェント起動、失敗伝播、再開判定を扱う場合に限定して採用する。

現時点の主対象は `case-run` の Epic Orchestrator モードである。

### ADR候補3: Requirement Source pipeline と RU boundary

intake / learning から要件化へ進む情報は、promoted artifact を直接 `req-define` に渡さず、以下のpipelineを通す。

```text
promoted artifact
→ backlog-review
→ backlog-save
→ RU
→ req-define
```

RU は Requirement Source として、`req-define` に渡せる粒度まで整理された、要件化前の正式入力とする。

## 主対象REQまたは変更対象候補

主対象REQ候補:

- REQ-0101: 文書・REQ管理基準
- REQ-0103: Artifact責任分界
- REQ-0104: Workflow / Command Protocol
- REQ-0105: Intake / Learning / Backlog

変更対象候補:

- `docs/adr/ADR-{NNNN}.md`
- `docs/adr/README.md`
- `docs/DOC-MAP.md`
- `docs/guides/workflow-overview.md`
- 必要に応じて `docs/guides/artifact-model.md`
- 必要に応じて `docs/guides/domain-state-lifecycle.md`

## 対象外

- コード実装
- command定義の即時修正
- skill定義の即時修正
- 既存ADRのaccepted化
- 個別commandの完了報告フォーマット修正
- `references/` / `reference/` の表記揺れ修正
- git retry回数などの個別運用要件
- 未合意の追加ADR候補
- 既存ADRの全面再構成

## 受け入れ条件

- HITL boundary による command 分割原則がADRとして記録されている
- manager / orchestrator pattern の限定採用がADRとして記録されている
- Requirement Source pipeline と RU boundary がADRとして記録されている
- 各ADRに背景、決定、代替案、結果・影響、関連する決定が含まれている
- `docs/adr/README.md` から各ADRへ到達できる
- 既存ADRとの重複ではなく、補完関係として説明されている
- 個別実装手順ではなく、今後の開発方針として読める粒度になっている
- `req-define` に渡した際、CREATE / APPEND / UPDATE の判断が可能な粒度になっている
