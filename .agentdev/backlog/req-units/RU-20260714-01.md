# ADF配布物の責務境界浄化と標準ワークフローの簡素化

## 背景

AgentDevFlowでは、開発ワークフロー、永続成果物、承認境界、Quality Gateを管理する本来の責務に加え、特定の実行ハーネスに起因する委譲方法、並列実行、timeout、retry、context管理、エラー解析、コンフリクト回復などが、command、skill、REQ、ADR、SPECへ段階的に取り込まれている。

これらには実障害への対処として導入された経緯があるが、ADFの恒久的なワークフロー契約と、ハーネス固有の実行制御との境界を曖昧にしている。

将来的な抜本リファクタリングでは、アーキテクチャと文書体系を含む再構成が検討されている。しかし、変更範囲が広く、一括変更では妥当性をレビューしにくい。

そのため、現行のディレクトリ構成、公開コマンド体系、成果物モデルを維持したまま、現時点でも明確に是正できる責務越境、重複、内部矛盾を先行して除去する。

本対応の目的は新機能の追加ではない。ADFが所有しない実行制御を標準配布物から除外し、既存ワークフローを単純化することである。

## 問題

現行ADFには、責務境界を定める要件・決定が存在する一方で、その境界と矛盾する記述がREQ、ADR、SPEC、command、skillに残っている。

主な問題は以下である。

1. ADF配布物が、進行条件、停止条件、成果物、結果契約だけでなく、固定timeout、固定並列数、retry回数、context圧縮、具体的な委譲起動方法まで規定している。
2. `case-auto`が工程制御への専念を宣言しながら、委譲失敗の解析、並列スケジューリング、再委譲、コンフリクト解析などを直接所有している。
3. `case-run`がIssue実行境界を越えて、agent-dev-flowリポジトリ固有の検査や実行アルゴリズムを標準処理として保持している。
4. execution adapterが最小限の接続契約ではなく、実行エージェント構成、実装手法、検証反復、timeoutなどを含む実行方式を規定している。
5. ADF標準と適用プロジェクト固有の規則が分離しきれておらず、repo-localな検査が標準commandから直接参照されている。
6. 同じ契約がREQ、SPEC、command、skillへ重複して記載され、変更時に不整合が発生しやすい。
7. ADFが直接観測できる壁時計タイムスタンプと、ハーネス内部の実行制御時間が区別されていない。

## Source Summary

本セッションでは以下を合意した。

- 抜本的なアーキテクチャ再編や文書体系変更は、現時点では実施しない。
- 添付された抜本リファクタリング方針のうち、現行構成のまま適用できる責務境界と簡素化の考え方を先行して取り込む。
- 主目的は新しい要件や機能を増やすことではなく、ADFの責務範囲を明確にし、過剰な責務と重複記述を削除することである。
- ADFは開発ワークフロー、永続成果物、承認境界、停止条件、Quality Gate、結果契約を所有する。
- ハーネスはエージェント選定、起動方法、並列数、timeout、retry、context管理、内部監視、ハーネス固有エラー解析を所有する。
- `case-auto`は工程選択、工程間状態引き継ぎ、停止・再開判定、結果報告を中心とする薄いオーケストレーターへ縮小する。
- `case-run`はIssue実行境界、worktree、実行依頼、結果処理を中心とし、ADFリポジトリ固有の検査は標準処理から分離する。
- execution adapterは入力、結果状態、永続化先、副作用境界を定める最小契約へ縮小する。
- プロジェクト固有の規則・検査はProject Extensionsまたはrepo-local領域へ分離する。
- ADFから直接観測できるタイムスタンプは維持する。
- ハーネス内部のtimeout、queue待機、retry単位時間、推論時間などはADF標準の管理対象としない。
- 公開コマンド体系、Epic/Wave、Issue/PRフロー、成果物形式は本対応では変更しない。

## 統合理由

本RUの各変更は、すべて「ADF標準が所有する責務と、ハーネスまたは適用プロジェクトが所有する責務の境界を是正する」という単一の関心に属する。

`case-auto`、`case-run`、execution adapter、Project Extensionsを個別に修正すると、同じ実行制御が別の層へ移動するだけになり、責務越境が再発する。

また、タイムスタンプについても、単純に削除または維持するのではなく、ADFが直接観測可能な壁時計時間と、ハーネス内部時間を横断的に区別する必要がある。

したがって、局所的なcommand修正ではなく、既存の責務境界をREQ、ADR、SPEC、command、skillへ一貫して適用する一つのRequirement Unitとして扱う。

## 要件化の方向

### 1. ADF標準の責務境界

ADF配布物は以下を定義対象とする。

- 工程の目的
- 入力
- 前提条件
- 進行条件
- 停止条件
- 永続成果物
- 許可される副作用
- 禁止される副作用
- Quality Gate
- 完了条件
- 再開条件
- 実行結果契約
- ADFから直接観測可能なタイムスタンプ

ADF配布物は以下を実現方法として規定しない。

- 具体的なエージェントまたはモデルの選定
- サブエージェント階層
- 委譲APIまたは起動コマンド
- 固定並列数
- 固定timeout
- 固定retry回数
- context生成、圧縮、破棄
- ハーネス内部queue
- heartbeatまたは内部監視方法
- ハーネス固有エラー文字列の解析
- ハーネス内部plan、task、監査ログ

### 2. `case-auto`の責務縮小

`case-auto`は以下を所有する。

- 入力成果物の解決
- `auto_gate`の確認
- `artifact_actions`に基づく必要工程の決定
- 各工程への入力引き渡し
- 各工程完了後の永続状態再読込
- 継続、停止、再開の判定
- 完了済み、進行中、未実行範囲の報告
- ADFから見える全体および構成工程の壁時計時間の計測

`case-auto`は以下を所有しない。

- 工程内部手順の再実装
- 実行エージェント選定
- 子Issueの具体的スケジューリング方式
- 固定並列数
- 委譲失敗理由のrawエラー文字列解析
- context圧縮
- 固定retry回数
- コンフリクト解消用実装プロンプトの構築
- Quality Gateの再評価
- capture処理の再実装

### 3. `case-run`の責務縮小

`case-run`は以下を所有する。

- 対象Issueまたは実行単位の解決
- 独立worktreeとbranchの準備・確認
- Issue、worktree root、branchを実行主体へ引き渡すこと
- 実行結果の受領と状態処理
- PRまたは停止情報の永続状態確認
- FindingsをPR本文経由で後工程へ引き継ぐこと
- 再開可能な状態を維持すること
- ADFから見えるworktree準備、実行依頼、結果受領、後処理の壁時計時間の計測

`case-run`標準処理から以下を分離する。

- agent-dev-flowリポジトリ固有の検査
- repo-local scriptの直接実行
- 特定ファイル群を前提とするgrep規則
- 固定timeout
- 固定並列数
- ハーネス内部の実装・検証反復アルゴリズム
- ハーネス固有の異常終了解析

### 4. execution adapterの最小契約化

execution adapterは以下の契約に限定する。

- 入力
  - Issue識別子
  - worktree root
  - branch
  - 必要な副作用境界
- 結果
  - `completed-pr`
  - `blocked`
  - `failed`
  - `delegation-unavailable`
- 永続化先
  - 成功結果はPR
  - blockerまたは失敗詳細はIssueコメント
- 禁止事項
  - 指定worktree外の変更
  - Issue完了判定
  - capture成果物への直接書込み
  - ハーネス中間成果物のADF永続状態化

具体的なエージェント構成、実行command、timeout、retry、内部planはadapter契約に含めない。

### 5. Project Extensionsとrepo-local境界

Project Extensionsは、プロジェクト固有の以下の情報を追加する宣言層として扱う。

- context
- rules
- checks
- acceptance_gates
- must_not

Project Extensionsは標準commandの上書きや、特定の実行skill・エージェントの指定を行わない。

agent-dev-flowリポジトリ固有の検査、整合性確認、grep規則は、標準ADFから分離し、Project Extensionsまたはrepo-local領域で管理する。

### 6. タイムスタンプ境界

ADFから直接観測可能な壁時計タイムスタンプは維持する。

対象には以下を含む。

- `case-auto`全体の開始、終了、停止時刻
- 構成工程の開始、終了、所要時間
- `case-run`のworktree準備、実行依頼、結果受領、後処理の開始、終了、所要時間
- 結果状態が確定した時刻
- 停止時点までの工程別経過時間

以下はADF標準の必須管理対象としない。

- ハーネス内部ツール呼び出しのtimeout値
- サブエージェント内部フェーズ時間
- モデル推論時間
- queue待機時間
- retry単位の内部時間
- context圧縮時間
- ハーネス内部の監視間隔

タイムスタンプ要件は削除せず、REQでは観測要件を保持し、形式・集計・表示方法の詳細はSPECへ集約する。

### 7. 重複と矛盾の解消

同一契約をREQ、ADR、SPEC、command、skillへ重複記載しない。

- REQは満たすべき恒久要求を記述する。
- ADRは責務境界を採用した判断理由を記述する。
- SPECは入力、出力、状態、表示形式などの詳細契約を記述する。
- commandは実行時に必要な手順とガードレールを記述する。
- skillは共有知識または最小プロトコルを記述する。

既存文書間で、インラインフォールバックの可否、`task()`等の具体実行手段、harness制御の配置先に矛盾がある場合は、ADF標準が実行方法を規定しない方向へ統一する。

- 旧文書体系の一括廃止

## 受け入れ条件

1. ADF配布command、skill、docsが、進行条件、停止条件、成果物、Quality Gate、結果契約、ADF観測可能なタイムスタンプを中心とする記述へ整理されている。
2. ADF配布物から、固定timeout、固定retry回数、固定並列数、context圧縮、ハーネス固有エラー文字列解析が除去されている。
3. `case-auto`が、入力解決、工程選択、状態引き継ぎ、停止・再開判定、結果報告、ADF可視タイムスタンプ計測に限定されている。
4. `case-auto`が各工程の内部処理、Quality Gate、capture、Issue階層判定を再実装していない。
5. `case-run`が、Issue実行境界、worktree、実行依頼、結果処理、ADF可視タイムスタンプ計測に限定されている。
6. agent-dev-flowリポジトリ固有の検査やrepo-local scriptが、標準`case-run`から直接実行されていない。
7. execution adapterが入力、4状態結果契約、永続化先、副作用境界を中心とする最小契約になっている。
8. execution adapterが具体的なエージェント構成、実行command、固定timeout、固定並列数を規定していない。
9. Project Extensionsが宣言データとして定義され、特定skill名、エージェント名、委譲方法を標準契約として要求していない。
10. ADFから直接観測できる開始、終了、停止、工程別所要時間が維持されている。
11. ハーネス内部の推論時間、queue待機時間、retry単位時間、context圧縮時間がADF標準の必須成果物になっていない。
12. タイムスタンプの観測要件がREQに残り、表示形式・集計方法の詳細がSPECへ集約されている。
13. インラインフォールバック、委譲起動不能、具体的起動手段に関するREQ、ADR、SPEC、command間の矛盾が解消されている。
14. 同一契約のREQ、SPEC、command、skill間の重複記述が削減されている。
15. 現行の公開コマンド体系が維持されている。
16. 現行のIssue／PRワークフロー、Epic／Wave、RU、draft、REQ、ADR、SPECの成果物形式が維持されている。
17. 新規機能追加ではなく、既存責務の削減、移管、重複解消を中心とした変更になっている。
18. 抜本的なdocs再構築を行わず、現行構成内で変更が完結している。
19. 変更後も、ユーザー承認境界、repo外副作用禁止、Quality Gate、再開可能性、Findings／Capture引き継ぎが維持されている。
20. ADF標準から特定harness固有の記述を除去しても、標準ワークフロー契約が単独で成立する。
