# ガバナンス統合再編後の恒久文書純化と完了状態の是正

## 背景

AgentDevFlowの基準体系、統治境界、索引管理を再編するため、以下の処置が段階的に実施された。

- 現行REQ、ADR、SPEC、command、skill、Project Extensions、索引類を対象とした三軸監査
- 一時監査台帳を用いた処置候補の分類
- REQ-0102の分割
- workflow-contractsの物理統合
- ADR群の再構成
- 索引類のAUTOGEN化
- IR-061による生成結果整合性検査
- SKILL.mdへの原本参照宣言の追加
- Phase 1監査台帳の後続入力への転記と廃棄

索引生成スクリプト、AUTOGEN領域、docs-checkによる整合性検証は実装され、手作業による件数・一覧更新の一部は機械生成へ移行した。

一方、適用後の最新状態を確認した結果、再編工程で使用した一時的な情報、作業計画、完了宣言が、現行REQ、accepted SPEC、accepted ADRへ残存していることが確認された。

このため、自動生成機構の実装は進んだものの、統合再編の本来の目的である以下の状態には到達していない。

- 現行REQが現在満たすべき状態だけを保持する
- ADRが現在有効な意思決定だけを保持する
- SPECが現在実装されている契約を矛盾なく表す
- one-timeの監査情報、作業計画、件数、Issue、PR、Wave、Phaseを恒久基準から分離する
- 完了宣言と実際の適用範囲を一致させる

## 問題

### 1. SC-002の内部状態が矛盾している

- 生成スクリプトやCI組込み等の実装は対象外であるという記述

- 本SPECはPhase Bの成果物であるという記述
- Phase Eを全索引展開としながら、複数の未対応索引を別途検討事項として残す記述

これにより、現在実装されている契約と、過去の導入計画が区別されていない。

### 2. Phase 3完了宣言が実際の適用範囲と一致していない

Phase 3はSC-002 Phase CからPhase Eまで完了したものとして扱われている。

### 3. REQ-0162に移行作業と一時情報が残っている

REQ-0162には、現在満たすべき恒久要件ではなく、特定の是正作業に関する以下の情報が残っている。

### 4. SC-003が今回固有の監査運用を恒久化している

監査台帳ライフサイクルSPECは、一時監査成果物を恒久管理しないことを目的としている。

### 5. ADR体系に現在状態との矛盾と重複が残っている

### 6. SKILL.mdのDERIVE完了表明が実態と一致していない

複数のSKILL.mdに、SPECを原本とする共通宣言が追加された。

これは正規所有者の明確化とREFERENCE強化としては有効である。

したがって、現在状態を「SKILL.md DERIVE機構完了」と表現することは過大である。

## 統合理由

これらの問題は個別の誤記ではなく、同一の根本原因を持つ。

根本原因は、再編を進めるために使用した以下の情報が、現在基準を表す恒久文書へ混入したことである。

- 監査上の識別子
- 作業フェーズ
- 実施Wave
- Issue、PR
- 一時的な件数
- 移行手順
- 完了判定の証拠
- 将来実施予定
- 再編工程固有の分類

そのため、各ファイルを個別修正するのではなく、「現行基準は現在満たすべき状態だけを保持する」という単一の要件単位として統合して扱う。

この統合により、REQ、ADR、SPECそれぞれで同じ是正原則を適用できる。

## 要件化の方向

### 1. 現行文書の現在状態純化

現行REQ、accepted ADR、accepted SPECは、現在有効な状態、契約、判断だけを保持する。

以下は現行基準の本文に保持しない。

- 作業Phase、Wave
- AG、CR、ACT、TS等の再編工程識別子
- Issue、PR、Epic番号
- RU番号
- intake、learningの個別ファイル名
- 特定時点の検出件数
- 移行手順
- 実施順序
- 作業完了の証拠
- 将来の実装計画
- 既に完了した導入段階

必要な履歴はgit履歴、Issue、PR、retired文書または作業記録で保持する。

### 2. SC-002の現在契約への再構成

SC-002は、導入計画ではなく現在稼働している索引生成契約として再構成する。

少なくとも以下を満たす。

- 生成スクリプトが存在する現在状態を記述する
- IR-061による検査契約を記述する
- 自動生成済み対象と未対応対象を分離する
- Phase AからEまでの導入履歴を現在契約から除去する
- 「実装は対象外」「Phase B成果物」等の旧状態を除去する
- 未対応対象が残る限り、全索引展開完了と表明しない
- 未対応対象を恒久的に人手管理とする場合、その判断を現在契約として明示する
- 自動生成対象、生成元、人手編集領域、検査方法を一意に定める

### 3. Phase 3完了状態の是正

Phase 3の完了表明は、実際に満たした範囲へ限定する。

以下を満たすまで「全索引展開完了」と扱わない。

- SC-002で対象とされた全領域が生成対象となっている
- または、未対応領域を対象外とする現在契約が正式に確定している
- 完了条件と現在の対象範囲が一致している
- 「別途検討事項」が完了条件内に残っていない

### 4. REQ-0162の恒久要件への縮約

REQ-0162は、配布物とharnessの責務境界、および最終的に満たすべき配布物状態だけを保持する。

以下はREQ本文から除去する。

- 特定Epicの完了条件
- 特定テスト項目
- 一時的な残留件数
- NG、WARNINGの基準時点件数
- RU番号
- intakeファイル参照
- 段階的な分割case実施方法
- grep証拠をPR本文へ書く等の作業記録方法
- 文脈保持型置換等の移行作業手段

最終状態として必要な以下は残す。

- 配布物にプロジェクト固有識別子を含めない
- 配布物に消費プロジェクト固有パスを含めない
- 対象検査で不整合が0件である
- 配布物とharnessの責務境界が維持される

### 5. SC-003の統合と退役

一時分析成果物のライフサイクルに必要な一般契約を、既存のdraft、backlog、artifact lifecycleを所有するSPECへ統合する。

統合対象は以下に限定する。

- 一時成果物は恒久状態として扱わない
- 後続入力が自足した時点で廃棄できる
- 廃棄前に移管漏れを検査する
- 廃棄の事実はgit履歴で追跡できる

今回固有のPhase、AG、CR、監査台帳パス、既知適用例は統合しない。

一般契約の統合後、SC-003は現行SPECから退役させる。

### 6. ADR体系の純化

ADR-0127は以下の状態へ再構成する。

- case-autoの構成工程委譲という意思決定に限定する
- case-runのインライン実行との関係を一度だけ記述する
- Epic Issue本文の単一書き手をcase-closeへ統一する
- 重複したコンテキスト分離効果を統合する
- 行数、具体的REQ行番号、作業履歴、ハーネス固有情報を除去する
- side effectの詳細契約はSPECへ移す
- 現在有効な決定、理由、代替案、影響だけを保持する

ADR-0134は`proposed`のまま残さない。

現行アーキテクチャ判断として独立して必要であればacceptedとし、既存REQ、SPEC、他ADRへ吸収済みであればdeprecatedまたはsupersededとする。いずれの場合も、現在の正規所有者と整合するstatusへ確定する。

### 7. SKILL.mdの現在分類の是正

SKILL.mdにSPEC原本参照を追加した現在状態は、REFERENCE強化として扱う。

DERIVE完了と表明できるのは、少なくとも以下のいずれかを満たす場合に限る。

- SPECから対象領域を機械生成する
- SPECとSKILL.mdの対応領域を機械検証する
- 共通テンプレートから原本宣言を生成する
- 手作業複製ではなく、変更追従を保証する仕組みが存在する

現在の原本宣言は維持できるが、「DERIVE機構完了」という表現は使用しない。
