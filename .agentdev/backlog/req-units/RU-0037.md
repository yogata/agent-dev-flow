# RU: intake / learning の promote コマンド統一

## 背景

現行の intake / learning パイプラインは、以下のように review/refine 系コマンドと promote 系コマンドが分離されている。

- `intake-review` → `intake-promote`
- `learning-refine` → `learning-promote`

ユーザー観点では、どちらも最終的に promote 対象を選ぶ操作であり、独立コマンドとして分ける必要性が薄い。

## 問題

`intake-review` / `learning-refine` が独立コマンドとして存在することで、以下の問題がある。

- コマンド数が増え、操作導線が複雑になる
- HITL の位置が分散する
- `intake/accepted/` のような中間状態が必要になる
- docs / commands / skills / templates に旧分割前提の記述が残る
- learning では refine と promote の責務境界がユーザー操作上わかりにくい

## Source Summary

このRUは、チャット内で合意した以下の内容を source とする。

- `intake-review` / `learning-refine` は独立コマンドとして扱わず、`*-promote` に統一する
- HITL は削除せず、promote コマンド内へ集約する
- `intake/accepted/` は不要とする
- `learning` の `evaluation-report.md` は廃止せず、`learning-promote` 内部の評価根拠 artifact として維持する
- `docs/` だけでなく、commands / skills / templates / REQ / SPEC / ADR / integrity-check も影響範囲に含める

## 統合理由

`intake-review` は `intake-promote` の前処理であり、採用済み item を `accepted/` に移すための中間コマンドになっている。

`intake-promote` に review 相当処理を統合すれば、`inbox/` から直接 promote / defer / reject を確定できるため、`accepted/` は不要になる。

`learning-refine` は単なる選別ではなく、正規化、問題クラス分類、8軸評価、`evaluation-report.md` 生成を担っている。そのため処理自体は残す。ただし独立コマンドとして分ける必要はなく、`learning-promote` 内部の refine phase として扱う。

## 要件化の方向

`intake-review` / `learning-refine` を独立コマンドとして廃止し、`intake-promote` / `learning-promote` に統合する。

HITL は削除しない。promote コマンド内の対象選択・判定確定ステップへ集約する。

主対象候補:

- intake pipeline を定義する REQ

## 要件詳細

### intake

`intake-review` を廃止する。

`intake-promote` は `intake/inbox/` から直接読み込み、ユーザーに promote / defer / reject 案を提示する。

ユーザーが確定した promote 対象のみ promoted artifact を生成する。

`intake/accepted/` は廃止する。

変更後の状態遷移は以下とする。

````text id="ptkj8f"
.agentdev/intake/inbox/
  ↓ intake-promote
.agentdev/intake/promoted/
.agentdev/intake/archive/promoted/
.agentdev/intake/archive/rejected/
````

defer された item は `intake/inbox/` に残す。

通常 intake と REQ再構成 intake は混在させない。`intake-promote` 内で区別して処理する。

### learning

`learning-refine` を廃止する。

`learning-promote` は refine 相当処理を内部フェーズとして持つ。

`learning-promote` は以下を行う。

1. learning item を読み込む
2. 正規化する
3. 問題クラス分類を行う
4. 8軸評価を行う
5. `evaluation-report.md` を生成または更新する
6. promote / defer / reject / duplicate 案を提示する
7. ユーザーが promote 対象と prune 対象を確定する
8. promoted artifact を生成する
9. promoted / rejected / duplicate 対象を active archive から prune する
10. deferred は active archive に残す

`evaluation-report.md` は廃止しない。`learning-promote` 内部の評価根拠 artifact として維持する。

promotion 判定は `evaluation-report.md` を根拠に行う。

promotion 判定フェーズで raw item を場当たり的に再分類してはならない。

### HITL

HITL は以下へ集約する。

- `intake-promote`: promote / defer / reject の確定
- `learning-promote`: promote / defer / reject / duplicate と prune 対象の確定

AI単独で promote や prune を確定してはならない。

## 対象外

以下は対象外とする。

- HITL の削除
- learning の `evaluation-report.md` 廃止
- AI単独での promote / prune 確定
- backlog-review / req-define / case-auto の責務変更
- 実装方式の詳細設計
- コード実装

## 受け入れ条件

- `learning-refine` が独立コマンドとして残っていない
- `intake-promote` が `intake/inbox/` から直接 promote / defer / reject 判定を行う
