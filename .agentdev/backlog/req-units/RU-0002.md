# ADR再編・ADR-01XX baseline化・旧ADR retired移動

## 背景

AgentDevFlow では ADR が過去に広く使われ、SPEC 相当の現在仕様、workflow 定義、command 動作仕様、運用方針、template 形式まで ADR に混入した。その結果、ADR の改変履歴が歪み、現行根拠として読むべき判断と歴史的経緯が分かりにくくなっている。

このリポジトリでは試行錯誤を伴う開発が避けにくく、ADR 自体も管理・改定対象になる。そのため、通常時は ADR を厳しく絞りつつ、ADR 体系の全面改定・責務再分類・圧縮時には統制された例外が必要である。

さらに、再編後の現行 ADR と再編前の歴史 ADR を明確に分離するため、移管後の現行 ADR は `ADR-01XX` 番台とし、既存 `ADR-00XX` は retired ADR として `docs/adr/retired/` に移動する方針とする。

## 問題

REQ-0112 には、既存 deprecated / superseded ADR の原則再編集禁止がある。しかし、全面改定時の例外が明示されていないため、誤分類された ADR を整理すべき局面でも、本文の再分類・圧縮・移管先注記が難しくなる。

また、既存 `ADR-0001` から `ADR-0023` は、accepted / proposed / deprecated / superseded が混在しており、現行根拠として読める ADR と歴史参照として読むべき ADR の境界が番号や配置から分かりにくい。

このまま `docs/adr/` 直下に旧 ADR と新 ADR が混在すると、全面改定後も読者や agent が旧 ADR を現行根拠として参照するリスクが残る。

## Source Summary

チャット内で以下を合意した。

- ADR の歪みの主因は、ADR に SPEC / workflow / command 仕様相当の内容まで入っていたことである。
- ADR に残すものは、採用した不可逆な技術判断、却下した主要代替案、その判断が必要だった理由、現在も有効な影響に限定する。
- 振る舞い・制約・状態は REQ、schema・path・lifecycle・現行仕様は SPEC、方針・案内は guide に逃がす。
- workflow、command 動作仕様、template 形式、非技術的合意は ADR 化禁止とする。
- 通常時は accepted ADR の意味変更を禁止し、deprecated / superseded ADR も原則再編集しない。
- ADR 体系の全面改定・責務再分類・圧縮時は、ユーザー承認済み req-save 入力に明示された場合のみ例外を許可する。
- accepted ADR の既存判断を覆す場合は、新規 ADR を作成し、旧 ADR を superseded または deprecated とする。
- 移管後の現行 ADR は `ADR-0101` 以降の `ADR-01XX` 番台とする。
- 既存 `ADR-00XX` はリネームではなく歴史 ADR として扱い、必要な判断を `ADR-01XX` に引き継いだ後、`docs/adr/retired/` に移動する。
- 旧 ADR への現行根拠参照は禁止し、履歴参照は `docs/adr/retired/ADR-00XX.md` として明示する。
- `docs/adr/README.md` は Current Baseline と Retired / Historical View を分離する。
- integrity-check は ADR 意味妥当性を strict NG にせず、誤分類兆候を warning / observation として扱う。
- 別エージェントが改変する前提で、本 RU は実行計画と影響対象の整理を提供する。

## 統合理由

ADR 再編の再発防止には、ADR 本文だけでなく、REQ-0112 の例外規定、ADR index、移管先 REQ/SPEC/guide、req-define / req-save の判断ゲート、integrity の検出方針を一体で整理する必要がある。

また、旧 ADR と新 ADR を同じ番号帯・同じディレクトリに残すと、全面改定後も現行根拠と歴史参照の混同が残る。`ADR-01XX` baseline と `docs/adr/retired/` を導入することで、再編断面を番号帯と配置の両方で明確化できる。

## 要件化の方向

REQ-0112 に、ADR 全面改定時の例外規定、`ADR-01XX` baseline、旧 ADR retired 移動を追加する。

要件化では以下を明確にする。

- ADR 体系の全面改定・責務再分類・圧縮を目的とする場合に限り、ユーザー承認済み req-save 入力で例外を許可する。
- 全面改定時は、編集目的、対象 ADR、変更種別、移管先 REQ/SPEC/guide、現行根拠として残す ADR を記録する。
- accepted ADR の決定内容を別判断へ意味変更してはならない。
- 既存判断を覆す場合は新規 ADR を作成し、旧 ADR を superseded または deprecated として扱う。
- 移管後の現行 ADR は `ADR-0101` 以降に作成する。
- 既存 `ADR-0001` から `ADR-0099` は再編前 ADR として `docs/adr/retired/` に移動する。
- `docs/adr/README.md` は Current Baseline ADR と Retired / Historical ADR を分離する。
- `docs/adr/ADR-00XX.md` から `docs/adr/retired/ADR-00XX.md` への参照更新を行い、旧 ADR を現行根拠として読ませない。
- 原則として旧パスに stub は残さず、参照を更新する。ただしリンク互換性を優先する場合は別途明示判断する。
- integrity-check は `docs/adr/ADR-01XX.md` を現行 ADR collection、`docs/adr/retired/ADR-00XX.md` を retired ADR collection として区別する。
- req-define / req-save は ADR 要否判断の根拠、ADR 不要理由、除外理由、移管先を構造化して保持する。
- learning-promote / backlog-review 由来の ADR 候補も adr-guidelines の禁止条件で絞る。
- repo-agentdev-integrity は ADR 意味妥当性を strict 判定せず、誤分類兆候を warning / observation として扱う。

### Phase 2: ADR-01XX baseline 作成

旧 ADR を移動する前に、現行根拠となる `ADR-0101` 以降を作成する。

- `ADR-0101+` は不可逆な技術判断のみを記録する。
- 旧 accepted ADR から必要な判断だけを引き継ぐ。
- workflow / command 仕様 / SPEC 詳細は ADR 本文に入れない。
- 各 `ADR-01XX` は、旧 `ADR-00XX` から何を引き継いだかを Related Decisions または歴史注記で最小限記録する。

### Phase 3: 旧 ADR retired 移動

`docs/adr/ADR-0001.md` から `docs/adr/ADR-0023.md` を `docs/adr/retired/` に移動する。

- 移動前に、現行判断が `ADR-01XX` に引き継がれていることを確認する。
- 旧 ADR の frontmatter status は、必要に応じて superseded / deprecated に整理する。
- 旧 accepted ADR は、現行根拠として残さず `ADR-01XX` へ接続する。
- 原則として旧パスに stub は残さず、参照元を retired path または `ADR-01XX` へ更新する。

### Phase 4: ADR index 再構成

docs/adr/README.md を全面再構成する。

- Current Baseline View: `ADR-0101+` のみを現行根拠として掲載する。

- Status View / Topic View / Decision Map / Related REQ を新構造に合わせて更新する。
- `ADR-00XX` が現行根拠の Topic View に混入しないようにする。

## 対象外

- この RU 自体ではファイル編集、保存、commit、push を行わない。
- 個別 ADR 本文の全面改定実作業は別エージェントが実施する。
- 新規 ADR の具体本文作成は本 RU の直接対象外。
- Issue / PR 作成は対象外。
- コード実装、テスト実装、integrity script の具体実装は対象外。
- retired REQ の大規模再編集は対象外。
- 旧 ADR パスに stub を残すかどうかの最終判断は、リンク互換性要件が必要な場合のみ別途判断する。
