## Source Summary

`docs/specs/` 直下の基盤SPEC群をドメイン別に体系化する合意内容:

1. **現状問題**: `docs/specs/README.md` は直下を「システム全体の構成、フォーマット、整合性検査など基盤SPEC」と定義するが、実際は分類軸がない。基盤モデル、設計原則、文書種別・粒度基準、品質・検査契約、ルールカタログ、メトリクス、成果物契約、ローカル版仕様、コマンド文書規約、移行済み旧SPEC が横並び。どの SPEC が何の原本か分かりにくい。
2. **推奨方針**: `commands/`, `skills/`, `workflows/` は維持。`docs/specs/` 直下だけを体系化。全面的な `behavior/` / `catalog/` 分割ではなく、基盤SPECのドメイン別サブディレクトリ化。SPEC という文書種別は増やさず、既存3層も壊さない。
3. **提案構造**（ドメイン別サブディレクトリ）:
   - `foundations/`（system, document-model, patterns, design-principles）
   - `responsibilities/`（document-type-responsibilities, artifact-responsibilities, artifact-contracts）
   - `quality/`（quality-specs, quality-gates, req-health-metrics, spec-health-metrics）
   - `integrity/`（integrity-contracts, integrity-rule-catalog, rule-ownership, docs-spec-rebuild-integrity, backticks-identifier-threshold, rules/IR-044-req-spec-boundary.md）
   - `local/`（runtime-package-boundary, local-case-file, local-generation, local-transform）
   - `authoring/`（command-file-format）
4. **段階移送（一括移動回避）**:
   - `README.md` に直下 SPEC の分類方針を追加
   - 新規 SPEC は新分類に従って作成
   - `spec-health-metrics.md` は `quality/` 配下に新設
   - `integrity-rule-catalog.md` の詳細は `integrity/rules/` へ分離
   - 既存直下ファイルは inspect/backlog で段階移送
   - 旧パス参照は移送時に更新

6. **SPEC 文書種別の増不加**: CATALOG 等の新文書種別は作らない。既存の SPEC 文書種別内でのディレクトリ整理。
7. **RU-1 との連携**: `spec-health-metrics.md` の概念・内容（REQ/SPEC 健全性の双方向メトリクス）は RU-1 で決定され、その配置先（`quality/`）は本 RU-2 で決定される。RU-1 で「何を品質メトリクス SPEC に入れるか」、RU-2 で「その SPEC をどこに置くか」。
