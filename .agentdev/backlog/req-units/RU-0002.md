### 望ましい変更

`docs/guides/domain-state-lifecycle.md` の Intake Lifecycle / RU Lifecycle 周辺に、以下の内容を追記する。

#### 1. raw intake item → accepted / rejected / hold

`intake-review` は raw intake item の採用・却下・保留のみを判定する。

- raw item 本文は編集しない
- review 結果を item 本文や frontmatter に書き込まない
- 状態はディレクトリ配置で表現する
- 採用: `.agentdev/intake/accepted/`
- 却下: `.agentdev/intake/archive/rejected/`
- 保留: `.agentdev/intake/inbox/` に残置

#### 2. accepted intake item → promoted artifact

`intake-promote` は accepted item を `req-backlog` が読める promoted artifact に整形する。

promoted artifact に含めるべき情報は以下。

- 元 item
- 観測内容
- 問題
- 影響
- 対象範囲
- 対象外
- 既存要件・仕様との関連
- 望ましい変更
- 完了条件候補
- 統合・分割メモ

ここでいう整形は、要件化ではない。
REQ要件行、Issue本文、Epic構成案までは作らない。

複数 item を束ねる場合は、同一の対象領域または変更意図を持つものに限定する。
独立した完了条件を持つ論点は、後続の RU で分割できるように promoted artifact 内で分けて記述する。

#### 3. promoted artifact → RU

`req-backlog` は promoted artifact を分析し、以下のいずれかで RU を生成する。

- 1:1: 1 promoted artifact → 1 RU
- N:1: 複数 promoted artifact → 1 RU
- 1:N: 1 promoted artifact → 複数 RU

RU は要件ではなく、`req-define` に渡す Requirement Source である。

RU には以下を含める。

- source_type
- sources
- source artifact の要点
- source summary
- 統合・分割理由
- 要件化の方向

RU 生成に成功した promoted artifact は削除する。
矛盾がある promoted artifact は RU 化せず残置し、ユーザー確認対象とする。

#### 4. RU → req-define → REQ / Issue

`req-define` は RU を入力として要件を壁打ちする。

- RU は要件化済みではない
- RU から既存REQ照合、CREATE / APPEND / UPDATE 判定、要件展開を行う
- REQ保存は `req-save`
- Issue作成は `case-open`
- Epic + 子Issue構成の判断は `case-open`
- intake-promote / req-backlog では Epic Issue 化を考慮しない

- REQ ファイルの新設・更新
- GitHub Issue の作成
- Epic Issue 構成設計
- intake-promote / req-backlog の実装変更
