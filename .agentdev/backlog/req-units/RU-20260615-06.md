# RU: req-define 全 work_type での draft 保存

## Sources

intake promoted artifact で観測された構造的ギャップ: req-define Step 9 が bugfix / maintenance / docs_chore の draft 保存を不要とし、セッション内完結としている。結果として maintenance OU の要件内容がファイルとして永続化されず、case-auto や case-open が入力として検出できない。

## Source Summary

req-define は work_type が feature の場合のみ draft ファイルを作成する。それ以外の work_type は「ドラフト保存不要。セッション内で完結」とし、要件内容をセッション内に留める。

この設計は case-auto が maintenance OU を処理する際に障害となる。case-auto Step 1 の入力解決は「明示パス→draft単一検出→セッション内要件doc」の優先順位だが、draft がない OU は第3優先のセッション内要件docに頼るしかなく、draft-meta がないため work_type 読取（Step 2）が機能しない。

セッション終了時の要件消失リスクもある。

## 統合理由

本RUは req-define の draft 保存対象の拡張に限定する。case-auto の逐次処理改善は別RU（RU-20260615-07）で扱う。両者は対象REQ（REQ-0102 vs REQ-0114）と対象command（req-define vs case-auto）が異なるため分離する。

## 要件化の方向

req-define は全 work_type で `.agentdev/drafts/req-draft-{topic-slug}.md` を保存する。work_type は draft-meta に記録し、後続工程の分岐制御に使用する。

ファイル保存の有無ではなく、どのコマンドが消費するかを work_type で決定する:
- feature: req-save → case-open → case-run → case-close
- bugfix / maintenance / docs_chore: case-open（req-save スキップ）→ case-run → case-close

draft の削除タイミングは全 work_type で共通: case-open の Issue 作成 + VERIFY 成功後。
