# RU: case-auto Epic flow での case-open 共通終了スキップ防止

## 背景

case-auto（`/agentdev/case-auto`）で Epic flow（scale=large）を実行した際、case-open の共通終了ステップ（Step 17〜18-2: コメント追加・ドラフト削除・RU削除・完了報告）がスキップされ、要件ドラフト（`.agentdev/drafts/req-draft-*.md`）と RU ファイル（`.agentdev/backlog/req-units/RU-*.md`）が削除されないまま Epic Issue がクローズされた。

具体的には、REQ-0124（inspect-* 完全直接移行）の case-auto 実行で、case-open task が Epic Issue 作成（Step 4-8）までを実行し、共通終了（Step 17〜18-2）を実行しなかった。結果として `req-draft-inspect-migration-completion.md` と `RU-20260615-01.md` が残存し、ユーザーが手動で指摘するまで気づかれなかった。

## 問題

本問題は複合的要因により発生した。単一のバグではなく、構造的な設計ギャップである。

### 1. case-open.md の構造的曖昧性

`src/opencode/commands/agentdev/case-open.md` のセクション構造が原因。

- L49: `### Epic flow（Step 4〜8）` — Epic flow が Step 8 で「終了」するかのように見える
- L75: `### Standard flow + 共通終了（Step 14〜）` — **2つの概念を1つのセクションに結合**。"Standard flow" と "共通終了" が分離されていない
- L85: `18. ドラフト削除` — `[Standard]` ラベルなし（共通終了の意図）だが、セクションタイトルの "Standard flow" に引きずられて Standard 専用と誤解されやすい
- L87: `18-1. RU ファイル削除` — 同上
- L83: `17. コメント追加...（Epic flowではEpic Issueにコメント追加）` — Epic flow 向け注記があるため、Step 17 は共通終了の意図と推測できるが、Step 18/18-1 には同様の注記がない

**核心**: Epic flow セクション（L49-73）から共通終了（L75-93）への**明示的なルーティングがない**。「Epic flow 完了後、Step 17〜18-2 を実行すること」という指示がない。

### 2. case-auto.md の分岐ロジックのギャップ

`src/opencode/commands/agentdev/case-auto.md` の実行パス定義が原因。

- L26: 「各工程の後段処理（**case-open の RU 削除**...）も含めて既存コマンド定義に従うこと」 — ✅ RU 削除を明示的に要求
- L29: 「case-open 相当処理の完了後、出力を確認して以下のいずれかに分岐する」 — **"case-open 相当処理" の範囲が不明確**
- L31: 「Epic Issue: Step 4-1〜4-3 のキュー処理に進む」 — Epic flow の場合、Step 4-1（キュー処理）に分岐し、共通終了がスキップされる

L26 の要求（RU 削除含めよ）と L29-31 の実行パス（"case-open 相当処理" を狭く解釈）の間にギャップがある。

### 3. 検証の欠如

case-close（Step 11「Domain state 永続化」）が `.agentdev/` を commit/push する際、削除されるべきファイル（ドラフト・RU）が残っていることを検出しない。case-close は learning/intake capture や PR マージ状態は検証するが、「ドラフト・RU が削除されているか」を検証するステップを持たない。

### 4. 過去の再発パターン

git log から、同様の手動補完パターンが複数確認される:

- `acc0302`: 「Case-close Step 11 did not capture the RU deletion performed by case-open Step 18-1. Completing the .agentdev domain state cleanup for #801.」
- `93432a9`: 「chore: delete RU-20260614-01 after case-open Issue #763 creation + VERIFY success」
- `e8520af`, `9903b64`, `4f29d0c` 等: 「remove consumed RU files」の手動実行

この問題は今回が初めてではなく、構造的に繰り返し発生している。

## Source Summary

case-auto の Epic flow 実行で、以下の合意的事実を確認した:

- case-open.md の Step 17〜18-2（コメント追加・ドラフト削除・RU削除・完了報告）は Standard flow と Epic flow の両方で実行されるべき共通終了処理である
- 現状の case-open.md 構造では、Epic flow から共通終了へのルーティングが明示されていない
- case-auto.md は L26 で「RU 削除含めよ」と要求しつつ、L29-31 の分岐ロジックで "case-open 相当処理" を狭く解釈させる構造になっている
- 過去に複数回の手動補完が発生しており、構造的問題の是正が必要

## 統合理由

3つの問題（case-open.md 構造・case-auto.md 分岐ロジック・検証欠如）は、いずれも「case-auto Epic flow でのファイルクリーンアップ漏れ」という同一の症状の原因であり、分離して扱うと中間状態の不整合が残る。したがって1つの Requirement Unit として統合し、後続の req-define で対象ファイル群と改善策を一括で扱えるようにする。

## 要件化の方向

case-auto Epic flow での case-open 共通終了スキップを防止する。以下の3軸で改善する:

### 軸1: case-open.md のセクション構造修正

"Standard flow" と "共通終了" を別セクションに分離し、Epic flow 完了後に共通終了（Step 17〜18-2）を実行することを明示する。

以下のいずれかの構造にする:

**案A（セクション分離）**:
```
### Epic flow（Step 4〜8）
...
### Standard flow（Step 14〜16-1）
...
### 共通終了処理（全フロー共通・Step 17〜18-2）
17. コメント追加...
18. ドラフト削除...
18-1. RU ファイル削除...
18-2. 完了報告...
```

**案B（ルーティング明記）**:
```
### Epic flow（Step 4〜8）
...
8-1. OU 結果の書き戻し...

**Epic flow 完了後、共通終了処理（Step 17〜18-2）を実行すること**

### Standard flow + 共通終了（Step 14〜）
...
```

### 軸2: case-auto.md の分岐ロジック修正

L29-31 の "case-open 相当処理" に共通終了を含めることを明記する。

```
case-open 相当処理の完了後、出力を確認して以下のいずれかに分岐する:
- **Standard flow（単一 Issue）**: 既存の直列フロー（case-run → case-close）をそのまま実行
- **Epic Issue（マルチREQ または 単一REQ Epic flow）**:
  1. case-open の共通終了処理（Step 17〜18-2: コメント追加・ドラフト削除・RU削除・完了報告）を必ず実行すること
  2. Step 4-1〜4-3 のキュー処理に進む
```

### 軸3: 検証ゲートの追加

case-auto の完了報告（Step 8）または case-close（Step 11）に、ファイルクリーンアップ状態の検証ゲートを追加する。

```
**クリーンアップ検証ゲート**:
- case-open 相当処理完了後（または case-close Step 11 実行前）、以下を検証:
  - ドラフトファイル（.agentdev/drafts/req-draft-{topic-slug}.md）が削除されていること
  - RU ファイル（.agentdev/backlog/req-units/RU-*.md）が削除されていること
  - 両者が残存する場合、停止して手動削除を依頼すること
```
