# RU: active REQ 全件の状態要件化

## 背景

本セッションでは、REQ-0123「workflow-lifecycle 宣言的純化」、REQ-0124「docs-review/skill-review/diagnostics-* → inspect-* 完全直接移行」、REQ-0120「Runtime Command 非必須参照除去」、REQ-0121「Runtime Command 規範語見直し + integrity 検査再定義」、REQ-0119「コマンド・スキル・サブエージェント責務分界の再基準化」、REQ-0115「docs-* command suite 定義」について、要件として不適切ではないかを確認した。

議論の結果、個別のREQだけでなく、active REQ 全件に対して「移行・変更・再定義・削除・改名・移管・除去などの作業手段を要件化してはならない」という方針を適用することで合意した。

retire は除外する。active REQ は retired せず、正しい姿へ修正する。例外は置かない。

## 問題

現行 active REQ には、満たすべき状態・責務・制約ではなく、以下のような作業手段が要件行、title、目的、適用範囲に混入している。

- 移行する
- 改名する
- 削除する
- 除去する
- 見直す
- 再定義する
- 移管する
- 抽出する
- 置換する
- 更新する
- 適用する
- 反映する
- 完了する
- 修正する
- 整理する
- 統合する
- 追加する
- 追記する

これは、REQ を「現在満たすべき要件」の canonical source として扱う方針と衝突する。

移行した後、再定義した後、改名した後、削除した後の状態が要件であり、移行・再定義・改名・削除そのものは手段である。

## Source Summary

本セッションで合意した内容は以下。

- REQ-0123 は、workflow-lifecycle を宣言的に純化する作業や、コマンド固有手順を目的別スキルへ移管する作業が要件化されている点が不適切である。
- REQ-0124 は、`inspect-*` が要件であり、`docs-review` / `skill-review` / `diagnostics-*` からの移行は要件ではない。
- REQ-0120 は、非必須参照を除去する作業ではなく、runtime command が保持できる参照範囲を要件化すべきである。
- REQ-0121 は、規範語見直しや integrity 検査再定義ではなく、command の制約表現と integrity 検査責務を要件化すべきである。
- REQ-0119 は、再基準化ではなく、command / skill / subagent の責務分界を要件化すべきである。
- REQ-0115 の `docs-* command suite` 概念は廃止し、文書整合性検査 command 責務境界として再定義すべきである。
- active REQ 全件を対象とする。特定REQだけに限定しない。
- retired REQ は対象外とする。
- active REQ は retire せず、正しい姿に修正する。
- 本件は「既存REQの見直し」と「再発防止」の2本立てで扱う。

## 統合理由

個別REQごとに「移行が要件ではない」と修正すると、同じ問題が別REQで再発する。

今回の問題は、REQ-0115 / REQ-0119 / REQ-0120 / REQ-0121 / REQ-0123 / REQ-0124 に限定されない。active REQ 全件で、要件行、title、目的、適用範囲、README active index を確認し、作業手段を状態要件へ変換する必要がある。

そのため、本RUでは active REQ 全件の状態要件化を一つの Requirement Unit として扱う。

## 要件化の方向

active REQ 全件を、以下の状態へ修正する。

- title は、作業名ではなく責務・状態・制約名を表す。
- 目的は、「なぜその状態が必要か」を説明する。
- 要件行は、「現在満たすべき状態・振る舞い・制約」だけを定義する。
- 適用範囲は、責務対象・禁止対象・検査対象を示す。
- 編集対象ファイル一覧、作業順序、移行経路、改名手順、削除手順を REQ 本文の要件として扱わない。
- 旧名称・旧構造に言及する場合は、現行体系に含めない状態、使用禁止状態、入力禁止状態、出力禁止状態、検出対象として記述する。
- active REQ は retire しない。
- retired REQ 本文は本件の修正対象外とする。
