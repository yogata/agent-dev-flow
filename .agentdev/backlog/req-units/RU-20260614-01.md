# RU: `.agentdev/drafts/` の用途・frontmatter・draft type registry の確立

## 背景

agent-dev-flow では、`.agentdev/` を永続 domain state、`.sisyphus/` を runtime workspace として扱う整理が行われた。これに伴い、agent-dev-flow が生成・保存・再読込・後続 command へ受け渡す draft は `.agentdev/drafts/` に配置する方針となった。

本セッションでは、`.agentdev/drafts/` には複数の位置づけの draft file が生成されうること、特に `req-define` が生成する保存前の要件ドラフトと、`skill-review` が生成しうる診断 finding draft は別種別として扱うべきことを確認した。

また、`skill-review → req-define` の受け渡しをセッションコンテキストだけで行うことの妥当性を検討した。結論として、診断結果・分類・根拠・推奨 route を後続 command に渡す場合、会話文脈だけでは compaction、再開、別エージェント実行、証跡確認に弱いため、`.agentdev/drafts/` に中間成果物として保存できる契約が必要であると整理した。

## 問題

`.agentdev/drafts/` の用途、利用方法、draft file 制約が曖昧なままだと、以下の問題が起きる。

- draft file の種別を filename だけで推測する必要がある
- 後続 command が誤った draft type を入力として受け取る可能性がある
- producer / consumer / next を各 draft file に重複記載すると、`draft_type` との不整合が発生する
- `skill-review` の診断結果を `req-define` に渡す標準的な durable artifact がない
- `skill-review` finding を直接 RU と誤認し、未合意事項や誤検知が要件化される可能性がある
- `.sisyphus/drafts/` 前提の記述が残ると、現行の `.agentdev/drafts/` 方針と衝突する
- drafts の標準ユースケースに含めない古い finding 形式が残ると、不要な command 間ループを正当化してしまう

## Source Summary

本セッションでは、以下を合意した。

- `.agentdev/drafts/` は、canonical artifact ではないが、後続 command に渡す必要がある中間成果物の置き場である。
- `.agentdev/drafts/` の利用方法と draft file 制約、主に frontmatter 制約を定義する必要がある。
- `.sisyphus/drafts/` は agent-dev-flow 管理下の draft 生成先・保存先・入力元・受け渡し先として使わない。
- drafts の標準ユースケースは、まず `req_draft` と `skill_review_finding` に絞る。
- `req_draft` は、`req-define` が生成し、`req-save` または `case-open` が読む保存前の要件ドラフトである。
- `skill_review_finding` は、`skill-review` が生成し、`req-define` が読む診断 finding draft である。
- `skill-review` の出力を直接 RU として保存しない。finding は診断結果であり、要件として合意済みとは限らない。
- `skill-review → req-define` はセッションコンテキストだけでつなぐのではなく、必要に応じて `.agentdev/drafts/skill-review-finding-{topic}.md` を介する。
- draft file の frontmatter は最小化し、`draft_type`、`topic`、`status`、`created_at` を基本とする。
- `producer`、`consumer`、`next` は、`draft_type` から導出できるため、個別 draft file の frontmatter には原則書かない。
- producer / allowed consumers / file pattern は draft type registry 側で定義する。
- `next` が一意に決まる場合は冗長であり、frontmatter に書かない。
- `req_draft` のように consumer が複数ありうる場合でも、実際の後続 command はユーザー指定や draft 本文の処理情報で判断すればよく、frontmatter の `next` は不要である。
- `req-save` が保存不能を検出した場合は、RU や draft を新規生成せず、保存を停止して差し戻し理由を報告するのが妥当である。
- ただし、「req-save → req-define の review finding は採用しない」という否定要件を新規要件として作らない。該当記述が既存にある場合は、標準 draft type から外し、不要な残存記述として削除する。

## 統合理由

本RUは、`.agentdev/drafts/` の配置先だけを決めるものではない。draft file を command 間の安全な受け渡し artifact として使うために、以下を一体で整理する必要がある。

- `.agentdev/drafts/` の位置づけ
- `.agentdev/drafts/` の利用方法
- draft file の種別識別方法
- draft file の frontmatter 制約
- draft type registry
- `skill-review` finding の draft 化
- `req-define` による finding draft の読み込み
- `.sisyphus/drafts/` 前提の排除
- drafts 標準ユースケースに含めない古い finding 形式の削除

`draft_type` と producer / consumer / next を個別 file の frontmatter に重複させると、個別ファイルと registry の間で不整合が生じる。そのため、draft file 自体は最小 frontmatter を持ち、route 定義は registry 側に集約する方が保守しやすい。

## 要件化の方向

### 1. `.agentdev/drafts/` の位置づけ定義

`.agentdev/drafts/` は、canonical artifact ではないが、後続 command に渡す必要がある中間成果物の置き場として定義する。

対象となる draft は、以下の条件を満たすものとする。

- 後続 command が読む
- canonical docs / REQ / ADR / SPEC / RU ではない
- セッションコンテキストだけでは失われる
- req-define、req-save、case-open、skill-review などの command 間受け渡しに使う
- `.sisyphus/` ではなく `.agentdev/` 管理下に置く必要がある

### 2. `.agentdev/drafts/` の利用方法定義

`.agentdev/drafts/` は、draft type registry に登録された draft type の中間成果物を配置する場所とする。

各 draft type には、少なくとも以下を registry 側で定義する。

- `draft_type`
- file pattern
- producer
- allowed consumers
- 位置づけ
- 必要に応じた lifecycle / removal condition

各 command は、入力 draft の `draft_type` が自 command の allowed input に該当するかを確認する。

### 3. 標準 draft type の初期定義

初期の標準 draft type は以下とする。

| draft_type | file pattern | producer | allowed consumers | 位置づけ |
|---|---|---|---|---|
| `req_draft` | `.agentdev/drafts/req-draft-{topic}.md` | `req-define` | `req-save`, `case-open` | 保存前の要件ドラフト |
| `skill_review_finding` | `.agentdev/drafts/skill-review-finding-{topic}.md` | `skill-review` | `req-define` | Skill / Command 診断結果の要件化入力 |

producer と allowed consumers は、各 draft file の frontmatter ではなく、draft type registry 側で定義する。

### 4. draft file frontmatter の最小制約

`.agentdev/drafts/` 配下の draft file は、原則として以下の frontmatter を持つ。

```yaml
---
draft_type: req_draft
topic: example-topic
status: draft
created_at: 2026-06-14T19:36:47+09:00
---
```

`skill_review_finding` の例:

```yaml
---
draft_type: skill_review_finding
topic: skill-reference-boundary
status: draft
created_at: 2026-06-14T19:36:47+09:00
---
```

frontmatter には、原則として以下を持たせない。

- `producer`
- `consumer`
- `next`

これらは `draft_type` から導出可能であり、個別 file に記載すると registry との不整合が発生しうるためである。

### 5. command 側の draft type 検証

各 command は、自分が受け付ける `draft_type` だけを確認する。

例:

- `req-save` は `req_draft` を受け付ける
- `case-open` は `req_draft` を受け付ける
- `req-define` は `skill_review_finding` を明示入力として受け付ける

producer / consumer / next を個別 file frontmatter から読んで整合性判定する必要はない。

### 6. `skill-review` finding draft の導入

`skill-review` は、必要に応じて以下の draft を生成できるようにする。

```text
.agentdev/drafts/skill-review-finding-{topic}.md
```

この draft は RU ではない。診断結果、分類、根拠、推奨 route、未確認事項を req-define に渡すための中間成果物である。

`skill-review-finding` には、少なくとも以下を含める。

- Summary
- Scope
- Findings
  - id
  - target
  - classification
  - evidence
  - recommended_route
  - confidence
  - unresolved_questions
- Initial Remediation Direction
- Out of Scope
- Suggested req-define Input

`req-define` は、`skill_review_finding` を read-only input として読み、未確認事項や採否未確定事項を要件本文に混入させない。

### 7. `skill-review` の side effect 境界

`skill-review` は、diagnostic-only command とする。

許可される side effect は、`.agentdev/drafts/skill-review-finding-*.md` の生成に限定する。

禁止事項は以下とする。

- canonical docs の変更
- REQ / ADR / SPEC の変更
- Command / Skill / Template / Script の変更
- RU 保存
- Issue 作成
- PR 作成
- commit
- push

### 8. 既存の標準外 draft 記述の削除

標準 draft type に含めない既存の draft 記述がある場合は、標準 contract から削除する。

特に、`requirements-review-finding` 相当の記述が既存に残っている場合は、draft type registry から外し、不要な残存記述として削除する。

ただし、本RUでは「req-save → req-define の review finding は採用しない」という否定要件を新規要件として作らない。対象は、採用する draft type を明確化し、標準外の残存記述を削除することに限定する。
