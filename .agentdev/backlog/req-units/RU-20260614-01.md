# RU: diagnostics workflow の導入と review 系 finding の RU 化経路整理

## 背景

`docs-review` / `skill-review` で発見された不整合は、現行では read-only 診断結果として扱われる。

しかし、診断で発見した問題が明確な不整合である場合、それを intake に未分類 item として遠回りさせるより、採用済み diagnostic artifact として RU 化対象へ進める方が自然である。

また、`review` という語は行為名として広く、`docs-review` / `skill-review` / `backlog-review` など既存コマンド名とも衝突しやすい。そのため、診断 workflow として `diagnostics` を一級化する。

## 問題

`docs-review` / `skill-review` の finding には、以下が混在する。

- 明確な不整合
- 不整合の疑い
- 誤検知
- 対応不要
- 今後の機械検査候補
- 再発防止知見

これらをすべて intake に入れると、intake が診断ログ置き場になり、未分類の作業候補という責務がぼやける。

一方で、明確な不整合を `req-define` 入力 draft や docs-check rule candidate として個別分類すると、RU 化経路が分断される。docs-check rule 追加も、不整合是正の一部であるなら RU の要件化方向または受け入れ条件に含めるべきである。

## Source Summary

本セッションでは以下を合意した。

- `review` ではなく `diagnostics` を用いる。
- コマンド体系は `diagnostics-*` に寄せる。
- `diagnostics-docs` / `diagnostics-skills` / 必要に応じて `diagnostics-commands` を診断実行コマンドとする。
- 診断 finding の未処理置き場は `.agentdev/diagnostics/inbox/` とする。
- `diagnostics-promote` は未分類 finding を分類し、採用 item を `.agentdev/diagnostics/promoted/` に送る。
- `promoted/` は intake / learning / diagnostics で共通して「採用済み・RU 化対象」を意味する。
- `backlog-review` は `.agentdev/intake/promoted/`、`.agentdev/learning/promoted/`、`.agentdev/diagnostics/promoted/` をまとめて吸い上げ、統合・分割して RU を生成する。
- `source_type` には diagnostics を追加する必要がある。
- `req-define 入力 draft 化` は独立分類としない。
- `docs-check rule candidate` は独立分類とせず、RU の要件化方向または受け入れ条件に含める。
- 不整合が明確であれば、promote 後は RU 化対象であり、intake に落とさない。
- intake / learning item になるのは、未成熟 finding または修正対象を持たない知見に限る。

## 統合理由

本件は、単なる `docs-review` / `skill-review` の出力形式変更ではない。

診断コマンド体系、`.agentdev/diagnostics/` の domain state、promoted artifact の意味、`backlog-review` の入力範囲、RU source_type を横断して整合させる必要がある。

そのため、diagnostics workflow 導入として一つの RU に統合する。

## 要件化の方向

以下を要件化する。

1. `.agentdev/diagnostics/` を diagnostics domain state として定義する。
2. `.agentdev/diagnostics/` は `inbox/`、`promoted/`、`archive/` を持つ。
3. `.agentdev/diagnostics/inbox/` は診断で発見された未分類 diagnostic finding を格納する。
4. `.agentdev/diagnostics/promoted/` は採用済み・RU 化対象の diagnostic artifact を格納する。
5. `promoted/` の意味を intake / learning / diagnostics で「採用済み・RU 化対象」に揃える。
6. `diagnostics-docs` を docs 系診断コマンドとして定義する。
7. `diagnostics-skills` を command/skill 参照妥当性・skill 構造診断コマンドとして定義する。
8. `diagnostics-commands` は command 定義単体の診断が必要になった場合のみ追加対象とする。
9. `diagnostics-promote` を diagnostic finding の分類・採用コマンドとして定義する。
10. `diagnostics-promote` の分類は promote / defer / reject を基本とする。
11. promote された diagnostic artifact は `.agentdev/diagnostics/promoted/` に保存する。
12. defer された finding は `.agentdev/diagnostics/inbox/` に残す。
13. reject された finding は `.agentdev/diagnostics/archive/rejected/` に移す。
14. `backlog-review` の入力対象に `.agentdev/diagnostics/promoted/*.md` を追加する。
15. RU frontmatter の `source_type` に `diagnostics` を追加する。
16. diagnostics promoted artifact を RU 化した場合、RU の `sources` に元 artifact path を記録する。
17. docs-check rule / fixture 追加候補は独立 route とせず、RU の要件化方向または受け入れ条件へ含める。
18. 明確な不整合 finding は intake に回さず、diagnostics promoted artifact として RU 化対象にする。
19. intake に落とすのは、不整合かどうか、採否、範囲、優先度、正とする情報源が未確定の finding に限る。
20. learning に落とすのは、具体的修正対象を持たない再発防止知見・判断基準候補に限る。
21. 既存 `docs-review` / `skill-review` は deprecated shim とし、対応する `diagnostics-docs` / `diagnostics-skills` へ誘導する方向で扱う。
