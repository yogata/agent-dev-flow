# RU: Skill 作成基本方針と Command→Skill 参照妥当性レビューの確立

## 背景

現行の agent-dev-flow では、Command / Skill / Template / Script の責務境界、および Command → Skill → Reference の依存方向は定義済みである。

一方で、本セッションの議論により、Skill の粒度、`USE FOR` / `DO NOT USE FOR` に基づく Skill 分割判断、`references/*` による段階的開示、Command 側の Skill 参照用途の妥当性検査については、基本方針として十分に明文化されていないことが確認された。

また、現行リポジトリを対象にした確認では、Command が Skill を参照する際に、参照先 Skill の `DO NOT USE FOR` に該当する用途で参照していると見られる事例が確認された。

## 問題

Command 作成・更新時に、参照先 Skill の `USE FOR` / `DO NOT USE FOR` と、Command 側の参照用途を照合する検査が不足している。

このため、以下の問題が起きうる。

- Command 固有の実行手順が Skill の責務に流れる
- Skill が判断基準ではなく実行手順の参照先として使われる
- Skill 全体参照が必要な場面と、`references/*` の段階的開示で足りる場面が区別されない
- Skill 分割すべき場合と、同一 Skill 内の reference 分割で足りる場合が混同される
- `USE FOR` / `DO NOT USE FOR` が異なる関心を、同一 Skill に保持し続ける可能性がある
- docs/*、skill authoring、review 観点の間で、Skill 作成・運用の基本方針が一貫しない

## Source Summary

本セッションでは、以下を合意した。

- Skill は、再利用可能な判断基準・domain knowledge を保持する。
- Command は、実行入口、入出力、ガードレール、高レベル Steps、実行順序、副作用を保持する。
- Template は、出力本文構造とプレースホルダーを保持する。
- Script は、決定的でテスト可能な処理・検査を保持する。
- Skill の粒度は、同じ関心、同じ責任境界、同じ判断モデル、矛盾しない `USE FOR` / `DO NOT USE FOR` を共有する単位とする。
- 複数の `USE FOR` があっても、同一関心・同一責任境界・同一判断モデルに属する場合は、同一 Skill として扱う。
- 同一 Skill 内で詳細関心が細分化される場合は、まず `references/*` による段階的開示を優先する。
- 同じ関心に見えても、`USE FOR` / `DO NOT USE FOR`、入力、出力、判断モデル、責任境界が分岐する場合は Skill 分割候補とする。
- Command 固有の実行順序、Issue作成、保存、更新、削除、完了報告などは Skill 化せず、Command / Template / 操作用 Skill / Script に配置する。
- Command が Skill を参照する場合、参照用途が Skill の `USE FOR` に合致し、`DO NOT USE FOR` に該当しないことを review できる必要がある。

本セッションで調査・整理した具体例は以下である。

### 事例1: `case-open` → `agentdev-workflow-lifecycle`

`agentdev-workflow-lifecycle` は lifecycle、work_type、scale、handoff、Epic 要否などの判断基準を扱う Skill であり、Issue本文生成、Issue作成、Epic/child Issue生成、RU削除後処理、特定 command の実行手順は `DO NOT USE FOR` に含まれる。

一方、`case-open` 側には、Issue本文生成、Epic本文生成、child Issue作成、RU削除後処理などの文脈で `agentdev-workflow-lifecycle` を参照する記述がある。

この事例の問題は、`agentdev-workflow-lifecycle` 内に禁止対象手順が混入していることではない。問題は、`case-open` が `agentdev-workflow-lifecycle` の `DO NOT USE FOR` に該当する用途で同 Skill を参照していることである。

整理方針は以下とする。

- `case-open` が `agentdev-workflow-lifecycle` を参照して work_type / scale / lifecycle / Epic要否を判断することは妥当。
- Issue本文生成、Issue作成、Epic/child Issue生成、RU削除後処理は `case-open` / command-local template / `agentdev-gh-cli` / 必要に応じた検査用 subagent に配置する。
- この事例は、Skill 自体の自己矛盾ではなく、Command→Skill の参照用途ミスマッチとして扱う。

### 事例2: `intake-from-github` / `intake-promote` → `agentdev-workflow-lifecycle`

本セッションでは、intake 系 command が、期間解釈、データ取得、構造検出、LLM解析、item生成、inbox処理などの intake pipeline 文脈で `agentdev-workflow-lifecycle` を参照している可能性が確認された。

`agentdev-workflow-lifecycle` が intake pipeline を `DO NOT USE FOR` とするなら、この参照は Command→Skill の参照用途ミスマッチ候補である。

この事例は、Skill 分割以前に、intake pipeline に関する判断基準・実行手順・保存処理の配置を再確認する必要がある。

### 事例3: `req-define` → `agentdev-req-file-manager`

`req-define` は既存REQ照合、APPEND-first 判断、要件行候補分類、template所在確認などを必要とする。

一方で、REQ採番、frontmatter更新、保存、README更新などの実ファイル操作は `req-save` 側の責務である。

このため、`req-define` が `agentdev-req-file-manager` 全体を必要としているのか、REQ照合・分類ゲート・template所在に関する `references/*` の段階的開示で足りるのかを review する価値がある。

この事例は、Skill 分割ではなく reference-disclosure-needed の候補である。

### 事例4: `agentdev-gh-cli`

GitHub 操作の安全手順は複数 command から参照されるため、Skill として保持する意義がある。

ただし、command ごとに必要な範囲が WRITE / READ / VERIFY / MERGE で異なる場合、Skill を分割する前に、`references/write.md`、`references/read.md`、`references/verify.md`、`references/merge-conflict.md` のような段階的開示を検討する。

この事例は、同一関心・同一責任境界を保ちつつ、詳細関心を `references/*` に分ける候補である。

### 事例5: `case-run` → `agentdev-workflow-orchestration`

`case-run` は再開ポイント、Epic検出、多重Issue、Wave実行、自律修正、CI loop、capture境界など、`agentdev-workflow-orchestration` の判断モデル全体を使う。

このため、Skill 全体参照が妥当であり、安易な Skill 分割は不要である。

この事例は、skill-review が「分割すべきでない参照」を識別する必要があることを示す。

### 事例6: `learning-promote` → `agentdev-learning-pipeline`

`learning-promote` は normalize、classify、8軸評価、evaluation-report、dispose、HITL、promoted artifact、archive move、prune まで learning pipeline 全体を扱う。

このため、`agentdev-learning-pipeline` の全体参照は妥当である。

この事例も、Skill 全体参照が適切なケースとして扱う。

## 統合理由

本RUは、個別の `case-open` 修正や個別 Skill 分割ではなく、Skill 作成・参照・レビューの基本方針を確立することを目的とする。

理由は以下である。

- `case-open` の事例は個別不備だが、根本原因は Command→Skill 参照用途照合の不足である。
- Skill 粒度、`references/*` 分割、Skill 分割、Command-local 配置、Template 配置、Script 配置の判断基準は共通化できる。
- docs/* に基本方針を明文化しない限り、skill authoring skill や review command が同じ判断基準を参照できない。
- 既存 Skill を修正する前に、どの修正が Skill 分割、reference 分割、Command への戻し、Template 化、Script 化に該当するかを分類する必要がある。
- 継続運用には、skill authoring 時の作成基準と、skill-review による read-only 診断の両方が必要である。

## 要件化の方向

### 1. docs/* への Skill 作成基本方針の明文化

docs/* に、少なくとも以下を追加する。

- Skill は、同一関心・同一責任境界・同一判断モデル・矛盾しない `USE FOR` / `DO NOT USE FOR` を共有する単位とする。
- 複数の `USE FOR` があっても、同一判断モデルに属する場合は同一 Skill として扱う。
- 同一 Skill 内の詳細関心は `references/*` に分離する。
- `references/*` は小さい Skill ではなく、同一 Skill 内の段階的開示である。
- `references/*` ごとに独自の `USE FOR` / `DO NOT USE FOR` が必要になる場合は Skill 分割候補とする。
- Command 固有の実行手順は Skill に置かず、Command に置く。
- 出力本文構造は Template に置く。
- 決定的に検査可能な処理は Script に置く。
- GitHub 操作などの横断的操作安全手順は操作用 Skill に置く。
- Command が Skill を参照する場合、参照用途が Skill の `USE FOR` に合致し、`DO NOT USE FOR` に該当しないことを review 可能にする。

### 2. skill authoring skill への取り込み

`agentdev-skill-authoring` に、上記の基本方針を取り込む。

取り込み対象は以下とする。

- Skill 粒度判断
- `USE FOR` / `DO NOT USE FOR` の設計基準
- `references/*` 分割基準
- Skill 分割基準
- Command / Template / Script / 操作用 Skill への配置判断
- Command が Skill 内部 section / protocol / reference に過度に依存しないための参照境界
- canonical Skill name の使用

### 3. skill-review の read-only 診断定義

`/agentdev/skill-review` 相当の read-only 診断を定義する。

skill-review は、少なくとも以下を診断する。

- Command→Skill 参照用途が Skill の `USE FOR` に合致するか
- Command→Skill 参照用途が Skill の `DO NOT USE FOR` に該当しないか
- Skill 全体参照が妥当か
- `references/*` による段階的開示で足りるか
- Skill 分割候補か
- Command 固有実行手順が Skill に流れていないか
- 出力本文構造が Command / Skill に混入していないか
- 決定的検査を Script 化できるか
- 操作安全手順が適切な操作用 Skill に置かれているか
- canonical Skill name が使われているか
- Command が Skill 内部の section / protocol / Step 名に依存していないか
- 既存 intake / learning / PR / RU 実績を、Skill 運用効果測定の補助 evidence として扱えるか

skill-review は read-only とし、以下を行わない。

- ファイル変更
- Skill 分割の実行
- Command 修正の実行
- Issue / PR 作成
- intake / learning / RU の保存
- commit / push

### 4. 既存 Command / Skill の適用対象洗い出し

skill-review または後続作業で、現行 Command / Skill を基本方針に照らして分類する。

分類は少なくとも以下を持つ。

- `full-skill-needed`
- `reference-disclosure-needed`
- `split-skill-candidate`
- `command-local-responsibility`
- `template-local-responsibility`
- `script-candidate`
- `operation-skill-reference`
- `wrong-skill-reference`
- `do-not-use-for-conflict`
- `canonical-name-violation`
- `skill-internal-reference-leak`

本RUで調査済みの初期候補は以下である。

- `case-open` → `agentdev-workflow-lifecycle`: `do-not-use-for-conflict` / `command-local-responsibility`
- `intake-from-github` / `intake-promote` → `agentdev-workflow-lifecycle`: `do-not-use-for-conflict` 候補
- `req-define` → `agentdev-req-file-manager`: `reference-disclosure-needed` 候補
- `agentdev-gh-cli`: `reference-disclosure-needed` 候補
- `case-run` → `agentdev-workflow-orchestration`: `full-skill-needed`
- `learning-promote` → `agentdev-learning-pipeline`: `full-skill-needed`

### 5. 既存 Command / Skill の修正

既存 Command / Skill の修正は、skill-review による分類結果に基づいて実施する。

修正方針は以下とする。

- `do-not-use-for-conflict` は、Command 側参照文脈または配置先を修正する。
- `command-local-responsibility` は、Skill ではなく Command に戻す。
- `template-local-responsibility` は、Template に移す。
- `script-candidate` は、決定的に検査可能な範囲に限って Script 化する。
- `reference-disclosure-needed` は、Skill 分割より先に `references/*` 分割を検討する。
- `split-skill-candidate` は、独立した `USE FOR` / `DO NOT USE FOR`、入力、出力、判断モデル、責任境界を持つ場合に限って Skill 分割する。
- `full-skill-needed` は、安易に分割せず、必要に応じて `SKILL.md` のナビゲーションだけ改善する。

- 個別 Skill の即時分割そのもの

## 後続エージェント向け実行方針

### 操作単位1: docs/* への基本方針追加

目的:
