# RU: コマンド・スキル・サブエージェント責務分界の再基準化

## 背景

コマンド実行において、intake / learning、関連REQ/ADR/SPEC、docs、スキル参照、検査結果を親エージェントのコンテキストへ広く取り込むと、処理の高度化に伴いコンテキスト消費が増大する。

このため、親エージェントから独立して実施できる探索・検査・分類・候補抽出を、スキルに定義したうえでサブエージェントへ委譲し、親エージェントのコンテキストには必要最小限の結果だけを戻す仕組みが必要である。

併せて、既存コマンドの Step 表記、委譲接続点、親エージェントの責務、スキル側へ移すガードレール、返却形式、capture 候補の扱いを再基準化する。

現時点で復元用タグが打たれているため、既存互換の維持よりも、現行方針への破壊的な再基準化を優先できる。

## 問題

現行のコマンド・スキル・関連文書には、以下の問題がある。

- コマンドが詳細手順、ガードレール、判定基準を持ちすぎる。
- 親エージェントが本来保持すべき最終判断と、サブエージェントへ委譲できる探索・検査・分類が混在している。
- 小数 Step や英字サブステップが混在し、再採番・参照・grep・文書同期で扱いにくい。
- `lightweight-delegation`、`manager-orchestrator`、`delegation_type`、`on_result`、旧 Findings/Intake 系見出しなどが、実装分類・委譲境界・capture 境界の中で十分に整理されていない。
- サブエージェントの出力を一律 verbatim とする制約は、親コンテキスト圧縮の目的と矛盾する。
- step 専用ガードレールをスキルへ移す際の移管先が明文化されていないと、制約が消えるリスクがある。

## Source Summary

チャット内では、以下を合意した。

- コマンドは step 見出しと委譲接続点だけを持つ。
- step 専用の詳細手順・ガードレールはスキルへ移す。
- 親エージェントは、進行判断、保存、Issue / PR 更新、commit、push、ユーザー確認を保持する。
- サブエージェントは、探索・検査・分類・候補抽出だけを行う。
- サブエージェントは、独立したコンテキストで処理し、親エージェントへ必要最小限の結果だけを返す。
- intake / learning として検出した事象は、保存せず capture 候補として親エージェントへ返す。
- 小さい確認や更新直後の VERIFY はサブエージェント化せず、親エージェントの step 内に含める。
- 次 step へ進めるかの判定は親エージェントが行う。
- Step はトップレベル整数に統一する。
- `Step 5.5` は独立した `Step 6` へ昇格する。
- `10-1` のようなサブステップ表記は許容する。
- `delegation_type` / `on_result` は必須 envelope から外す。
- command 本文では英語ラベルを増やさず、日本語の説明文で記述する。
- ADR / SPEC では、必要最小限の設計語彙として `inputs`、`side_effect_boundary`、`output_contract`、`capture_handoff` を扱う。
- ADR-0111 は superseded、ADR-0112 は accepted な包括 ADR として再定義する変更対象候補とする。
- 「実施済み更新」は使わず、「変更対象候補」として扱う。

## 統合理由

本RUは session-sourced RU である。

対象論点は、コマンドの薄型化、スキルへの詳細移管、サブエージェントへの委譲境界、Step 表記の再基準化、capture 候補の返却形式、ADR / SPEC / command authoring standard の整合化にまたがる。

これらは個別に修正すると、コマンドだけ薄くなりスキル側の契約が欠落する、または ADR / SPEC と実装コマンドが不整合になるため、1つのRUとして扱う。

## 要件化の方向

1. コマンドは、公開API、入力、出力、親エージェントの責務、Step 見出し、委譲接続点に限定する。
2. Step 専用の詳細手順、確認手順、禁止事項、返却形式、失敗時の扱いはスキルへ移す。
3. 長い判定基準、例、分類表、詳細な失敗時手順はスキルの `references/` に移す。
4. トップレベル Step は整数のみとし、小数 Step を廃止する。
5. サブステップは `N-M` 形式を許容する。
6. 既存の `Step 5.5` は独立 Step へ昇格し、case-run では実装後の関連文書更新漏れ確認として `Step 6` 相当に再配置する。
7. 親エージェントは、次 Step へ進む判断、保存、Issue / PR 更新、commit、push、ユーザー確認を保持する。
8. 小さい確認、作成・更新直後の VERIFY、Issue / PR 更新に密接な確認は、サブエージェント化せず親エージェントの Step 内に含める。
9. 広い探索、局所検査、候補抽出、分類、大きなファイル確認はサブエージェント化候補とする。
10. サブエージェントは、親エージェントから渡された限定入力だけを使い、処理結果を `pass` / `warn` / `fail` / `partial` を基本として返す。
11. サブエージェントは、保存、Issue / PR 更新、commit、push、ユーザー確認を行わない。
12. サブエージェントの返却には、副作用がないことを示す情報を含める。
13. サブエージェントの最終出力を一律 verbatim とする制約は廃止し、親に戻す情報は要約、成果物パス、判定、capture 候補へ圧縮する。
14. intake / learning 候補は、保存せず、capture 候補として親エージェントへ返す。
15. `lightweight-delegation` は primary pattern とせず、主要な実装分類に重ねる委譲の扱いとして整理する。
16. `manager-orchestrator` は、複数子タスクを統括する特殊な委譲形態として扱い、軽量な単一タスク委譲と混同しない。
17. `delegation_type` / `on_result` は必須 envelope から外す。必要な場合のみ、簡潔なラベルや親側の扱いとして記述する。
18. ADR / SPEC 上では、委譲時の最小契約を `inputs`、`side_effect_boundary`、`output_contract`、`capture_handoff` を中心に整理する。
19. command 本文では、英語ラベルを増やさず、日本語の説明文で委譲接続点を記述する。
20. current docs / source の旧 Findings/Intake 系見出しは `Findings / Capture候補` へ統一する。
21. projection 側や integrity rule の旧語検出文字列は、旧語検出用途であれば残存を許容する。
22. ADR-0111 は superseded 化し、ADR-0112 を後継として参照する変更対象候補とする。
23. ADR-0112 は accepted 化し、実装分類と委譲境界を包括する ADR として再定義する変更対象候補とする。
24. 既存コマンド、既存スキル、関連 SPEC、DOC-MAP、必要な検査ルールを同時に再基準化する。

## 受け入れ条件

- トップレベル Step が整数に統一され、小数 Step が current command docs に残っていない。
- `10a` / `10b` / `10c` のような英字サブステップが、必要に応じて `10-1` / `10-2` / `10-3` 形式へ統一されている。
- case-run の旧 Step 5.5 が独立 Step へ昇格し、Step 5 は実装中心、昇格後 Step は実装後の関連文書更新漏れ確認として分離されている。
- コマンド本文が、委譲接続点、入力、戻り値、親エージェントの最終責務を中心とした薄い記述になっている。
- step 専用の詳細手順、禁止事項、確認手順、返却形式がスキルまたは `references/` に移管されている。
- 親エージェントが、次 Step へ進む判断、保存、Issue / PR 更新、commit、push、ユーザー確認を保持することが明記されている。
- サブエージェントが、保存、Issue / PR 更新、commit、push、ユーザー確認を行わないことが明記されている。
- サブエージェントの返却形式が `pass` / `warn` / `fail` / `partial` を基本に整理されている。
- サブエージェントの返却に、副作用がないことを示す情報が含まれる。
- intake / learning 候補は、保存されず、capture 候補として親エージェントへ返す形式に整理されている。
- サブエージェント出力の一律 verbatim 制約が current docs/source に残っていない。
- `lightweight-delegation` が current docs/source で primary pattern として扱われていない。
- `manager-orchestrator` が軽量な単一タスク委譲と混同されず、複数子タスクを統括する特殊形として整理されている。
- `delegation_type` / `on_result` が command / authoring standard の必須 envelope になっていない。
- ADR / SPEC 上の委譲時の最小契約が `inputs`、`side_effect_boundary`、`output_contract` を中心に整理され、capture 候補を扱う場合は `capture_handoff` が明示されている。
- current docs / source の旧 Findings/Intake 系見出しが `Findings / Capture候補` に統一されている。
- 旧語検出用途の projection 側または integrity rule の検出文字列は、検出目的が分かる形で残存を許容されている。
- ADR-0111 が `status: superseded` となり、ADR-0112 を後継として参照している。
- ADR-0112 が `status: accepted` となり、実装分類と委譲境界を包括する ADR として再定義されている。
- 関連 SPEC / DOC-MAP / 必要な検査ルールが、変更後の責務分界と Step 表記に追従している。
- 変更後のコマンド・スキル・関連文書が、コマンドは薄く、詳細はスキルまたは `references/` に置く構造になっている。
