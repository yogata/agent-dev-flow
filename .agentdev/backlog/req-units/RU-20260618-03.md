# RU: Epic / Wave / Issue 構成生成責務を case-open に整理する

## 背景

議論の中で、Epic / Wave / Issue の構成責務が req-define 側に寄りすぎている可能性が明確になった。

実装順序、Issue分解、Wave構成、技術的依存関係は、ユーザーが満たしたい機能・非機能・制約そのものではない。ユーザーが明示的に実装順序を制約として述べた場合を除き、実装順序はユーザー要件ではなく、要件を満たすために実行エージェントが自律的に決める実行構造である。

したがって、req-define はユーザー要件の定義に集中し、case-open がその要件を満たすための GitHub Issue 構造を自律生成する責務を持つべきである。

## 問題

現行整理では、operation_units / execution_groups / depends_on / recommended_order などにより、req-define が実装順序やEpic候補構成まで保持する形になっている。

しかし、実装順序自体は要件ではない。req-define で実装順序やWave構成を壁打ち必須にすると、要件定義に実行計画が混入する。

一方で、case-open が Issue / Epic / Wave / 依存関係を生成できなければ、case-auto は大規模作業を自律的に進行できない。

そのため、case-open は req-define の要件を変更せず、要件を満たすための実行構造を自律生成する工程として定義し直す必要がある。

## Source Summary

チャット内で以下を合意した。

- 実装順序は、ユーザーが明示しない限り要件ではない。
- 要件があるならユーザーが述べているため、未明示の実装順序をあえてREQ/SPECへ書く必要はない。
- req-define は、機能・非機能・制約・対象外・受け入れ条件を扱う。
- case-open は、req-define の要件を満たすために Epic / Wave / Issue / 依存関係 / 初期status を自律的に作る。
- case-open は、新しい要件を作らない。
- case-open は、実装順序やIssue分解についてユーザーに聞かない。
- case-open がユーザーに確認してよいのは、要件が曖昧で「何を満たすべきか」が決められない場合だけである。
- skipped status は不要。
- pending / ready / blocked は、Issue実行状態として意味を分ける。
- Wave status は保存せず、Issue状態から導出する方がよい。

## 統合理由

このRUは、req-define / case-open の責務境界を整理するものである。

case-auto の実行モデル変更とは関係するが、対象は異なる。case-auto は case-open が生成した構造を進める側であり、本RUはその構造を誰が生成するかを扱う。

## 要件化の方向

### REQ concern

REQでは、req-define と case-open の責務境界を定義する。

要件化の方向は以下。

- req-define は、ユーザー要件を定義する。
- req-define は、機能要件、非機能要件、制約、対象外、受け入れ条件を扱う。
- case-open は、req-define の要件を変更せず、要件を満たすための GitHub Issue 構造を生成できる。
- case-open は、Epic / Wave / Issue / 依存関係 / 初期status を実行構造として生成できる。
- case-open は、実装順序・Issue分解・Wave構成をユーザー確認事項にしない。
- case-open は、新しい機能要件・非機能要件・対象外・受け入れ条件を作成しない。
- case-open は、要件が曖昧で実行構造を生成できない場合に停止できる。
- case-auto は、case-open が生成した実行構造に従って進行できる。

「ユーザーが明示しない実装順序は要件ではない」という再説明は、既存のREQ/SPECの禁止・分離原則から導けるため、新規REQ行として追加しない。

### SPEC concern

SPECでは、case-open が生成する実行構造の表現と判定規則を定義する。

仕様化の方向は以下。

- OU / Epic / Wave / Issue の関係定義。
- Epic Issue 本文の状態表。
- 子Issueのstatus enum。
- pending / ready / running / completed / blocked / failed の意味。
- skipped をstatus enumに含めないこと。
- Wave status を保存せず、Issue状態から導出する規則。
- case-open がIssue分解を行う基準。
- case-open が依存関係を設定する基準。
- case-open が初期statusを設定する規則。
- case-open が停止する条件。
- case-auto が読み取る状態表の形式。

## OU / Epic / Wave / Issue の関係

本RUで採用する関係は以下。

```text
OU:
  要件変更の処理単位。
  req-define / backlog-review 由来の「何を実現するか」のまとまり。
  実装順序やIssue分解そのものではない。

Epic:
  case-open がOUを実行可能にするために作る上位Issue。
  大きなOUを複数Issueで処理する必要がある場合に作る。
  子Issue全体の進捗SSoTを持つ。

Wave:
  Epic内の実行段階。
  case-open が技術的依存関係から作る。
  Wave内Issueは同じ前提条件のもとで実行可能。
  Wave間には順序がある。

Issue:
  case-run が1回で扱う最小実装単位。
  サブエージェント委譲の単位。
  PR作成・検証・case-close の単位。
```

構造は以下。

```text
OU
└── Epic
    ├── Wave 1
    │   ├── Issue 1
    │   └── Issue 2
    ├── Wave 2
    │   ├── Issue 3
    │   └── Issue 4
    └── Wave 3
        └── Issue 5
```

小さいOUでは Epic / Wave を省略できる。

```text
小さいOU:
  OU → Issue

中規模OU:
  OU → Epic → Issue

大規模OU:
  OU → Epic → Wave → Issue
```

## status の扱い

主statusは Issue に付与する。

```text
pending:
  依存Issueまたは前Waveの完了待ち。
  異常ではない。

ready:
  依存が満たされ、case-auto が次に case-run へ渡せる状態。

running:
  case-auto が選択し、case-run が実行中の状態。

completed:
  Issueの実装・検証・必要なcase-closeが完了した状態。

blocked:
  要件曖昧性、外部副作用、権限不足、矛盾などにより自動継続できない状態。

failed:
  実装・検証・CI・PR作成などの実行結果として失敗した状態。
```

`skipped` は採用しない。

Wave status は保存しない。Wave内Issueの状態から導出する。

OU / Epic のstatusは進捗集約として扱う。主たる実行状態は Issue status とする。

## 主対象REQまたは変更対象候補

変更対象候補は以下。

- req-define の責務を定義するREQ。
- case-open の責務を定義するREQ。
- case-auto の入力前提を定義するREQ。
- workflow lifecycle / orchestration / issue structure に関するSPEC。
- req-define / case-open / case-auto コマンド定義。
- Epic Issue / child Issue テンプレート。
- 状態表・Issue構成を扱う関連スキル。

正確なREQ番号・SPEC名は、後続工程で最新リポジトリを確認して確定する。

## 対象外

以下は本RUでは扱わない。

- case-auto が子Issueをどのように順次実行するか。
- case-run のサブエージェント委譲方式。
- コンパクション耐性。
- 具体的なコード実装。
- PR作成・merge戦略。
- 並列実行。
- ユーザーが明示した実装順序制約の個別扱い。
- Issue作成、PR作成、既存Issueコメントの実施。

## 受け入れ条件

- req-define は、実装順序・Issue分解・Wave構成を要件として扱わない。
- req-define は、ユーザー要件、制約、対象外、受け入れ条件に集中する。
- case-open は、req-define の要件を変更せず、Issue構造を生成できる。
- case-open は、Epic / Wave / Issue / 依存関係 / 初期status を自律生成できる。
- case-open は、実装順序やIssue分解についてユーザー確認を要求しない。
- case-open は、新しい機能要件・非機能要件・対象外・受け入れ条件を作成しない。
- case-open は、要件が曖昧でIssue構造を生成できない場合に停止できる。
- pending / ready / running / completed / blocked / failed の意味がSPEC上で区別されている。
- skipped status が導入されていない。
- Wave status は保存対象ではなく、Issue状態から導出される。
- case-auto が読み取れるIssue構造と状態表が定義されている。
