# RU: case-auto の停止条件外ユーザー確認禁止

## 背景

`case-auto` は、要件docから `req-save → case-open → case-run → case-close` を順次自走実行し、repo 内変更に限りマージまで進める最大自走モードとして定義されている。

既存定義では、repo 内にファイルとして残る変更、および GitHub Issue / PR / comment / merge / close は自走対象である。一方で、停止条件に該当しない通常の repo 内変更に対して、commit / push 前にユーザー確認を挟んでしまう運用が発生した。

## 問題

`case-auto` 実行中、停止条件に該当しないにもかかわらず「コミットしてよろしいですか？」と確認し、処理が停止した。

これは `case-auto` の「最大自走モード」と矛盾する。停止条件に該当しない repo 内変更であれば、commit / push まで自走するべきであり、任意確認で止まってはならない。

また、ユーザーが旧本文や既存内容を貼り付けたことを rollback 指示と誤認し、明示的な `戻して` / `revert` / `取り消して` がないにもかかわらず変更を取り消すリスクがある。

## Source Summary

チャット内で以下を合意した。

- 今回の直接原因は、`case-auto` の自走境界を守らず、不要なユーザー確認で停止した運用ミスである。
- 旧 `case-auto` には複数操作単位の継続処理不足もあったが、今回の「コミット前確認で止まった」問題の直接原因ではない。
- 再発防止には、`case-auto` に「停止条件以外でユーザー確認を挟まない」ルールを追加する必要がある。
- `case-auto` の完了は、変更作成ではなく commit / push 完了までを含む。
- rollback は、ユーザーが明示的に `戻して` / `revert` / `取り消して` 等を指示した場合のみ行う。

## 統合理由

本件は `case-auto` の実行制御・停止条件・自走完了条件に関する問題であり、既存の `/agentdev/case-auto 最大自走モード` 要件に APPEND するのが妥当である。

新規REQを作成するほど独立した機能ではなく、REQ-0114 の停止条件および自走境界を補強する内容である。

## 要件化の方向

REQ-0114 に以下を APPEND する。

- `case-auto` は、停止条件に該当しない repo 内変更について、commit / push 前にユーザー確認を求めてはならない。
- `case-auto` は、停止条件に該当しない限り、変更作成後に commit / push まで継続する。
- `case-auto` の完了報告は、commit / push 完了後に行う。
- ユーザーが既存ファイル本文や旧本文を貼り付けただけでは rollback 指示と解釈しない。
- rollback / revert / 変更取り消しは、明示的な取り消し指示がある場合のみ実行する。
- 停止する場合は、REQ-0114 の停止条件に基づき、停止理由・現在地点・再開可能な次コマンドを報告する。
