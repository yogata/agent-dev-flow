# RU: コマンド・スキル肥大化是正と再発防止ガバナンス

## 背景

ユーザーは、AgentDevFlow の command / skill が肥大化し、実行時に指示へ従わないケースがあると問題提起した。特に command が多数の skill を読み込み、実行時コンテキストを圧迫している疑いがある。

2026-06-01 時点の repository snapshot を静的確認した結果、Command / Skill / Template / Script の責務分界という設計方針自体は妥当である一方、実体としては command 側に詳細仕様・失敗処理・共通手順・完了報告・診断仕様が積み戻され、progressive disclosure が十分に効いていない状態が確認された。

既存の command authoring standards には command 行数目安、Steps 数目安、Command に書くべきでない内容が定義されている。しかし、その基準が実運用・lint・integrity-check・review checklist に十分反映されておらず、基準が存在しても肥大化を止められていない。

## 問題

### 1. Command が薄い dispatcher になっていない

現状の command は、公開入口・Input / Output・Guardrails・高レベル Steps に留まらず、以下を直接保持している。

- 詳細な判定表・分類表
- 大きな状態機械や再開条件
- 共通 Git 同期・push 失敗時の詳細手順
- 完了報告本文または完了報告に近い文面
- integrity-check の検査カテゴリ詳細
- workflow / reporting / capture の局所ルール

これにより、Command / Skill / Template / Script の責務境界が曖昧になり、command 修正時に似た記述が再増殖しやすい。

### 2. command の行数

が既存基準を大きく超えている

既存基準では、command は原則 100 行以内、101〜150 行は分割検討、151〜200 行は分割推奨、200 行超は orchestration skill / reference / script への切り出し必須検討である。

確認時点では以下の状態であった。

| 指標              |      値 |
| ----------------- | ------: |
| 実行系 command 数 |      17 |
| command 合計行数  |   2,924 |
| 100 行超 command  | 16 / 17 |
| 150 行超 command  |  9 / 17 |
| 200 行超 command  |  5 / 17 |

特に大きい command は以下である。

| command                                          | 行数 | load_skills 数 |
| ------------------------------------------------ | ---: | -------------: |
| `.opencode/commands/agentdev/integrity-check.md` |  306 |              4 |
| `.opencode/commands/agentdev/case-close.md`      |  289 |             10 |
| `.opencode/commands/agentdev/req-backlog.md`     |  258 |              3 |
| `.opencode/commands/agentdev/req-define.md`      |  223 |              6 |
| `.opencode/commands/agentdev/case-run.md`        |  209 |             14 |

### 3. load_skills が実行時の必要量を超えている

`case-run` は 14 skills、`case-close` は 10 skills、`case-open` は 9 skills を読み込んでいる。Command の目的に対して常時必要ではない skill が含まれており、実行時の注意散逸・指示競合・古い参照の混入リスクが高い。

特に以下は常時 load ではなく、必要時参照または出力生成段階の局所参照に寄せる余地がある。

- `agentdev-no-ai-slop-writing`
- `agentdev-workflow-reporting`
- `agentdev-req-file-manager`
- `agentdev-adr-file-manager`
- `agentdev-conventional-commits`
- `agentdev-workflow-templates`

### 4. Skill 側も入口カード化が不十分である

大きい SKILL.md が複数存在し、Skill 本体が詳細仕様・例・長い表を抱えている。

| skill                             | SKILL.md 行数 |
| --------------------------------- | ------------: |
| `agentdev-skill-authoring`        |           491 |
| `agentdev-req-file-manager`       |           460 |
| `agentdev-learning-pipeline`      |           394 |
| `agentdev-workflow-orchestration` |           362 |
| `agentdev-adr-file-manager`       |           275 |
| `agentdev-req-analysis`           |           257 |

Skill は再利用可能な判断基準を持つべきだが、SKILL.md 本体は実行時に最初に読まれる入口であるため、詳細は `references/` や scripts に逃がし、SKILL.md は progressive disclosure の入口カードとして薄く保つ必要がある。

### 5. 共通手順が複数 command に重複している

`git pull --ff-only`、`git push`、push 失敗時の構造化エラー、読込時 hash と pull 後 hash の照合などの類似記述が複数 command に重複している。

確認時点では、`git pull --ff-only` または `git push` 関連の詳細記述が以下のように複数 command に出現していた。

- `intake-review.md`
- `intake-promote.md`

- `integrity-check.md`

この重複はコンテキスト圧迫だけでなく、将来の手順変更時のドリフト原因になる。

### 6. canonical path と旧表記が混在している

`references/` が canonical であり、`reference/` は obsolete structure とされている一方、active docs / commands / skill references 内に `reference/` 表記や旧リンクが残っている。

例:

- `README.md`
- `docs/guides/artifact-model.md`
- `docs/guides/domain-state-lifecycle.md`
- `docs/specs/patterns.md`
- `docs/specs/system.md`
- `.opencode/commands/agentdev/case-run.md`
- `.opencode/commands/agentdev/case-close.md`
- `.opencode/commands/agentdev/integrity-check.md`
- `.opencode/skills/agentdev-workflow-lifecycle/references/*.md`

LLM は古い導線を見て obsolete path を辿る可能性があるため、active guidance から旧表記を除去する必要がある。

### 7. Script 化済みの処理を command が再説明している

`agentdev-integrity/scripts/check_integrity.ts` などの検査 script が存在するにもかかわらず、`integrity
-check.md` 側にも検査カテゴリ・対象表・レポート仕様・一部の検査仕様が長く記述されている。

Script-backed な read-only diagnostic command は、対象確認・script 実行・レポート提示・必要時の intake 化に絞り、検査仕様の詳細は skill reference または script / tests に集約するべきである。

### 8. 非推奨 command が長いまま残っている

`req-backlog.md` は deprecated な互換入口として扱うべきだが、258 行の詳細処理を持っている。deprecated command は短い shim とし、canonical command への移行導線だけを提供するべきである。

## Source Summary

この RU は、以下の観察とユーザー承認済み方針を Requirement Source として整理したものである。

- ユーザーは、command / skill の肥大化、過剰な skill load、コンテキスト圧迫、指示不遵守の発生を問題として明示した。
- repository snapshot の静的確認により、command 行数、load_skills 数、SKILL.md 行数、旧 path 表記、共通 Git 手順の重複が確認された。
- 既存の `agentdev-command-authoring` には行数目安と責務境界が既にあるが、実体が基準を満たしていない。
- ユーザーは、単なる修正ではなく、同じ過ちを繰り返さないように今回の学びを反映することを求めた。

## 統合理由

本件は、個別 command の局所修正だけでは解決しない。

問題は、以下の複合である。

1. command を薄く保つ基準が実装・検査・レビューに結びついていない
2. skill を入口カードとして扱う基準が弱い
3. load_skills に prompt budget の考え方がない
4. 共通処理の置き場所が曖昧なため、似た手順が command に再コピーされる
5. obsolete path / reporting / integrity / git 同期のような横断ルールが複数 artifact に散っている

したがって、
個別修正要件と再発防止要件を同じ RU で扱う。ただし req-define / req-save では、既存 REQ への APPEND / UPDATE を優先し、CREATE は原則不要と考える。

## 要件化の方向

### 推奨分類

- Pattern: B（enhancement / maintenance を含む改善だが、command / skill / validation の振る舞いを変更するため Pattern B 扱いを推奨）
- Scale: large
- ADR required: false を推奨
- req-operation: APPEND / UPDATE

ADR は原則不要である。Command / Skill / Template / Script 責務境界は既に ADR-0001 にあり、orchestration skill 作成基準は ADR-0002 にあるため、本件は既存ADRの具体化・運用強化として扱うのが自然である。

## 変更後仕様候補

以下は req-define で要件行候補として評価してほしい。既存REQに吸収できるものは APPEND / UPDATE とする。

### A. Command 薄型化基準

1. agentdev command は、公開入口、Input / Output、Guardrails、高レベル Steps、固有制約、参照先だけを保持し、詳細な判断表・状態機械・共通手順・長い失敗時手順を直接保持しないこと（SHALL）
2. agentdev command は、原則 100 行以内、150 行以内を実運用上限、200 行超を例外状態として扱うこと（SHOULD）
3. command が 150 行を超える場合、切り出し候補と残置理由を review 可能な形で示すこと（SHOULD）
4. command が 200 行を超える場合、orchestration skill / reference / script への切り出し、または明示的な例外理由なしに維持してはならないこと（SHALL）
5. command の Steps は原則 5〜12 個に保ち、細かなサブ手順は skill reference または script に移すこと（SHOULD）
6. deprecated command は、互換入口・移行先・最小限の警告に限定し、詳細処理を保持しない
   こと（SHALL）

### B. load_skills prompt budget

1. command の `load_skills` は、当該 command の主責務を実行するために即時必要な skill に限定すること（SHALL）
2. 必要時にだけ参照すれば足りる skill は、`load_skills` ではなく manual reference として扱うこと（SHOULD）
3. command は implementation pattern ごとの `load_skills` 目安上限を持つこと（SHOULD）
4. 初期目安上限は、capture-only は 2、wall-session は 4、file-pipeline は 5、read-only-diagnostic は 3、manager-orchestrator は 8 とすること（SHOULD）
5. `load_skills` が pattern 目安上限を超える場合、超過理由と必要性を明示すること（SHOULD）
6. `agentdev-no-ai-slop-writing` は、自然言語成果物の最終生成・レビュー段階で参照することを基本とし、すべての command に常時 load してはならないこと（SHOULD）
7. `agentdev-workflow-reporting` は、完了報告 variant の解決と出力段階に必要な範囲で参照し、すべての variant 詳細を command 実行時に常時読ませないこと（SHOULD）
8. command が skill を load する場合、その skill の USE FOR / DO NOT USE FOR と command の implementation pattern が矛盾しないこと（SHALL）

### C. Skill 入口カード化

1. SKILL.md は progressive disclosure の入口として、目的、使用条件、禁止条件、最重要ルール、参照先を中心に構成すること（SHOULD）
2. 長い例、詳細な判定表、スキーマ、分類表、失敗時手順、コマンド固有の分岐は SKILL.md 本体ではなく `references/` または script に移すこと（SHOULD）
3. SKILL.md が 200 行を超える場合、分割候補として integrity-check または authoring review で報告すること（SHOULD）
4. SKILL.md が command 固有の Step 番号、局所ファイルパス、局所エラーメッセージを一次情報として保持しない
   こと（SHALL）
5. Skill reference は `references/` を canonical とし、`reference/` 表記や旧リンクを active guidance に残さないこと（SHALL）

### D. 共通処理の中央集約

1. 複数 command で使う Git 同期、push、pull 後 hash 照合、push 失敗時の構造化エラーは、共通 skill reference または script に集約すること（SHALL）
2. Git 同期共通処理は、既存の `agentdev-git-worktree` 配下へ reference / script として集約することを優先し、新規 skill 作成は既存 skill では責務を表現できない場合に限定すること（SHOULD）
3. command 側には、共通処理の詳細本文ではなく、使用する共通処理名、入出力、許可パス、停止条件のみを保持すること（SHALL）
4. 共通処理の変更時は、参照元 command を一括更新せずに挙動を変えられる構造にすること（SHOULD）

### E. 完了報告の軽量参照

1. command 定義は、完了報告本文・追加ブロック・次コマンド文面を直接保持せず、variant file path、選択条件、置換変数のみを保持すること（SHALL）
2. `agentdev-workflow-reporting` は、variant registry / resolver として薄く保ち、variant の最終出力全文は command 別・variant 別ファイルに分離すること（SHALL）
3. integrity-check は、command 側に完了報告本文相当の長文が残っていないことを検査すること（SHALL）

### F. integrity-check の script-backed 化

1. `integrity-check` command は read-only diagnostic の薄い入口として、対象確認、script 実行、レポート提示、ユーザー承認時の intake 化に限定すること（SHALL）
2. integrity-check の検査カテゴリ詳細、分類表、report schema、fixture 判断は、skill reference、script、tests のいずれかへ集約すること（SHALL）
3. integrity-check は、command line count、Steps count、load_skills budget
   、SKILL.md size、obsolete `reference/` 表記、共通手順重複、completion report 直接保持、deprecated command 肥大化を検出対象に含めること（SHOULD）
4. integrity-check は、検査結果に recommendation candidate を付与し、修正作業の入力に使えるようにすること（SHALL）
5. integrity-check の実行環境は repository 内で再現可能であること（SHALL）
6. integrity-check の新規検査には regression fixture / test を追加すること（SHALL）

### G. req-restructure-review と integrity-check の役割分担

1. `integrity-check` は機械的・再現可能な整合性検査を担当すること（SHALL）
2. `req-restructure-review` は REQ 体系の意味的レビュー、SPLIT / MERGE / MOVE / DUPLICATE / RETIRE / DRIFT 判断、req-define 入力案の作成を担当すること（SHALL）
3. `req-restructure-review` は、link validation、obsolete path 検出、command line count などの機械的検査詳細を重複保持しないこと（SHOULD）
4. `req-restructure-review` は、必要に応じて integrity-check の結果を前提情報として参照するが、integrity-check の代替にはならないこと（SHOULD）

### H. 再発防止としての authoring / review gate

1. command / skill 変更時の Definition of Done は、行数、Steps 数、load_skills 妥当性、共通処理の重複、canonical path、完了報告variant、script-backed 化可否を含むこと（SHALL）
2. command / skill authoring standards は、prompt budget と progressive disclosure を品質軸として明示すること（SHALL）
3. 新しい command / skill / reference を追加する場合、既存 artifact への APPEND / UPDATE で足りるか、新規 artifact が必要かを確認すること（SHALL）
4. 長い説明を追加する場合、command / SKILL.md へ直接追加する前に、reference / template / script のいずれに置くべきかを判定すること（SHALL）
   5
   . 同種の重複記述が複数箇所に現れた場合、個別修正だけでなく、共通化または検査ルール追加を検討すること（SHALL）
5. 今回のような「基準はあるが守られていない」状態を防ぐため、基準文書の更新と integrity-check の検査追加を同じ Case または連続 Case で扱うこと（SHOULD）

## 学びとして反映する観点

今回の根本原因は、単に個別 command が長くなったことではない。以下の学びを今後の設計・レビュー・検査に反映する必要がある。

1. **Prompt budget は品質要件である**: command / skill の内容量と `load_skills` 数は、実行品質に直接影響するため、可読性だけでなく実行安定性の観点で管理する。
2. **基準だけでは不十分である**: command authoring standards が存在しても、integrity-check と review checklist に接続されていなければ肥大化は止まらない。
3. **例外処理は command に積み戻されやすい**: 失敗時手順や共通 Git 処理は便利なため command にコピーされやすい。共通処理の正式な置き場所と検査を用意する。
4. **常時 load は便利だが危険である**: 汎用 skill を常時 load すると、指示競合・古い参照・注意散逸が起こる。manual-reference と output-stage reference を区別する。
5. **progressive disclosure は SKILL.md にも必要である**: Skill を分離しても SKILL.md が巨大なら、実行時コンテキスト圧迫は解消しない。
6. **修正と再発防止は同時に扱う**: 既存 command を薄くするだけでなく、次回の変更で再び太らないよう lint / integrity-check / authoring checklist を更新する。
7. **obsolete guidance は少量でも危険である**: `reference/` のような旧表記が active guidance に残ると、LLM が誤った導線を採用する。canonical path drift は機械的に検出する。

## 受け入れ条件

### 要件定義・保存の受け入れ条件

- 既存 active REQ への APPEND / UPDATE として整理され、CREATE しない場合の理由が明記されている
- `REQ-0103`, `REQ-0107`, `REQ-0108` のどこに何を反映するかが明確である
- 変更後仕様と反
  映作業が分離されている
- prompt budget、load_skills budget、SKILL.md入口カード化、共通処理集約、integrity-check 検査強化が要件として表現されている
- ADR 不要または ADR 必要の判断理由が記録されている

### 実装修正の受け入れ条件

- `integrity-check.md`, `case-close.md`, `case-run.md`, `req-define.md`, `req-backlog.md` が薄型化され、command authoring standards の基準に近づいている
- `case-run` の `load_skills` が大幅に削減され、manager-orchestrator として必要な skill に限定されている
- `case-close` の `load_skills` が削減され、常時不要な learning / writing / reporting 詳細の load が外れている
- `req-backlog` が deprecated shim として短くなり、canonical route を案内している
- 共通 Git 同期・push 失敗時処理の重複本文が command から削除されている
- active guidance 内の obsolete `reference/` リンクが `references/` へ修正されている
- 完了報告本文が command 側に残らず、variant path / 選択条件 / 変数だけになっている
- integrity-check の新規検査に regression test がある
- integrity scripts の実行方法が repository 内で再現可能である

### 再発防止の受け入れ条件

- command / skill authoring standards に prompt budget と load_skills budget が追加されている
- integrity-check が command size、Steps count、load_skills budget、SKILL.md size、重複共通手順、deprecated command 肥大化を少なくとも warning として報告できる
- command / skill 変更時の review checklist に、詳細を command / SKILL.md へ積み戻していないかを確認する項目がある
- 同種の重複が複数 command に出た場合、learning / integrity-check gap / common reference 化のいずれかに接続できる
- 修正後に、今回の問題が単発修正ではなく基準・検査・レビューへ反映さ
  れていることが確認できる

## 対象外

- AgentDevFlow 以外の汎用 OpenCode 設計論の追加
- 既存 command 名の変更
- 既存 workflow route の全面変更
- retired REQ の再構成や削除
- promoted artifact の直接修正
- ユーザー承認なしの GitHub Issue / PR 作成
- 新規 ADR 作成。ただし req-define の過程で ADR 閾値に達すると判断された場合は別途扱う
- すべての SKILL.md を一度に完全分割すること。優先度の高いものから段階的に行う
- line count だけを目的に可読性や安全性を落とすこと

## 推奨分解計画

Scale は large を推奨する。1 Issue / 1 PR に詰め込むとレビュー困難になり、再び command に詳細を積み戻すリスクが高いため、以下の分解を推奨する。

1. **REQ / SPEC / authoring standards 更新**: prompt budget、load_skills budget、thin dispatcher、SKILL.md入口カード、共通処理集約、integrity-check 強化を基準化する
2. **integrity-check 薄型化 + 検査追加**: command を script runner 化し、line count / load_skills / SKILL.md size / obsolete path / duplicated git sync / reporting body 検査を追加する
3. **runtime command 薄型化**: `case-run`, `case-close`, `req-define`, `req-backlog` を優先して削減する
4. **skill 入口カード化**: 大きい SKILL.md から順に references へ詳細を移し、必要時参照へ寄せる
5. **canonical path / reporting / docs drift cleanup**: active guidance の `reference/` 旧表記、completion report 参照、docs/specs/README の導線を整理する
