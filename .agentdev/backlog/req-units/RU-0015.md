# RU: 診断コマンドの検出精度改善

## 背景

`integrity-check` と `req-restructure-review` は、どちらも AgentDevFlow の整合性・健全性を人間が明示的に実行して確認する診断系コマンドである。

今回の方針として、AGENTS.md には導線や実行ルールを追加しない。人間が必要に応じて明示的にコマンドを実行する前提を維持する。

そのうえで、診断結果を次の作業に使いやすくするため、以下の方針で見直す。

1. `integrity-check` は過検出を削り、壊れているもの・高確度に直すべきものを中心に出す。
2. `req-restructure-review` は抽象的な観点名だけでなく、具体的な検出シグナルと出力基準を持つ。
3. 両コマンドの責務は統合しない。`integrity-check` は機械的・準機械的な整合性検査、`req-restructure-review` は REQ体系の意味的な再構成レビューとして分ける。
4. ファイル変更・自動修正・自動起票・AGENTS.md 変更は、この見直しの対象外とする。

## 問題

現行の診断観点には妥当な部分が多いが、以下の過検出と観点漏れがある。

### integrity-check の過検出

1. `missing-load-skills` が implementation pattern ごとの固定キーワードや substring matching に依存しており、実際の command 責務や side effect を確認せず大量の info / warning を出しやすい。
2. `excess-load-skills` が command 本文と skill 説明の語彙 overlap の少なさで過剰参照を推定しており、意味的に必要な analysis / guideline 系 skill まで不要候補にしやすい。
3. retired REQ 参照を current primary reference と historical reference に分けず、ADR・RE
   ADME・mapping-table・履歴説明に残る妥当な参照まで warning / NG 化しやすい。
4. ADR↔REQ 参照検査が active REQ だけを存在確認対象にすると、retired に実在する REQ を broken reference と誤判定する。
5. 完了報告検査が旧形式の command 別セクションを前提にし、Variant Registry + `completion-reports/{command}/{variant}.md` 形式を正しく扱えない場合がある。
6. legacy namespace / bare slash command 検査が、実際の public command 呼び出しではない path 断片や履歴記述まで拾う。
7. root README / `docs/specs/system.md` に全 command 掲載を要求すると、代表導線・仕様概要にすぎない文書に対して過剰な inventory drift を出す。
8. インライン完了報告検査が、例示コードブロックやエラー表示例まで完了報告本文の残存として検出しやすい。

### integrity-check の観点漏れ

1. retired REQ の frontmatter / filename / id の一致検査が active REQ と同等に扱われていない。
2. active / retired をまたぐ frontmatter id 重複検査が不足している。
3. mapping-table の正規パスは `docs/requirements/mapping-table.md` だが、検査実装・記述が古い path を前提にすると coverage が落ちる。
4. mapping-table について、retired coverage、phantom entry、successor existence、status enum、successor consistency の検査が不足している。
5. command 定義本文が参照している completion report variant path の実在確認・registry 登録確認が不足している。
6. skill frontmatter `name` と directory 名の一致、`description` の使用条件、`references/` / `reference/` 混在の検査が不足している。
7. command frontmatter について、`implementation_pattern` 必須、`secondary_pattern` 定義済み値、`load_skills` 配列形式、既知 agent、deprecated command の inventory 分離などの検査が不足
   している。

### req-restructure-review の課題

1. SPLIT / MERGE / MOVE / DUPLICATE / RETIRE / DRIFT の観点名は妥当だが、検出シグナルが抽象的で、実行者ごとに判断が揺れやすい。
2. 機械的なリンク切れ一覧・ID整合性一覧と、REQ体系の再構成判断が混ざると、`integrity-check` と責務が重複しやすい。
3. 1つの弱いシグナルだけで再構成 finding として出すと、過検出になりやすい。
4. 未処理 artifact の存在確認は必要だが、処理・生成・移送に踏み込むと read-only 診断の責務を超える。

## 統合理由

本RUは、個別の不具合修正ではなく、診断コマンドの検出精度・出力品質・責務境界を改善する一連の変更である。

主な反映先は以下の2系統に分かれる。

1. `integrity-check` の検出ルール整理と観点追加
2. `req-restructure-review` の診断シグナル具体化と出力基準追加

両者は同じ診断系コマンドだが、責務は統合しない。`req-define` では、必要に応じて REQ-0108 と REQ-0109 への APPEND / UPDATE として分けて扱う。

## 要件化の方向

### 推奨される要件操作

- CREATE よりも、既存REQへの APPEND / UPDATE を優先する。
- 主対象候補:
  - REQ-0108: `integ
rity-check` の検査精度、過検出削減、観点追加、regression fixture
  - REQ-0109: `req-restructure-review` の検出シグナル、finding 化基準、出力スキーマ、責務境界
- 関連候補:
  - REQ-0107: completion report variant registry / variant file 参照検査の前提が不足している場合のみ
  - REQ-0103: command / skill frontmatter と implementation pattern / load_skills 境界の要件が不足している場合のみ
  - REQ-0101: retired REQ の current / historical reference 境界が不足している場合のみ

### integrity-check: 標準検査から削る・弱めるもの

1. `missing-load-skills` は標準 finding から削除する。残す場合は、明示的な任意診断モード、または主 finding ではない observation とする。
2. implementation pattern ごとの固定 concern と substring matching だけで、load_skills 不足を断定してはならない。
3. command 本文と skill 説明の語彙 overlap が少ないことだけで、`excess-load-skills` を出してはならない。
4. load_skills の過剰参照 finding は、少なくとも以下のいずれかに限定する。
   - implementation pattern の禁止 side effect に明確に抵触する skill を読み込んでいる。
   - skill の DO NOT USE FOR と command の責務が明示的に衝突している。
5. retired REQ 参照は、current primary reference と historical reference を区別する。
6. mapping-table、retired index、履歴説明、置換関係説明に含まれる retired REQ 参照は、原則として current-use warning の対象外とする。
7. ADR / ADR README / README に retired REQ を残す場合でも、履歴参照または retired path であることが読める表記なら warning 対象外とする。
8. naked `REQ-NNNN` が現行要件根拠のように読める retired REQ 参照のみ、current-use finding として扱う。
9. ADR↔REQ の存在確認では、active または
   retired のどちらかに REQ が存在すれば broken reference とは扱わない。retired の current-use 判定は別 finding とする。
10. completion report 検査は、旧式の command 別見出し必須判定を削除する。
11. completion report 検査は、Variant Registry と `completion-reports/{command}/{variant}.md` の組み合わせを基準にする。
12. bare slash command 検査は public command 呼び出し表現に限定し、path 断片、completion-reports path、reference path、履歴記述、検出対象リスト、テスト fixture は標準 finding 対象外とする。
13. legacy namespace 検査が複数関数で重複している場合は統合し、同一根拠から重複 finding を出さない。
14. root README / `docs/specs/system.md` に全 command 掲載を要求しない。存在しない command を案内している場合、または明確に現行 command と矛盾する場合のみ finding とする。
15. インライン完了報告検査は、単に `完了` や `次のコマンド` が code block にあるだけでは検出しない。共通必須フィールドのうち複数項目が同一 code block に揃うなど、完了報告本文らしさが高い場合に限定する。

### integrity-check: 追加する観点

1. retired REQ についても、filename と frontmatter `id` の一致を検査する。
2. retired REQ についても、必須 frontmatter の欠落を検査する。
3. active / retired をまたぐ frontmatter `id` の重複を NG として検出する。
4. mapping-table の正規パスは `docs/requirements/mapping-table.md` とする。
5. retired REQ 全件が mapping-table に記録されているかを検査する。
6. mapping-table に、active / retired のどちらにも存在しない旧REQ ID がないかを検査する。
7. `migrated` の移行先が active REQ として存在するかを検査する。
8. mapping-table の status enum を検査する。
9. `retired-no-succ
essor` / `historical-only` など successor を持たない status で、移行先が不自然に設定されていないかを検査する。
10. command 定義本文が参照している completion report variant path が実在するかを検査する。
11. command 定義本文が参照している completion report variant が registry に載っているかを検査する。
12. command 名と completion report variant directory 名が一致するかを検査する。
13. `.opencode/skills/{dir}/SKILL.md` の frontmatter `name` が `{dir}` と一致するかを検査する。
14. skill description に使用条件または USE FOR / DO NOT USE FOR 相当の境界説明がない場合は warning または observation とする。
15. `references/` と `reference/` の混在、および `reference/` の残存を検査する。
16. command frontmatter の `implementation_pattern` 必須を検査する。
17. `secondary_pattern` が定義済み implementation pattern のみを使っているかを検査する。
18. `load_skills` が配列形式であるかを検査する。
19. `agent` が既知 agent 名かを検査する。
20. deprecated command が通常 command inventory に混ざっていないかを検査する。

### integrity-check: finding レベル設計

1. strict / heuristic / observation を区別する。
2. strict は存在、参照、frontmatter、index、registry、path など再現可能な検査を対象とする。
3. heuristic は意味判断を含むが、明確な根拠を持つ warning として扱う。
4. observation は参考情報であり、標準レポートの主 finding や NG カウントに含めない。
5. load_skills の意味的な過不足は、標準では strict にしない。
6. natural language interpretation を含むものは warning / recommendation に留める。
7. regression fixture / test は、削った過検出が再発しないことと、追加した観点が検出されることの両方を含める。

###

req-restructure-review: 具体シグナル

SPLIT / MERGE / MOVE / DUPLICATE / RETIRE / DRIFT は、以下の検出シグナルを使って診断する。

| 観点      | 検出シグナル                                                                                                              |
| --------- | ------------------------------------------------------------------------------------------------------------------------- |
| SPLIT     | 1つのREQに複数の関心対象、複数 artifact 種別、複数 command family、複数 lifecycle が混在している                          |
| MERGE     | 複数REQが同じ目的、同じ対象 artifact、同じ command、同じ責務を扱っている                                                  |
| MOVE      | REQ行が変更後仕様ではなく、ファイル移動、README更新、参照修正、テンプレート本文など反映作業そのものになっている           |
| DUPLICATE | active REQ 間、または REQ と spec / guide / command の間で、同じ SHALL / MUST 相当の責務が重複している                    |
| RETIRE    | active REQ が DOC-MAP / README / specs / command から現行要件として参照されず、関心対象も既存 active REQ に吸収済みである |
| DRIFT     | REQ が要求する command / skill / variant / path / lifecycle と、実体ファイルまたは command 定義が矛盾している             |

### req-restructure-review: finding 化基準

1. 1シグナルのみの場合は観察メモに留め、問題候補 finding としては出さない。
2. 2シグナル以上が揃った場合に、問題候補として出す。
3. 3シグナル以上、または active / retired 判断に影響する場合は高優先度候補として出す。
4. 機械的なリンク切れ・ID不整合の一覧化を主目的にしない。それらは `integrity-check` の責務とし、`req-restructure-review` では再構成判断に影響する場合だけ根拠として使う。
5. 未処理 intake / learning / RU / requirements-review finding は、存在と診断への影響可能性だけを報告する。処理、accept / reject、promote、RU生成は行わない。

### req-restructure-review: 出力スキーマ

問題候補を出す場合は、少なくとも以下を含める。

| フィールド | 内容
|
|---|---|
| 観点 | SPLIT / MERGE / MOVE / DUPLICATE / RETIRE / DRIFT |
| 対象 | REQ ID、関連 ADR、spec、command、skill、variant path など |
| 根拠 | REQ行、関連 artifact、参照導線、実体ファイルとの差分 |
| シグナル数 | finding 化に使ったシグナルの数 |
| 確信度 | high / medium / low |
| 推奨アクション | SPLIT / MERGE / MOVE / RETIRE / UPDATE / APPEND / no-action |
| req-define入力案 | 要件化する場合に壁打ちすべき最小内容 |

## req-define 初期判定メモ

- 推奨 Pattern: maintenance / refactor 系（Pattern C）候補。ただし、実装対象が command 定義・integrity script・fixture/test・REQ更新にまたがるため、`req-define` で scale を確認する。
- ADR: 不要候補。既存の責務境界・診断系コマンドの設計を変更するというより、既存方針の精緻化である。
- CREATE: 原則不要。REQ-0108 と REQ-0109 への UPDATE / APPEND を優先する。
- SPLIT: 保存時は、`integrity-check` 精度改善と `req-restructur
e-review` シグナル具体化を別Issueに分ける可能性がある。
