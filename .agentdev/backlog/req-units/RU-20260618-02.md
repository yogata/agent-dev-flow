# RU: case-auto の大規模Epic実行モデルを子Issue単位オーケストレーションへ修正する

## 背景

case-auto で大規模な作業に対応するため、Epic Issue に複数の子Issueを紐づけ、各子Issueを実装可能なサイズに分割し、順次実装する方式が必要である。

当初は、コンテキスト肥大を防ぐためにコンパクション境界をどこに置くかを議論した。しかし、コマンド仕様が直接保証すべき対象はコンパクション実行そのものではなく、子Issue単位で親コンテキストに実装詳細を累積させず、永続状態から進行・再開できる実行モデルであると整理した。

## 問題

現行整理では、case-auto が大規模Epicを処理する際に、Epic内の全Wave・全子Issueの実行を一括的に case-run 側へ委譲する形になり得る。この形では、Epic内の子Issue単位で作業コンテキストを切ることが難しく、親セッションに実装詳細や累積判断が溜まる。

大規模Issueへの対応では、case-auto は実装主体ではなく、Epic / Issue / PR / 状態表を読み、次の1 Issueを選択する薄いオーケストレーターであるべきである。実装作業は case-run から子Issue単位のサブエージェントへ委譲する必要がある。

## Source Summary

チャット内で以下を合意した。

- case-auto は全体の進行管理に寄せる。
- case-auto は一度に1つの子Issueだけを選択し、case-run に渡す。
- case-run は指定された1 Issueの実装を扱う。
- 実装作業は子Issue単位でサブエージェントへ委譲する。
- サブエージェントは1 Issueの実装に閉じ、親のEpic全体や他Issueの履歴を抱えない。
- コンパクション自体は主要件にしない。
- 要件化する対象は、親コンテキスト非累積、子Issue単位委譲、永続状態ベースの進行・再開である。
- 子Issue単位に実装詳細を閉じることで、コンパクション耐性は副次的に成立する。

## 統合理由

このRUは case-auto の実行モデルに関する変更であり、Epic構成を誰が生成するかという責務変更とは分離する。

本RUの対象は、case-open が生成済みの実行構造を case-auto がどのように進めるかである。Epic / Wave / Issue の生成責務は別RUで扱う。

## 要件化の方向

### REQ concern

REQでは、case-auto が大規模Epicを親コンテキストへ実装詳細を累積させずに進行できる状態を定義する。

要件化の方向は以下。

- case-auto は、Epic全体の実装を直接保持・実行せず、進行管理に徹する。
- case-auto は、実行可能な子Issueを1件ずつ選択し、case-run へ渡せる。
- case-auto は、子Issue単位の実装結果を GitHub Issue / PR / `.agentdev/` などの永続状態から把握できる。
- case-auto は、子Issueの実装過程ログを次Issue選択の前提にしない。
- case-auto は、子Issue完了後に永続状態を再読込し、次の実行対象を選択できる。
- case-run は、case-auto から指定された1 Issueの実装実行に閉じる。
- case-run は、1 Issue単位で実装作業をサブエージェントへ委譲できる。
- サブエージェントは、渡された1 Issueの実装に必要な文脈だけを扱う。

### SPEC concern

SPECでは、上記REQを実現するための契約・状態遷移・入力出力を定義する。

仕様化の方向は以下。

- case-auto が読み取る永続状態の種類。
- case-auto が次Issueを選択する入力。
- case-auto が case-run に渡す最小入力。
- case-run がサブエージェントへ渡す1 Issue単位の入力。
- case-run / subagent の結果を Issue / PR / `.agentdev/` へ永続化する形式。
- 子Issue完了後に case-auto が再読込する状態。
- blocked / failed / completed の扱い。
- 親コンテキストに実装過程ログを持ち越さないための出力契約。

コンパクション実行そのものはSPEC対象にしない。SPEC対象は、コンパクションが発生しても再開・継続できる永続状態契約である。

## 主対象REQまたは変更対象候補

変更対象候補は以下。

- case-auto の自律実行・大規模作業処理を定義するREQ。
- case-run の実装実行単位を定義するREQ。
- workflow orchestration / artifact contract / case lifecycle に関するSPEC。
- case-auto / case-run / case-close コマンド定義。
- サブエージェント委譲契約を記述する関連スキルまたはSPEC。

正確なREQ番号・SPEC名は、後続工程で最新リポジトリを確認して確定する。

## 対象外

以下は本RUでは扱わない。

- case-open が Epic / Wave / Issue をどのように生成するか。
- req-define が実装順序やIssue分解を扱うかどうか。
- コンパクション実行そのものをコマンド要件にすること。
- 1 Issue内の詳細な実装手順。
- PRの具体的な差分作成手順。
- 並列実行の導入。
- Issue作成、PR作成、既存Issueコメントの実施。
- コード実装。

## 受け入れ条件

- case-auto は、Epic全体の実装詳細を保持せず、進行状態から次の1 Issueを選択できる。
- case-auto は、一度に1つの子Issueだけを case-run に渡す。
- case-run は、渡された1 Issueの実装に閉じる。
- case-run は、1 Issue単位でサブエージェントへ実装作業を委譲できる。
- サブエージェントは、他Issueの実装履歴やEpic全体の実装過程を前提にしない。
- 子Issueの完了・失敗・停止結果は、親コンテキストではなく永続状態に記録される。
- case-auto は、子Issue完了後に永続状態を再読込して次対象を選択できる。
- コンパクションそのものをREQ/SPECの主対象としていない。
- case-open のEpic構成生成責務を本RUに混入していない。
