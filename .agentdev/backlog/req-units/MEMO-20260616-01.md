# 添付RU群適用時の補足コンテキストメモ

## 1. このメモの位置づけ

このメモは Requirement Unit ではない。

添付RU群を `req-define` / `case-auto` で処理する際に、実行エージェントが推論の中で参照してよい補足コンテキストである。

一次入力は、添付RUファイル群と最新レポジトリ `agent-dev-flow-main-2026-06-16.zip` である。このメモは、添付RU本文にない新しい要件、受け入れ条件、対象ファイル、実装作業を追加するための根拠ではない。

## 2. 優先順位

解釈の優先順位は以下とする。

1. 添付RU本文
2. 最新レポジトリの実状態
3. 既存の REQ / ADR / SPEC / docs / command / skill の定義
4. この補足コンテキストメモ

このメモとRU本文が矛盾する場合は、RU本文を優先する。

このメモから新規Issue、新規REQ、新規ADR、新規SPEC、追加受け入れ条件を直接作成してはならない。追加すべきに見える事項は、今回のRU適用範囲には含めず、finding / backlog / intake 候補として記録する。

## 3. 今回のRU群の全体像

今回のRU群は、単なる文言修正ではなく、agent-dev-flow の体系再編である。

主な関心は以下である。

- REQ体系全体の再編成
- active REQ から移行履歴、改名作業、廃止作業、バグ修正履歴を退避すること
- command-level REQ の所有関係を明確化すること
- docs-check / inspect-* / integrity 系の責務境界を整理すること
- ADR の current baseline を accepted ADR 中心に整理すること
- `case-run` の具体実装責務を外部実行バックエンドへ委譲し、agentdev 側を workflow state orchestration に寄せること

目指す姿は、LLMに長大な手順を抱えさせる command 集ではなく、Issue / PR / `.agentdev/` domain state を中心に、薄い command、再利用 skill、機械検査、外部実行バックエンドを接続する self-hosting workflow system である。

## 4. 推奨される実行順序

### 4.1 REQ再編の親子関係

`RU-0001.md` は REQ体系再編の親RU、または Epic 相当の統合コンテキストとして扱う。

`RU-0002.md` から `RU-0007.md` は、`RU-0001.md` の子RUとして扱うのが自然である。

`RU-0001.md` と子RU群を同時に入力する場合、同じ作業を親RUと子RUの両方から重複してIssue化しないこと。親RUは全体方針・依存関係・完了像のコンテキストとして扱い、具体作業は子RUで分担する。

### 4.2 推奨順序

推奨順序は以下である。

1. `RU-0002.md`
   - REQ分類、粒度ルール、新設/更新/分割/統合/退役判断の基準を先に固める。
2. `RU-0003.md`
   - ワークフロー全体REQと command map を確立する。
3. `RU-0004.md`
   - req/case/intake/learning/backlog 系 command REQ を分割・再配置する。
4. `RU-0005.md`
   - 文書統治、ADR、artifact、runtime/skill 責務の横断REQを整理する。
5. `RU-0006.md`
   - docs-check / inspect-* / integrity 系REQを再編する。
6. `RU-0007.md`
   - retire / merge / index / mapping-table / docs-check 整合を最後に完了する。

`RU-0007.md` は仕上げ工程である。前段の統合・分割・退役判断が完了する前に実施すると、index / mapping / rule ownership が再度ずれる可能性がある。

### 4.3 ADR再編RU

`RU-0008.md` は ADR体系再編であり、REQ再編とは独立に実行可能である。

ただし、`document-model.md`、REQ-0101、REQ-0112 の文書種別責務境界と整合させる必要があるため、可能であれば `RU-0002.md` による分類・粒度ルール確定後に実施する。

`diagnostics-* -> inspect-*` について新規ADRを作らないこと。これは現行状態・命名・移行整理として REQ / SPEC 側で扱う。

ADR-0113 は `deprecated` とし、現行根拠として扱わない。

ADR README の Current Baseline View は accepted ADR のみを現行根拠として読める構成にする。

### 4.4 OMO委譲RU

`RU-20260616-01.md` は、`case-run` の実装責務を外部実行バックエンドへ委譲する大きな境界変更である。

このRUは、`RU-0004.md` によって `case-run` / `case-close` の command-level REQ責務が整理された後に適用するのが自然である。

特に以下を混同しないこと。

- `case-run` の成功成果は PR 作成である。
- `case-run`、driver subagent、OMO は Issue本文の完了条件チェックボックスを更新しない。
- Issue本文の完了条件チェックボックスの評価・更新は `case-close` の責務である。
- agentdev の正式完了判定は、`case-close` が Issue本文の完了条件を再読込して行う。

## 5. 解釈上の重要前提

### 5.1 docs/ は RU を根拠として参照しない

RUは case-open で削除される入力であり、永続文書ではない。

そのため、docs/、REQ、ADR、SPEC には、特定RUを設計根拠として参照する記述を残さないこと。

RUに含まれる恒久的内容は、REQ / ADR / SPEC / docs の適切な場所へ転記・再構成する。RU-IDやこの補足メモを durable design rationale として記録しない。

### 5.2 active REQ は現在満たすべき状態を書く

active REQ には、以下を主題として残さない。

- 移行作業そのもの
- 改名作業そのもの
- 削除作業そのもの
- 廃止作業そのもの
- 一度限りのバグ修正
- 過去の検討経緯
- 統合・退役の理由説明そのもの

これらは必要に応じて、retired REQ、mapping-table、Update Notes、履歴説明へ退避する。

active REQ は「今後も満たすべき状態・制約・責務境界」を表す。

### 5.3 REQ / ADR / SPEC / Guide / DOC-MAP の境界

原則は以下である。

- REQ: 満たすべき WHAT、保証、制約、責務境界
- ADR: 将来を制約する設計判断と理由
- SPEC: 現在仕様、schema、lifecycle、command topology、rule catalog、fixture detail
- Guide: 利用者向け案内、手順、読み方
- DOC-MAP: 文書探索と参照経路の入口

ADRは、削除・廃止・移行・統合・完全削除そのものを主題にしない。

SPECに置くべき schema / lifecycle / command topology / rule catalog / fixture detail を active REQ に抱え込ませない。

### 5.4 command と skill の基本姿勢

command は薄く保つ。

command は入口、入力解決、workflow上の責務、停止条件、次の接続を担う。

skill は再利用可能な判断基準、検査観点、実行補助、サブエージェント委譲時の限定タスクを担う。

長大な実装手順、重複したチェック観点、他command詳細の再掲を command に抱え込ませない。

### 5.5 原本と配置先の用語

日本語文書では以下の用語を用いる。

- `src/opencode/` は「原本」
- `.opencode/` は「配置先」
- command / skill file を原本から配置先へ置くことは「配置する」

`正本`、`authoring source`、`projection`、`実行用配置`、`反映` を主要表現として使わない。

### 5.6 docs-check と inspect-* の違い

`/repo/docs-check` は repo-local な機械検査である。

`/agentdev/inspect-docs` は意味レビュー、REQ体系整合、文書分類、再構成 finding の検出である。

`/agentdev/inspect-skills` は skill / command / reference の検出・評価である。

`/agentdev/inspect-promote` は inspect finding を req-define 可能な入力へ昇格する lifecycle を担う。

`docs-check` が意味レビューを代替しない。`inspect-docs` が機械検査を代替しない。

### 5.7 case-run / case-close / OMO の境界

`case-run` は、Issue単位の実装詳細を自分で抱え込まず、driver subagent を通じて外部実行バックエンドへ委譲する方向に整理する。

OMO固有の処理は driver skill 側に閉じ込める。

OMO plan artifact は agentdev から不透明な外部成果物として扱う。

最小 driver result は以下の3状態とする。

- `completed(pr)`
- `blocked`
- `failed`

成功時の詳細SSoTは PR本文である。

blocked / failed の詳細SSoTは Issueコメントである。

Findings / Capture候補が存在する場合は PR本文に記述する。

`case-close` は PR本文を capture 入力源として扱い、Issue本文の完了条件を再読込して正式完了判定する。

## 6. 適用時に避けること

以下は避ける。

- このメモをRUとして扱うこと
- このメモから新規要件を起こすこと
- RU本文にない追加受け入れ条件を作ること
- 親RUと子RUから同じ作業を二重にIssue化すること
- active REQに「なぜ退役したか」「何を削除したか」を主題として残すこと
- docs/に特定RU-IDを設計根拠として残すこと
- `diagnostics-* -> inspect-*` のために新規ADRを作ること
- `case-run`、driver subagent、OMO が Issue本文の完了条件チェックボックスを更新すること
- OMO内部の plan artifact 構造を agentdev 側の正式仕様として定義すること
- command を再び巨大な手順書にすること

## 7. finding 候補として扱うべき追加案

以下は有用かもしれないが、添付RU本文の必須実装対象として扱わない。

### command ownership map

RU適用後、command ownership を継続的に検査しやすくするため、command / primary REQ / input SSoT / output SSoT / GitHub副作用 / local file副作用 / capture責務 / next command を一覧化する SPEC 表が有用かもしれない。

ただし、これはこのメモから直接実装しない。

必要と判断した場合は、backlog / intake / inspect finding 候補として記録し、別途 req-define 可能な入力にする。

## 8. 完了像の読み方

添付RU群の適用後、目指す状態は以下である。

- active REQ の各ファイルが、関心対象の総体として説明できる。
- 各公開commandに、親ワークフローREQまたはcommand-level REQ上の明確な責務所有先がある。
- 移行・改名・削除・バグ修正そのものが active REQ の主題として残らない。
- REQ / ADR / SPEC / Guide / DOC-MAP の責務分界が document-model と矛盾しない。
- docs index、DOC-MAP、AGENTS.md、mapping-table の active REQ 範囲と概要が一致する。
- docs-check / integrity 検査で、再編に起因する stale range、旧command名、壊れたリンク、retired参照が残らない。
- accepted ADR が current baseline として読め、deprecated / superseded ADR が現行根拠として混入しない。
- `case-run` は Issue実行の orchestration と driver結果処理へ責務を絞る。
- `case-close` は別コンテキストで Issue完了条件を評価・更新する。

## 9. 実行エージェントへの最終注意

このメモは、判断の補助である。

作業範囲を決めるときは、必ずRU本文を優先すること。

RU本文にないが重要に見える事項は、その場で実装範囲に追加せず、finding / backlog / intake 候補として記録すること。

最終的な永続成果物には、このメモやRU-IDを根拠として残さず、REQ / ADR / SPEC / docs の適切な文書体系へ内容を落とし込むこと。
