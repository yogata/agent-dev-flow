# RU: 複数RU入力時の要件doc分割とIssue階層展開の安定化

## 背景

req-define に複数RUを渡した場合、1つまたは複数の要件docへ整理される可能性がある。

後続で req-save / case-open / case-auto を実行した場合、複数REQ docに対応するIssue構造が文脈・意味ベースで処理されており、動作が安定していない。

現行の責務境界として、req-define は要件docへの整理と scale / decomposition 等の判断材料の記録までを担い、Issue階層の最終決定は case-open が担う。case-auto は req-save / case-open / case-run / case-close の既存責務を再実装せず、同じ成果物と判定ロジックを使う前提で扱う。

## 問題

複数RU入力時に、req-define が以下を安定して明示する契約が不足している。

- 複数RUを1つの要件docへ統合するか、複数REQ操作として分けるか
- 複数REQ操作を後続がどう識別するか
- large / Epic 規模として扱うべきか
- REQ間または子Issue間に順序制約があるか
- case-open が Epic / 子Issue / 孫Issue / wave を判断するために参照すべき情報

その結果、case-open / case-auto が文脈から補完して動作するが、同じ入力に対するIssue構造が安定しない。

## Source Summary

チャット内で以下を合意した。

- Issue階層の最終判定は req-define の責務ではない。
- req-define は、複数RU入力時に後続が判断できる情報を要件docまたは draft-meta に残す責務を持つ。
- req-save は、複数REQ操作を保存し、後続へ参照可能な保存結果を残す責務を持つ。
- case-open は、REQ doc群、scale、decomposition、wave候補を読み、Issue構造を決定する責務を持つ。
- case-auto は、req-save / case-open の責務を再実装せず、同じ成果物・同じ判定ロジックを使う。
- 現状の case-auto 方針はこの前提を満たしているため、主修正対象ではない。
- 安定化の主対象は req-define の複数RU入力契約、req-save の後続引き継ぎ、case-open のIssue展開判断である。

## 統合理由

複数RU入力、複数REQ doc保存、Epic / 子Issue / 孫Issue / wave 展開は別々の振る舞いではなく、1つの後続処理契約として扱う必要がある。

req-define の出力が曖昧なままだと、req-save / case-open / case-auto 側で文脈補完が発生し、動作が安定しない。

したがって、複数RU入力時の出力契約と、後続コマンドが参照する情報を一体で要件化する。

## 要件化の方向

### req-define

req-define は、複数RUを入力として扱う場合の整理規則を明文化する。

- 複数RU入力を許可する。
- 各RUを Requirement Source として扱う。
- 同一主題・同一主対象REQ・同一変更理由に収まるRU群は、1つの要件docまたは1つのREQ操作へ統合する。
- 主対象REQ、変更理由、受け入れ条件、対象外、実行順序が分かれるRU群は、複数REQ操作として整理する。
- 複数REQ操作がある場合、後続が参照できる形で明示する。
- large / Epic 規模の場合、`scale: large` と `decomposition` を記録する。
- REQ間または子Issue間に順序制約がある場合、wave候補または依存関係を記録する。
- Issue階層そのものは確定しない。

### req-save

req-save は、複数REQ操作を保存した結果を後続が参照できるように明示する。

- 保存したREQ doc一覧を保持する。
- 各REQ操作と保存先REQ docの対応を保持する。
- source RUとの対応を保持する。
- case-open が入力として扱える保存結果を明示する。
- Issueは作成しない。

### case-open

case-open は、req-define / req-save で明示された情報をもとにIssue構造を決定する。

- 単一REQ docの場合は、既存どおりREQ doc対応Issueを作成する。
- 複数REQ docまたは `scale: large` の場合は、Epic Issueを作成する。
- Epic配下にREQ doc単位の子Issueを作成する。
- 必要な場合のみ、子Issue配下に孫Issueを作成する。
- wave候補または依存関係がある場合、Epic本文または子Issue本文に反映する。
- REQ doc単位のトレーサビリティを崩して、wave単位だけの子Issue構造にはしない。

### case-auto

case-auto は、既存の req-save / case-open / case-run / case-close の責務を再実装しない。

- req-save の保存結果を case-open 相当処理へ引き継ぐ。
- case-open と同じIssue Plan規則を使う。
- case-auto独自のIssue階層判定を持たない。
- 必要な修正は、引き継ぎ仕様の明確化に留める。

## 主対象REQまたは変更対象候補

- req-define command 定義
- req-save command 定義
- case-open command 定義
- case-auto command 定義
- 必要に応じて、workflow / artifact contract 系docs

## 対象外

- case-auto に独自のIssue階層判定を追加すること
- wave単位をREQ doc単位より上位の主トレーサビリティ単位にすること
- req-define がGitHub Issue構造を確定すること
- 実装手順の詳細化
- GitHub Issue作成APIの具体実装
- 未合意のIssueテンプレート変更
- 未合意のIssue / PR作成

## 受け入れ条件

- 複数RU入力時に、req-define が統合・分割・scale・decomposition・依存関係の判断結果を後続参照可能な形で出力すること。
- 複数RUが1つの要件docに統合される条件が明文化されていること。
- 複数RUが複数REQ操作に分かれる条件が明文化されていること。
- req-save が複数REQ操作の保存結果を後続へ渡せること。
- case-open が単一REQ doc、複数REQ doc、scale: large の各ケースでIssue構造を判断できること。
- 複数REQ docの場合、Epic Issue配下にREQ doc単位の子Issueを作成する方針が明文化されていること。
- 孫Issueは必要な場合のみ作成する方針が明文化されていること。
- waveはEpic本文または子Issue本文に反映され、REQ doc単位のトレーサビリティを置き換えないこと。
- case-auto が case-open と異なるIssue階層判定を持たないこと。
- case-auto の修正が必要な場合でも、req-save から case-open 相当処理への引き継ぎ明確化に限定されていること。
