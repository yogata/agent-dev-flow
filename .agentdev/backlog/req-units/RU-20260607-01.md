# RU: case-auto における Epic Issue 状態引き継ぎの明確化

## 背景

`case-auto` は自走コマンドとして、`req-save`、`case-open`、`case-run`、`case-close` 相当の工程を連続実行する想定である。

今回、Wave 1 の PR 作成後に、次の進め方として「Wave 1 PR を merge してから Wave 2 に進むか」「Wave 2 も同じブランチに追加するか」という確認が発生した。

しかし、Epic Issue には Wave 構成と子Issueが作成済みであり、さらに既存コメント上で PR / branch 方針が合意済みだった。したがって、`case-auto` はユーザー確認ではなく、Epic Issue の本文・コメント・子Issue・PR状態を読み取り、合意済み方針に沿って進行判断すべきだった。

## 問題

`case-auto` が Epic Issue を制御単位として保持できていない。

具体的には、`case-open` によって作成された Epic Issue、Wave 構成、子Issue、関連コメント、関連PR状態が、後続の `case-run` / `case-close` 相当処理へ十分に引き継がれていない。

その結果、既に Epic 側で決まっている Wave 順序や PR / branch 方針を再確認してしまう。

## Source Summary

チャット内で以下を合意した。

- 直接原因は `case-open` 単体ではなく、`case-auto` における Epic Issue の状態引き継ぎ不足である。
- `case-open` は Epic Issue、子Issue、Wave 構成の作成という基本責務は果たしている。
- ただし、PR集約方針、branch方針、検証専用Waveの扱いが Epic 本文の構造化項目として残らない場合、後続工程で落ちる可能性がある。
- `case-auto` は `case-open` 後、Epic Issue を制御単位として保持し、Epic本文、コメント、子Issue状態、関連PR状態を読んで次工程を決める必要がある。
- 合意済み方針がある場合、`case-auto` はユーザーに再確認しない。
- 停止して確認すべきなのは、scope拡大、CI失敗、merge conflict、repo外変更、認証・権限変更など、既存方針では解決できない場合に限る。

## 統合理由

この問題は単なる確認文言の不備ではない。

`case-auto` が Epic Issue を状態管理の中心として扱わない場合、Wave分割、子Issue、PR、case-close の完了条件が工程間で分断される。これは `case-auto` の自走性と、既存コマンド成果物の再利用方針に直接影響する。

また、`case-open` 側にも Epic artifact の構造化不足があるため、`case-auto` の修正だけでは再発を完全に防げない。主対象は `case-auto` とし、補助的に `case-open` / Epic artifact contract の明確化を含める。

## 要件化の方向

`case-auto` は、`case-open` が Epic Issue を作成または返却した場合、その Epic Issue を case 全体の制御単位として保持しなければならない。

`case-auto` は後続工程で、少なくとも以下を読み取る。

- Epic Issue 本文の Wave 構成
- Epic Issue 本文の子Issue一覧
- Epic Issue コメント上の合意済み実行方針
- 子Issueの open / closed 状態
- 関連PRの open / merged 状態
- PR本文に記載された close 対象Issue
- case-close に必要な残タスク

合意済み方針が読み取れる場合、`case-auto` は同じ判断をユーザーに再確認しない。

`case-open` は、後続の `case-auto` / `case-run` が読み取れるように、Epic Issue に以下の実行制御情報を構造化して残す。

- Wave順序
- Waveごとの対象子Issue
- PR集約方針
- branch方針
- 検証専用Waveの扱い
- case-close 到達条件

## 主対象REQまたは変更対象候補

主対象候補:

- `case-auto` の要件またはコマンド定義
- `case-run` の Epic Issue / Wave orchestration 定義
- `case-open` の Epic Issue artifact contract
- Epic Issue template または関連する artifact contract

変更対象候補:

- `.opencode/commands/.../case-auto.md`
- `.opencode/commands/.../case-open.md`
- `.opencode/commands/.../case-run.md`
- `docs/specs/artifact-contracts.md`
- 関連REQの APPEND または UPDATE

正確なファイルパスとREQ番号は、後続の `req-define` でリポジトリ最新状態を確認して確定する。

## 対象外

以下は対象外とする。

- 既存Issue / PRへのコメント投稿
- 既存PRのmerge
- 既存Issueのclose
- 実装手順の策定
- コード差分案の作成
- GitHub Actions / CI の挙動変更
- Wave分割アルゴリズム自体の全面見直し
- `case-open` のIssue作成責務そのものの変更
- ユーザー未承認の保存、commit、push

## 受け入れ条件

- `case-auto` が `case-open` 後の Epic Issue 番号を後続工程へ引き継ぐことが定義されている。
- `case-auto` が Epic本文、Epicコメント、子Issue状態、関連PR状態を進行判断に使うことが定義されている。
- Epic Issue 上に合意済み方針がある場合、`case-auto` が同じ判断をユーザーに再確認しないことが定義されている。
- `case-auto` が停止してユーザー確認する条件が、scope拡大、CI失敗、merge conflict、repo外変更、認証・権限変更などに限定されている。
- `case-open` が PR集約方針、branch方針、検証専用Waveの扱い、case-close 到達条件を Epic Issue に構造化して残すことが定義されている。
- `case-run` または `case-auto` が Epic Issue の Wave 構成を読み取り、Wave順序と子Issue状態に基づいて次工程を判断することが定義されている。
- `case-close` 相当処理に到達する前に、関連PRと子Issueの状態を確認することが定義されている。
- 未合意の実装手順、コード差分、既存Issue操作が要件本文に混入していない。
