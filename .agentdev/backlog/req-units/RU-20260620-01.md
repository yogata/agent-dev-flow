# ローカル版生成仕様とローカルCaseファイル運用の管理方式

## 背景

AgentDevFlow は GitHub Issue / PR を前提にした `src/opencode/` を持つ。一方で、ローカルで完結する個人利用版では GitHub Issue / PR を使えないため、Issue / PR 相当の情報をローカルファイルで扱う必要がある。

ローカル版は暫定的・個人利用向けであり、GitHub 版の原本である `src/opencode/` をローカル版のために改変しない。また、バックエンド抽象化や GitHub 互換ローカルサーバの導入は、現時点では過剰対応として扱う。

## 問題

GitHub 版の `case-open` / `case-run` / `case-close` は、GitHub Issue 作成、PR 作成、PR 取り込み、Issue のクローズを前提にしている。

ローカル版では GitHub Issue / PR を使えないため、GitHub Issue本文、GitHub PR本文、Issueコメント、PR取り込み結果、Issue のクローズ結果が担っていた情報をローカルファイルへ置換する必要がある。

特に PR本文は、実行結果、SPEC確定候補、Findings / Capture候補をローカル版 `case-close` に引き継ぐ媒体である。PR を独立オブジェクトにしない場合、これらの記録先をローカルCaseファイルに定義する必要がある。

また、ローカル版コマンド / スキルをリポジトリ管理すると、GitHub版原本である `src/opencode/` との二重管理が発生する。したがって、リポジトリ管理対象はローカル版生成に必要な仕様・定義・変換プロンプトに限定する必要がある。

さらに、`.opencode/` がジャンクション / シンボリックリンク等により `src/opencode/` へ接続されている環境でローカル版を生成すると、`src/opencode/` を変更しないという要件と衝突する。このため、ローカル版生成は、`.opencode/` が `src/opencode/` に接続されていない導入先リポジトリで実行する必要がある。

## 統合理由

本件は、ローカル版のファイル運用、GitHub Issue / PR の置換、ローカルCaseファイルの構造、変換仕様、変換プロンプト、ひな形変換、生成先の安全確認、リポジトリ管理範囲が不可分である。

`case-open` / `case-run` / `case-close` の個別差分だけを扱うと、PR本文が担っていた SPEC確定候補や Findings / Capture候補の引き継ぎが欠落する。

また、生成済みコマンド / スキルをリポジトリ管理対象にすると、GitHub版原本である `src/opencode/` との二重管理が発生する。

さらに、`.opencode/` が `src/opencode/` へ接続された環境で生成を許すと、ローカル版生成が GitHub版原本を変更する危険がある。

したがって、ローカル版を「仕様・定義・変換プロンプトをリポジトリ管理し、生成物は安全確認後に導入先リポジトリの `.opencode/` へ直接配置する方式」として一体で要件化する。

## 要件化の方向

ローカル版では、仕様管理リポジトリと生成先リポジトリを分けて扱う。

仕様管理リポジトリは AgentDevFlow 本体リポジトリであり、`src/opencode/` と `src/opencode-local/` を持つ。

生成先リポジトリは、ローカル版を導入する利用側リポジトリであり、`.opencode/commands/`、`.opencode/skills/`、`.agentdev/cases/` を持つ。

ローカル版生成は AgentDevFlow 本体リポジトリでは実行しない。ただし、生成時の入力元として AgentDevFlow 本体リポジトリの `src/opencode/` と `src/opencode-local/` を参照する。

`src/opencode-local/` は、ローカル版生成のための要件、仕様、定義ファイル、変換プロンプトを保持する。

`src/opencode-local/` 配下には、生成済みコマンド / スキル / ひな形を置かない。

変換スクリプトはリポジトリ管理対象にしない。AI エージェントが、仕様管理リポジトリの `src/opencode/` と `src/opencode-local/` 配下の仕様・定義・変換プロンプトを読み、生成先リポジトリの `.opencode/commands/` および `.opencode/skills/` にローカル版コマンド / スキル / ひな形を直接生成する。

ローカル版生成前には、生成先リポジトリの `.opencode/` の実パスと既存同名ファイルを確認する。`.opencode/` が `src/opencode/` 配下へ解決される場合、または安全に上書きできない同名ファイルが存在する場合は生成を停止する。

後続の `req-define` では、ローカル版生成方式を単一の要件単位として扱い、既存REQへの追記は既存契約との矛盾解消に必要な範囲に限定する。

## 仕様管理リポジトリのディレクトリ構成

ローカル版仕様管理の構成は以下を要件とする。

```text
src/opencode-local/
  README.md

  requirements/
    local-case-file.md
    local-generation.md

  specs/
    local-case-file.md
    local-transform.md

  rules/
    case-frontmatter.yaml
    case-status.yaml
    case-labels.yaml
    case-headings.yaml

  prompts/
    generate-local-transform.md
    review-local-transform.md
```

`src/opencode-local/` は、ローカル版生成のための仕様・定義・変換プロンプトを管理する領域である。

`src/opencode-local/` 配下には、生成済みコマンド / スキル / ひな形を置かない。

`src/opencode-local/_conv/`、`src/opencode-local/commands/`、`src/opencode-local/skills/` は作成しない。

## 生成先リポジトリの配置方針

ローカル版コマンドは、生成先リポジトリの `.opencode/commands/` に直接配置する。

ローカル版スキルは、生成先リポジトリの `.opencode/skills/` に直接配置する。

ローカル版コマンド / スキルが参照するひな形は、参照元からの相対参照が解決できる場所に配置する。

既存の相対参照構造を維持することを優先する。維持できない場合は、参照元から解決できる配置へ変換し、その変更理由と変更内容を変換レポートに記録する。

`src/opencode-local/` は生成物の配置先ではない。`src/opencode-local/` は、ローカル版生成に必要な仕様・定義・変換プロンプトのみを保持する。

生成後の `.opencode/commands/` および `.opencode/skills/` はリポジトリ管理対象にしない。

生成前には、生成先リポジトリの `.opencode/` の実パスを確認する。`.opencode/` が `src/opencode/` 配下へ解決される場合、ローカル版生成は停止する。

生成前には、生成先リポジトリの `.opencode/commands/` および `.opencode/skills/` の同名ファイルを確認する。

同名ファイルが存在しない場合は、生成できる。

同名ファイルに `generated_by: local-opencode-transform` の識別情報がある場合は、過去のローカル版生成物として再生成・上書きを許可する。

同名ファイルに識別情報がない場合は、安全確認不能な既存ファイルとしてローカル版生成を停止する。

同名ファイルに異なる識別情報がある場合は、ローカル版生成を停止する。

ローカル版は GitHub 版と同じ `/agentdev/*` および `agentdev-*` の名前で生成する前提とする。このため、GitHub 版とローカル版を同じ `.opencode/` に同居させる利用環境は対象外とする。

## 生成物識別情報

ローカル版生成物には、過去のローカル版生成物かどうかを判定するための識別情報を持たせる。

Markdown または YAML前書きを持つ生成物には、以下を記録する。

```yaml
generated_by: local-opencode-transform
```

YAML前書きを持たない生成物には、同等の先頭コメントを記録する。

識別情報が存在しない同名ファイルは、安全確認不能として扱う。

## ローカルCaseファイル要件

ローカルCaseファイルは、生成先リポジトリの以下のパスで管理する。

```text
.agentdev/cases/case-{NNNN}.md
```

`.agentdev/cases/` 配下のローカルCaseファイルは、Issue / PR 相当の永続情報としてリポジトリ管理対象にする。

`{NNNN}` は 4 桁ゼロ埋め番号とする。

YAML前書きは以下を必須とする。

```yaml
id:
title:
status:
created_at:
updated_at:
closed_at:
labels:
```

`id` は `case-{NNNN}` 形式とし、ファイル名 `case-{NNNN}.md` と一致させる。

`closed_at` は `status: closed` または `status: cancelled` の場合のみ値を持つ。それ以外では空とする。

ブランチ情報は YAML前書きに持たせない。

## ローカルCaseファイルラベル

`labels` は配列とする。

`labels` は `rules/case-labels.yaml` から選定する。

`labels` は補助分類であり、状態遷移やワークフロー状態の代替として扱わない。

`rules/case-labels.yaml` は以下の値域を定義する。

```yaml
label_enum:
  - feature
  - bugfix
  - maintenance
  - docs
  - refactor
  - chore
  - epic
```

## ローカルCaseファイル採番規則

`{NNNN}` は 4 桁ゼロ埋め番号とする。

新規作成時は `.agentdev/cases/case-*.md` の既存最大番号 + 1 を使用する。

欠番は再利用しない。

同一番号のファイルが既に存在する場合、ローカル版 `case-open` は停止する。

暫定ローカル版では、複数プロセスによる同時作成の排他制御は対象外とする。

## ローカルCaseファイル状態

`status` は以下のいずれかとする。

```text
open
running
blocked
review
closed
cancelled
```

## ローカルCaseファイル状態遷移

状態遷移は以下とする。

| 操作 | 変更前 | 変更後 |
|---|---|---|
| ローカル版 `case-open` | なし | `open` |
| ローカル版 `case-run` 開始 | `open` / `blocked` | `running` |
| ローカル版 `case-run` 完了 | `running` | `review` |
| ローカル版 `case-run` 停止 | `running` | `blocked` |
| ローカル版 `case-close` 停止 | `review` | `blocked` |
| ローカル版 `case-close` 再開 | `blocked` | `review` |
| ローカル版 `case-close` 完了 | `review` | `closed` |
| 明示中止 | `open` / `running` / `blocked` / `review` | `cancelled` |

`closed` と `cancelled` は終端状態とする。

ローカル版 `case-run` 停止後は、`blocked` → `running` → `review` の経路で再開する。

ローカル版 `case-close` 停止後は、`blocked` → `review` → `closed` の経路で再開する。

`blocked` から `closed` へ直接遷移しない。

## ローカルCaseファイル見出し

ローカルCaseファイルは以下の見出しを持つ。

```markdown
## 入力
## 背景
## 問題
## 目的
## 対象範囲
## 対象外
## 受け入れ条件
## 作業ログ
## マージ前確認
## SPEC確定候補
## Findings / Capture候補
### intake
### learning
## マージ結果
## 残課題
## 完了判定
```

`SPEC確定候補` と `Findings / Capture候補` は必須とする。これらは、GitHub 版で PR本文が担っていた引き継ぎ情報の代替である。

## GitHub Issue / PR 置換方針

ローカル版では、GitHub Issue / PR が担う情報を `.agentdev/cases/case-{NNNN}.md` に集約する。

| GitHub版 | ローカル版 |
|---|---|
| GitHub Issue本文 | ローカルCaseファイル本文 |
| GitHub Issueコメント | `## 作業ログ` |
| GitHub Issueの状態 | ローカルCaseファイルの `status` |
| GitHub Issueのラベル | ローカルCaseファイルの `labels` |
| GitHub PR本文 | `## マージ前確認`、`## SPEC確定候補`、`## Findings / Capture候補` |
| GitHub PR取り込み結果 | `## マージ結果` |
| GitHub Issue のクローズ | `status: closed` と `closed_at` |

ローカル版 `case-run` は PR を作成しない。

ローカル版 `case-run` は PR本文相当の情報をローカルCaseファイルへ追記する。

ローカル版 `case-close` は PR本文ではなく、ローカルCaseファイルから SPEC確定候補、intake 候補、learning 候補を読み取る。

構造的実行以降の唯一の情報源は、GitHub Issue / PR ではなくローカルCaseファイルである。

## ひな形変換方針

ローカル版生成では、GitHub Issue本文、GitHub PR本文、Issueコメント、完了報告に関係するひな形を変換対象に含める。

GitHub Issue本文向けのひな形は、ローカルCaseファイル本文向けに変換する。

GitHub PR本文向けのひな形は、ローカルCaseファイルの `## マージ前確認`、`## SPEC確定候補`、`## Findings / Capture候補` 向けに変換する。

Issueコメント向けのひな形は、ローカルCaseファイルの `## 作業ログ` 向けに変換する。

GitHub PR 取り込み後の完了報告向けひな形は、ローカル版完了報告または `## 完了判定` 向けに変換する。

ローカル版コマンド / スキルが参照するひな形は、既存の相対参照構造を維持して配置する。

既存の相対参照構造を維持できない場合は、参照元から解決できる配置へ変換し、その変更理由と変更内容を変換レポートに記録する。

## マージ結果の記録方針

ローカル版 `case-close` は GitHub PR 取り込みを実行しない。

ローカル版 `case-close` は、ローカルGit上で実施済みの取り込み・反映結果を `## マージ結果` に記録する。

`## マージ結果` には少なくとも以下を記録する。

- 実行した操作
- 関連するコミットハッシュ
- 実行日時
- 結果: `PASS` / `FAIL`

ブランチを使った場合は、以下も記録する。

- 取り込み先ブランチ
- 取り込み元ブランチ

ブランチを使わない場合は、ブランチ名の記録を必須としない。

ローカルGit上で実施済みの取り込み・反映結果が失敗または未完了の場合、ローカル版 `case-close` は `status` を `blocked` に更新し、理由を `## 残課題` に記録する。

YAML前書きにブランチ情報は持たせない。

## 変換プロンプト要件

`prompts/generate-local-transform.md` は、AI エージェントがローカル版コマンド / スキル / ひな形を生成先リポジトリの `.opencode/commands/` および `.opencode/skills/` に生成するためのプロンプトである。

このプロンプトは、仕様管理リポジトリの `src/opencode/` と `src/opencode-local/` 配下の仕様・定義を入力として使用する。

このプロンプトは、`src/opencode/` を変更しない。
