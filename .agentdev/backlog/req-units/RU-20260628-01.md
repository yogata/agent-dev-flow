# RU: 配布物内の導入先未解決参照を検出する docs-check ルール追加

## 背景

AgentDevFlow 配布対象の command 定義および skill 定義は、導入先プロジェクトで解決可能な参照のみを含む原則が既に定義されている。

チャット内確認により、対象原則は REQ-0103-079 / REQ-0103-080 / REQ-0103-081 および ADR-0104 の実行時独立性に基づくものとして整理された。

しかし現状では、配布対象 command / skill に含まれる導入先未解決参照を継続検出する仕組みが不十分である。特に、REQ/ADR ID 固定参照だけでなく、`src/opencode/`、`docs/specs/`、`docs/guides/`、`/repo/*`、`repo-*`、本体 docs への GitHub URL、line number 付き内部参照も同じ問題圏として扱う必要がある。

## 問題

配布物内に導入先プロジェクトで解決できない参照が残っても、作成時・レビュー時・docs-check 実行時に継続的に検出されない。

その結果、実行時配布物が本体リポジトリ内部文書や repo-local 要素に依存し、導入先プロジェクトで自己完結しない状態が発生する。

既存違反が多数存在するため、full audit を即 fail gate 化すると既存負債により通常作業が停止する。したがって、既存違反を baseline として扱い、新規増加を止める delta guard が必要である。

## Source Summary

チャット内で以下に合意した。

- 問題の主題は「REQ/ADR ID 残留」ではなく「配布物内の導入先未解決参照」である。
- 主担当は inspect-skills ではなく `/repo/docs-check` / integrity rule である。
- 理由は、対象が正規表現とパススコープで機械的に検出可能なためである。
- `check_integrity.ts` に `checkRuntimeUnresolvedReferences(root)` 相当の検出関数を追加する。
- 既存の `walkMarkdown` / `collectAgentdevSkillMarkdown` 相当の走査パターンを再利用する。
- `cli_utils.ts` の `STRICT_CHECKS` / `CHECK_TO_FINDING_CATEGORY` / route 連携に新検出ルールを接続する。
- 既存違反が多いため、full audit はまず report / intake 出力とし、delta guard で新規増加を fail 可能にする。
- 既存違反解消は別 RU として扱う。

## 統合理由

REQ/ADR ID、`src/opencode/`、`docs/specs/`、`/repo/*` などは表面的なパターンは異なるが、いずれも「導入先プロジェクトで解決不能な参照を配布物が保持する」という同一の配布境界違反である。

個別パターンごとに RU を分けると、検出責務・走査対象・baseline / delta guard 設計が重複するため、まず共通の検出ルールとして統合する。

## 要件化の方向

`/repo/docs-check` の integrity rule として、配布物内の導入先未解決参照を検出する機械的ルールを追加する。

ルール名は `runtime-unresolved-reference` 相当とする。

検出対象は以下とする。

- `src/opencode/commands/agentdev/**/*.md`
- `src/opencode/skills/agentdev-*/**/*.md`
- `src/opencode/skills/agentdev-*/references/**/*.md`
- `src/opencode/skills/agentdev-*/SKILL.md`

検出パターンは少なくとも以下を対象にする。

- `REQ-NNNN`
- `REQ-NNNN-NNN`
- `ADR-NNNN`
- `src/opencode/`
- `docs/specs/`
- `docs/guides/`
- `/repo/*`
- `repo-*`
- 本体 docs への GitHub URL
- line number 付き内部参照

severity はパターンごとに分ける。

- `REQ-*` / `ADR-*`: 原則違反として NG
- `src/opencode/`: 原則違反として NG
- `/repo/*` / `repo-*`: 原則違反として NG
- `docs/specs/` / `docs/guides/` / 本体 docs URL / line number 付き内部参照: 既存例外確認までは warning または configurable baseline 対象

継続検査は以下の段階で実現する。

1. full audit: `/repo/docs-check` で全件検出し、report / intake に出す
2. baseline: 既存違反数または既存違反一覧を固定する
3. delta guard: 変更差分で新規増加した導入先未解決参照を fail にする
4. baseline 0 到達後: full audit を fail gate 化できる状態にする

## 受け入れ条件

- `runtime-unresolved-reference` 相当の検出ルールが docs-check / integrity check に追加されている。
- 検出対象が `src/opencode/commands/agentdev/**/*.md` および `src/opencode/skills/agentdev-*/**/*.md` を含む。
- skill references 配下も検出対象に含まれている。
- `REQ-NNNN`、`REQ-NNNN-NNN`、`ADR-NNNN` を検出できる。
- `src/opencode/`、`/repo/*`、`repo-*` を検出できる。
- `docs/specs/`、`docs/guides/`、本体 docs URL、line number 付き内部参照の扱いが severity または baseline 方針として定義されている。
- `cli_utils.ts` の finding classification / route / exit code 連携に新ルールが接続されている。
- report に file / line / evidence / expected / route が出力される。
- regression test または fixture により、少なくとも REQ/ADR ID、`src/opencode/`、`/repo/*`、`repo-*` の検出が確認されている。
- 既存違反が存在する状態でも full audit 導入で通常作業を即停止させない baseline 方針が存在する。
- delta guard により、新規増加分を fail 可能にする方針が定義されている。
