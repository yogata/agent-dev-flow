# RU: command 本体の RFC2119 規範語を token 最適化観点で全面見直しする

## 背景
AgentDevFlow では RFC 2119 語の運用見直しが進み、REQ 要件行に RFC 2119 語の使用は必須でなくなった。加えて、禁止・停止条件・完了不可条件は RFC 2119 語ではなく自然文で保持する方針が追加された。

一方、runtime command 本体には `SHALL` / `MUST` / `SHOULD` / `MAY` が多数残存している。特に `case-run`、`case-close`、`req-save` など高頻度 command に残存件数が多く、token 消費と読解負荷の両面でコストになっている。

## 問題
command 本体に残る RFC 2119 規範語の多くは、実行時に機械解釈される強制条件ではなく、説明文上の強調に留まっている。これにより以下の問題がある。

- token を消費するが、実行挙動や停止条件の追加情報になっていない箇所が多い
- 「禁止」「停止」「完了扱いにしない」など自然文で書くべき条件と混在し、現行 REQ 方針と不整合が生じる
- command 本体の責務が「何をするか」より「規範語での再宣言」に寄り、可読性が下がる
- integrity 側に残る RFC2119 marker 検査と command 本体の語彙依存が、不要な保守コストを生む

## Source Summary
チャット上で以下を合意した。

- RFC 2119 語必須運用は廃止済みである
- `SHALL` 自体が実行上の強制力を持つとは言えない
- command 本体には `SHALL` が多数残っている
- token 最適化の観点から全面的な見直しが必要である

確認した一次情報では、現行 REQ は以下を示している。

- REQ 要件行に RFC 2119 語の使用は必須でない
- SHOULD/MAY 相当は REQ 要件行から整理対象である
- 禁止・停止・完了不可条件は自然文で保持する
- SPEC 側の `SHALL/MUST` は「現在有効な契約記述」に限定する

## 統合理由
RFC 2119 語必須廃止と token 最適化を別件として扱うのではなく、「runtime command 本体から規範語依存を外し、必要な強制は自然文と検証手順に寄せる」一つの要件として統合する方が整合的である。

これにより、
- command 本体の token 削減
- command の責務明確化
- REQ / SPEC / integrity の語彙ポリシー整合
を同時に達成できる。

## 要件化の方向
- runtime command 本体では RFC 2119 語を原則使用しない
- command 本体に残す条件文は、実行手順・停止条件・禁止事項・検証条件として自然文で記述する
- `SHALL/MUST/SHOULD/MAY` が単なる強調に留まる箇所は削除する
- `SHOULD` / `MAY` は command 本体から原則排除し、必要なら注記・guide・SPEC・REQ に責務分離する
- `MUST NOT` / `SHALL NOT` 相当の禁止事項は「〜してはならない」「禁止」として自然文へ置換する
- integrity の RFC 2119 marker 検査は、command 本体に対する語彙検査から、責務境界違反または残存移行漏れ検査へ再定義する
- 高頻度 command（case-run, case-close, req-save）を優先して削減対象とする

- command 本体に残る禁止・停止・完了不可条件が自然文で表現されている
- `case-run`、`case-close`、`req-save` の高頻度 command で、規範語削減による token 削減が確認できる

- REQ / SPEC / command / integrity 間で語彙ポリシーの矛盾が残らない
