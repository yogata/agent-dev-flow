## Sources

- 背景: Epic #1736（REQ-0155 関連）の Wave 1（PR #1745、Issue #1738）および Wave 2（PR #1746、Issue #1739）で、REQ 相互参照追加のみ、SPEC 内参照表記是正（ADR-0139 参照追記、相対パス是正）のみ等の軽微な変更に対し、`check_changed_docs.ts` が `spec_readme_update_required: true`、`requirements_readme_update_required: true`、`extensions_check_required: true` の各フラグを立てる誤検知が連続して発生。
- 問題: Wave 1 と Wave 2 で同一問題クラスの誤検知が再発し、毎 Wave の agent が誤検知判定と処理を強いられている。対象ファイル（SPEC 一覧表、extensions、REQ README 等）への影響がない変更でも、機械的差分ベースでフラグが立つため、QG-3 の信頼性が低下している。
- 根本原因: `check_changed_docs.ts` のフラグ判定が「変更ファイルパス・差分有無」ベースであり、変更の意味内容を考慮しない。Wave 1→Wave 2 で連続再発しており、対策しない限り継続再発する。
- ギャップ分類: fix gap。意味差分考慮ロジックの未実装が根本原因。

## 要件化の方向

req-define で要件を確定する。`check_changed_docs.ts` のフラグ判定ロジックを、機械的差分ベースから意味差分考慮ベースへ改善する。具体的実装候補（いずれか、または組み合わせ）:

1. **行レベル差分ベース判定への変更**: ファイル単位の差分有無ではなく、行レベルで意味のある変更（frontmatter 変更、新規エントリ追加、実質的な本文変更等）がある場合のみフラグを立てる。REQ 相互参照追記、SPEC 内参照表記是正等の軽微な行変更はフラグ対象外とする。
2. **README 更新必須判定を frontmatter 変更時に限定**: `requirements_readme_update_required` / `spec_readme_update_required` は、REQ/SPEC ファイルの frontmatter（id、title、status、depends_on 等）が変更された場合のみ true とする。本文中の相互参照追記や表記是正ではフラグを立てない。
3. **false positive 抑止フラグの追加**: PR 本文の frontmatter（例: `docs-check-skip: [spec_readme_update_required, extensions_check_required]`）または commit message trailer で、agent が明示的にフラグを抑止できる仕組みを追加。ただし濫用防止のため抑止理由の記述を必須とする。

制約:
- 後方互換性: 既存のフラグ判定に依存する PR（SPEC 一覧表の実質的更新を含む変更等）は、新ロジックでも正しくフラグが立つ必要がある。意味差分考慮の閾値設計が重要（過剰な抑止は真正なフラグまで消す）。
- テスト: 新ロジックには、誤検知抑止の回帰テスト（Wave 1/2 と同種の軽微変更でフラグが立たないこと）と、真正検知の維持テスト（実質的な README/ extensions 変更でフラグが立つこと）をペアで追加する。
- 運用移行: 新ロジック導入後、既存の誤検知運用（agent が毎回フラグを解釈する手順）は段階的に廃止する。

反映先候補:
- script（配布物）: `scripts/check_changed_docs.ts`（フラグ判定ロジック変更、必要に応じて false positive 抑止フラグ受付追加）
- command: `.opencode/commands/agentdev/docs-check.md`（QG-3 手順で新フラグ判定ロジックの解釈と FP 抑止フラグの運用を明記）
- command: `.opencode/commands/agentdev/case-close.md`（targeted docs guard、QG-3/QG-4 での新フラグ判定ロジックの取り扱いを更新）
- spec: `docs/specs/integrity/` 配下（該当 SPEC が既存の場合、意味差分考慮ロジックを追記。未整備の場合は新規起票を検討）

推奨Issue分類: fix（既存機能の不具合修正、誤検知の是正）。優先度は medium を想定。
