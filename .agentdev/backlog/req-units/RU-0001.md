# RU: case-auto 実行時の case-open / case-close 相当処理漏れ修正

## 背景

`/agentdev/case-auto` は、req-define 完了後の後続工程を自走実行する入口であり、既存コマンド定義（req-save / case-open / case-run / case-close）を authoritative source として読み込み、各コマンドの Steps / Guardrails / Error handling に従って実行する仕様である。

現行確認では、case-auto 実行時に以下が実施されていない。

- case-open 相当処理での RU ファイル削除
- case-close 相当処理での learning / intake 補足

## 問題

### case-open 相当処理で RU ファイルが削除されない

case-open には、Issue 作成 + VERIFY 成功後、要件docまたはセッション内要件docの `Requirement Source` から `.agentdev/backlog/req-units/RU-*.md` を抽出して削除する処理がある。

しかし case-auto の工程間引き継ぎは Issue番号・PR番号のみが明示されており、RU削除に必要な `Requirement Source` / RUパス / 一時メタ情報の引き継ぎが明記されていない。

feature 経路では req-save を挟むが、req-save は RU パスを docs 永続文書へ保存しない仕様であるため、case-auto が元の要件docに含まれる RU 情報を保持しないと、case-open 相当処理で RU を削除できない。

### case-close 相当処理で learning / intake 補足が漏れる

case-close には、学びの検知・抽出、`.agentdev/learning/inbox.md` への追記、Post-run intake capture、learning と intake の分離、`.agentdev/` 配下の commit/push が定義されている。

しかし `docs/specs/workflow-contracts.md` の参照フローでは、case-close は learning / intake の両方を WRITE する一方、case-auto は learning のみ WRITE で intake が対象外のように記載されている。

case-auto は case-close 相当処理を実行する仕様であるため、learning だけでなく intake も case-close と同条件で扱う必要がある。

## Source Summary

本RUは、チャット内で確認した以下の一次情報に基づく。

- `case-auto.md`: case-auto は既存コマンド定義を authoritative source として読み、req-save / case-open / case-run / case-close の相当処理を順次実行する。
- `case-open.md`: Issue 作成 + VERIFY 成功後、Requirement Source から RU パスを抽出して RU ファイルを削除する。
- `req-save.md`: RU パスは docs 永続文書へ保存せず、RU 削除は case-open 成功後に実行する。
- `case-close.md`: learning capture / Post-run intake capture / `.agentdev/` commit-push を実行する。
- `workflow-contracts.md`: case-auto の intake 欄が `—` となっており、case-close 相当処理との整合が取れていない。

## 統合理由

RU削除漏れと learning/intake 補足漏れは、どちらも case-auto が既存コマンド相当処理を実行する際の工程間状態引き継ぎ不足に起因する。

別々の不具合ではなく、case-auto orchestration が case-open / case-close の後段処理に必要な入力・状態を保持しきれていない同一領域の問題であるため、1つのRUとして扱う。

## 要件化の方向

case-auto は、case-open / case-close 相当処理を省略・簡略化せず、既存コマンド定義に従って実行することを明確化する。

特に、工程間の状態引き継ぎとして、Issue番号・PR番号だけでなく、case-open / case-close の後続処理に必要な一時メタ情報を保持する。

## 要件

### case-open 相当処理

- case-auto は、入力要件docまたはセッション内要件docに含まれる `Requirement Source` を case-open 相当処理まで保持すること。
- case-auto の case-open 相当処理は、Issue 作成 + VERIFY 成功後、case-open.md と同一条件で `.agentdev/backlog/req-units/RU-*.md` を削除すること。
- Issue 作成または VERIFY に失敗した場合、RU ファイルを削除しないこと。
- `Requirement Source` に RU 以外の source が含まれる場合、RU パターンに一致しない source は削除しないこと。
- RU 削除後は、case-open.md と同一条件で同期確認を行うこと。

### case-close 相当処理

- case-auto の case-close 相当処理は、case-close.md の learning capture を実行すること。
- case-auto の case-close 相当処理は、case-close.md の Post-run intake capture を実行すること。
- case-auto は learning と intake を混合せず、別々の成果物として扱うこと。
- case-auto は learning / intake の変更を `.agentdev/` 配下の domain state として commit/push すること。
- `.agentdev` push に失敗した場合、case-close.md と同様に完了扱いにしないこと。

### workflow-contracts 修正

- `docs/specs/workflow-contracts.md` の参照フローにおける `/agentdev/case-auto` の intake 欄を、case-close と整合するように `WRITE（capture）` へ修正すること。
- case-auto の参照フローは、case-auto が case-close 相当処理を含むことと矛盾しない内容にすること。

## 対象外

- case-open 単体実行時の RU 削除仕様の変更
- case-close 単体実行時の learning / intake capture 仕様の変更
- req-save が docs 永続文書に RU パスを保存する変更
- promoted artifact を req-define で直接読む変更
- backlog-review / backlog-save の責務変更
- case-auto に case-open / case-close の詳細手順を重複実装する変更

## 受け入れ条件

- req-define で RU ファイルを参照した後、case-auto を実行すると、Issue 作成 + VERIFY 成功後に該当 RU ファイルが削除される。
- Issue 作成または VERIFY に失敗した場合、該当 RU ファイルは残置される。
- RU 削除後、main 作業ディレクトリの HEAD と origin/main の一致、および `git status --porcelain` に削除済み RU が残っていないことが確認される。
- case-auto 経由の case-close 相当処理で、learning capture が実行される。
- case-auto 経由の case-close 相当処理で、Post-run intake capture が実行される。
- learning と intake は別成果物として保存される。
- learning / intake の `.agentdev/` 変更が commit/push される。
- `docs/specs/workflow-contracts.md` の case-auto 行で intake が `WRITE（capture）` として表現される。
- case-auto 定義内に case-open / case-close の詳細手順を重複実装しない。
