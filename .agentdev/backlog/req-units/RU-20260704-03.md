# .agentdev artifact lifecycle の処理済み削除と living pool 明文化

## 背景

`.agentdev/intake/`、`.agentdev/inspect/`、`.agentdev/learning/` は、未処理成果物、採用済み成果物、保留・再評価対象を扱う lifecycle 領域である。

チャット上で、reject 確定済み成果物、backlog-review で RU 化に成功した promoted 成果物、learning の staged / rejected / duplicate entry は、処理済みとして削除または prune する方針に合意した。

一方で、defer / 保留 / 未処理 / 再評価対象、RU 化失敗、矛盾除外、未判断の成果物は、後続判断のために保持する。

## 問題

現行 lifecycle の説明には、処理済み成果物を保持するように読める記述、または過去配置を補足する記述が残っている。

この状態では、読み手が以下を判別しにくい。

- reject 確定済み成果物を削除するのか、保持するのか
- promoted 成果物を RU 化成功後に削除するのか、保持するのか
- learning の保留・未処理・再評価対象をどこに保持するのか
- 現行の正規 lifecycle と履歴的な説明のどちらを基準にすべきか

## Source Summary

チャット上で以下に合意した。

- REQ 保存先は新規 REQ CREATE ではなく、REQ-0127 と REQ-0128 への APPEND とする
- REQ-0124 と REQ-0129 は要件行としては既に整合しており、新規要件追加は不要とする
- REQ-0124 は active REQ であるため、現行 lifecycle と無関係な対象外記述は削除対象とする
- ADR は新規作成しない
- intake の reject item は分類確定後に削除する
- learning の staged / rejected / duplicate entry は判定確定時に `deferred.md` から prune する
- `deferred.md` は deferred / 未処理 / 再評価対象の living pool として扱う
- backlog-review で RU 化に成功した promoted 成果物は削除する
- RU 化失敗、矛盾除外、未判断の promoted 成果物は残置する
- inspect の reject finding 削除、defer finding の `inbox/` 残置、promoted の RU 化対象化は既存 REQ で定義済みと扱う
- 歴史的文書である ADR-0113、retired REQ、retired ADR の履歴参照は保持する
- active REQ / SPEC / command / skill / guide / README は、正規 lifecycle と処理済み削除ルールだけを現行説明として記述する
- 存在しないものを否定する要件や、過去配置の廃止経緯を現行 lifecycle 要件として書かない

## 統合理由

本 RU は、`.agentdev/{intake,inspect,learning}/` の全体方針を新規 REQ として作るものではない。

最終判断では、既存ドメイン REQ が既に大半の lifecycle 契約を持っているため、不足する intake / learning の処理済み削除境界だけを既存 REQ に APPEND するのが正しい。

また、active 文書に残る履歴的な説明は、要件として否定形で保存するのではなく、現行説明を正規 lifecycle と処理済み削除ルールへ整理する作業として扱う。

## 要件化の方向

REQ-0127 には、`intake-promote` で reject 判定された item の元 inbox file は、分類確定後に削除することを APPEND する。

REQ-0128 には、`learning-promote` で staged / rejected / duplicate 判定された entry は、判定確定時に `deferred.md` から prune することを APPEND する。`deferred.md` は deferred / 未処理 / 再評価対象の living pool として保持する。

REQ-0124 と REQ-0129 には新規要件行を追加しない。REQ-0124 については、active REQ 内に残る現行 lifecycle と無関係な対象外記述を削除する。REQ-0129 は、RU 化成功時の promoted 削除、RU 化失敗・矛盾時の promoted 残置が既に定義済みであるため、変更不要とする。

SPEC / command / skill / guide / README は、現行 lifecycle 説明として、正規配置、処理済み削除、保留・未処理・再評価対象の保持先だけを記述する。履歴文書はこの整理対象に含めない。
