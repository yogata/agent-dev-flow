# .agentdev lifecycle archive directory の廃止と処理済み成果物削除ルールの統一

## 背景

`.agentdev/intake/`、`.agentdev/inspect/`、`.agentdev/learning/` は、未処理成果物、採用済み成果物、保留・再評価対象を扱う lifecycle 領域である。

チャット上で、reject 済み成果物や backlog-review 済み成果物は削除する方針に合意した。さらに、`.agentdev/learning/archive/active.md` の配置見直しを契機に、`learning` 固有ではなく `.agentdev/{intake,inspect,learning}/archive/` 全体に同型の問題があることを確認した。

この方針の本質は、`archive/` を終端保管先として残すことではなく、active queue、promoted staging、deferred living pool、処理済み削除の lifecycle に統一することである。

## 問題

`.agentdev/{intake,inspect,learning}/archive/` が残ると、処理済み成果物を削除する方針と衝突する。

特に、`archive/promoted/`、`archive/rejected/`、`archive/active.md` のような配置や、それらの廃止経緯説明が残ると、以下の問題が発生する。

- reject 済み成果物を削除するのか、終端保管するのかが曖昧になる
- promoted artifact を backlog-review 後に削除するのか、別ディレクトリに移すのかが曖昧になる
- learning の living pool が `deferred.md` なのか `archive/active.md` なのか揺れる
- lifecycle の正規配置より、過去の archive 配置の説明が読み手の判断を妨げる
- `.agentdev/` 配下に処理済み artifact が滞留し、再検出・再処理・確認ノイズの原因になる

## Source Summary

チャット上で以下に合意した。

- reject 済み成果物は削除する
- backlog-review で RU 化に成功した promoted artifact は削除する
- defer / 保留 / 未処理 / 再評価対象は残す
- learning の living pool は `.agentdev/learning/deferred.md` に統一する
- `.agentdev/learning/archive/` は削除する
- `.agentdev/learning/archive/active.md` は廃止する
- `.agentdev/learning/archive.md` は採用しない
- `archive/` に関する記述は残さない
- intake / inspect の `archive/` も同型問題として扱う
- 「archive/ を参照しない」「archive/ を前提にしない」のような否定形の要件にはしない
- ユーザーの一点指摘は局所修正ではなく、同型問題と横断影響を確認して整理する

## 統合理由

本 RU は、`learning` の living pool 配置だけではなく、`.agentdev/{intake,inspect,learning}/` 全体の lifecycle 配置方針を統一する必要がある。

`learning` だけを対象にすると、`intake/archive/` や `inspect/archive/` に残る同型の archive 記述が未整理となる。これは、reject 済み成果物や backlog-review 済み成果物を削除する合意と不整合である。

そのため、本 RU は `.agentdev` lifecycle 全体における archive directory 廃止、処理済み成果物削除、living pool の正規配置を一体として扱う。

## 要件化の方向

`.agentdev/{intake,inspect,learning}/` の lifecycle は、archive directory を終端保管先として使わない構造に統一する。

`intake` は、未処理 item を `.agentdev/intake/inbox/`、採用済み item を `.agentdev/intake/promoted/` に置く。採用 item の元 inbox file は promoted artifact 保存後に削除する。却下 item の元 inbox file は分類確定後に削除する。

`inspect` は、未処理 finding を `.agentdev/inspect/inbox/`、採用済み finding を `.agentdev/inspect/promoted/` に置く。採用 finding の元 inbox file は promoted artifact 保存後に削除する。却下 finding の元 inbox file は分類確定後に削除する。defer された finding は `.agentdev/inspect/inbox/` に残す。

`learning` は、未整理 entry を `.agentdev/learning/inbox.md`、採用済み成果物を `.agentdev/learning/promoted/`、保留・未処理・再評価対象を `.agentdev/learning/deferred.md` に置く。`staged / rejected / duplicate` は `learning-promote` の判定確定時に `deferred.md` から prune する。`deferred / 未処理 / 再評価対象` は保持する。

`backlog-review` は、RU 化に成功した `.agentdev/{intake,inspect,learning}/promoted/` 配下の採用済み成果物を削除する。RU 化に失敗した成果物、矛盾により除外された成果物、未判断の成果物は削除しない。

`archive/` に関する過去配置、移動先、廃止済み経緯説明は、現行 lifecycle の説明から削除する。正規配置と削除ルールだけを記述する。

- `src/opencode/commands/agentdev/learning-promote.md`

## 対象外

- intake item の抽出ロジック変更
- inspect-docs の診断ロジック変更
- learning の 8軸評価ロジック変更
- promote / defer / reject / duplicate の判定基準変更
- backlog-review の RU 統合・分割判定ロジック変更
- 新規 ledger、履歴ファイル、終端アーカイブの追加
- GitHub Issue / PR 作成ルールの変更
- `archive/` を参照しないこと、または前提にしないことを独立した否定要件として定義すること

## 受け入れ条件

- `.agentdev/intake/` の正規 lifecycle が `inbox/`、`promoted/`、処理済み削除で説明されている
- `.agentdev/inspect/` の正規 lifecycle が `inbox/`、`promoted/`、処理済み削除、defer 残置で説明されている
- `.agentdev/learning/` の正規 lifecycle が `inbox.md`、`promoted/`、`deferred.md` で説明されている
- `.agentdev/learning/deferred.md` が deferred / 未処理 / 再評価対象を保持する living pool として説明されている
- intake の reject 済み item は分類確定後に削除される
- inspect の reject 済み finding は分類確定後に削除される
- learning の `staged / rejected / duplicate` は判定確定時に `deferred.md` から prune される
- backlog-review で RU 化に成功した promoted artifact は削除される
- RU 化失敗、矛盾除外、defer、未処理、未判断の成果物は削除されない
- `archive/` に関する過去配置、移動先、廃止済み経緯説明が現行 lifecycle 説明から削除されている
- 変更後の記述で、処理済み成果物削除と終端アーカイブ保管が混同されていない
