# 受け入れ条件中心のcommand・skill責務是正

## 背景

AgentDevFlowでは、commandを公開入口と高位の実行契約に限定し、再利用可能な判断をskill、詳細情報をreference、決定的処理をscriptへ配置する設計を採用している。

チャットでは、最近のモデルに対して結果へ影響しない内部手順を細かく固定するより、公開成果物、安全境界、承認境界、停止条件、受け入れ条件を明確にし、実現手段に裁量を与える方針を採用することに合意した。

この方針をAgentDevFlowの全command、全skill、関連REQ、SPEC、reference、script、integrity ruleへ一貫して適用する。

適用対象の現状は、GitHub上の最新HEAD `612c09a726ff0abdc55efd5b37fbc1b86be9c99f`を基準に確認した。

## 問題

既存設計は責務分離の方向性を持っているが、実体には次の問題が残っている。

- commandに、詳細手順、探索順、個別分岐、ツール引数、待機条件、Step番号依存が残っている
- commandから他skillの内部reference、内部見出し、内部Stepを直接参照している
- command SPECがcommandのStep番号構成を複製している
- REQに、Step番号、ツール引数、待機間隔、reference内見出しなどの実装詳細が含まれている
- command、skill、REQ、SPEC間で同じ契約や手順が重複している
- SPEC操作とREQ操作の所有境界が`agentdev-req-file-manager`へ集中している
- `inspect-docs`の横断診断を一次所有する専用skillが存在しない
- skill件数などの固定値が現在の構成と一致していない
- ハーネス責務として分離された実行制御パラメータが、配布commandまたはskillへ残っている箇所がある

行数やtoken数は問題検出の補助指標になるが、短縮そのものを目的または合格条件にすると、安全条件や必要な契約を失うおそれがある。

## 統合理由

command、skill、reference、script、REQ、SPEC、integrity ruleは、同じ実行契約を異なる責務で表現している。

commandだけを短縮すると、REQに残る実装詳細、SPECによるStep複製、skillへの責務集中、referenceへの直接依存が残る。後続変更で詳細手順が再びcommandへ流入する可能性もある。

このため、全command、全skill、関連REQ、SPEC、authoring規約、integrity checkerを一つの変更関心として扱う。

ただし、変更は一括移送を意味しない。各記述について正規な所有者を特定し、公開契約と安全性を維持できることを確認したものだけを移す。

## 要件化の方向

### command

commandは次の内容を所有する。

- 公開目的
- 入力
- 成果物
- 許可される副作用
- 安全境界
- 承認境界
- 停止条件
- 後続commandとの接続契約
- 順序を変えると成果物または安全性が変わる必須順序
- 利用するskillの名前と委譲する責務

commandから次の内容を除く。

- 結果へ影響しない内部手順
- 探索順
- 詳細な分類表
- 重複する例
- ツール固有の引数
- 他skillの内部referenceパス
- 他skillの内部Step、Section、Phase、見出し名
- ハーネス固有の起動方法、待機時間、並列度、再試行条件

### skillとreference

skillは、同一の判断モデルと責任境界を共有する再利用可能な判断および操作契約を所有する。

SKILL.mdは、使用条件、対象外、入力、出力、副作用、主要な不変条件、必要なreferenceの選択条件を中心にする。

詳細な判定表、スキーマ、例、失敗時手順は、必要な場合に限り当該skill自身のreferenceへ配置する。

異なる判断モデル、入力、出力、責任境界を持つ内容は、単なるreference分割ではなくskill分割候補として扱う。

### script

採番、構文解析、見出し検索、整合性検査など、同じ入力に対して同じ結果を返せる処理はscriptが所有する。

scriptは操作責務を持つskillの配下に配置する。同じscriptを複数skillへ複製せず、正規な所有者を一つに定める。

SPEC固有の処理は`agentdev-spec-file-manager`、REQおよびADR固有の処理はそれぞれの操作skillが所有する。複数文書種別で利用する共通scriptは、責務台帳で正規な所有者と利用境界を確定する。

### REQ

REQは次の内容へ限定する。

- 外部から観測できる振る舞い
- 公開成果物
- ドメイン状態
- 安全境界
- 承認境界
- 停止条件
- 後続工程との安定した接続契約
- ハーネス非依存などの恒久的制約

Step番号、ツール引数、待機間隔、探索順、reference内見出し、scriptの詳細な入出力形式は、安定した外部契約でない限りSPEC、skill、referenceへ移す。

### SPEC

SPECは、現在の振る舞い、スキーマ、判定規則、状態遷移、実装上必要な順序とパラメータを所有する。

command SPECはcommandのStep番号を複製せず、成果物、副作用、停止状態、必須順序を対応付ける。

### ハーネス境界

REQ-0162、ADR-0136、REQ-0103-163を回帰基準とする。

実行エージェントの選定、起動方法、待機時間、並列度、再試行、実行基盤固有の救済処理はハーネス側文書が所有する。

配布command、skill、reference、docsは業務ワークフロー契約と抽象的な委譲結果だけを保持する。
