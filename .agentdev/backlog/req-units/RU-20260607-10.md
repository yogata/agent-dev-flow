# RU: ADR responsibility boundary cleanup

## 背景

ADR群に、技術的な設計判断ではなく、REQ/SPECに属する仕様変更・command動作・artifact contract・作業指示が混入している。

ADR-0017では、ADRは「取り返しのつかない技術判断の記録」、REQは「変更後に満たすべき振る舞い・制約・状態」、SPECは「現在仕様の基準」として整理されている。

この責務境界に基づくと、既存ADRの一部はADRとして扱うべきではない。

## 問題

ADR-0023は、`backlog-save` 廃止、`backlog-review` へのRU生成統合、review draft廃止、RU frontmatter `depends_on` 追加、`learning-promote` 後の直行禁止などを扱っている。

これらは技術的な選択ではなく、workflow仕様、command仕様、artifact仕様の変更である。

同様に、ADR-0022、ADR-0021、ADR-0012、ADR-0010、ADR-0003にも、ADRではなくREQ/SPECとして扱うべき内容が含まれている。

このまま accepted/proposed ADRとして残すと、仕様変更をADR化する前例になり、ADRの責務が曖昧になる。

## Source Summary

チャット内で以下を合意した。

- ADR-0023はADRではない。
- 仕様変更・仕様言及だけで技術的な決定を含まない文書は、新規ADRとして作成しない。
- ADR-0023の内容はREQ/SPECに移管する。
- 同種の再編成対象として、ADR-0022、ADR-0021、ADR-0012、ADR-0010、ADR-0003を扱う。
- ADRとして成立する文書は、技術判断・代替案・影響に圧縮し、REQ/SPEC相当の詳細を分離する。
- 既存の deprecated / superseded ADRは、原則として歴史的参照として維持する。
- 新規ADRは作らない。

## 統合理由

対象ADRは個別の表現差はあるが、根本問題は同じである。

ADR本文に以下が混入している。

- commandの動作仕様
- artifact schema / frontmatter仕様
- workflow lifecycle
- path配置・namespaceの現在仕様
- integrity-check検査条件
- 実施タスク・更新指示

これらを個別RUに分けると、ADR責務境界の統一判断が分散し、再編成結果が不整合になりやすい。

そのため、ADR責務境界の是正を1つのRequirement Unitとして扱う。

## 要件化の方向

ADRを以下の3区分に再編成する。

1. ADRとして残す  
   技術判断が存在するADRは残す。ただし、REQ/SPEC相当の詳細は本文から分離する。

2. deprecated化する  
   技術判断ではなく仕様変更のみのADRは、現行根拠から外す。本文は「ADRとしての誤分類」「REQ/SPECへ移管済みまたは移管対象」を示す内容へ圧縮する。

3. 既存 deprecated / superseded を維持する  
   既に deprecated / superseded のADRは、原則として本文を再編集しない。README分類や参照不整合がある場合のみ修正する。

新規ADRは作成しない。

## 受け入れ条件

- ADR-0023が現行ADR根拠として扱われない。
- ADR-0023の仕様変更内容がREQ/SPECへ移管される、または移管対象として明示される。
- ADR-0022、ADR-0021、ADR-0012、ADR-0010、ADR-0003について、ADRとして残すか、deprecated化するか、REQ/SPECへ移管するかが明確になる。
- ADRとして残す文書は、技術判断、代替案、採否理由、影響を中心に構成される。
- command仕様、artifact schema、workflow lifecycle、path配置、検査条件、作業指示がADR本文の主内容として残らない。
- `docs/adr/README.md` の一覧、Status View、Topic Viewが各ADR本文のstatusと矛盾しない。
- 既存 deprecated / superseded ADRを現行仕様として参照しない。
- 新規ADRが作成されない。
- 移管後も、REQ/SPEC側で必要な仕様情報が欠落しない。
