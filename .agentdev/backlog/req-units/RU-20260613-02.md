# RU: active REQ 本文からの見え消し除去方針

## 背景

active REQ は現行要件の一次情報源として参照される。現行要件を明確に読むことが主目的であり、履歴や過渡状態の解釈を本文読解に持ち込まないことが望ましい。

本セッションでは、REQ ファイルに残っている見え消し記述は、現行要件か履歴かの文脈判断を読み手に要求するため、削除してよいのではないかという問題提起があった。あわせて、履歴が必要なら別の場所で管理すべきであり、active REQ 本文に見え消しを残す必要は薄いという認識が共有された。

## 問題

active REQ 本文に見え消しが残ると、以下の問題が生じる。

- 現行要件と履歴情報が同じ本文層に混在する
- 読み手に「この行は今も有効か」の文脈判断を要求する
- active REQ を現行 SSoT として読む際の明瞭性が下がる
- 後続の要件整理や `req-define` 入力整理でノイズになる

## Source Summary

本セッションで合意した内容は以下の通り。

- active REQ 本文の見え消しは、原則として削除してよい
- 見え消しを残すと、現行要件か履歴かの判断を読み手に委ねることになり、一次情報源として不適
- 履歴が必要な場合は、active REQ 本文ではなく `Update Notes` 等の履歴管理箇所へ移すべき
- 現行判断に必要な境界がある場合でも、見え消しで残すのではなく、現行文として再記述すべき
- したがって、active REQ 本文では「見え消しで履歴を残す」運用は採らない方向が適切である

## 統合理由

本RUでは、見え消しを個別に削る話ではなく、active REQ 本文における見え消しの扱いを統一方針として整理する。

理由は以下の通り。

- 個別対応にすると、ある REQ では削除、別の REQ では残置となり運用がぶれる
- 見え消し残置の可否を本文読解時に判断させる設計自体が、active REQ の役割と相性が悪い
- 履歴の必要性そのものは否定せず、履歴の置き場を本文外に分離することで、現行要件の明確性を維持できる

## 要件化の方向

- active REQ 本文には見え消し記述を残さない
- 廃止、更新、移行の履歴を残す必要がある場合は、本文ではなく `Update Notes` 等の履歴管理箇所に移す
- 現行判断に必要な制約、例外、境界は、見え消しではなく現行文として記述する
- 見え消しが単なる履歴保持である場合は削除対象とする
- active REQ は現行要件の一次情報源として読める状態を優先する

## 主対象REQまたは変更対象候補

- 主対象: `docs/requirements/REQ-0101.md` から `docs/requirements/REQ-0121.md` までの active REQ 全体
- 変更対象候補:
  - active REQ 本文中に残っている見え消し要件行
  - active REQ 本文中の見え消し見出し、見え消し注記
- 履歴移管先候補:
  - `Update Notes`
  - 必要に応じて retired 側または後継要件への参照

## 対象外

- retired REQ の履歴表現方式
- 実装計画、編集手順、差分案の具体化
- `docs-check` 実装や自動検出ルール追加の具体設計
- 見え消し以外の冗長表現削減全般
- 履歴自体の削除を目的とした整理

## 受け入れ条件

- active REQ 本文に見え消しを残さない方針が明記されている
- 見え消しを削除する理由として、現行要件と履歴の混在回避が示されている
- 履歴が必要な場合の移管先として `Update Notes` 等が示されている
- 現行判断に必要な内容は、見え消しではなく現行文として再記述する方針が示されている
- 実装手順や未合意の個別差分を含めていない
