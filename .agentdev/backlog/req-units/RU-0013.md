# RU: 完了報告フォーマットの責務分離・重複排除・variant最適化

## 背景

`/agentdev/case-open` の完了報告で、以下の2行が同じ事実を重複して表現していた。

- `RU-0012.md を削除・永続化（commit/push完了）`
- `git 永続化: RU-0012.md削除 commit/push済み（HEAD = origin/main 同期確認OK）`

現行の完了報告フォーマットは、共通必須フィールドとして `完了コマンド`、`対象`、`結果`、`検証結果`、`git 永続化`、`次のコマンド` を要求している。

一方で、各variantでは `結果` 欄にgit永続化の詳細を書ける構造が残っており、`git 永続化` 欄との責務境界が曖昧になっている。

## 問題

完了報告フォーマットに以下の問題がある。

1. `結果` 欄と `git 永続化` 欄で同じ事実を重複して書ける。
2. `case-open/standard.md` と `case-open/epic.md` が `git 永続化: 該当なし` 固定であり、RU削除後のcommit/pushを表現できない。
3. `case-close` の部分失敗variantで、`.agentdev 永続化` の詳細と `git 永続化` が重複している。
4. `req-define` の `bug-or-lightweight`、`maintenance`、`docs-or-chore` が同一内容で、variant分割の意味が薄い。
5. `agentdev-workflow-reporting/SKILL.md` のvariant数説明とregistry実体にドリフトがある。

## Source Summary

- `completion-reports.md` は全agentdevコマンドの完了報告で共通6フィールドを必須としている。
- `case-open.md` はRU削除後にcommit/pushした場合、HEADとorigin/mainの同期確認を要求している。
- `case-open/standard.md` と `case-open/epic.md` は現状 `git 永続化: 該当なし` 固定である。
- `case-close/agentdev-push-failed.md` は結果欄に `.agentdev 永続化` のcommit/push詳細を書き、さらに `git 永続化` 欄にも同じ情報を書く構造になっている。
- `req-define/bug-or-lightweight.md`、`req-define/maintenance.md`、`req-define/docs-or-chore.md` は同一内容である。
- `agentdev-workflow-reporting/SKILL.md` では `backlog-review` / `backlog-save` が1 variantと説明されているが、registryでは各3 variantが定義されている。

## 統合理由

本件は個別コマンドの表記ゆれではなく、完了報告フォーマット全体の責務分離不備である。

`結果`、`検証結果`、`git 永続化` の境界を明確化しない限り、他コマンドでも同様の重複が再発する。

したがって、個別variantの文言修正ではなく、完了報告フォーマット全体の最適化として扱う。

## 要件化の方向

完了報告フォーマットを以下の責務で再定義する。

| フィールド | 責務 |
|---|---|
| `結果` | ユーザー視点・ドメイン視点の成果 |
| `検証結果` | 成果物・書き込み操作の検証結果 |
| `git 永続化` | commit / push / HEAD同期確認などgit操作結果 |
| `次のコマンド` | 具体的な後続コマンドまたは `なし` |

`結果` 欄ではcommit/push/HEAD同期確認を書かない。
`git 永続化` 欄ではIssue作成、コメント追加、RU削除などの業務成果を主語にしない。

## 要件

### RU-0013-001: 結果欄の責務分離

完了報告の `結果` 欄は、ユーザー視点・ドメイン視点の成果のみを記載すること。

含める例:

- Issue作成
- PR作成
- REQ/ADR保存
- intake item保存
- promoted artifact生成
- RU生成
- RU削除
- コメント追加
- worktree削除
- Findings回収

含めない例:

- commit hash
- push成否
- HEAD同期確認
- git status
- git diff

### RU-0013-002: git永続化欄の責務分離

完了報告の `git 永続化` 欄は、git操作結果のみを記載すること。

記載形式は以下を基本とする。

- `該当なし`
- `変更なし（commit/push スキップ）`
- `✅ OK（commit {hash}, push 済み）`
- `✅ OK（commit {hash}, push 済み, HEAD = origin/main 同期確認OK）`
- `⚠️ 注意（commit {hash}, push 失敗）`
- `❌ NG（commit失敗: {reason}）`

### RU-0013-003: 結果欄とgit永続化欄の重複禁止

同じ事実を `結果` 欄と `git 永続化` 欄の両方に記載してはならない。

NG例:

- `結果: RU-0012.md を削除・永続化（commit/push完了）`
- `git 永続化: RU-0012.md削除 commit/push済み`

OK例:

- `結果: RU-0012.md を削除`
- `git 永続化: ✅ OK（RU-0012.md削除をcommit/push済み。HEAD = origin/main 同期確認OK）`

### RU-0013-004: case-open variantのgit永続化可変化

`case-open/standard.md` と `case-open/epic.md` は、`git 永続化: 該当なし` 固定を廃止すること。

RU削除が発生しない場合は `該当なし` とする。
RU削除が発生しcommit/pushした場合は、commit/push結果およびHEAD同期確認結果を `git 永続化` 欄に記載する。

### RU-0013-005: case-close部分失敗variantの重複排除

`case-close/agentdev-push-failed.md` と `case-close/worktree-cleanup-failed.md` は、`結果` 欄に `.agentdev 永続化` のcommit/push詳細を記載しないこと。

git操作結果は `git 永続化` 欄に集約する。

### RU-0013-006: req-define重複variantの統合

`req-define/bug-or-lightweight.md`、`req-define/maintenance.md`、`req-define/docs-or-chore.md` は、同一内容であるため統合対象とする。

統合先候補:

- `req-define/simple.md`

ただし、後続コマンドや出力内容にPattern別差分を持たせる必要がある場合は、統合せず差分を明示する。

### RU-0013-007: registry説明のドリフト修正

`agentdev-workflow-reporting/SKILL.md` の対象コマンド表は、実際のvariant registryと一致させること。

少なくとも以下を修正する。

- `backlog-review` は1 variantではなく3 variant
- `backlog-save` は1 variantではなく3 variant

### RU-0013-008: integrity-check検査観点への追加

`integrity-check` または完了報告フォーマット検査観点に、以下の検査を追加すること。

- `結果` 欄にcommit/push/HEAD同期確認が含まれていないこと
- `git 永続化` 欄が存在する場合、git操作結果に限定されていること
- 同一のgit永続化事実が `結果` 欄と `git 永続化` 欄で重複していないこと
- `git 永続化: 該当なし` 固定variantが、実行時にgit永続化を発生させるコマンドに使われていないこと
