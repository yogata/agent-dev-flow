# RU: 利用者向けワークフロー説明と `.agentdev/` 状態説明の再整理

## 背景

AgentDevFlow のドキュメントは、`req-*` / `case-*` に加えて `intake-*`、`learning-*`、`backlog-*`、`integrity-check`、`req-restructure-review` が追加されたことで、利用者が次の判断をしづらくなっている。

- いま持っている入力から、どのコマンドを使えばよいか分かりにくい
- `intake`、`learning`、`backlog`、`RU` の境界が読み取りにくい
- `.agentdev/` 配下の各ファイルが「未処理」「処理済み」「次工程待ち」「履歴」のどの状態なのか分かりにくい
- `.agentdev/` と `.sisyphus/` の違いは説明されているが、現行REQ再基準化後の参照に更新されていない箇所がある
- README、command README、workflow-overview、artifact-model、domain-state-lifecycle、`.agentdev/README.md` の説明が重複し、どれを入口として読めばよいか分かりにくい

ユーザーとの見直しで、説明軸を「本流 / 支流」のような上下関係に置くのではなく、**入力の成熟度** と **成果物の状態** に置く方針で合意した。

## 問題

現行の利用者向けドキュメントは、概念説明やコマンド系統の説明が先行しており、利用者が知りたい以下の問いに直接答えにくい。

1. いま自分が持っている材料は何か
2. その材料に対して次に実行するコマンドは何か
3. コマンド実行後に生成された `.agentdev/` 配下のファイルはどの状態か
4. そのファイルは次にどのコマンドで読まれるか
5. そのファイルはいつ削除・移動・保持されるか

また、`.agentdev/README.md` には旧 `REQ-0017`、旧 `R
EQ-0039`、`req-backlog` 参照が残っており、現行 active REQ 前提とずれている。root README には `reference/command-map.md` への obsolete path 参照が残っている。

## Source Summary

チャットで検討した結果、次の方針に整理した。

- `req/case` を「本流」と呼ぶ説明に寄せない
- `intake/learning/backlog` を「支流」や「供給路」としてではなく、入力・成果物状態の違いとして説明する
- 新規の `start-here.md` や `use-cases.md` をすぐ追加するのではなく、まず既存ドキュメントの責務を整理し重複を減らす
- 利用者向けには「入力の状態 → 次コマンド → 生成物 → 次工程」を先に見せる
- `.agentdev/README.md` は概念説明ではなく、domain state の状態表に寄せる
- `artifact-model.md` は利用者入口ではなく、保守者向け・内部成果物モデルとして位置づけを下げる
- `.opencode/commands/agentdev/README.md` は概念説明ではなく、コマンド一覧・入出力・次コマンドのリファレンスに寄せる

## 統合理由

このRUは、複数ドキュメントの文言修正ではなく、利用者向け説明の責務再配置である。

対象は README、workflow guide、domain state README、command README、DOC-MAP にまたがるが、いずれも同じ問題、すなわち「利用者が次の操作と状態遷移を判断しづらい」ことに起因する。そのため、個別のリンク修正や旧REQ参照修正として分割せず、1つの要件化単位として扱う。

## 要件化の方向

`req-define` では、新規REQを作るよりも、既存 active REQ への UPDATE / APPEND として扱うことを優先的に検討する。

主な候補は以下。

- `REQ-0101`: 文書・REQ管理基準。guides / README / DOC-MAP の責務境界に関係する
- `REQ-0105`: Intake / Learning / Backlog。`.agentdev/` domain state と RU lifecycle
  に関係する
- `REQ-0107`: Reporting / Writing Quality。利用者向け文書の簡潔性・AI-slop抑止に関係する
- `REQ-0108`: Integrity / Validation / Tests。旧参照、壊れリンク、obsolete path 検出に関係する
- 必要に応じて `REQ-0103`: Artifact責任分界。command README / artifact-model の責務境界に関係する

独立した新規REQは、既存REQでは「利用者向けドキュメント責務」「入力状態に基づく入口判断」「`.agentdev/` 状態表」の扱いを表現できないと判断した場合のみ検討する。

## 必須要件候補

### 1. 利用者入口は「入力の状態」で判断できること

README または `workflow-overview.md` の冒頭に、利用者が現在持っている入力から次コマンドを選べる表を置く。

表には少なくとも以下を含める。

| 入力の状態                                     | 次のコマンド                   |
| ---------------------------------------------- | ------------------------------ |
| 実装したい内容を説明できる                     | `/agentdev/req-define`         |
| 具体的な作業候補だが採否・粒度が未整理         | `/agentdev/intake-capture`     |
| GitHub Issue / PR から未回収事項を拾いたい     | `/agentdev/intake-from-github` |
| 失敗・手戻り・判断ミスから得た知見を整理したい | `/agentdev/learning-refine`    |

| `.agentdev/intake/prom
oted/*.md` または `.agentdev/learning/promoted/*.md` がある | `/agentdev/backlog-review` |
| `.agentdev/backlog/req-units/RU-*.md` がある | `/agentdev/req-define` |
| 文書・コマンド・参照の整合性を診断したい | `/agentdev/integrity-check` |
| REQ体系の肥大・重複・分散を診断したい | `/agentdev/req-restructure-review` |

この表では「本流」「支流」「合流」のような比喩的な上下関係を構造名として使わない。

### 2. コマンド間の受け渡しを生成物中心で説明すること

`workflow-overview.md` では、概念説明より先に以下の関係を示す。

| 前工程             | 生成物                                                   | 次工程                        |
| ------------------ | -------------------------------------------------------- | ----------------------------- |
| `intake-promote`   | `.agentdev/intake/promoted/*.md`                         | `backlog-review`              |
| `learning-promote` | `.agentdev/learning/promoted/*.md`                       | `backlog-review`              |
| `backlog-review`   | `.sisyphus/drafts/backlog-review-*.md`                   | `backlog-save`                |
| `backlog-save`     | `.agentdev/backlog/req-units/RU-*.md`                    | `req-define`                  |
| `req-define`       | `.sisyphus/drafts/req-draft-*.md` またはセッション内要件 | `req-save` または `case-open` |
| `req-save`         | `docs/requirements/REQ-*.md` / `docs/adr/ADR-*.md`       | `case-open`                   |
| `case-open`        | GitHub Issue                                             | `case-run`                    |
| `case-run`         | PR                                                       | `case-close`                  |

### 3. `.agentdev/README.md` は状態表に寄せること

`.agentdev/README.md` は、旧REQ参照や子issue参照を削除し、現行 active REQ 前提の domain state 早見表にする。

少なくとも以下の列を持つ表を含める。

- path
- 状態
- producer
- consumer / next command
- retention / removal condition
- manual edit policy（必要に応じて）

表には少なくとも以下のパスを含める。

- `.agentdev/intake/inbox/*.md`
- `.agentdev/intake/accepted/*.md`
- `.agentdev/intake/promoted/*.md`
- `.agentdev/intake/archive/**`
- `.agentdev/learning/inbox.md`
- `.agentdev/learning/archive/active.md`
- `.agentdev/learning/evaluation-report.md`
- `.agentdev
/learning/promoted/*.md`
- `.agentdev/backlog/req-units/RU-*.md`
- `.agentdev/integrity/reports/*.md`

`.agentdev/backlog/req-units/RU-*.md` の producer は `backlog-save` または明示的な session-sourced RU とし、`req-backlog` を現行producerとして説明しない。

### 4. `.agentdev/` と `.sisyphus/` の境界を短く現行化すること

`.agentdev/README.md` では以下を明記する。

- `.agentdev/` は AgentDevFlow の永続 domain state
- `.sisyphus/` は runtime workspace
- `.sisyphus/drafts/req-draft-*.md` や `.sisyphus/drafts/backlog-review-*.md` は working draft であり、canonical domain state ではない
- `.sisyphus/` の詳細命名規則や旧設計説明を `.agentdev/README.md` に長く置かない

### 5. command README はリファレンスに徹すること

`.opencode/commands/agentdev/README.md` は、概念説明の重複を減らし、各コマンドについて以下を一覧化する。

- command
- primary input
- primary output
- next command / next action
- side effect category（必要に応じて）

`/agentdev/req-backlog` は非推奨であることを残してよいが、通常の利用者導線や `.agentdev/backlog/req-units/` の現行producerとして強調しない。

### 6. `artifact-model.md` は保守者向けとして位置づけること

`docs/guides/artifact-model.md` は、利用者入口ではなく、AgentDevFlow の内部成果物・責務分界を編集する保守者向けの参照文書として位置づける。

README や DOC-MAP からの説明文も、利用者向けの最初の入口ではなく、内部構造・成果物モデルの参照先として読める表現にする。

### 7. `domain-state-lifecycle.md` は詳細ライフサイクルに集中すること

`docs/guides/domain-state-lifecycle.md` は、`.agentdev/README.md` の早見表と矛盾しない詳細版にする。

重複する概念説明は削り、状態遷移・producer / consumer / retention / failure han
dling に集中する。

### 8. README は短く、入口とリンクに絞ること

root `README.md` は以下に絞る。

- AgentDevFlow のidentity
- 入力の状態から次コマンドを選ぶための短い入口表
- 詳細参照先へのリンク
- 最小限のクイックスタート

長い概念説明、複数guideと同じフロー説明、command README と重複するコマンド詳細は削る。

### 9. stale reference を除去すること

対象文書から以下を除去または現行化する。

- 現行要件のように読める旧 `REQ-0017` / `REQ-0039` 参照
- `req-backlog` を現行の RU producer とする説明
- `.opencode/skills/**/reference/` への obsolete path 参照
- `REQ-0101`〜`REQ-0109` のような stale active REQ range 表記が残っている場合は `REQ-0101`〜`REQ-0111` へ更新

## req-define への補足

このRUは、実装機能の追加ではなく利用者向けドキュメントの再整理を求めるものである。要件化時は、機能要件として過度に展開せず、以下の観点で既存REQに反映するのが望ましい。

- 利用者入口は入力状態で選べること
- domain state は path ごとに状態・producer・consumer・retention を説明すること
- README / guide / command README / artifact model の責務を重複させないこと
- 旧REQ・obsolete path・deprecated command を現行導線として案内しないこと
