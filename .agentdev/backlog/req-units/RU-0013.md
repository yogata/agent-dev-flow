# RU: 完了報告フォーマットの責務分離・重複排除・現行registry追従

## 背景

`/agentdev/case-open` の完了報告で、以下の2行が同じ事実を重複して表現していた。

- `RU-0012.md を削除・永続化（commit/push完了）`
- `git 永続化: RU-0012.md削除 commit/push済み（HEAD = origin/main 同期確認OK）`

完了報告フォーマットは、共通必須フィールドとして `完了コマンド`、`対象`、`結果`、`検証結果`、`git 永続化`、`次のコマンド` を要求している。

一方で、各variantでは `結果` 欄にgit永続化の詳細を書ける構造が残っており、`git 永続化` 欄との責務境界が曖昧になっている。

現行mainを確認した結果、当初RUに含まれていた `req-define/bug-or-lightweight.md`、`req-define/maintenance.md`、`req-define/docs-or-chore.md` は現行registry上の前提ではなくなっている。現行registryでは `req-define/lightweight.md` に統合済みの扱いとし、残論点は `agentdev-workflow-reporting/SKILL.md` のvariant数説明とregistry実体のドリフトに置き換える。

## 問題

完了報告フォーマットに以下の問題がある。

1. `結果` 欄と `git 永続化` 欄で同じ事実を重複して書ける。
2. `case-open/standard.md` と `case-open/epic.md` が `git 永続化: 該当なし` 固定であり、RU削除後のcommit/pushを表現できない。
3. `case-close` の部分失敗variantで、`.agentdev 永続化` の詳細と `git 永続化` が重複している。
4. `agentdev-workflow-reporting/SKILL.md` のvariant数説明と `completion-reports.md` のVariant Registry実体にドリフトがある。
5. `REQ-0103` のRequirement Source表で、RU-0013の反映先が完了報告フォーマット最適化ではなく別要件に見える記録ドリフトがある。
6. `integrity-check` の検査観点として、完了報告の責務混在・重複・registry driftを検出する必要がある。

## 統合理由

本件は個別コマンドの表記ゆれではなく、完了報告フォーマット全体の責務分離不備である。

`結果`、`検証結果`、`git 永続化` の境界を明確化しない限り、他コマンドでも同様の重複が再発する。

また、registry実体とSKILL.mdの説明がずれると、完了報告variantの選択・検査・保守で誤判定が発生する。

したがって、個別variantの文言修正ではなく、完了報告フォーマット全体の最適化とregistry追従として扱う。

## 要件

### RU-0013-001: 結果欄の責務分離

完了報告の `結果` 欄は、ユーザー視点・ドメイン視点の成果のみを記載すること。

含める例:

- Issue作成
- PR作成
- REQ/ADR保存
- intake item保存
- promoted artifact生成
- RU生成
- RU削除
- コメント追加
- worktree削除
- Findings回収

含めない例:

- commit hash
- push成否
- HEAD同期確認
- git status
- git diff

### RU-0013-002: git永続化欄の責務分離

完了報告の `git 永続化` 欄は、git操作結果のみを記載すること。

記載形式は以下を基本とする。

- `該当なし`
- `変更なし（commit/push スキップ）`
- `✅ OK（commit {hash}, push 済み）`
- `✅ OK（commit {hash}, push 済み, HEAD = origin/main 同期確認OK）`
- `⚠️ 注意（commit {hash}, push 失敗）`
- `❌ NG（commit失敗: {reason}）`

`git 永続化` 欄では、Issue作成、コメント追加、RU削除、worktree削除などの業務成果を主語にしない。

### RU-0013-003: 結果欄とgit永続化欄の重複禁止

同じ事実を `結果` 欄と `git 永続化` 欄の両方に記載してはならない。

NG例:

- `結果: RU-0012.md を削除・永続化（commit/push完了）`
- `git 永続化: RU-0012.md削除 commit/push済み`

OK例:

- `結果: RU-0012.md を削除`
- `git 永続化: ✅ OK（commit {hash}, push 済み, HEAD = origin/main 同期確認OK）`

### RU-0013-004: case-open variantのgit永続化可変化

`case-open/standard.md` と `case-open/epic.md` は、`git 永続化: 該当なし` 固定を廃止すること。

RU削除が発生しない場合は `該当なし` とする。

RU削除が発生しcommit/pushした場合は、commit/push結果およびHEAD同期確認結果を `git 永続化` 欄に記載する。

`結果` 欄には、RU削除というドメイン成果のみを記載し、commit/push/HEAD同期確認は記載しない。

### RU-0013-005: case-close部分失敗variantの重複排除

`case-close/agentdev-push-failed.md` と `case-close/worktree-cleanup-failed.md` は、`結果` 欄に `.agentdev 永続化` のcommit/push詳細を記載しないこと。

git操作結果は `git 永続化` 欄に集約する。

`結果` 欄に `.agentdev 永続化` セクションを残す場合でも、記載内容は「domain state更新対象の有無」などのドメイン成果に限定し、commit hash、push成否、git status、HEAD同期確認は含めない。

### RU-0013-006: registry説明の現行main追従

`agentdev-workflow-reporting/SKILL.md` の対象コマンド表は、`completion-reports.md` のVariant Registryと一致させること。

少なくとも以下を修正する。

- `req-define` は5 variantsではなく3 variants
- `backlog-review` は1 variantではなく3 variants
- `backlog-save` は1 variantではなく3 variants

### RU-0013-007: 存在しないreq-define旧variant参照の排除

完了報告フォーマット最適化の変更対象・説明・受け入れ条件から、現行mainに存在しない以下のvariant参照を排除すること。

- `req-define/bug-or-lightweight.md`
- `req-define/maintenance.md`

これらを復活させてはならない。

現行mainでは `req-define/lightweight.md` を軽量系の統合variantとして扱う。

### RU-0013-008: Requirement Source記録の補正

`REQ-0103` のRequirement Source表において、RU-0013の反映先が完了報告フォーマット最適化として追跡できるように補正すること。

少なくとも、RU-0013の反映先として `REQ-0103-046` が追跡できる状態にする。

`REQ-0108` についても、RU-0013の反映先として `REQ-0108-113` / `REQ-0108-114` が追跡できる状態にする。

### RU-0013-009: integrity-check検査観点への追加・追従

`integrity-check` または完了報告フォーマット検査観点に、以下の検査を追加または追従すること。

- `結果` 欄にcommit/push/HEAD同期確認が含まれていないこと
- `git 永続化` 欄が存在する場合、git操作結果に限定されていること
- 同一のgit永続化事実が `結果` 欄と `git 永続化` 欄で重複していないこと
- `git 永続化: 該当なし` 固定variantが、実行時にgit永続化を発生させるコマンドに使われていないこと
- `agentdev-workflow-reporting/SKILL.md` のvariant数説明がVariant Registryと一致していること
- 完了報告referenceが存在しないvariantを参照していないこと
