# req-define 引数なし単体実行時の入力優先順位明確化

## 背景

`req-define` を引数なしで単体実行した場合、当該セッション履歴および現在コンテキストを入力として要件定義を進める挙動を期待する。

現行の整理では、セッション履歴検知と `.agentdev/backlog/req-units/RU-*.md` の自動検出が併存しており、実行エージェントが `req-units` を先に扱う余地が残っている。

## 問題

引数なし単体実行時の入力優先順位が明確でないため、次の問題が発生し得る。

- 当該セッションに十分な要件合意が存在するにもかかわらず、`req-units` 自動検出が先に処理される
- セッション履歴・現在コンテキストが Requirement Source 候補として扱われない
- `req-define` がユーザーの直前合意を入力として利用せず、不要なファイル候補提示または壁打ちに進む
- Confirmed / Inferred / Unknown の分類前に外部入力へフォールバックする

## Source Summary

チャット内で次の内容が合意された。

- `req-define` を引数なしで単体実行したとき、当該セッション履歴・現在コンテキストをインプットとして処理を進める
- 入力優先順位の最優先はセッション履歴・現在コンテキストである
- 次点は `.agentdev/backlog/req-units/RU-*.md` である
- `req-units` の自動検出は、セッション履歴・現在コンテキストから有効な Requirement Source を構成できない場合の後続処理とする
- セッション履歴・現在コンテキストに不足がある場合は、直ちに `req-units` へ進まず、必要に応じて壁打ち対話で不足論点を解消する

## 統合理由

本 RU は、`req-define` の引数なし単体実行時における入力解決順序の明確化を扱う単一論点である。

セッション履歴優先、`req-units` 次点、壁打ち対話へのフォールバックは同一の入力解決仕様として不可分であり、分割すると実行順序の解釈が再び曖昧になる。

## 要件化の方向

`req-define` の入力解決仕様として、引数なし単体実行時の優先順位を明確化する。

優先順位は次の通りとする。

1. 当該セッション履歴・現在コンテキスト
2. `.agentdev/backlog/req-units/RU-*.md`
3. 通常の壁打ち対話

`req-define` は、引数なし単体実行時に最初に当該セッション履歴・現在コンテキストを Requirement Source 候補として評価する。

評価対象は少なくとも次を含める。

- 要件内容
- work_type
- scale
- ADR 要否
- 構造化状態
- 適用範囲

抽出した内容は Confirmed / Inferred / Unknown に分類する。

Confirmed のみで req-draft を自足できる場合、セッション履歴・現在コンテキストを入力として処理を継続する。

Confirmed のみでは自足しないが、補足質問により解消可能な場合、`req-units` 自動検出へ進まず、壁打ち対話を継続する。

セッション履歴・現在コンテキストから有効な Requirement Source を構成できない場合に限り、 `.agentdev/backlog/req-units/RU-*.md` を自動検出する。

`req-units` が 1 件のみ存在する場合は、その RU を read-only input として扱う。

`req-units` が 2 件以上存在する場合は候補一覧を提示し、実行エージェントが勝手に選択しない。

セッション履歴・現在コンテキストも `req-units` も有効な入力にならない場合、通常の壁打ち対話を開始する。
