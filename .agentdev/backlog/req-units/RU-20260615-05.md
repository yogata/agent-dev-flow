# RU: REQ authoring policy の再発防止強化

## 背景

本セッションでは、active REQ 全件に「移行・変更・再定義・削除・改名・移管・除去などの作業手段が要件化されている」問題が確認された。

既存REQを修正するだけでは、今後 `req-define`、`backlog-review`、`inspect-promote`、関連skill、integrity rule などが再び作業手段を要件候補として生成・保存する可能性がある。

そのため、既存REQの見直しとは別に、REQ authoring policy と保存前ゲートを強化し、再発を防ぐ必要がある。

## 問題

ADR に方針を記録するだけでは、再発防止として不十分である。

ADR は判断記録であり、運用上の強制点は以下に分担して配置する必要がある。

- REQ
- SPEC
- command
- skill
- integrity rule
- README / index

`req-define` だけを修正しても、`backlog-review` や `inspect-promote` が手段要件を生成すれば再発する。

`req-save` だけを修正しても、生成段階で「移管すること」「改名すること」が要件候補として出続ける。

integrity だけを修正しても、保存後に検出するだけで、生成・レビュー時の品質は上がらない。

## Source Summary

本セッションで合意した内容は以下。

- 今回の問題は、既存REQの見直しと再発防止の2本立てで扱う。
- ADR には「active REQ は状態要件のみを保持する」という判断を記録する必要がある。
- ADRだけでは維持できない。
- REQ-0102 を基軸に、全 active REQ へ適用される状態要件化ルールを明文化する必要がある。
- SPEC には、REQ文書の content contract として、title / purpose / requirement rows / scope の書き方を契約化する必要がある。
- command には、`req-define`、`req-save`、`backlog-review`、`inspect-promote` などで、手段要件を生成・保存しないゲートを持たせる必要がある。
- skill には、要件分析・REQ作成支援の判断基準として、状態要件と反映作業の分離基準を持たせる必要がある。
- integrity には、active REQ の title / 目的 / 要件行 / 適用範囲 / README active index に作業手段語が混入していないか検査する rule を追加する必要がある。

## 統合理由

再発防止は、単一ファイルの文言修正では成立しない。

REQ生成、REQ保存、backlog統合、inspect finding の昇格、skill判断、integrity検査、README active index の維持まで、複数の入口で同じ基準を使う必要がある。

そのため、本RUでは「REQ authoring policy の再発防止強化」として、ADR / REQ / SPEC / command / skill / integrity への反映を一体の要件化対象として扱う。

## 要件化の方向

AgentDevFlow は、REQ authoring において次の方針を維持する。

- active REQ は、現在満たすべき状態・振る舞い・制約だけを保持する。
- 移行・変更・再定義・削除・改名・移管・除去などの作業手段を、active REQ の要件行として保存しない。
- 作業手段は、case / Issue / 受け入れ条件 / 作業記録で扱う。
- `req-define` は、入力内容を状態要件と反映作業に分離する。
- `req-save` は、反映作業だけの要件行が残る場合に保存を停止する。
- `backlog-review` と `inspect-promote` は、promoted artifact や finding を RU 化する際に、作業手段を要件化しない。
- 要件分析・REQ作成支援 skill は、状態要件と反映作業の分離基準を判断基準として持つ。
- integrity は、active REQ の作業手段語混入を継続検査する。
- README active index は、作業名ではなく責務・状態・制約名を保持する。

## 受け入れ条件

- ADR に、active REQ は状態要件のみを保持し、作業手段を要件化しないという判断が記録されている。
- REQ-0102 に、全 active REQ に適用される状態要件化ルールが定義されている。
- SPEC に、REQ文書の title / purpose / requirement rows / scope の content contract が定義されている。
- `req-define` が、要件候補を状態要件と反映作業に分離する責務を持っている。
- `req-save` が、反映作業だけの要件行が残る場合に保存を停止する責務を持っている。
- `backlog-review` が、promoted artifact を RU 化する際に作業手段を要件化しない責務を持っている。
- `inspect-promote` が、inspect finding を要件化する際に作業手段を要件行へ混入させない責務を持っている。
- 要件分析・REQ作成支援 skill が、状態要件と反映作業の分離基準を持っている。
- integrity rule が、active REQ の title / purpose / requirement rows / scope / README active index に作業手段語が混入していないか検査対象にしている。
- README active index が、作業名ではなく責務・状態・制約名を保持する契約になっている。
- 再発防止ルールは、ADRだけに閉じず、REQ / SPEC / command / skill / integrity に分担反映されている。
