## 影響

横断パターン X-1〜X-7 は 7/7 ディレクトリで共通出現する。X-2（em-dash）は docs/specs/ だけで 496 件 / 59 ファイルに上り、放置すれば文書品質が持続的に低下する。X-6 は規範を定める REQ-0140 自身が「において」を使っており、規範の権威を損なう。

ただし、ソースカタログ `.agentdev/inspect/inbox/compliance-catalog-*.md`（7 件）は現時点で存在しない（`inspect/inbox/` には `.gitkeep` のみ）。各 finding のファイルパス・行番号・件数の裏付けをカタログから再取得できず、本成果物が依拠する証拠は inbox 項目の引用のみになった。横断 Issue を起票する前に、各ディレクトリで改めて `inspect-docs` を実行してカタログを再生成する必要がある。

加えて、Wave 2 で既に起票された Issues #1076/#1078/#1079/#1080（および #1086）が、X-1 中黒・X-2 em-dash・X-3 LLM 表現・X-4 一文一行の横断修正と重複する可能性がある（参考: PR #1088〜#1091 が各 Issue の OU に対応）。新規に横断 Issue を起票する前に、これら既存 Issue の消化状況・スコープとの重複を照合しなければ、二重修正や修正漏れを生む。

## 課題

7 ディレクトリに共通する文書規範逸脱 X-1〜X-7 を横断的に是正する作業を、どのように RU 化するか。ただし作業開始前に二つの制約を解決する必要がある。第一に、裏付けカタログが消失しているため再生成が前提となる。第二に、Wave 2 既存 Issue とのスコープ重複を整理しなければ、どのパターンが未対応か判断できない。

## 整形の方向性

backlog-review では以下を検討する。

1. **事前確認（必須）**: Wave 2 Issues #1076/#1078/#1079/#1080/#1086 の消化状況とスコープを照合し、X-1〜X-7 のうち未対応パターンを特定する。
2. **カタログ再生成**: 裏付けカタログが消失しているため、`inspect-docs` を 7 ディレクトリで再実行し、finding の実体を再確認してから RU 化する。
3. **RU 構成案**: 重複確認後、X-1〜X-5（既存 Issue で未対応分）と X-6・X-7（新規）を分けるか、単一横断 RU に集約するかを判断する。X-2（em-dash）は docs/specs/ で 496 件と高頻度のため、機械的修正 RU として独立させると並列実行しやすい。
4. **優先度**: X-2 > X-6 > X-1・X-3・X-4・X-5 > X-7 を目安とする。X-6 は REQ-0140 の自己準拠矛盾を含み、規範の権威回復の観点で早めの対応が望ましい。X-7 は SPEC 運用基準の策定を伴うため、やや後回しでよい。
5. **バンドル**: 本成果物は 3 inbox 項目を 1 RU 群に束ねる。X-8（ファイル数精度）は learning 系列へ分離済みであり、本 RU 群には含めない。
