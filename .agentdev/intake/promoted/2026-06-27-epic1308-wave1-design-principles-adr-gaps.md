## 観測内容
Phase C（PR #1315 / Issue #1311）で `design-principles.md` の判断理由・歴史的経緯を既存 ADR 参照へ置換した（AG-005、CR-003 Q3=A）。しかし以下 5 項目は判断理由（トレードオフ）を直接覆う ADR が存在せず、Phase C の範囲外（CR-003 Q3=A）として報告された:
1. work_type 分類の設計意図（バグ修正オーバーヘッド最小化 + 機能追加の品質ゲート確保）
2. 実装先行の原則、SSoT 段階的遷移の判断理由（対話的開発の自然なフロー、変動性への対応）
3. エージェント責務分離（Prometheus / Sisyphus 2 エージェント体制）の判断理由（認知負荷分離）
4. gh CLI 安全手順の判断理由（Windows PowerShell 文字化け問題への実用的回避策）
5. git worktree 採用理由（並列開発基盤、作業ディレクトリ分離）

## 影響
5 つの重要な設計判断について ADR カバレッジが不足しており、設計意図の可視性・再利用性に影響。

## 課題
- 上記 5 項目それぞれについて ADR 作成要否を評価
- 作成する場合、既存要件（REQ-0104、ADR-0109、ADR-0130）との整合性を整理
- 判断理由が SPEC に残置されたまま運用するか、ADR 化して SPEC は参照化するか

## 既存要件との関連
- REQ-0104: work_type 分類を規定（判断理由は ADR 化されていない）
- ADR-0109: Epic Issue SSoT（実装先行原則に関連のみ、判断理由は扱わない）
- ADR-0130: gh CLI I/O 境界確立（文字化け対応の技術判断理由は直接扱わない）
- CR-003 Q3=A: 既存 ADR で覆えない判断内容は新規 ADR 化せず後続評価候補とすることを規定
