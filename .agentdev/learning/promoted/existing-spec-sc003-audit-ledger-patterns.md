# SC-003 監査台帳ライフサイクル SPEC への適用例追記

## 背景

Epic #1601（AgentDevFlow 統合再編）の Phase 2 で、監査台帳（`.agentdev/drafts/audit-ledger-governance-system-audit.md`）を前提情報とする大規模変更計画を4 Wave に渡って実施した。この過程で、SC-003（`docs/specs/local/audit-ledger-lifecycle.md`）が定める5ステップライフサイクルと次フェーズ用入力の運用において、4つの具体的適用パターンが実証された。これらのパターンは SC-003 SPEC の「既知の適用例」「次フェーズ用入力」節に未記載で、将来の大規模変更計画での再利用性が損なわれている。

## 問題

SC-003 SPEC は5ステップライフサイクル（生成→参照→参照専用化→次フェーズ転記→廃棄）を定義するが、具体的な適用パターンが未整備。大規模変更計画で監査台帳を利用する際、フェーズ分割判断、書き戻し形式、ライフサイクル完遂の commit 構成、次フェーズ用入力の要素構成を毎回ゼロから設計する必要がある。

## 望ましい変更

SC-003 SPEC の「既知の適用例」節へ以下4パターンを追記する:
1. 監査フェーズを one-time ライフサイクルとして独立させるフェーズ分割パターン
2. 実装 PR close 時の監査台帳書き戻し（解消状況記録）の標準形式
3. 中間成果物ライフサイクル完遂を 2 commit 構成で証明するパターン
4. 次フェーズ用入力の自足性を6要素構成（背景/対象候補/正規所有者/自動化後状態/対象外/受け入れ条件）で確保する転記パターン

## 対象範囲
