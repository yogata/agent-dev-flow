# サブエージェント委譲指示のガードレール強化

## 背景

case-run でサブエージェントにタスクを委譲する際、プロンプトの指示が不完全なため、サブエージェントが表面的な変更のみを行い、実際の実行パスや境界を正確に把握できない問題が2件発生した。関数シグネチャ変更時の呼び出し箇所未列挙と、ドキュメント削除時の境界未明示が主な事例。

## 問題

workflow-orchestration skill の「サブエージェント実行プロトコル」セクションのプロンプト構成に、以下のガードレールが欠落している:

1. **関数シグネチャ変更時**: 定義箇所の変更のみで、全ての呼び出し箇所（SQL/RPC/API）での新パラメータ渡しが漏れる
2. **ドキュメント削除時**: 削除範囲の開始・終了マーカーが未指定で、隣接セクションが誤削除される

## 望ましい変更

workflow-orchestration skill の「サブエージェント実行プロトコル → プロンプト構成」に以下のガードレールを追加:

- 関数シグネチャ変更を伴うタスクでは、委譲プロンプトの MUST DO に「定義箇所以外に全ての呼び出し箇所（SQL/RPC/API）での新パラメータ渡しを含めること」を明記
- ドキュメントの部分削除を伴うタスクでは、委譲プロンプトの MUST DO に「削除範囲の開始・終了マーカー（行番号または見出し文字列）を明記すること」「削除対象外の隣接コンテンツを明示すること」を追加

## 元learning item / 根拠

- **要約**: サブエージェントへの委譲プロンプトで具体的な操作範囲が明示されておらず、不完全な変更が実行された
- **根拠**: RPC関数のSQL呼び出しに新パラメータが含め忘れた事例（Issue #813）と、ドキュメント削除で隣接セクションが誤削除された事例（Issue #816）の2件
- **再発条件**: 委譲プロンプトで作業範囲の境界や全ての影響箇所を列挙せずにタスクを依頼した場合
- **横展開可能性**: 委譲パターンは全プロジェクト・コンテキストで発生し得る（横展開性 5/5）
