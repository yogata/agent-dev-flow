# VERIFY 実装観点（ローカル版）

agentdev-gh-cli ローカル版の VERIFY 手続きの実装観点と検査項目（REQ-0150-001, ADR-0130 decision #4, #5）。
書き込み操作（Case ファイル作成、本文更新、作業ログ追記、PR 相当セクション追記、マージ結果記録、close）の直後に実施し、内容が正しく反映されたかを検証する。各書き込み操作ごとに個別に実行すること（一括検証は不可）。

観点は標準版（[../../skills/agentdev-gh-cli/references/verify.md](../../../skills/agentdev-gh-cli/references/verify.md)）と同じ4観点である。対象が Case ファイル（ローカルファイル）になるため、エンコーディング検証の失敗要因（gh CLI パイプライン、コンソールコードページ）は発生しないが、書き込み tool の誤使用（BOM 付き書き出し等）を検出するため観点を維持する。

## 検証手順

### 1. 読み戻し

対象 Case ファイル `.agentdev/cases/case-{NNNN}.md` を Read tool で読み込む。

- Case ファイル全体: Case ファイル読込手続き
- PR 相当セクションのみ: PR 本文読込手続きで該当セクションを抽出

### 2. 4観点の比較検証

書き込み元テキストと読み戻しテキストを以下の4観点で比較する。

#### (a) エンコーディング検証

- 書き込み元テキストと読み戻しテキストの日本語文字列が一致するか確認
- BOM が付与されていないか確認（U+FEFF が先頭にないこと）
- 改行コードが LF であることを確認（CRLF でないこと）
- 制御文字（U+0000-U+001F, U+007F-U+009F）が混入していないか確認（改行 U+000A, タブ U+0009 は除外）

#### (b) Markdown 構造検証

- テーブル: 行ごとの列数が一致しているか
- チェックボックス: `- [ ]`, `- [x]` 構文が保持されているか
- コードブロック: `` ``` `` の開閉ペアが一致しているか
- リスト: インデント階層が保持されているか

#### (c) 必須セクション検証

Case ファイルの用途に応じて以下を検証する。

- PR 作成（PR 相当セクション追記）時: `## SPEC確定候補` と `## Findings / Capture候補`（`### intake`, `### learning` サブ見出し含む）が存在するか
- `## Findings / Capture候補` セクションは SPEC [local/local-case-file.md](../../../../../docs/specs/local/local-case-file.md) の「見出し一覧」で Case ファイルに必須と定義されている。新規作成後に欠落していないか確認する

#### (d) リポジトリ参照リンク検証

- 読み戻しテキスト内の Markdown リンク `[text](url)` で、リポジトリ内パス（`docs/`, `.opencode/` 等で始まる）が相対パスとして使用されていないか確認
- ローカル版では GitHub URL 形式を必須としないが、相対パスが意図せず混入していないかを検出する
- コードブロック内、テンプレート変数プレースホルダー（`{xxx}`）は検証対象外

## 検証結果報告

完了報告に以下の形式で検証結果を含める:

```
検証結果: OK（N件の書き込み操作を検証済み）
```

リトライが発生した場合:

```
検証結果: リトライあり（操作: Case ファイル本文更新 case-0042, リトライ: 2回目で成功, 差分: テーブル列ずれ）
```

## 各コマンドでの適用

ローカル版の各コマンドの書き込み操作後に本 VERIFY を適用する。各コマンドが参照する agentdev-gh-cli（ローカル版）の VERIFY 参照箇所で実行する。

- `case-open`（ローカル版）: Case ファイル作成、作業ログ追記
- `case-run`（ローカル版）: PR 相当セクション追記
- `case-close`（ローカル版）: マージ結果記録、Case ファイル本文更新（status: closed）、親 Case ファイル本文更新（Epic の場合）
- `case-update`（ローカル版）: Case ファイル本文更新、作業ログ追記

VERIFY 失敗時のリトライロジックは [retry.md](retry.md) 参照。
