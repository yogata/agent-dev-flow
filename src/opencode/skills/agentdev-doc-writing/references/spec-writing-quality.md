# SPEC 本文の合格基準

> **原本**: [docs/specs/writing-style.md](../../../../../docs/specs/writing-style.md)「文書種別責務」。本ファイルは運用ビュー（詳細仕様の置き場としての成立性判定）である。

SPEC は現在の実装構成を成立させるスキーマ・ライフサイクル・コマンド構成・ルールカタログ・判定表・enum・format・内部パラメータを記述する。要件（満たすべき成果）でも意思決定（ADR）でもない。

## 詳細仕様の置き場としての成立性

SPEC 本文が合格するためには、以下を満たす必要がある。

- [ ] **現在形で記述されている**: 「現在どうなっているか」を記述していること。「すべき」「しなければならない」等の要件表現は REQ に、決定の経緯は ADR に属する
- [ ] **検証可能な構成要素を含む**: スキーマ・ライフサイクル・コマンド構成・ルールカタログ・判定表・enum・format・内部パラメータのいずれかが具体化されていること
- [ ] **内部設計文書として位置づけられている**: 実行時配布対象ではなく、リポジトリ内部設計文書であることが前提である

## SPEC 不合格の典型

以下に該当する場合、修正または移送を提示する。

| 問題 | 分類 | 対応 |
|---|---|---|
| 要件（満たすべき成果）を記述している | 移送 | REQ へ移送候補を提示 |
| 意思決定の経緯・代替案の却下理由を記述している | 移送 | ADR へ移送候補を提示 |
| ナビゲーション・案内のみで構成されている | 移送 | guide へ移送候補を提示 |
| 抽象的で構成要素が具体化されていない | 分割 | [rewrite-patterns.md](rewrite-patterns.md) の抽象語置換パターンを適用 |
| 英語混じり抽象語が残留している | 分割 | [rewrite-patterns.md](rewrite-patterns.md) の書き換えパターンを適用 |

## SPEC と他文書種別の境界

- **REQ が記述するのは「満たすべき成果」**: SPEC 相当内容（スキーマ・enum・判定表等）のみを主たる文意とする REQ 要件行は、SPEC への移送候補となる。安定契約例外（公開 command 名・接続契約等）は REQ に要約として記述できる。
- **ADR が記録するのは「意思決定」**: SPEC に決定の経緯を書かず、決定結果（現在の構成）のみを記述する。
- **guide はナビゲーション層**: SPEC の内容を代替しない。guide に詳細仕様が混入している場合は SPEC への移送候補となる。

## 查読出力

不合格の SPEC 本文は [review-output.md](review-output.md) の形式で、修正文案・移送先候補として提示する。
