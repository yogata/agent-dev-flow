# 機械判定可能な文書品質是正ルール

中黒、em-dash、LLM 表現、一文一行の 4 規範について、文脈判断を伴わず機械的に適用できる判定アルゴリズムを集約する。
判定基準が実行者に依存せず、inspect-docs 自動検出、CI 検証、横断修正で再現性が確保されることを目的とする。

> **原本**: 中黒、em-dash、一文一行の機械判定アルゴリズムは本ファイルが原本。
> LLM 表現の検出、書き換え辞書は [llm-expression-patterns.md](llm-expression-patterns.md) を参照。
> 執筆規範の SSoT は `japanese-tech-writing` スキル（AGENTS.md 経由）、用語政策の原本は [docs/specs/document-type-responsibilities.md](../../../../../docs/specs/document-type-responsibilities.md) を参照。

## 対象読者

docs/ 配下の REQ/ADR/SPEC/guides/DOC-MAP/README、および docs を生成、編集する command/skill の自然言語記述に対し、機械判定可能な是正を適用する agentdev-doc-writing スキル利用者、inspect-docs 実装者、CI 検証設定者。

## 対象範囲と適用除外

機械判定は以下の 4 規範に限定する。
文脈判断が必要な表現（「最も」「重要な」等）は機械判定対象外とし、サンプリング査読へ委譲する（後述「3. LLM 表現機械判定」の対象外リスト参照）。

適用除外（機械判定を実行しない領域）:

- ユーザー発言の引用、既存文書の引用
- 他ファイルの見出し、節名の引用（`「...」` で括られた参照）
- 禁止表現そのものの批評、説明（本ファイル等の規範記述）
- コードブロック、インラインコードの内部
- YAML frontmatter のキー名、識別子値

## 1. 中黒（・）許容範囲判定

中黒（`・`）は原則として日本語並列に使わない（japanese-tech-writing L18）。
ただし以下の 4 条件のいずれかを満たす場合は許容し、それ以外の日本語並列は是正対象とする。

### 許容条件（いずれかを満たす場合は許容）

| 条件 | 機械判定方法 | 例 |
|---|---|---|
| SPEC 明示例 | SPEC `document-type-responsibilities.md`「中黒使用の許容範囲」節に列挙された固定複合名詞の内部と一致 | `実行時・編集時`、`コマンド・スキル・テンプレート・スクリプト` |
| 時系列対比ペア | 中黒の左右がいずれも `時` で終わる時刻表記（`NN時・NN時` 形式） | `9時・17時`、`開始時・終了時` |
| code block 内 | fenced code block（``` 〜 ```）の内部行 | `・` を含むコード例 |
| YAML title | YAML frontmatter の `title:` フィールド値 | `title: REQ・ADR・SPEC 横断品質` |

固定複合名詞、単一固有名詞の許容範囲は SPEC「中黒使用の許容範囲」節が原本。
本ファイルは SPEC 明示例との照合で機械判定可能な部分のみを扱う。

### 判定手順

1. 行単位で `・` を検索する
2. 当該行が code block 内または YAML frontmatter 内であれば許容（検出A、B）
3. 中黒の左右が `NN時`（数字 + `時`）であれば許容（検出C）
4. 中黒を含む前後文字列が SPEC 明示例リストと完全一致すれば許容（検出D）
5. 上記いずれにも該当しない `・` を含む行を是正対象として抽出する

### 是正対象の例

- 流動的並列: `要件・設計・実装を分担する` → 読点（、）または箇条書きへ
- 識別子の並列: `REQ-0101・REQ-0102` → スラッシュ（`/`）または箇条書きへ
- コード値の並列: `feature・fix・chore` → スラッシュまたは箇条書きへ

## 2. em-dash（—）置換判定

em-dash（`—`、`―`）は japanese-tech-writing L17 に従い、同格、補足は括弧（`（）`）で、言い換え、敷衍は句点で2文分割または読点でつなぐ。
コロン（`:`）による置換は行わない。

### 検出パターン

| パターン | 検出方法 | 置換先 |
|---|---|---|
| 本文中の em-dash | 半角空白挟み ` — ` （` ` + U+2014 + ` `）を行単位で検索 | 文脈に応じて `（）` または句点分割、読点接続 |
| テーブルセル N/A プレースホルダ | `| — |` （パイプ + 半角空白 + em-dash + 半角空白 + パイプ）を検索 | `| - |` （ハイフン1文字へ置換） |

テーブルセル `| — |` は N/A プレースホルダ用途であり、本文中の em-dash とは意味が異なる。
そのため機械的に `| - |` へ置換し、文脈判断を要しない。

### 置換手順

1. テーブルセル `| — |` をすべて `| - |` へ置換する（安全な機械置換）
2. 本文中 ` — ` を検索する
3. 各検出箇所の前後関係を確認し、同格、補足なら括弧（`（）`）へ、言い換え、敷衍なら句点で2文分割または読点でつなぐ
4. コロン（`:`）による置換は行わない

本文中 ` — ` は文脈判定が残るため、置換先の選択（括弧か句点か）は査読対象とする。
検出自体は機械的だが、置換先の決定は文脈依存であることを明示する。

## 3. LLM 表現機械判定

LLM 表現は、表層形式置換で cover できるもの（機械判定対象）と、文脈判断が必要なもの（機械判定対象外）に区分する。
包括的な表現パターン辞書は [llm-expression-patterns.md](llm-expression-patterns.md) を参照。
本節は機械判定可能な代表例のみを列挙する。

### 機械判定対象（表層形式置換で cover 可能）

検出パターンに対して書き換え先が一意に定まる表現。
文脈によらず機械的に置換できる。

| 検出パターン | 書き換え | 解説 |
|---|---|---|
| `において` | `で` | 「において」は文語調。「で」で足りる |
| `非常に` | 削除 | 主観的強調で根拠を伴わない |
| `極めて` | 削除 | 同上 |
| `することとしている` | `する` | 述語の先送りをやめる |
| `について` | 直接内容を書く | 「関して」は迂遠 |
| `に関して` | 直接内容を書く | 同上 |

包括的な検出、書き換え辞書は [llm-expression-patterns.md](llm-expression-patterns.md) の「接続の型」「空虚な動詞」「空虚な形容・副詞」「ラベル前置き」節を参照。
本表は機械判定で即時適用可能な代表例である。

### 機械判定対象外（文脈判断が必要、サンプリング査読へ委譲）

以下の表現は文脈によって正当な使用があり、一律置換すると文意を損なう。
機械判定の対象外とし、サンプリング査読で運用する。

| 表現 | 機械判定対象外の理由 |
|---|---|
| `最も〜` | 比較軸が文脈で明示されている場合は正当。機械判定では比較対象の有無を判定できない |
| `重要な〜` | 重要性の理由が本文で示されている場合は正当。機械判定では根拠の有無を判定できない |
| `必要な〜` | 必要性が本文で説明されている場合は正当。同上 |
| `適切に〜` | 基準が文脈で明示されている場合は正当。同上 |
| `〜を行う` | 固定複合名詞の一部（「決定事項を行う」等でない場合）は正当。複合名詞か否かの判定が文脈依存 |

これらの表現は [llm-expression-patterns.md](llm-expression-patterns.md) に検出パターンとして掲載されるが、本ファイルの機械判定アルゴリズムでは自動置換の対象としない。
サンプリング査読で文脈を確認した上で書き換え要否を判断する。

## 4. 一文一行機械判定

一文一行規範（japanese-tech-writing 整形規範）を機械判定する。
対象は prose line（本文行）であり、テーブル行、リスト項目、見出しは本アルゴリズムの対象外とする。

### 検出条件

1行に複数の `。`（句点）を含む prose line を検出する。

### 例外規則（括弧内 `。` の除外）

括弧（`（）`、`()`）内の `。` は文境界ではなく、括弧内の補足情報の区切りである。
文境界判定から除外するため、括弧の内側にある `。` はカウントしない。

括弧の種類と扱い:

| 括弧 | 扱い |
|---|---|
| `（）` （全角丸括弧） | 内部の `。` は文境界から除外 |
| `()` （半角丸括弧） | 同上 |
| `「」` （かぎ括弧） | 引用内部の `。` は対象行の文境界から除外 |
| `【】`、`『』` | 同上 |

### 分割手順

1. 対象行が prose line であることを確認する（テーブル行 `|...|`、リスト行 `- ...`/`* ...`、見出し `# ...` は除外）
2. 括弧の内側にある `。` を一時的にマスクする（文境界カウントから除外）
3. 残った `。` が2個以上であれば是正対象とする
4. 各 `。` の直後で改行し、1行1文へ分割する
5. 括弧内の `。` は元の位置を保持する（マスク解除）

### 判定例

| 入力行 | 判定 | 理由 |
|---|---|---|
| `本規範は機械判定を前提とする。文脈判断は査読に委譲する。` | 是正対象 | `。` が2個、改行で分割 |
| `本規範（機械判定を前提とする。文脈判断は査読に委譲する）を採用する。` | 許容 | 括弧内 `。` は除外、文境界は1個のみ |
| `| 列A | 列B |` | 対象外 | テーブル行 |
| `- 項目1` | 対象外 | リスト行 |

## 4 規範の分業と相互参照

中黒、em-dash、一文一行は判定手順（検出、許容/是正の二値判定）であり、LLM 表現は検出、書き換え辞書である。
判定アルゴリズムの性質が異なるため、両者を混載せず分業する。

| 規範 | 性質 | 本ファイルでの扱い | [llm-expression-patterns.md](llm-expression-patterns.md) での扱い |
|---|---|---|---|
| 中黒 | 許容範囲判定（検出→二値判定） | 機械判定アルゴリズム（原本） | 扱わない |
| em-dash | 置換判定（検出→テーブルは機械置換、本文は査読） | 機械判定アルゴリズム（原本） | 扱わない |
| LLM 表現（表層形式置換） | 検出→書き換え（一意） | 機械判定対象の代表例 | 検出→書き換え辞書（包括的） |
| LLM 表現（文脈判断必要） | 検出→書き換え（文脈依存） | 機械判定対象外、サンプリング査読への委譲を明示 | 検出→書き換え辞書（包括的） |
| 一文一行 | 改行分割（検出→分割位置） | 機械判定アルゴリズム（原本） | 扱いません |

既存 [llm-expression-patterns.md](llm-expression-patterns.md) は表現パターンの検出→書き換え辞書として現状維持する。
本ファイルから同ファイルへの相互参照を張り、機械判定アルゴリズム側に LLM 表現の機械判定対象を集約する。

## 再現性の担保

機械判定アルゴリズムは同一入力に対して常に同一の判定結果を返す。
判定結果が実行者に依存しないことが、inspect-docs 自動検出、CI 検証、横断修正での再発防止の前提である。

Wave 2 既存 OU（#1076 em-dash、#1078 中黒、#1079 LLM 表現、#1080 一文一行）の修正サンプルへ本ルールを再適用する手順:

1. 各 OU の修正後 prose を入力とする
2. 上記 4 アルゴリズムを順に適用する（中黒 → em-dash → LLM 表現 → 一文一行）
3. 各アルゴリズムの判定結果が0件（是正対象なし）であることを確認する
4. 判定結果が1件以上出る場合は、Wave 2 修正が不完全か本アルゴリズムが過剰検出のいずれかである。両者を突き合わせて原因を特定する

手順3で0件となることで、Wave 2 修正が本アルゴリズムと整合し、再現性が確認されたことになる。

### 機械横断是正 PR における Step 3-4 の必須化（REQ-0153）

上記 Step 3-4（再 grep 0 件確認）は推奨ではなく**必須**である（L-012、PR #1090 / #1122 由来。PR #1090 が「7 ディレクトリ完全対応」と完了宣言したが 3 件残存し、後続 PR #1122 で回帰修正した事象への対策）。

- 機械横断是正（機械的テキスト置換・複数ディレクトリ横断是正）を含む PR の完了に先行して、Step 3-4 を必ず実行すること
- 再 grep 結果（0 件）を PR 本文の品質メトリクス欄に記載することを運用化する
- Step 3-4 の実行と PR 本文記載は QG-3（観点 5）/ QG-4（観点 7）で担保される
