# ADR 本文の合格基準

> **原本**: 配置基準、用語政策は [docs/specs/responsibilities/document-type-responsibilities.md](../../../../../docs/specs/responsibilities/document-type-responsibilities.md)「文書種別責務」、執筆規範は `japanese-tech-writing` スキル（AGENTS.md 経由）。本ファイルは運用ビュー（意思決定文書としての成立性判定）である。

ADR は将来の設計、運用、文書システムを制約する意思決定の記録である。
要件（満たすべき成果）でも仕様（現在どうなっているか）でもない。

## 意思決定文書としての成立性

ADR 本文が合格するためには、以下を満たす必要がある。

- [ ] **決定内容が明示されている**: 何を決定したかが1文で表現されていること
- [ ] **文脈（Context）がある**: この決定が必要になった状況、制約、前提が記述されていること
- [ ] **根拠がある**: なぜこの決定に至ったか、検討した代替案と却下理由があること
- [ ] **影響（Consequences）がある**: この決定が将来の設計、運用、文書システムに与える制約が記述されていること
- [ ] **判断主体が明確**: 誰がこの決定を承認したか（status: accepted 等）が明記されていること

## ADR 不合格の典型

以下に該当する場合、修正または移送を提示する。

| 問題 | 分類 | 対応 |
|---|---|---|
| 要件（満たすべき成果）を記述している | 移送 | REQ へ移送候補を提示 |
| 現在の実装構成の詳細を記述している | 移送 | SPEC へ移送候補を提示 |
| 決定内容がなく背景説明のみ | 削除候補 | 決定内容の追記、または不要なら削除を提示 |
| 根拠なしで結論のみ | 分割 | 決定と根拠を分離し、根拠の追加を提示 |
| 「アーキテクチャに影響する」等の英語混じり抽象語が残留 | 分割 | [rewrite-patterns.md](rewrite-patterns.md) の書き換えパターンを適用 |

## ADR と他文書種別の境界

- **REQ**: ADR ではなく REQ に記述すべきは「満たすべき状態、振る舞い、制約」。ADR に要件を書かない。
- **SPEC**: ADR ではなく SPEC に記述すべきは「現在の実装構成（スキーマ、ライフサイクル、コマンド構成等）」。ADR に仕様を書かない。
- **ADR が記録するのは意思決定のみ**: 作業手段（どう実装したか）は ADR の対象外である。

## ADR 要否の動的判断

ある変更が ADR を要するかの判定は、本スキル（静的査読）の対象外である。
実行時の動的判断は agentdev-adr-guidelines が担う。
本スキルは「書かれた ADR が意思決定文書として成立しているか」のみを査読する。

## 查読出力

不合格の ADR 本文は [review-output.md](review-output.md) の形式で、修正文案、移送先候補として提示する。
