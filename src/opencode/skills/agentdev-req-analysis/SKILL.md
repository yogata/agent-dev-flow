# 要件分析スキル

要件分析のための**知識ベース**。分析観点・品質基準・定義を提供し、壁打ちフェーズでの壁打ち品質を担保する。

- **このスキル（知識）**: 分析観点・品質基準・定義
- **適用先**: 要件定義プロセス全般（壁打ち・Issue作成・実装検証）

---

## 要件分析

機能要件・非機能要件の区別なく、統一的に要件を展開する際の観点。

### ユーザーストーリー

「[誰]が[何]を[なぜ]する」形式で記述する。

- 明確な主体（ユーザー、システム、外部サービス等）
- 具体的なアクション
- ビジネス価値・目的

### 完了条件

Given-When-Then形式で記述し、測定可能であること。

- **Given**: 事前条件・前提状態
- **When**: 実行するアクション・イベント
- **Then**: 期待される結果・状態変化

客観的な完了判定が可能であること。

### 境界条件

正常系・異常系・エッジケースを網羅する。

- **正常系**: 期待されるフロー
- **異常系**: エラー・例外・拒否パターン
- **エッジケース**: 境界値・空入力・同時実行等

### 必達要件記述ガイダンス

要件行は検証可能な必達要件として記述する。推奨・任意・将来候補は要件行に含めない。

**記述ルール**:
- 各要件行は「満たす必要がある検証可能な要件」として記述する
- 禁止・停止条件・完了不可条件は自然文で記述する（「〜禁止」「〜停止する」「〜完了扱いにしない」等）
- 曖昧な表現（「適切に」「効率的に」等）は、具体的な測定可能な条件に置き換える

### 状態要件と反映作業の分離基準

要件行候補を「状態要件（変更後仕様）」と「反映作業」に分類する判断基準（REQ-0102-008, 050, ADR-0103）。

**状態要件（要件行として採用）**:
- システムが満たすべき振る舞い・制約・状態の宣言
- 「〜すること」「〜を禁止」「〜しない」等の検証可能な記述
- 変更後に満たされるべき条件

**反映作業（要件行として不採用、移送候補）**:
- 既存成果物への操作: 更新・削除・移動・名称変更・参照修正・移行・再定義・改名・移管・除去・廃止・置換
- 作業手順の記述（ステップバイステップの実行指示）
- 既存文書の修正内容そのもの

**混合候補の処理**:
- 状態要件の記述を含み、かつ反映作業も含む場合 → 要件行として採用（反映作業部分は移送候補としてマーク）
- 反映作業のみの場合 → 要件行として不採用、関連ドキュメント更新候補へ移送

**移送先**:
- 反映作業は case / Issue / 受け入れ条件 / 作業記録で扱う（REQ-0102-049）
- active REQ の title / 目的 / 要件行 / 適用範囲に作業手段語を混入させない（REQ-0102-006, 007）

### 用語定義

要件内の専門用語を明確化する。

- ドメイン固有の用語
- 技術的な前提条件
- 業務用語・略語の統一定義

## 壁打ちメソドロジー

要件確定前の未決分岐を証拠ベースで解消し、壁打ち品質を担保するメソッドロジー。未決分岐解消を親概念とし、既存の深掘り・合意形成・ステークホルダー視点を再構造化する。

### 未決分岐の抽出と整理

壁打ち中の論点を未決分岐として扱い、体系的に整理する。

**観点**:
- 目的・対象・対象外・粒度・永続化・責務境界・判定基準

**依存順解消ルール**:
- 優先順: 目的 → 対象 → 対象外 → 責務境界 → 粒度 → 永続化 → 判定基準
- 既存資料（明示入力ファイル、関連REQ、関連ADR、specs、command/skill定義）により上位分岐が確定済みの場合は、未確定の最上位分岐から開始する

### 質問運用ルール

**1問1論点**:
- 1回の質問で1つの未決分岐のみを扱う
- 複数の論点を含む質問は分割する

**evidence-first**:
- ユーザーへ質問する前に、以下の証拠で解決可能か確認する:
  - 明示入力ファイル
  - 関連REQ
  - 関連ADR
  - specs（system.md / patterns.md）
  - command/skill定義
- 証拠で解決できる事項はユーザーに質問しない

### 回答分類と反映

**分類**:
壁打ち結果を以下の5種に分類する:

| 分類 | 説明 |
|------|------|
| Confirmed | ユーザーが明示的に確認・承認した事項 |
| Inferred | 証拠から推定した事項（ユーザーの明示的確認なし） |
| Unknown | 証拠からも推定できず、ユーザー確認も未完了の事項 |
| User Decision | ユーザーの判断に委ねるべき事項 |
| Out of Scope | 今回の要件化対象外の事項 |

**要件docへの反映**:
- **Confirmed のみ**を要件doc（目的・要件・適用範囲）に反映する
- Inferred / Unknown / User Decision / Out of Scope は要件doc外の壁打ち出力で区別して提示する

**Inferred の取扱い**:
- Inferred を要件doc本文・要件テーブル・適用範囲に混入させない
- 推定事項として根拠付きで壁打ち出力に提示する
- ユーザーが Inferred を明示的に承認または修正した場合のみ、Confirmed に昇格して要件docへ反映する

**射影ルール**:
壁打ち結果を以下の3セクションに射影する:

| 射影先 | 対象分類 |
|--------|----------|
| 目的 | Confirmed のうち、要件の存在理由・背景・狙い |
| 要件（テーブル） | Confirmed のうち、検証可能な必達要件として表現される個別要件 |
| 適用範囲（対象/対象外） | Confirmed のうち、スコープ境界に関する事項 |

### Unknown の取扱い

**残存禁止**:
- 要件成立に必要な Unknown が残存している場合、壁打ちを終了しない
- 以下のいずれかで処分する:
  - 追加調査による解決
  - ユーザーへの判断依頼
  - 要件化対象外への移行（Out of Scope への再分類）

**doc混入禁止**:
- Unknown を要件doc本文・要件テーブル・適用範囲の対象に混入させない

**Out of Scope 再分類**:
- Unknown が今回の要件化対象外と判断された場合、Unknown のまま残さず Out of Scope に再分類する

### 既存メソドロジーの再構造化

未決分岐解消を親概念とし、既存の壁打ちメソッドを再構造化する。

#### 未決分岐の抽出（深掘り）

ユーザーの真の意図 vs 表面的な要望を区別し、未決分岐を抽出する。

- 表面的な要望: 具体的な解決策の提示（「ボタンを追加して」）
- 真の意図: 解決したい課題・実現したい価値（「効率的にデータを入力したい」）

「5回のなぜ」を用いて根本的な目的を明らかにし、未決分岐の観点（目的・対象・対象外・粒度・永続化・責務境界・判定基準）に沿って整理する。

#### 未決分岐の解消（合意形成）

未決分岐を解消し、曖昧さゼロ（全項目が測定可能・一意・実装可能）の状態に到達する。

- **測定可能**: 完了判定が客観的にできる
- **一意**: 解釈の余地がない
- **実装可能**: 開発者がコードに落とし込める

あいまいな表現（「適切に」「効率的に」等）を具体化し、回答分類（Confirmed / Inferred / Unknown / User Decision / Out of Scope）に従って処理する。

#### 確定判定（ステークホルダー視点）

開発者・運用者・エンドユーザーの各観点から、未決分岐の解消結果を確定判定する。

- **開発者**: 技術的実現可能性・工数・リスク
- **運用者**: 監視・トラブルシューティング・変更の容易さ
- **エンドユーザー**: UX・操作性・学習コスト

異なる視点からの質問・懸念を洗い出し、全ステークホルダーの観点で合意形成を確認する。

### 既存REQ/ADRの定量的照合

文書記載の番号レンジ（例: AGENTS.md の "REQ-0101〜REQ-0133"）を鵜呑みにせず、実ファイル列挙により active REQ/ADR を照合する。文書記載のみに依存すると、Issue #786（行数前提の誤り）や REQ-0123 の見落とし等のリスクを生む。

**列挙コマンド**:
- `glob docs/requirements/REQ-*.md` — active REQ ファイルを列挙
- `glob docs/adr/ADR-*.md` — ADR ファイルを列挙

**乖離解消手順**:
1. 文書レンジと実ファイル一覧に乖離がある場合、原因を調査する
2. 実ファイルの追加・削除が文書に反映されていない場合 → 文書を修正する
3. 文書のレンジ記述が誤っている場合 → 文書を修正する

REQ レンジ・ADR レンジの双方に適用する。詳細な照合方法論（CREATE/APPEND/UPDATE 評価等）は `agentdev-req-file-manager` を参照。

### 責務境界

壁打ちメソッドロジーの責務範囲を明確にする。

**扱わないこと**:
- REQファイルの採番、CREATE/APPEND/UPDATE、frontmatter更新、README更新（→ `agentdev-req-file-manager`）
- 実装計画、タスク分割、コード変更方針の確定（→ `/agentdev/case-run` の work plan）
- ADRファイルの作成・更新（→ `agentdev-adr-file-manager`）。ADR閾値に達する可能性がある判断候補の抽出は行うが、ファイル操作は扱わない

## チェックボックス品質基準

Issue本文のチェックボックスが満たすべき品質条件。

### 測定可能

完了/未完了が客観的に判定できること。

- ❌ 「処理を最適化する」
- ✅ 「レスポンス時間を200ms以下にする」

### 一意

1つのチェックボックス = 1つの要件。複合要件は分割する。

- ❌ 「認証機能の実装とテスト」
- ✅ 「認証APIの実装」「認証フローのテスト」

### 実装可能

1つのチェックボックス = 1つのPRで実装可能な粒度。

- ❌ 「システム全体のリファクタリング」
- ✅ 「ユーザーモジュールのリファクタリング」「決済モジュールのリファクタリング」

## ADR閾値判定ブリッジ

技術判断がADR閾値に達するかの基準。

### ADRが必要な判断

- アーキテクチャ変更: システム全体の構造・主要コンポーネント間の関係に影響
- 複数システム影響: 技術スタック選択・データモデル・認証スキームなど
- 長期間有効な決定: 将来の開発・運用に長期的影響を与える
- 取り返しがつかない変更: 後で変更するコストが非常に高い・困難

### ADR閾値未満

- 単一モジュール内の実装選択
- ライブラリのマイナーバージョン選択
- コーディングスタイル・命名規約

### ADR参照

ADRが必要と判定された場合 → `agentdev-adr-guidelines` スキルを参照する。ADR候補記録までを責務とし、ADRファイル作成は req-save + agentdev-adr-file-manager に委譲する。
