# 要件分析スキル

要件分析のための**知識ベース**。
分析観点、品質基準、定義を提供し、壁打ちフェーズでの壁打ち品質を担保する。

- **このスキル（知識）**: 分析観点、品質基準、定義
- **適用先**: 要件定義プロセス全般（壁打ち、Issue作成、実装検証）

---

## skill read contract 参照方針

本スキルは以下の方針に従う（ADR-0133）。

1. プロジェクト固有の `docs/specs/**` 内部構成を仮定しない
2. 呼び出し元コマンドから渡された解決済み文脈を優先する
3. 解決済み文脈がなく skill read contract が存在する場合のみ読む
4. read contract に列挙されていない `docs/specs/**` 内部パスを固定知識として参照しない
## 要件分析

機能要件、非機能要件の区別なく、統一的に要件を展開する際の観点。

### ユーザーストーリー

「[誰]が[何]を[なぜ]する」形式で記述する。

- 明確な主体（ユーザー、システム、外部サービス等）
- 具体的なアクション
- ビジネス価値、目的

### 完了条件

Given-When-Then形式で記述し、測定可能であること。

- **Given**: 事前条件、前提状態
- **When**: 実行するアクション、イベント
- **Then**: 期待される結果、状態変化

客観的な完了判定が可能であること。

### 境界条件

正常系、異常系、エッジケースを網羅する。

- **正常系**: 期待されるフロー
- **異常系**: エラー、例外、拒否パターン
- **エッジケース**: 境界値、空入力、同時実行等

### 必達要件記述ガイダンス

要件行は検証可能な必達要件として記述する。
推奨、任意、将来候補は要件行に含めない。

**記述ルール**:
- 各要件行は「満たす必要がある検証可能な要件」として記述する
- 禁止、停止条件、完了不可条件は自然文で記述する（「〜禁止」「〜停止する」「〜完了扱いにしない」等）
- 曖昧な表現（「適切に」「効率的に」等）は、具体的な測定可能な条件に置き換える

### 状態要件と反映作業の分離基準

要件行候補を「状態要件（変更後仕様）」と「反映作業」に分類する判断基準（050）。

**状態要件（要件行として採用）**:
- システムが満たすべき振る舞い、制約、状態の宣言
- 「〜すること」「〜を禁止」「〜しない」等の検証可能な記述
- 変更後に満たされるべき条件

**反映作業（要件行として不採用、移送候補）**:
- 既存成果物への操作: 更新、削除、移動、名称変更、参照修正、移行、再定義、改名、移管、除去、廃止、置換
- 作業手順の記述（ステップバイステップの実行指示）
- 既存文書の修正内容そのもの

**混合候補の処理**:
- 状態要件の記述を含み、かつ反映作業も含む場合 → 要件行として採用（反映作業部分は移送候補としてマーク）
- 反映作業のみの場合 → 要件行として不採用、後続工程への移送候補として振り分け

**移送先**:
- 反映作業は case/ Issue/ 受け入れ条件/ 作業記録で扱う
- 現行 REQ の title/ 目的/ 要件行/ 適用範囲に作業手段語を混入させない（007）

### REQ/SPEC 境界判定基準

状態要件と判定された要件行候補を、REQ に記述する内容と SPEC 等に配置する内容に振り分ける判断基準（-069, -055）。
「状態要件と反映作業の分離基準」と併用する。

**REQ に記述する内容（外部契約、状態要件）**:
- システム、コマンドが満たすべき状態、振る舞い、制約の宣言
- 外部から見える責務、公開 command 名、公開入口
- ドメイン状態の位置づけ、他 command との接続契約
- 安全境界、停止条件の大枠、利用者に見える分類体系
- 恒久的な文書体系制約

**SPEC 等（SPEC/ rule catalog/ command reference/ skill reference/ test docs）に配置する内容（詳細、内部パラメータ）**:
- schema field、enum 値一覧、route/category/status 詳細判定表、file pattern、template variant、report format
- fixture detail、regression test 条件、checker 個別ルール、false positive 抑制方式
- retry 回数、token 目安、行数上限、Step 番号、Phase 番号、内部アルゴリズム
- 作業履歴、Case/RU/Issue/PR/OU 由来の作業記録

**安定契約例外**:
要件行候補がパラメータ、分類、値の形式をとる場合であっても、公開 command 名、公開入口、ドメイン状態の位置づけ、他 command との接続契約、利用者に見える分類体系、安全境界、停止条件の大枠、後続工程が依存する安定した外部契約のいずれかに該当する場合は、REQ に要約として記述する。
詳細な値一覧、判定表、内部処理条件は SPEC 等に配置する。

**必要性テスト**:
「当該行が存在しない場合、対象 REQ が何を満たすべきか不明になるか？」を問う。
不明になる場合は REQ 行として残し、対象 REQ の必達要件の理解に寄与しない場合は SPEC 等へ移送する。

**記述形式**:
主たる文意は肯定文で記述する（-066 準拠）。
否定文は境界条件、例外、補足として併記する場合に限る。

**判定タイミング**:
req-define/ req-save は保存前に REQ/SPEC 配置判定を行う。
QG-1 の観点 6（REQ/SPEC 配置判定）がこれを担保する。

**SPEC 候補の分離経路**:
req-define は SPEC 等に配置すべきと判定した要件行候補を REQ 要件行に残留させず、ドラフト内の `## SPEC候補` 補助セクションと `draft-meta.spec-candidates`（SC-ID、content、intended_spec、classification、source）に分離する。
各 SPEC 候補には想定配置先 SPEC（既存 SPEC ファイル名または新規 SPEC 候補）と分離根拠（の該当種別）を記録する。
この分離経路により、SPEC 相当行が REQ ファイルへ混入することを構造的に防止する。
req-save は REQ ファイル保存時に SPEC候補セクションを除去し、内容は spec-save が `docs/specs/*.md` へ保存する。

### 用語定義

要件内の専門用語を明確化する。

- ドメイン固有の用語
- 技術的な前提条件
- 業務用語、略語の統一定義

## 壁打ちメソドロジー

要件確定前の未決分岐を証拠ベースで解消し、壁打ち品質を担保するメソッドロジー。
未決分岐解消を親概念とし、既存の深掘り、合意形成、ステークホルダー視点を再構造化する。

req-define Step1 のセッションコンテキスト検知の詳細手順は `references/session-context-detection.md` を参照。

### 未決分岐の抽出と整理

壁打ち中の論点を未決分岐として扱い、体系的に整理する。

**観点**:
- 目的、対象、対象外、粒度、永続化、責務境界、判定基準

**依存順解消ルール**:
- 優先順: 目的 → 対象 → 対象外 → 責務境界 → 粒度 → 永続化 → 判定基準
- 既存資料（明示入力ファイル、関連REQ、関連ADR、specs、command/skill定義）により上位分岐が確定済みの場合は、未確定の最上位分岐から開始する

### 質問運用ルール

**1問1論点**:
- 1回の質問で1つの未決分岐のみを扱う
- 複数の論点を含む質問は分割する

**evidence-first**:
- ユーザーへ質問する前に、以下の証拠で解決可能か確認する:
 - 明示入力ファイル
 - 関連REQ
 - 関連ADR
 - specs（system.md/ patterns.md）
 - command/skill定義
- 証拠で解決できる事項はユーザーに質問しない

### 回答分類と反映

**分類**:
壁打ち結果を以下の5種に分類する:

| 分類 | 説明 |
|------|------|
| Confirmed | ユーザーが明示的に確認、承認した事項 |
| Inferred | 証拠から推定した事項（ユーザーの明示的確認なし） |
| Unknown | 証拠からも推定できず、ユーザー確認も未完了の事項 |
| User Decision | ユーザーの判断に委ねるべき事項 |
| Out of Scope | 今回の要件化対象外の事項 |

**要件docへの反映**:
- **Confirmed のみ**を要件doc（目的、要件、適用範囲）に反映する
- Inferred/ Unknown/ User Decision/ Out of Scope は要件doc外の壁打ち出力で区別して提示する

**Inferred の取扱い**:
- Inferred を要件doc本文、要件テーブル、適用範囲に混入させない
- 推定事項として根拠付きで壁打ち出力に提示する
- ユーザーが Inferred を明示的に承認または修正した場合のみ、Confirmed に昇格して要件docへ反映する

**射影ルール**:
壁打ち結果を以下の3セクションに射影する:

| 射影先 | 対象分類 |
|--------|----------|
| 目的 | Confirmed のうち、要件の存在理由、背景、狙い |
| 要件（テーブル） | Confirmed のうち、検証可能な必達要件として表現される個別要件 |
| 適用範囲（対象/対象外） | Confirmed のうち、スコープ境界に関する事項 |

### Unknown の取扱い

**残存禁止**:
- 要件成立に必要な Unknown が残存している場合、壁打ちを終了しない
- 以下のいずれかで処分する:
 - 追加調査による解決
 - ユーザーへの判断依頼
 - 要件化対象外への移行（Out of Scope への再分類）

**doc混入禁止**:
- Unknown を要件doc本文、要件テーブル、適用範囲の対象に混入させない

**Out of Scope 再分類**:
- Unknown が今回の要件化対象外と判断された場合、Unknown のまま残さず Out of Scope に再分類する

### 既存メソドロジーの再構造化

未決分岐解消を親概念とし、既存の壁打ちメソッドを再構造化する。

#### 未決分岐の抽出（深掘り）

ユーザーの真の意図 vs 表面的な要望を区別し、未決分岐を抽出する。

- 表面的な要望: 具体的な解決策の提示（「ボタンを追加して」）
- 真の意図: 解決したい課題、実現したい価値（「効率的にデータを入力したい」）

「5回のなぜ」を用いて根本的な目的を明らかにし、未決分岐の観点（目的、対象、対象外、粒度、永続化、責務境界、判定基準）に沿って整理する。

#### 未決分岐の解消（合意形成）

未決分岐を解消し、曖昧さゼロ（全項目が測定可能、一意、実装可能）の状態に到達する。

- **測定可能**: 完了判定が客観的にできる
- **一意**: 解釈の余地がない
- **実装可能**: 開発者がコードに落とし込める

あいまいな表現（「適切に」「効率的に」等）を具体化し、回答分類（Confirmed/ Inferred/ Unknown/ User Decision/ Out of Scope）に従って処理する。

#### 確定判定（ステークホルダー視点）

開発者、運用者、エンドユーザーの各観点から、未決分岐の解消結果を確定判定する。

- **開発者**: 技術的実現可能性、工数、リスク
- **運用者**: 監視、トラブルシューティング、変更の容易さ
- **エンドユーザー**: UX、操作性、学習コスト

異なる視点からの質問、懸念を洗い出し、全ステークホルダーの観点で合意形成を確認する。

### 既存REQ/ADRの定量的照合

文書記載の番号レンジ（例: AGENTS.md の "〜"）を鵜呑みにせず、実ファイル列挙により現行 REQ/ADR を照合する。
文書記載のみに依存すると、Issue #786（行数前提の誤り）や の見落とし等のリスクを生む。

**列挙コマンド**:
- `glob docs/requirements/REQ-*.md`（現行 REQ ファイルを列挙）
- `glob docs/adr/ADR-*.md`（ADR ファイルを列挙）

**乖離解消手順**:
1. 文書レンジと実ファイル一覧に乖離がある場合、原因を調査する
2. 実ファイルの追加、削除が文書に反映されていない場合 → 文書を修正する
3. 文書のレンジ記述が誤っている場合 → 文書を修正する

REQ レンジ、ADR レンジの双方に適用する。
詳細な照合方法論（CREATE/APPEND/UPDATE 評価等）は `agentdev-req-file-manager` を参照。

### explore 委譲スコープの絞り込み（REQ-0102-072）

req-define Step 5-1（変更影響候補抽出）で explore 委譲へ渡す調査スコープを、決定的前処理で絞り込む。
RU の frontmatter、本文から対象領域キーワードを抽出し、glob/grep で関連 REQ/ADR/SPEC を事前特定して、調査優先対象リスト（priority targets）を explore 委譲へヒントとして渡す。

**ヒントでありハードフィルタではない**。
絞り込みは explore 委譲の調査優先対象リストのみに適用し、リスト外のファイルを調査対象から除外しない。
REQ-0102-002 が要求する実ファイル完全列挙は維持し、本前処理が完全列挙を代替、省略しない。

キーワード抽出、glob/grep 前処理、調査優先対象リスト構築、explore 委譲への引き渡し契約の詳細は `references/explore-scope-refinement.md` を参照。

### REQ 健全性メトリクスと SPLIT 予兆検知

req-define は既存 REQ（Step 3-2）および生成ドラフト（Step 10-2）の健全性を、req-health-metrics SPEC の定量閾値で計測し SPLIT 予兆として `draft-meta.split-forecast` に記録する。

**計測メトリクス**（計測対象は要件テーブル行 `^| REQ-NNNN-MMM |` のみ。目的、適用範囲の散文は除外）:

| メトリクス | 計測方法 |
|---|---|
| 要件行数 | `^| REQ-NNNN-\d{3} |` に一致する行数 |
| 関心分類数 | 後述4シグナル（関心対象、成果物種別、command family、lifecycle 段階の複数混在）の検出数 |
| 成果物種別数 | `req-impact-map.md` の「影響する Artifact」列、または要件本文の対象成果物記述から集計 |

**SPLIT シグナル閾値**（詳細は req-health-metrics SPEC を正とする）:

| メトリクス | 閾値 | シグナル |
|---|---|---|
| 要件行数 | 0〜50/ 51〜80/ 81+ | +0/ +1/ +2 |
| 関心分類数 | 0〜1/ 2+ | +0/ +1 |
| 成果物種別数 | 1〜2/ 3+ | +0/ +1 |
| SPEC分離基準違反（high-specificity） | 0/ 1+ | +0/ +1（安定契約例外 は対象外） |

**推奨アクション**: 合計シグナル 0〜1 → APPEND 許可、2 → SPLIT 検討（ユーザーへ提案）、3+ → SPLIT 推奨。

`split-forecast` は計測対象（既存 REQ ID または `draft`/`new`）、各メトリクス値、各シグナル値、合計、推奨アクション、`thresholds_ref: req-health-metrics SPEC` を構造化して記録する。
計算式の詳細、ドラフト適用時の注意は `references/req-define-detailed-gates.md`「SPLIT 予兆計算」を参照。

### APPEND 操作での precedent 利用

APPEND 操作で要件行を追加する際、既存 REQ ファイルに同種の要件行が存在する場合は precedent として書式を機械的に再利用することを推奨する。

- **対象操作**: 廃止宣言、ステータス変更、相互参照、置換（supersede）宣言
- **再利用方法**: precedent 行の書式を読み取り、相違点（対象 REQ 番号、理由、移行先など）のみ調整する
- **位置付け**: 推奨（必須ではない）。書式設計のコストを削減し、REQ 間の表記揺れを防ぐ
- **注意**: 特定の precedent を「唯一の正解」として固定化しない。あくまで出発点としての参照用である

### 責務境界

壁打ちメソッドロジーの責務範囲を明確にする。

**扱わないこと**:
- REQファイルの採番、CREATE/APPEND/UPDATE、frontmatter更新、README更新（→ `agentdev-req-file-manager`）
- 実装計画、タスク分割、コード変更方針の確定（→ `/agentdev/case-run` の work plan）
- ADRファイルの作成、更新（→ `agentdev-adr-file-manager`）。ADR閾値に達する可能性がある判断候補の抽出は行うが、ファイル操作は扱わない

## チェックボックス品質基準

Issue本文のチェックボックスが満たすべき品質条件。

### 測定可能

完了/未完了が客観的に判定できること。

- ❌ 「処理を最適化する」
- ✅ 「レスポンス時間を200ms以下にする」

### 一意

1つのチェックボックス = 1つの要件。
複合要件は分割する。

- ❌ 「認証機能の実装とテスト」
- ✅ 「認証APIの実装」「認証フローのテスト」

### 実装可能

1つのチェックボックス = 1つのPRで実装可能な粒度。

- ❌ 「システム全体のリファクタリング」
- ✅ 「ユーザーモジュールのリファクタリング」「決済モジュールのリファクタリング」

## ADR閾値判定ブリッジ

技術判断がADR閾値に達するかの基準。

### ADRが必要な判断

- アーキテクチャ変更: システム全体の構造、主要コンポーネント間の関係に影響
- 複数システム影響: 技術スタック選択、データモデル、認証スキームなど
- 長期間有効な決定: 将来の開発、運用に長期的影響を与える
- 取り返しがつかない変更: 後で変更するコストが高い、困難

### ADR閾値未満

- 単一モジュール内の実装選択
- ライブラリのマイナーバージョン選択
- コーディングスタイル、命名規約

### ADR参照

ADRが必要と判定された場合 → `agentdev-adr-guidelines` スキルを参照する。
ADR候補記録までを責務とし、ADRファイル作成は req-save + `agentdev-adr-file-manager` に委譲する。
