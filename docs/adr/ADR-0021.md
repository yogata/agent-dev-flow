## Context

AgentDevFlow consumer projects may discover bugs or improvements in AgentDevFlow itself during their daily workflow.
Without a formal protocol, consumers might attempt to modify AgentDevFlow artifacts directly, creating divergence
between the consumer copy and the upstream repository. Six or more commands (req-define, backlog-review,
case-open, case-run, case-close) need to detect upstream handoff situations and stop local execution accordingly.
Duplicate long-form rules across commands degrades maintainability and increases the risk of inconsistent behavior.

## Decision

1. Adopt `apply_in_current_project` and `handoff_target` as RU frontmatter metadata fields for upstream handoff
   identification. Default values: `handoff_target: agent-dev-flow`, `apply_in_current_project: false`.
2. Commands that detect `apply_in_current_project: false` in requirement documents, RUs, or Issue bodies must stop
   local execution (Issue creation, implementation) and report the item as a manual upstream handoff candidate.
3. Centralize common upstream handoff policy in
   `agentdev-workflow-lifecycle/references/upstream-handoff.md` so commands reference a single source of truth
   rather than embedding long-form duplicate rules.
4. The agent-dev-flow repository itself is exempt from handoff stops because AgentDevFlow artifacts are its own
   deliverables (normal req/case workflow applies).

## Consequences

### Positive

- Prevents direct AgentDevFlow artifact modification in consumer projects
- Clear upstream reporting path with structured metadata
- Single source of truth for handoff policy; updating one reference file covers all commands
- Commands remain thin: each only needs a stop-condition check and a reference to the shared policy

### Negative

- Consumer improvement requests depend on upstream response time
- Additional metadata fields in RU frontmatter require tooling awareness
