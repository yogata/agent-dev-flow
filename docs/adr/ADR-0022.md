# ADR-0022: review/refine 系中間コマンドを promote 内フェーズへ統合

## 背景

intake / learning パイプラインには中間確認コマンドとして `intake-review` と `learning-refine` が存在した。ユーザーは review → promote または refine → promote の 2 ステップを踏む必要があり、「どのコマンドを使うべきか」の迷いが発生していた。HITL（Human-in-the-Loop）確定はプロモーションの最終ステップとして自然な位置づけであり、AI 単独確定の禁止は promote 内のフェーズ境界で強制できる。

## 決定

以下の変更を行う:

1. `intake-review` を廃止し、`intake-promote` 内に review 相当の採用・却下・保留判定フェーズを統合する
2. `learning-refine` を廃止し、`learning-promote` 内に normalize / classify / 8-axis evaluation フェーズを統合する
3. `.agentdev/intake/accepted/` ディレクトリを廃止し、inbox → intake-promote → promoted/archive に一本化する
4. HITL 確定ステップを各 promote コマンド内に集約し、AI 単独での最終確定を禁止する

## 代替案

### 代替案 A: review / refine を残したまま HITL を追加

review / refine コマンドを維持しつつ、各コマンドに HITL 確定を追加する。コマンド数は増えないが、2 ステップ運用の複雑さは残る。

**却下理由**: ユーザーの「どのコマンドを使うべきか」という迷いを解消できない。

### 代替案 B: review / refine に compatibility period を設ける

review / refine を deprecated とし、移行期間を設けてから promote に統合する。

**却下理由**: 2 つの並行経路が存在する期間中、ユーザーと agent の両方に混乱が生じる。新旧パスの整合性維持コストが大きい。

## 結果・影響

- **コマンド数削減**: `intake-review` と `learning-refine` が削除され、パイプラインが単純化される
- **HITL 集約**: 確定ポイントが promote コマンドに集約され、ユーザーは promote のみを意識すればよい
- **データフロー単純化**: `accepted/` 経由の中間状態がなくなり、inbox → promoted/archive の直接フローになる
- **SPEC 影響**: `workflow-contracts.md`、`system.md`、`integrity-contracts.md` の更新が必要
- **ADR 影響**: ADR-0010（HITL boundary）の対象コマンド一覧更新、ADR-0003 / ADR-0005 のコンテキスト更新
