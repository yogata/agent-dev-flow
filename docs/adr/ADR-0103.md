# ADR-0103: 文書種別責務境界

## 背景

REQ/ADR/SPEC/Guides/DOC-MAP の責務境界が曖昧だった。SPEC に「なぜそう決めたか」が混入し、ADR に「現在の仕様」が記述されることがあった。Guides が規範的権限を持つかのように読める記述も存在した。ADR-0014 は ADR/SPEC 再分類基準を定めたが、文書種別全体の責務境界を包括的に宣言していなかった。

さらに、ADRの主題として「削除・廃止・移行・完全削除」などの作業手段を含めたADR（ADR-0110, ADR-0113等）がacceptedとして作成される問題があった。作業手段は意思決定ではなく実行手段であり、ADRの主題として不適切である。同様に、active REQでも移行・改名・削除などの作業手段が要件行として混入する問題があった。

## 決定

以下の文書種別責務境界を定める:

- **REQ**: 変更後に満たすべき振る舞い・制約・状態を定義する。現行判断の第一参照。規範的権限を持つ
- **ADR**: 将来の設計・運用・文書システムを制約する決定の記録。`accepted` のみ現行根拠。`proposed`/`deprecated`/`superseded` は歴史的参照 <!-- REQ-0101 -->
- **SPEC**: `docs/specs/` は repo-internal 設計文書。現在仕様の基準。agent-dev-flow レポジトリの管理文書（runtime 配布物の依存先は ADR-0104 で分離）
- **Guides**: 人間向け navigation 層。必達要件・規範表現は REQ/ADR/SPEC に配置する
- **DOC-MAP**: 探索経路インデックス。REQ/ADR/SPEC 各文書への参照経路を提供する

各文書種別の記述対象は以下の境界に従う:

- **ADR** の主題は『あるべき状態』の意思決定と理由の記録とし、削除・廃止・移行・統合・再構築・完全削除という作業手段は case/Issue で扱う
- **REQ** は現在満たすべき状態・振る舞い・制約を要件行として定義し、作業手段（移行・改名・削除・除去等）は case/Issue で扱う
- **作業手段**（移行・改名・削除等）の配置先は case / Issue / PR / 作業記録とする

過去の判断を現行基盤から外すだけの場合は、対象ADRのstatus遷移（retire / supersede）で処理する。新規ADRを作成できるのは、旧判断を置き換える新しい「あるべき状態」の意思決定が存在する場合のみである。

ADR作成前に既存accepted ADRとの内容重複を確認し、重複時は既存ADRのUPDATEを優先する。

## 代替案

1. 各文書種別に「第一参照」「第二参照」の順位付け → 順位より責務の排他性が重要と判断
2. Guides に軽い規範的権限を持たせる → REQ/ADR と重複するため却下
3. SPEC を runtime 配布物に含める → agent-dev-flow 内部文書であり配布不要

## 結果・影響

- SPEC は agent-dev-flow レポジトリ専用管理文書として扱う
- Guides に 必達要件 規範表現を含めない
- ADR-0014 の範囲を核判断に縮小し、詳細仕様は SPEC/REQ へ委譲
- ADRの適用範囲は、技術判断に限定されず、将来の設計・運用・文書システムを制約する決定を含む（REQ-0101）。
- docs-check が Guides の規範表現混入を検出可能
- accepted ADR で作業手段（削除・廃止・移行・完全削除）を主題にしたものは retire / supersede または「あるべき状態」の意思決定へ再構成の対象
- req-define の ADR 作成可否ゲートが作業手段 ADR を拒否する（REQ-0101-047）
- integrity / docs-check が ADR 主題妥当性（作業手段混入・retired ADR の現行基盤参照・index整合性）を検出する（REQ-0108-249-251）
