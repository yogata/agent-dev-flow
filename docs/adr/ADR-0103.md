# ADR-0103: 文書種別責務境界

## 背景

REQ/ADR/SPEC/Guides/DOC-MAP の責務境界が曖昧だった。SPEC に「なぜそう決めたか」が混入し、ADR に「現在の仕様」が記述されることがあった。Guides が規範的権限を持つかのように読める記述も存在した。ADR-0014 は ADR/SPEC 再分類基準を定めたが、文書種別全体の責務境界を包括的に宣言していなかった。

## 決定

以下の文書種別責務境界を定める:

- **REQ**: 変更後に満たすべき振る舞い・制約・状態を定義する。現行判断の第一参照。規範的権限を持つ
- **ADR**: 取り返しのつかない技術判断の記録。`accepted` のみ現行根拠。`proposed`/`deprecated`/`superseded` は歴史的参照
- **SPEC**: `docs/specs/` は repo-internal 設計文書。現在仕様の基準。runtime 配布物の依存先ではなく、agent-dev-flow レポジトリの管理文書
- **Guides**: 人間向け navigation 層。規範的権限を持たない。MUST/SHALL 規範表現を含めない
- **DOC-MAP**: 探索経路インデックス。REQ/ADR/SPEC の内容を代替しない

## 代替案

1. 各文書種別に「第一参照」「第二参照」の順位付け → 順位より責務の排他性が重要と判断
2. Guides に軽い規範的権限を持たせる → REQ/ADR と重複するため却下
3. SPEC を runtime 配布物に含める → agent-dev-flow 内部文書であり配布不要

## 結果・影響

- SPEC は agent-dev-flow レポジトリ専用管理文書として扱う
- Guides に SHALL/MUST 規範表現を含めない
- ADR-0014 の範囲を核判断に縮小し、詳細仕様は SPEC/REQ へ委譲
- integrity-check が Guides の規範表現混入を検出可能
