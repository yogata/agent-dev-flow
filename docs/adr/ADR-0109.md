# ADR-0109: Epic Issue 本文を実行順序 SSoT とする設計

## 背景

Epic 内の子 Issue に実行順序・前提関係があるが、現状は人間が手動で順番を管理している。実行順序をコード化する方法として、以下の3案が考えられる:

- (A) Epic Issue 本文の Wave テーブルを SSoT とする
- (B) 別設定ファイル（YAML/JSON等）で管理する
- (C) 依存関係分析のみに頼る（明示的な順序定義なし）

## 決定

Epic Issue 本文の `## 実行順序` セクション内 Wave テーブルを実行 SSoT とする。

理由:
1. GitHub Issue が既に SSoT の置き場所として機能している
2. 人間とエージェントの両方が同じ場所を参照できる
3. 別ファイル管理は同期コストが高い
4. 依存関係分析のみでは意図を反映できない（暗黙の順序を推測することになる）

## 代替案

### (B) 別設定ファイルで管理

YAML/JSON で実行順序を定義する案。利点は機械可読性と検証の容易さだが、GitHub Issue との二重管理が発生し同期コストが高い。またローカルファイルとリモート Issue の不整合リスクがある。

### (C) 依存関係分析のみに頼る

LLM による依存関係分析から動的に Wave を構成する案。利点は明示的な定義が不要なことだが、人間の意図する順序と LLM の推測が一致する保証がない。特に「並列可能だが直列で実行したい」意図を反映できない。

## 結果・影響

- case-open は Epic Issue 本文に `## 実行順序` セクションを生成する
- case-run は Epic Issue を入力とした場合、本文から Wave テーブルを読み取ってオーケストレータとして動作する
- 既存の `## 分解（Decomposition）` テーブルと `## ステータス追跡` テーブルは変更しない
- 子 Issue 番号直接指定の既存動作は維持される
