# ADR-0009: REQ体系再基準化 — 旧REQ分類モデル・対応表・分類ゲート導入

## 背景

現行REQ群（REQ-0001〜REQ-0040, retired）には、現行仕様・旧仕様・既存成果物への反映作業・実装分解・個別ファイル修正指示が混在している。実装者が旧仕様をactive要件として読むリスクがあり、req-define / req-save / case-run の各工程で入力品質が担保されていない。

また、REQ-0023 (retired)（learning staging stub の取り込み追跡と取り込み後アーカイブ）と REQ-0039 (retired)（req-backlog/RU削除モデル）の間で promoted artifact の扱いが衝突しており、再基準化で解消が必要である。

## 決定

1. **旧REQを履歴として保持し、新基準REQ群を別途作成する**: 既存REQファイルを削除せず、3分類（retained / partially superseded / superseded）で管理する。新基準REQ群は現行AgentDevFlowの仕様基準として、現行command・domain state・workflow・artifact lifecycle を基準に記述する。

2. **3分類モデルを導入する**:
   - `retained`: 旧REQのID・本文をそのまま現行基準として維持
   - `partially superseded`: 旧REQの一部だけを後継REQへ移し、残りは履歴または限定的基準として残す
   - `superseded`: 旧REQを現行基準として読まず、後継REQまたは履歴参照に置き換える

3. **旧REQ↔新基準REQの対応表を作成する**: 旧REQごとに継承・置換・廃止・未継承の範囲を明記し、反映作業をUPDATE/APPEND/関連ドキュメント更新候補に分離する。

4. **req-define / req-save に分類ゲートを追加する**: 要件行候補が変更後仕様か既存成果物への反映作業かを分類し、反映作業のみを表す候補を新規REQの独立要件行として扱わない。

5. **REQ-IDの数値形式を統一する**: `REQ-{NNNN}-{MMM}` の数値形式に統一し、英字suffix（REQ-0041-A等）を新規に使用しない。

## 代替案

### A: 既存REQをインプレース更新

旧REQの本文を直接書き換えて現行仕様に合わせる。履歴が失われるリスクと、更新中の一貫性維持コストが高い。

**不採用理由**: 履歴保持（REQ-0041-001, retired）の要件に反する。git履歴はあるが、要件としての履歴管理とは異なる。

### B: 旧REQをすべてsupersededに統一

partially supersededを設けず、retainedかsupersededの2分類にする。

**不採用理由**: 多くの旧REQ（19件）が一部だけ現行仕様として有効であり、全面的supersededは情報損失になる。partially superseded は後継REQへの移行範囲を明示するために必要。

### C: 反映作業を要件行に含める

新基準REQの要件行に反映作業を記述し、実装者が一覧性を持てるようにする。

**不採用理由**: 反映作業は要件ではなく作業指示であり、要件行に混入すると読み手保護に反する（REQ-0041-004, retired）。

## 結果・影響

- 新基準REQ群（REQ-0101〜0109）が現行仕様の基準として機能する
- 旧REQは履歴・参照用として保持され、実装者が旧仕様を誤読するリスクが減少する
- req-define / req-save に分類ゲートが追加され、以後の要件定義で反映作業の混入が防止される
- REQ-0023 (retired) / REQ-0039 (retired) の promoted/RU lifecycle 衝突が対応表を通じて解消される
- README / DOC-MAP が新基準REQ群を探索入口として示す構造に再設計される
- 旧command名・旧directory名・旧workflow名がactive仕様として残らない
