# ADR-0129: 複数 execution_unit 並列実行モデル（複数 SSoT 並立と Epic 間並列 orchestration）

## 背景

複数RU群をバッチ処理する要件で、関連しないRU群が不適切に1つの Epic に集約される問題（REQ-0114-088 旧版「独立 OU → 1 Epic」）を解消するため、case-open が連結成分ベースで複数 Standard/Epic を生成し、case-auto が複数 execution_unit を並列実行するモデルを採用する。

この変更は ADR-0109（Epic Issue 本文 = 実行順序 SSoT）の単一 SSoT 前提を「複数 SSoT 並立」へ拡張し、ADR-0125（Wave 内並列子Issue実行）の並列モデルを「Epic 間並列」へ拡張する構造的変更である。

Oracle 助言（bg_11f2d958, High 確信度）に基づき、ADR-0125 UPDATE ではなく新規 ADR とした（scope が直交する別次元のため）。

## 決定

- 複数 execution_unit（standard issue | epic issue）が並立可能。各 Epic は独立した SSoT（ADR-0109 原則の per-Epic 適用）
- case-auto は依存関係のない execution_unit を並列 orchestration する（blocked 単位停止、ready 継続）
- 安全性境界の拡張: Epic Issue 本文の単一書き手は per-Epic-Issue-body で維持（複数 case-close が異なる Epic 本文にそれぞれ書く）
- case-auto レベルでのグローバル並列上限は設定しない（case-run 単位5件上限のみ、REQ-0130-026 踏襲）
- case-run スコープは単一 standard/epic のまま（ADR-0128 変更なし）

### ADR-0129 UPDATE — depends_on 意味の再定義

Decision 追記: depends_on は必須依存（インターフェース契約、データモデル、schema、公開API、検証不能な前提成果物）のみを記録する。
弱依存、関連依存、推奨実行順、同一ファイル変更可能性、同一SPECセクション変更可能性、同一機能領域、クリティカルパス優先度は depends_on から除外し、推奨順は recommended_order に記録する。

従来の「技術的依存＋機能的依存（ヒント）」定義（REQ-0138-019、REQ-0148-002、2026-06-23/24確定）を「必須依存のみ（直列化ゲート）」に再定義する（RU-20260625-01、チャット合意 session:2026-06-25-maximal-parallel-execution）。

理由: accepted SPEC（epic-wave-model.md 連結成分セクション L185-238）は既に新モデル移行済み。
本変更は SPEC 整合を REQ/command へ波及させる作業。

## 代替案

1. ADR-0125 UPDATE: Wave 内並列（同一 Epic 内）の scope と Epic 間並列の scope は直交する別次元のため、単一 ADR の拡張では責務境界が曖昧になる。
新規 ADR で分離する本案を採用。
2. グローバル並列上限の導入: case-auto レベルで N×5 件の同時起動を制限する案。
運用監視で十分という判断（REQ-0148-018、AG-005）により不採用。
3. 技術的依存レベル（L0-L3）を並列可否判定軸に残す: ファイル衝突（L2）があっても worktree 分離により作業自体は並列可能であり、マージコンフリクト解決コストを受容する本案（CR-002）により不採用。

## 結果、影響

ポジティブ:
- 関連しないRU群が不適切な Epic に集約される問題が解消
- 並列実行によりバッチ処理のスループット向上
- blocked 部分停止により、一部失敗が全体停止を引き起こさない

ネガティブ:
- 複数 case-close 並列書き込みによる状態管理の複雑化（per-body 単一書き手は維持されるが、横断追跡負荷増）（技術spike Issue #1294 にて REQ-0137 明示パスステージング + Step 9-1 重複ファイルチェック + G17 `.agentdev/` 限定スコープの3層構造で並列安全性が担保されていることを確認。データ損失、巻き込みコミット、原子性破壊の全てを防止可能。設計変更不要、新規ADR不要）
- グローバル並列上限なしによる N×5 件の task() 同時起動リスク（運用監視必要）
- クロス Epic 実行時依存の発生リスク（連結成分解析は case-open 時点ベース）

## 関連する決定

- relates-to ADR-0109（Epic Issue 本文を実行順序 SSoT とする設計）: SSoT 原則の per-Epic 拡張。各 Epic は独立した SSoT として並立する。
- relates-to ADR-0125（case-auto Wave 内並列子Issue実行モデル）: Wave 内並列の維持、Epic 間並列の追加。Wave 内並列と Epic 間並列は直交する別次元の並列制御である。
- relates-to ADR-0127（case-auto 構成工程の task() 委譲によるスケーラビリティ確立）: case-auto orchestration 拡張。本 ADR は複数 execution_unit 並列 orchestration を追加し、task() 委譲モデルは維持する。
- relates-to ADR-0128（case-run の実行モデル: task() による実行担当サブエージェント委譲）: case-run スコープ維持。case-run は引き続き単一 standard/epic の現在 Wave を実行する。
