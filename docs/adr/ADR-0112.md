# ADR-0112: サブエージェント委譲の一般化と委譲時最小契約

## 背景

ADR-0107（Command/Skill/Template/Script 責任分界）は Command の薄さを定めた。ADR-0111（Manager/orchestrator パターンの限定採用）は manager/orchestrator の適用条件を限定した。しかし以下の未解決問題があった。

1. **文脈圧迫の未解消**: Skill に詳細を分離しても、親エージェントが同じ文脈で Skill を読み込むため、検査観点が親文脈を圧迫する。独立した文脈で完結できる処理を委譲する手段がない。
2. **委譲パターンの限定**: manager-orchesterator が唯一の委譲パターンとされ、軽量な読み取り専用委譲が定義されていなかった。
3. **出力の一律 verbatim 制約**: サブエージェント出力を一律 verbatim とする制約が、中間成果としての採否判断を阻害していた。
4. **Command の肥大化**: 検査観点の詳細が Command に展開され、薄く保つ方針が減衰していた。

## 決定

### 1. サブエージェント委譲の一般概念

サブエージェント委譲を一般概念として定義する。manager-orchestrator は委譲の一形態（特殊型）に位置づける。

### 2. manager-orchestrator の位置づけ

manager-orchestrator は複数サブエージェント統制・Wave scheduling・障害伝播を行う特殊型である（旧 ADR-0111 の採用条件 3 項目を維持）:
1. 複数 Issue の並列・依存管理（Wave scheduling）
2. サブエージェント起動・障害伝播
3. 状態機準の複雑性（再開ポイント、self-healing loop）

軽量な単一タスク委譲は、より軽い委譲形式（後述）を使用する。

### 3. 読み取り専用委譲の許容

読み取り専用委譲は file-pipeline / read-only-diagnostic 実装パターンでも許容する。書き込みを伴わない限り、委譲先の実装パターンを制限しない。

### 4. 親コマンドの最終判断保持

親コマンドは最終判断・副作用（保存、Issue/PR 更新、commit、push、ユーザー確認）・完了報告を保持する。サブエージェントは判断の入力のみを提供し、最終決定は親が行う。

### 5. 委譲時の最小契約

委譲時の最小契約を以下の 4 要素を中心に整理する:

- **`inputs`**: 委譲先に渡す入力（限定されたスコープ）
- **`side_effect_boundary`**: 委譲先の副作用境界（読み取り専用、保存禁止等）
- **`output_contract`**: 委譲先の返却形式（`pass` / `warn` / `fail` / `partial` を基本とし、成果物パス・根拠・capture 候補を含む）
- **`capture_handoff`**: intake / learning 候補を capture 候補として親へ返す形式

委譲時の最小契約は上記 4 要素を中心とする。`delegation_type` / `on_result` は必要な場合のみ、簡潔なラベルや親側の扱いとして記述する。

### 6. 成果物本文の verbatim 扱い

成果物本文（Issue 本文、PR 本文、commit message、保存対象ファイル本文、テンプレート成果物）は verbatim で返す。判定結果、調査過程、中間ログ、読解メモは要約・成果物パス・根拠・親判断事項・capture 候補へ圧縮して返す（出力形式は成果物種別に応じて切替える）。

### 委譲種別の参考分類

委譲種別の参考分類（旧 delegation_type taxonomy）は SPEC（workflow-contracts.md / artifact-contracts.md）に記載する。Command 本文での使用は任意とし、実際の入力・副作用境界・返却内容を分類ラベルより優先する。

### lightweight-delegation の扱い

`lightweight-delegation` は補助的分類とし、主要な実装分類に重ねる委譲の扱いとして整理する。

## 代替案

1. **全委譲を manager-orchestrator に統合**: ADR-0111 の管理下に統一。一貫性はあるが軽量委譲に過剰な抽象化。
2. **委譲パターンを定義せず現場判断**: 現状維持。Command 肥大化と文脈圧迫が継続。
3. **delegation_type を必須 envelope に維持**: 分類の一貫性はあるが、Command の薄さと矛盾。
4. **一律 verbatim 制約を維持**: 親文脈の圧迫が解消されない。

## 結果・影響

- Command は検査観点の詳細を持たず、使用する Skill/reference と戻り値処理のみを持つ。Command の薄さ（ADR-0107）が回復する。
- 親文脈の圧迫が軽減し、独立した検査観点を維持できる。
- file-pipeline / read-only-diagnostic で読み取り専用委譲が可能になり、実装パターンの制約が緩和される。
- 委譲の最小契約が標準化され、新規コマンドでの委譲定義が一貫する。
- delegation_type 分類は SPEC に降格し、Command 本文の必須項目とはしない。
- 旧 ADR-0111（manager/orchestrator 限定採用）の内容は本 ADR に包括された。3 採用条件は本 ADR の manager-orchestrator 特殊型の制約として維持される。
- 単純な決定的検査は Script 優先であることを明記し、サブエージェント委譲の過剰適用を防ぐ。
