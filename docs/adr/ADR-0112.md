# ADR-0112: サブエージェント委譲の一般化と委譲時最小契約

## 背景

ADR-0107（コマンド・スキル・テンプレート・スクリプト責任分界）はコマンドの薄さを定めた。ADR-0111（マネージャー・オーケストレータパターンの限定採用）はマネージャー・オーケストレータの適用条件を限定した。しかし以下の未解決問題があった。

1. **文脈圧迫の未解消**: スキルに詳細を分離しても、親エージェントが同じ文脈でスキルを読み込むため、検査観点が親文脈を圧迫する。独立した文脈で完結できる処理を委譲する手段がない。
2. **委譲パターンの限定**: マネージャー・オーケストレータが唯一の委譲パターンとされ、軽量な保存・更新を親に残す検査・分類委譲（対象を直接修正せず、保存・Issue/PR 更新・commit・push・ユーザー確認・完了報告を親コマンドが保持する委譲）が定義されていなかった。
3. **出力の一律逐語制約**: サブエージェント出力を一律逐語とする制約が、中間成果としての採否判断を阻害していた。
4. **コマンドの肥大化**: 検査観点の詳細がコマンドに展開され、薄く保つ方針が減衰していた。

## 決定

### 1. サブエージェント委譲の一般概念

サブエージェント委譲を一般概念として定義する。マネージャー・オーケストレータは委譲の一形態（特殊型）に位置づける。

### 2. マネージャー・オーケストレータの位置づけ

マネージャー・オーケストレータは複数サブエージェント統制・Wave スケジューリング・障害伝播を行う特殊型である（旧 ADR-0111 の採用条件 3 項目を維持）:
1. 複数 Issue の並列・依存管理（Wave スケジューリング）
2. サブエージェント起動・障害伝播
3. 状態機械の複雑性（再開ポイント、自己修復ループ）

軽量な単一タスク委譲は、より軽い委譲形式（後述）を使用する。

### 3. 保存・更新を親に残す検査・分類委譲の許容

検査・分類・候補抽出・根拠提示に限定し、保存・Issue/PR 更新・commit・push・ユーザー確認・完了報告を親コマンドに残す委譲は、file-pipeline / 検査対象を直接修正しない診断（read-only-diagnostic）実装パターンでも許容する。書き込みを伴わない限り、委譲先の実装パターンを制限しない。

### 4. 親コマンドの最終判断保持

親コマンドは最終判断・副作用（保存、Issue/PR 更新、commit、push、ユーザー確認）・完了報告を保持する。サブエージェントは判断の入力のみを提供し、最終決定は親が行う。

### 5. 委譲時の最小契約

委譲時の最小契約を以下の 4 要素を中心に整理する:

- **`inputs`**: 委譲先に渡す入力（限定されたスコープ）
- **`side_effect_boundary`**: 委譲先の副作用境界。許可操作（read_files, inspect_content, return_evidence 等）と禁止操作（file_write, issue_pr_update, commit, push, user_confirmation）を具体的に列挙し、包括値 `read_only` は使用しない
- **`output_contract`**: 委譲先の返却形式（`pass` / `warn` / `fail` / `partial` を基本とし、成果物パス・根拠・キャプチャ候補を含む）
- **`capture_handoff`**: intake / learning 候補をキャプチャ候補として親へ返す形式

委譲時の最小契約は上記 4 要素を中心とする。`delegation_type` / `on_result` は必要な場合のみ、簡潔なラベルや親側の扱いとして記述する。

### 6. 成果物本文の逐語扱い

成果物本文（Issue 本文、PR 本文、commit message、保存対象ファイル本文、テンプレート成果物）は逐語で返す。判定結果、調査過程、中間ログ、読解メモは要約・成果物パス・根拠・親判断事項・キャプチャ候補へ圧縮して返す（出力形式は成果物種別に応じて切替える）。

### 委譲種別の参考分類

委譲種別の参考分類（旧 delegation_type 分類体系）は SPEC（workflow-contracts.md / artifact-contracts.md）に記載する。コマンド本文での使用は任意とし、実際の入力・副作用境界・返却内容を分類ラベルより優先する。

### 軽量委譲の扱い

`lightweight-delegation`（軽量委譲）は補助的分類とし、主要な実装分類に重ねる委譲の扱いとして整理する。

## 代替案

1. **全委譲をマネージャー・オーケストレータに統合**: ADR-0111 の管理下に統一。一貫性はあるが軽量委譲に過剰な抽象化。
2. **委譲パターンを定義せず現場判断**: 現状維持。コマンド肥大化と文脈圧迫が継続。
3. **delegation_type を必須枠に維持**: 分類の一貫性はあるが、コマンドの薄さと矛盾。
4. **一律逐語制約を維持**: 親文脈の圧迫が解消されない。

## 結果・影響

- コマンドは検査観点の詳細を持たず、使用するスキル・参照資料と戻り値処理のみを持つ。コマンドの薄さ（ADR-0107）が回復する。
- 親文脈の圧迫が軽減し、独立した検査観点を維持できる。
- file-pipeline / 検査対象を直接修正しない診断（read-only-diagnostic）で保存・更新を親に残す検査・分類委譲が可能になり、実装パターンの制約が緩和される。
- 委譲の最小契約が標準化され、新規コマンドでの委譲定義が一貫する。
- delegation_type 分類は SPEC に降格し、コマンド本文の必須項目とはしない。
- 旧 ADR-0111（マネージャー・オーケストレータ限定採用）の内容は本 ADR に包括された。3 採用条件は本 ADR のマネージャー・オーケストレータ特殊型の制約として維持される。
- 単純な決定的検査はスクリプト優先であることを明記し、サブエージェント委譲の過剰適用を防ぐ。
