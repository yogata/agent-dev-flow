# ADR-0112: サブエージェント委譲の一般化と manager-orchestrator の位置づけ

## 背景

ADR-0107（Command/Skill/Template/Script 責任分界）と ADR-0111（Manager/orchestrator パターンの限定採用）は、Command の薄さと manager/orchestrator の限定適用を定めた。しかし以下の未解決問題がある。

1. **文脈圧迫の未解消**: Skill に詳細を分離しても、親エージェントが同じ文脈で Skill を読み込むため、検査観点が親文脈を圧迫する。独立した文脈で完結できる処理を委譲する手段がない。
2. **委譲パターンの限定**: manager-orchestrator（ADR-0111）が唯一の委譲パターンとされ、軽量な読み取り専用委譲が定義されていない。file-pipeline / read-only-diagnostic 実装パターンではサブエージェント起動が不可という制約が過剰。
3. **出力の verbatim 制約**: サブエージェント出力を verbatim で出力する制約が、中間成果としての採否判断を阻害している。親コマンドが部分的に採用・修正・却下する柔軟性がない。
4. **Command の肥大化**: 検査観点の詳細が Command に展開され、薄く保つ方針（ADR-0107）が減衰している。

## 決定

### 1. サブエージェント委譲の一般概念

サブエージェント委譲を一般概念として定義する。manager-orchestrator は委譲の一形態（特殊型）であり、委譲全体と同義ではない。

### 2. manager-orchestrator の位置づけ

manager-orchestrator は複数サブエージェント統制・Wave scheduling・障害伝播を行う特殊型である（ADR-0111 の採用条件を維持）。軽量な単一タスク委譲は manager-orchestrator を必要としない。

### 3. 読み取り専用委譲の許容

読み取り専用委譲（gate_check, semantic_review, log_analysis, classification, extraction, draft_generation）は file-pipeline / read-only-diagnostic 実装パターンでも許容する。書き込みを伴わない限り、委譲先の実装パターンを制限しない。

### 4. 親コマンドの最終判断保持

親コマンドは最終判断・副作用・完了報告を保持する。サブエージェントは判断の入力を提供し、最終決定は親が行う。

### 5. 委譲インタフェースの標準化

委譲インタフェースは「共通エンベロープ + 委譲種別 typed payload + コマンド別 reference」とする:

- **共通エンベロープ**: delegation_type, skill, reference, inputs, on_result
- **typed payload**: 委譲種別ごとの構造化入力
- **コマンド別 reference**: 委譲先が参照する skill reference / SPEC section

自由記述欄は補足情報に限定し、制御情報は構造化フィールドに格納する。

### 6. サブエージェント出力の中間成果扱い

サブエージェント出力は中間成果である。親コマンドは出力の一部を採用・修正・却下できる。verbatim 出力制約は置き換える。

### delegation_type taxonomy

| delegation_type | 用途 | 書き込み |
|---|---|---:|
| `gate_check` | 完了判定・ガードレール充足確認・保存前/close前検査 | 禁止 |
| `semantic_review` | 文書・差分・REQ/ADR/SPECの意味レビュー | 禁止 |
| `log_analysis` | テストログ・CIログ・review結果解析 | 禁止 |
| `classification` | artifact / finding / intake / learning の分類 | 禁止 |
| `extraction` | 候補・論点・未回収事項の抽出 | 禁止 |
| `draft_generation` | Issue本文・PR本文・report案などの草案生成 | 禁止 |
| `controlled_case_execution` | case-run Epic / 複数Issue実行 | 条件付き |

### 初期適用対象（優先度A 6コマンド）

| コマンド | 初期適用する委譲 |
|---|---|
| req-define | 入力整理、既存文書照合、関連文書候補抽出 |
| case-run | 検査・解析系ステップ |
| docs-review | 意味レビュー、分類一貫性確認 |
| backlog-review | artifact分析、統合/分割、矛盾検出 |
| learning-promote | 分類、評価、既存対策確認 |
| intake-promote | itemレビュー、分類案生成 |

### controlled_case_execution の扱い

既存 case-run manager-orchestrator（ADR-0111）の再位置づけに限定。新規展開は後続イテレーションで扱う。

## 代替案

1. **全委譲を manager-orchestrator に統合**: ADR-0111 の管理下に統一。一貫性はあるが軽量委譲に過剰な抽象化。
2. **委譲パターンを定義せず現場判断**: 現状維持。Command 肥大化と文脈圧迫が継続。
3. **Skill 内委譲のみ（サブエージェント不使用）**: Skill 内部で処理を分離。親文脈の圧迫は解消されない。
4. **Script への完全移譲**: 決定的処理のみ対応可能。意味レビュー・分類・抽出などの非決定的処理には不適。

## 結果・影響

- Command は検査観点の詳細を持たず、使用する Skill/reference と戻り値処理のみを持つ。Command の薄さ（ADR-0107）が回復する。
- 親文脈の圧迫が軽減し、独立した検査観点を維持できる。
- file-pipeline / read-only-diagnostic で読み取り専用委譲が可能になり、実装パターンの制約が緩和される。
- 委譲インタフェースが標準化され、新規コマンドでの委譲定義が一貫する。
- ADR-0111（manager/orchestrator 限定採用）は維持される。本 ADR は委譲全体の一般化であり、manager/orchestrator の適用条件は変更しない。
- 単純な決定的検査は Script 優先であることを明記し、サブエージェント委譲の過剰適用を防ぐ。
