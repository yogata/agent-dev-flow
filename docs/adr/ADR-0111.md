# ADR-0111: マネージャー・オーケストレータパターンの限定採用

## 背景

ADR-0108（オーケストレーションスキル作成基準の導入）は、オーケストレーションスキルの作成条件（大きな状態機械、再開ポイント、CI ループ、Wave スケジューリング、サブエージェントプロトコル）を定義した。しかし、スキル化の基準とマネージャー・オーケストレータパターンの採用可否は別の問題である。全コマンドにマネージャー・オーケストレータパターンを適用すべきか、特定ケースに限定すべきかが明文化されていない。

現状、マネージャー・オーケストレータパターンを必要とするのは case-run の Epic Orchestrator モード（複数 Issue の Wave スケジューリング、サブエージェント起動、障害伝播）が主たる対象である。

## 決定

マネージャー・オーケストレータパターンを標準構造とはしない。以下の条件に該当する場合にのみ採用する:

### 採用条件

1. **複数 Issue の並列・依存管理**: 複数 Issue を Wave 単位で順次実行する必要がある場合
2. **サブエージェント起動・障害伝播**: 複数サブエージェントを起動し、障害を伝播・ハンドリングする必要がある場合
3. **状態機械の複雑性**: 再開ポイント検出や自己修復ループを持つ場合

### 主対象

- **case-run Epic Orchestrator**: 上記 3 条件を全て満たすため、マネージャー・オーケストレータパターンの主たる適用対象である

### 採用しないケース

- 単一 Issue の case-run（標準フロー）
- intake / learning / backlog / integrity / requirement の各コマンド
- 上記条件を満たさない新規コマンド

### ADR-0108 との補完関係

ADR-0108 は「オーケストレーションスキルを作成する条件」を定義する。本 ADR は「マネージャー・オーケストレータパターンを採用する条件」を定義する。スキル化 ≠ パターン採用であり、この 2 つを区別する。

- オーケストレーションスキルを持つコマンドが必ずしもマネージャー・オーケストレータパターンを採用するとは限らない
- マネージャー・オーケストレータパターンを採用するコマンドは、オーケストレーションスキルを持つことが多い（が必須ではない）

## 代替案

1. **全コマンドにマネージャーパターン適用**: 一貫性はあるが過剰な抽象化
2. **パターン採用基準なし**: 現場判断 → 一貫性なし、過剰適用のリスク
3. **スキル化 = パターン採用**: 区別を設けない → 小さいコマンドに不要な抽象化

## 結果・影響

- マネージャー・オーケストレータパターンの適用が明確に限定される
- case-run Epic Orchestrator 以外への不用意な適用を防ぐ
- 新規コマンド設計時に「このパターンが必要か」の判断基準が提供される
- ADR-0108 のスキル化基準と本 ADR のパターン採用基準が明確に区別される

## 上書きに関する備考

本 ADR は ADR-0112（サブエージェント委譲の一般化とマネージャー・オーケストレータの位置づけ）に包括された。ADR-0112 は本 ADR の採用条件（3条件）を `controlled_case_execution` の制約として維持しつつ、委譲全体の一般化フレームワークを定義する。本 ADR の内容は ADR-0112 に吸収されているため、参照は ADR-0112 を優先すること。
