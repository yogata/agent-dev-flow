# ADR-0124: req_draft soft-contract 原則: LLM推論消費・厳格schemaなし

## 決定

コマンド間ドラフト引き継ぎ成果物（req-define が生成し req-save/spec-save/case-open/case-auto/case-run/case-close が消費する req_draft）は、ソフトコントラクト（提供側が意味付けを定める基準）として設計する。後続コマンドは構造化 YAML フィールドを LLM 推論で解釈し、機械的なスキーマ検証ベースのパース処理は行わない。

## 背景

req-define が生成するドラフトは、従来人間向け Markdown と draft-meta セクションの形式だった。後続コマンドは本文セクションと draft-meta を解釈して処理を進めていた。ワークフロー全体の構造化と case-auto 自走の最大化のため、ドラフトを構造化合意結果ファイルへ再設計する必要が生じた。

## 判断基準

- 7 コマンドの統合結合モデルを決定する（アーキテクチャ上重要）
- ソフトコントラクト + LLM 推論は将来の全コマンド設計を制約する（長期影響）
- ハードコントラクトとソフトコントラクトの切替は全コマンドの入力処理変更を伴う（逆転困難）

## 妥協事項

### 受入済みリスク（ユーザー承認: 2026-06-19）

1. **QG 決定性との矛盾**: auto_gate/operation_units 等の運用フィールドが LLM 推論で非決定的に解釈されるリスク。同じドラフトでも実行や LLM バージョンごとにゲート判定が変動する可能性がある。
2. **提供側・利用側の暗黙のズレ**: スキーマバージョン管理がないため、req-define の出力進化時に利用側へ破壊的変更の信号が伝わらない。
3. **テスト困難性**: ドラフト解釈の決定的単体テストが困難（LLM 呼出または並行するパーサーテストハーネスが必要）。
4. **デバッグ不透明性**: 不正解釈時の根本原因が曖昧（プロンプトや命名の調整が必要であり、スキーマ編集ではない）。

### 軽減策

- docs-check/integrity-check は主要フィールド名の存在をズレ検出の対象とする（厳格な検証器ではなくドリフト検出として運用）。
- SPEC（docs/specs/artifact-contracts.md）が提供側基準の詳細を定義し、フィールド進化時の参照点を提供する。
