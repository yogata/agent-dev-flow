## 決定

Inter-command draft handoff artifacts（req-define が生成し req-save/spec-save/case-open/case-auto/case-run/case-close が消費する req_draft）は、soft contract — producer-side semantic standard — として設計する。後続コマンドは構造化 YAML フィールドを LLM 推論で解釈し、機械的 schema-validated parsing を行わない。

## 背景

req-define が生成する draft は、従来人間向け Markdown + draft-meta セクション形式だった。後続コマンドは本文セクションと draft-meta を解釈して処理を進めていた。ワークフロー全体の構造化と case-auto 自走の最大化のため、draft を構造化合意結果ファイルへ再設計する必要が生じた。

## 判断基準

- 7 コマンドの統合結合モデルを決定する（アーキテクチャ上重要）
- soft contract + LLM 推線は将来の全コマンド設計を制約する（長期影響）
- hard contract ↔ soft contract 切替は全コマンドの入力処理変更を伴う（逆転困難）

## 妥協事項（Consequences）

### 受入済みリスク（ユーザー承認: 2026-06-19）

1. **QG 決定性との矛盾**: auto_gate/operation_units 等の運用フィールドが LLM 推論で非決定的に解釈されるリスク。同じ draft でも実行/LLM version ごとにゲート判定が変動する可能性がある。
2. **Producer/consumer サイレントドリフト**: schema version 管理がないため、req-define の出力進化時に consumer への breaking change 信号がない。
3. **テスト困難性**: draft 解釈の決定的単体テストが困難（LLM 呼出または並行 parser harness が必要）。
4. **デバッグ不透明性**: 不正解釈時の根本原因が曖昧（prompt/naming 調整が必要、schema edit でない）。

### 軽減策

- docs-check/integrity-check は主要フィールド名の存在をドリフト検出対象とする（厳格 validator ではなく drift detection として運用）。
- SPEC（docs/specs/artifact-contracts.md）が producer-side standard の詳細を定義し、フィールド進化時の参照点を提供する。
