## Context

AgentDevFlow のワークフローは従来 `req-define → req-save → case-open → case-run → case-close` の順で実行されてきた。このワークフローには以下の問題があった:

1. **SPEC 分離経路の不在**: req-define は Step 4-2/4-3 で REQ/SPEC 分類判定を行うが、SPEC 相当の要件行をドラフト内に残留させるのみで、SPEC ファイルへ分離・保存する経路が存在しなかった
2. **req-save の編集スコープ制約**: req-save は Guardrail G02 により `docs/specs/**` の編集を禁止しており、SPEC ファイルへの保存ができない状態だった
3. **case-* フェーズでの SPEC 確定不在**: case-run で実装時に SPEC レベルの詳細が発見されても、それを SPEC ファイルへ反映する標準化された経路がなく、PR 本文の Findings セクションに記録されるのみだった
4. **REQ/SPEC 責務分離の形骸化**: REQ/SPEC 境界基準（REQ-0101-067〜069）は完備されていたが、自動適用経路が3層（事前予防・実装時・事後監視）すべてで途切れており、結果として SPEC 相当の詳細が REQ ファイルに混入し続ける状態だった

この問題を解決するため、新コマンド `/agentdev/spec-save` を導入し、ワークフローを `req-define → req-save → spec-save → case-open → case-run → case-close` に修正する。

## Decision

1. **SPEC lifecycle の導入**: SPEC ファイルに `status` frontmatter フィールド（`draft` / `accepted`）を導入し、SPEC のライフサイクルを管理する
   - `draft`: req-define で分離され、spec-save で保存された直後の状態。IR-044 等、境界違反検査の対象外
   - `accepted`: case-close で SPEC 確定チェックを通過した状態。すべての integrity rule の検査対象

2. **`/agentdev/spec-save` コマンドの新設**: req-define で分離された SPEC 候補を SPEC ファイルに保存・確定するコマンド
   - Input: `.agentdev/drafts/req-draft-{topic-slug}.md` の SPEC候補セクション + draft-meta の spec-candidates
   - Output: `docs/specs/*.md`（既存 SPEC への追記 or 新規 SPEC 作成）
   - 編集スコープ: `docs/specs/**`, `.agentdev/drafts/**`（status 更新用）のみ
   - 責務: REQ-0101-055「SPECに置くべき schema/lifecycle 等 を active REQ から切り出す基準」に準拠

3. **case-auto ワークフローの更新**: case-auto のステップ順序を `req-save → spec-save → case-open → case-run → case-close` に変更
   - draft-meta の `spec-candidates` が空の場合は spec-save をスキップ（条件分岐）
   - 旧形式 draft（`spec-candidates` フィールドなし）は後方互換性を維持し、従来ワークフローで実行

4. **case-* フェーズでの SPEC 確定フロー**:
   - case-run で実装時に発見された SPEC 詳細は、PR 本文の `## SPEC確定候補` セクション（`## Findings / Capture候補` とは別）に記録
   - case-close Step 3（docs/ 検証）を強化し、SPEC 確定候補のリストを提示
   - ユーザー承認で spec-save 相当の処理を case-close 内で実行可能、または spec-save 再起動を提案
   - SPEC status が `draft` から `accepted` に昇格するタイミングを規定

5. **operation_units の operation type 拡張**: `spec-create` / `spec-update` を追加し、SPEC 操作も OU として管理可能にする。`target_spec` フィールドを新設

## Consequences

### Positive

- **REQ/SPEC 責務分離の徹底**: req-define 段階で SPEC 候補を分離し、spec-save で独立保存することで、REQ ファイルへの SPEC 混入を構造的に防止
- **SPEC のライフサイクル明確化**: draft / accepted の2状態により、未確定 SPEC と確定 SPEC の区別が明確に
- **case-* フェーズでの SPEC 反映経路の確立**: 実装時に発見された SPEC 詳細が標準化された経路で SPEC ファイルに反映される
- **IR-044（REQ/SPEC 境界違反検出）との協調**: spec-save で分離された SPEC は IR-044 の検出対象となり、REQ への SPEC 混入を自動検出可能

### Negative

- **ワークフローの複雑化**: case-auto のステップ数が1つ増える（spec-save の挿入）
- **学習コスト**: 新コマンド spec-save の使い方、SPEC status の意味をチームが学ぶ必要
- **後方互換性維持のコスト**: 旧形式 draft と新形式 draft の両方をサポートする必要

### Mitigation

- 後方互換性は `spec-candidates` フィールドの有無で判定し、旧形式 draft は従来ワークフローで実行
- spec-save は SPEC 候補がない場合はスキップ可能で、すべての feature で必須ではない
- ドキュメント整備（Task 6.1）で新ワークフローを明記し、学習コストを低減

## Alternatives Considered

1. **req-save の G02 制約を緩和して SPEC 編集を許可**: 却下。req-save の責務（REQ保存）と SPEC 保存を混在させると責務分離が曖昧になる。新コマンドで責務を分離する方が明確
2. **case-close で SPEC を保存**: 却下。case-close は完了処理が責務で、SPEC 保存を追加すると責務が膨らむ。ただし case-close で SPEC 確定チェックを行い、必要に応じて spec-save を再起動する提案は採用
3. **SPEC をドラフト内に残留させ、case-run で実装時に反映**: 却下。これでは現状の問題（SPEC 分離経路の不在）を解決できない
