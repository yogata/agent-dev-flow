# ADR-0128: case-run の実行モデル: 実行担当サブエージェント委譲

## 背景

case-run は ADR-0114 に基づき外部実行基盤を CLI subprocess（外部実行バックエンド）
として起動していた。このモデルでは外部実行基盤内部の bash コマンドがハングアップした場合、
case-run 側から検知不可能だった（case-run #971 事例: tui-state cancelled/lastTool:bash/toolCalls:129）。

外部実行基盤は CLI ツールとしてだけでなく、エージェント型（実行担当サブエージェント）
を提供している。委譲機構経由で実行担当サブエージェントを起動すると、外部実行基盤の実行エンジンを
ネイティブに利用でき、各ツール呼び出しが120秒 timeout で保護されるためハングは構造的に検知される。

外部実行基盤の重複オーケストレーション問題（issue #4027、PR #4071、2026-05-21 クローズ）は
解決済みであり、委譲機構への移行の技術的障壁はない。

さらに、case-run のスコープを単一 Issue から「単一 Issue または単一 Wave」に拡張し、
case-run #epic で Epic Issue の現在 Wave を並列実行できるようにする。
Epic 全体（複数 Wave）は Wave 境界で PR マージ（case-close 責務）が必要なため扱わない。

## 決定

### 1. 実行担当サブエージェント委譲

case-run は委譲により実装作業を実行担当サブエージェントへ委譲する。実行担当サブエージェントの選定、起動方式、timeout、retry などの実行制御は harness 側の責務とし、配布物である case-run 本文には依存させない（ADR-0136 決定2、REQ-0162-001/002）。
委譲契約の詳細は SPEC（docs/specs/commands/case-run.md）に配置する。

### 2. 実行担当サブエージェントによる evidence-backed 実行

実行担当サブエージェントは委譲 prompt 内で指定された command を使用し、Issue を success criteria
に分解、各 criterion に observable evidence を要求、品質ゲート（code review + QA review + gate
review）を実行する。監査トレイルは worktree 配下に配置され、worktree 削除時に破棄される
（REQ-0139-007）。

### 3. case-run スコープ拡張: 単一 Wave 実行

case-run は Epic Issue 番号を受け取った場合、Epic Issue 本文から現在 ready な Wave の
子Issue を特定し、各子Issue を委譲起動で実行担当サブエージェントに並列委譲する（最大5件）。
case-run は全委譲起動の完了を待ち、結果を収集して return する。
1 Wave の実行（PR作成まで）で return し、Wave 境界（マージ）は扱わない。
同一コマンド再実行で次 Wave に進む（べき等、Epic Issue 本文から進行状況判定）。

### 4. case-close スコープ拡張: Wave クローズ

case-close は Epic Issue 番号を受け取った場合、現在 Wave の PR マージ、子Issue クローズを
行う。
最終 Wave で Epic クローズ + 最終処理。
それ以外は残 Wave 通知。

### 5. 単独 case-run と case-auto 経由 case-run の読み分け（ADR-0137、ADR-0138 反映）

case-run は以下の2形態のいずれかで起動される。処理の内部ロジックは共通だが、委譲起点と結果処理の所有者が異なる。

- **単独 case-run**: ユーザーまたは外部起動が case-run を直接起動する。case-run は実行担当サブエージェントへ委譲し、結果（completed-pr / blocked / failed / delegation-unavailable）を処理する。スコープは単一 Issue または単一 Wave。
- **case-auto 経由 case-run**: case-auto が case-run の準備（Issue 番号解決、worktree 設定）と結果処理（Epic Issue 本文読込、次工程引き継ぎ）をインライン実行し、実行担当サブエージェントへの委譲を case-auto から直接行う（委譲起点の折りたたみ、ADR-0137）。Phase 2 並列実行制御、固定並列数5、bg task 状態管理は ADR-0138 で AgentDevFlow 側制御点として規定。

Epic Issue 本文の単一書き手は case-close（REQ-0114-081、ADR-0125）。Phase 分離（Phase 1 case-open 順次、Phase 2 case-run 並列、Phase 3 case-close 順次、REQ-0114-102）は ADR-0138 で規定される。

### 6. result 契約（拡張）

実行担当サブエージェントの result 契約は ADR-0114 の3状態を ADR-0136 に基づき4状態へ拡張:
completed-pr / blocked / failed / delegation-unavailable。

## 代替案

1. 外部実行バックエンド + tui-state 監視: tui-state の CLI モードでの更新が未検証であり、
 ハッシュアルゴリズムも不明。
確実性に欠けるため不採用。
2. 外部実行バックエンド + timeout 短縮: ハング検知が改善されるが、正常な長時間処理も
 timeout で kill されるリスクがあり、また外部実行バックエンドのブラックボックス性は変わらない。
 不採用。
3. 外部実行基盤廃止: スコープが大きすぎる。外部実行基盤は委譲機構経由で
 利用可能なため不要。

## 結果、影響

- 外部実行バックエンド（CLI 経由）は委譲の結果として使用されない。外部実行バックエンドのハングアップ問題が構造的に解消される（各ツール呼び出し120秒 timeout）
- case-run が Epic Wave 実行をサポートし、手動実行時のユーザー体験が改善される
- case-auto が簡素化され、Epic/Wave orchestration 責務が case-run/case-close に集約される
- `agentdev-case-run-execution-adapter` スキルが CLI 起動プロトコルから
 委譲プロトコルに全面移行する
- AGENTS.md のハーネス選定セクションが更新される
- result 契約が ADR-0136 に基づき3状態から4状態へ拡張され、delegation-unavailable が追加される
- REQ-0130-010/016/017/020/021/022/024/025、REQ-0139-013、REQ-0114-084 が更新される
