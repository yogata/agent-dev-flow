# ADR-0113: diagnostics workflow 導入と review 系コマンド完全削除

> **status: deprecated** (2026-06-16, agentdev-system-reorganization OU-07)
>
> 本 ADR は `diagnostics-*` 命名を採用したが、その後 inspect-* への全面改名（REQ-0124）により現行判断の根拠として読めなくなった。`diagnostics` 命名自体がすでに `inspect` に置き換えられており、本 ADR の決定内容は現行基盤に反映されていない。現行の検出コマンド群の状態は REQ-0103 / REQ-0124 / SPEC 側に記録されている。本 ADR は履歴参照としてのみ扱い、現行根拠として引用しないこと。

## 背景

AgentDevFlow の診断機能は `docs-review` と `skill-review` の2コマンドで構成されていた。これらには以下の問題があった。

1. **語の衝突**: `review` は行為名として広く、`docs-review` / `skill-review` / `backlog-review` など既存コマンド名と衝突しやすい
2. **検出事項の RU 化経路の分断**: 診断検出事項の行き先が intake / req-define 入力ドラフト / docs-check ルールに分断され、経路が一貫していなかった
3. **採用済み概念の不統一**: intake / learning は採用済み（promoted）の概念を持つが、diagnostics には同等の概念がなかった

## 決定

### 1. diagnostics を一級概念として導入

`diagnostics` を診断ワークフローの一級概念として導入する。コマンド体系を `diagnostics-*` に統一する:

- `diagnostics-docs`: docs 系意味整合診断（旧 `docs-review`）
- `diagnostics-skills`: コマンド・スキル参照妥当性・スキル構造診断（旧 `skill-review`）
- `diagnostics-promote`: 診断検出事項の分類・採用
- `diagnostics-commands`: 予約拡張点（本件では対象外）

### 2. docs-review / skill-review の完全削除

非推奨ラッパーなしで完全削除する（ドキュメント記述 + 実体ファイル）。REQ-0115-039「互換ラッパーを前提とした段階移行は行わず、即座にあるべき姿へ移行すること」に合致する。

### 3. diagnostics ドメイン状態の導入

`.agentdev/diagnostics/` をドメイン状態として導入する:

- `inbox/`: 未分類の診断検出事項
- `promoted/`: 採用済み・RU 化対象
- `archive/rejected/`: 却下済み

### 4. 採用済み（promoted）概念の統一

`promoted/` の意味を intake / learning / diagnostics で「採用済み・RU 化対象」に統一する。

### 5. 検出事項経路の整理

- 明確な不整合 → diagnostics promoted → RU 化対象（intake に回さない）
- 未確定検出事項 → intake
- 修正対象を持たない知見 → learning
- docs-check ルール / テストデータ候補 → 独立経路とせず RU の要件化方向または受け入れ条件に含める

## 根拠

### 完全削除 vs 非推奨ラッパー

REQ-0115-039 が「互換ラッパーを前提とした段階移行は行わず、即座にあるべき姿へ移行すること」を定めている。非推奨ラッパーを選択すると REQ-0115-039 との競合が発生し、ラッパーの保守コストも発生する。完全削除は REQ-0115-039 に合致し、diagnostics ワークフロー導入の全体構造を単純化する。

### review → diagnostics への改名

`review` は `backlog-review` 等、既存コマンドと衝突する。`diagnostics` はドメイン固有であり、衝突を回避できる。
