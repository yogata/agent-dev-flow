# ADR-0004: 要件管理構造の area-based 移行方針

## 背景

現状の `docs/requirements/` は `REQ-{NNNN}.md` の per-file 構造をとっている。この構造は要件IDとファイル名が1:1で対応するためシンプルだが、area（領域）単位での要件管理・レビュー・退役を行う将来の運用に合致しない。area-based current document 構造への段階的移行が必要である。

## 決定

以下の方針で段階的に移行する:

1. **area-based SSoT**: `docs/requirements/` を area 単位の current document の SSoT とする。将来の標準構造は `README.md` / `INDEX.md`（generated） / `core.md`（暫定area） / `{area}.md`（安定area）とする
2. **REQ-ID の安定ID化**: REQ-ID（`REQ-{NNNN}`）はファイル名から分離し、安定IDとして扱う。area情報はREQ-IDに含めず、ファイル配置と frontmatter / generated INDEX で管理する
3. **core.md を暫定area**: 初期状態では全要件を `core.md` に配置する。areaの分割は自然な凝集度が確認できた段階で実施する
4. **unclassified.md / sources/ / areas/ は不採用**: 未分類要件を正式要件として固定する `unclassified.md`、外部ソース参照用の `sources/`、area格納用の `areas/` は作成しない
5. **INDEX.md は generated artifact**: `INDEX.md` は自動生成され、手動編集は行わない
6. **一括移行は行わない**: 既存 `REQ-{NNNN}.md` ファイルの一括移行は実施しない。移行は別 Issue で段階的に実施する

## 代替案

1. **即時一括移行**: 全REQ-{NNNN}.mdをarea.mdに再構築 → 移行コストが高く、リスクが大きい
2. **unclassified.md 採用**: 未分類要件を一時ファイルに集約 → 分類できていないものが正式要件として固定される問題
3. **現状維持**: per-file構造を維持 → area単位の運用が不可能

## 結果・影響

- REQ-IDが安定IDとして機能し、ファイル配置の変更に影響されない
- req-file-manager スキルにarea管理の責務が追加される
- INDEX.md生成の自動化が必要（将来の実装対象）
- 既存REQファイルは当面現状のまま運用される
