## Context

AgentDevFlow の backlog パイプラインは `backlog-review`（promoted artifact の分析、統合、ユーザー承認）と `backlog-save`（承認済み review 結果から RU 生成）の2段構えだった。しかし以下の問題があった:

1. **中間成果物の不要な複雑さ**: `backlog-review` が review draft（`.sisyphus/drafts/`）を生成し、`backlog-save` がそれを読んで RU を生成する2段階は、ユーザー承認後の RU 生成という単純な操作に不要な間接層だった
2. **status 管理の過剰**: review draft の `status: reviewed` / `status: saved` による状態管理は、backlog-review → backlog-save 間の受け渡しに過ぎないが、REQ-0108-123 が禁止する workflow status 管理との境界が曖昧だった
3. **depends_on の統合先**: RU 間依存関係は backlog-review の分析段階で扱うのが自然であり、独立したコマンドで処理する必要性がなかった

## Decision

1. `backlog-save` コマンドを廃止する
2. `backlog-review` に RU 生成機能を統合する（ユーザー承認後に直接 RU を生成）
3. review draft を必須中間成果物から廃止する
4. RU frontmatter に `depends_on` フィールドを追加し、backlog-review が依存順処理を行う
5. `learning-promote` 完了後の次コマンドを `/agentdev/backlog-review` に明示し、`req-define` / `case-run` 直行を禁止する

## Consequences

- **正の影響**: パイプラインが単純化され、backlog-review → req-define の2段階に集約される。ユーザー承認から RU 生成までが1コマンド内で完結する
- **負の影響**: backlog-review の責務が拡大する。ただし RU 生成は承認済み内容の構造化出力であり、分析、統合、承認という既存フローからの自然な延長である
- **リスク**: backlog-save に依存する既存の completion report template やガイド記述の更新が必要
