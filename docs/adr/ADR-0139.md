# ADR-0139: REQ/SPEC 意味分類と正規所有モデル

## 背景

REQ と SPEC の責務境界は ADR-0103（文書種別責務境界）、ADR-0123（SPEC lifecycle と spec-save の導入）、ADR-0107（コマンド・スキル・テンプレート・スクリプト責任分界）で定義されてきた。これらの ADR は「REQ は要件、SPEC は現在仕様」という大枠を確立しているが、learning/intake 由来の変更を REQ と SPEC のどちらへ配置すべきかの判定基準、SPEC 内部の論理区分の数、正規所有の単位、分類根拠の伝播経路については明示していなかった。

その結果、既存の SPEC 分離基準（REQ-0101-068）と安定契約例外（REQ-0101-069）を満たすか否かの判定が現場に委ねられ、同一仕様関心について複数 SPEC が正規所有者を主張する事態、パラメータ調整が実装詳細 SPEC と混在する事態、learning/intake から流入した変更種別が工程間で伝播されず req-define で再分類を強いられる事態が生じていた。

## 決定

REQ/SPEC 責務分界を意味分類と正規所有の一体モデルとして再構築し、以下の5決定を追加する。ADR-0103/0123/0107 の意味（文書種別境界、SPEC lifecycle、責任分界）は維持し、新規概念を追加するのみとする。supersede ではなく relates-to 関係とする（document-model.md L226「accepted ADR の意味変更を禁止、変更時は新規 ADR」、REQ-0119-033 自身も「適用条件の精緻化のため新規 ADR は作成しない」の先例）。

### 決定1: REQ はステークホルダー視点の4妥当性基準を満たす要件行のみを受け入れる

REQ は識別可能なステークホルダーが求める状態、振る舞い、制約、外部契約を記述する文書種別とし、4妥当性基準（要求元ステークホルダー特定、観測可能成果、内部実装非依存、要求文存立）を満たす要件行のみを受け入れる（REQ-0101-079）。内部成果物（command、skill、script、ファイル）だけを主語とし、具体パス、フィールド、enum、閾値、アルゴリズム、作業履歴が主題となる記述は原則 SPEC へ配置する（REQ-0101-068）。安定契約例外（REQ-0101-069）は維持する。

### 決定2: SPEC 論理区分を5区分とし、パラメータ責務を実装詳細から区別する

SPEC 論理区分を現行4区分（挙動SPEC、カタログSPEC、横断契約SPEC、実装詳細SPEC）から5区分（挙動SPEC、カタログSPEC、横断契約SPEC、パラメータSPEC、実装詳細SPEC）へ拡張する（REQ-0155-009、REQ-0156-013）。パラメータ責務（retry 回数、timeout、閾値、重み、優先順位、上限、下限、fallback、許容範囲等）を独立区分として実装詳細から区別し、learning/intake 由来のパラメータ調整の正規所有先として機能させる。パラメータは単一の全体パラメータファイルへ集約せず、対象 command、skill、workflow、品質ルール、整合性ルール等の所有責務に基づいて配置先を決定する。

### 決定3: 正規所有の単位を「安定した関心キー」とする

正規所有の単位は「安定した関心キー」である（REQ-0119-038）。これは REQ-0119-033「責務ごとに最も安定した最小の定義元を正規とする」の延長であり、適用条件を精緻化する。1ファイルが複数関心を参照することは許容するが、正規定義だけを単一所有とし、同一仕様関心について複数 SPEC が正規所有者を主張しない。各 SPEC は frontmatter または冒頭宣言節で主論理区分（5区分のいずれか）と正規所有対象を識別可能にする（REQ-0156-013）。

### 決定4: learning/intake 由来変更は8種別で分類し、REQ 拡張を2種別に限定する

learning/intake 由来の知見は次の8変更種別で分類する: 新しい利用者要求、利用者から見える外部契約の変更、既存要求を満たすバリエーション追加、エッジケース対応、パラメータ調整、既存REQ/SPECへの不適合修正、外部挙動を変えない内部再構成、文書記述の訂正。REQ の作成または拡張を候補とするのは「新しい利用者要求」または「外部契約変更」に該当する場合のみとする（REQ-0136-033）。それ以外（バリエーション追加、エッジケース、パラメータ調整、不適合修正、内部再構成、文書訂正）は、既存 REQ が要求を既に保持している限り REQ を拡張しない。learning/intake 成果物は分類根拠（変更の性質、REQ影響の有無、対象ステークホルダー、利用者から見える変更の有無、SPEC論理区分、正規所有対象、追記先を選択した理由、根拠となる観測事実）を後続工程（RU、req-define、spec-save）へ伝播させる（REQ-0136-033）。分類根拠は soft-contract（ADR-0124）とし、欠落時は unknown 既定値で警告する後方互換運用をとる。

### 決定5: req-define は最終分類を確定し、spec-save は配置一貫性検証を実施する

req-define は backlog-review の暫定分類（tentative_classification）を暫定入力とし、最終分類を自身で確定する（REQ-0102-087）。REQ 影響なしと確定した変更からは REQ 保存操作を生成しない（REQ-0102-088）。spec-save は対象 SPEC の主論理区分・正規所有対象と保存内容の整合を「配置一貫性検証」として検証する（REQ-0136-034）。配置一貫性検証は確定済み分類・所有情報と保存先の整合確認であり、内容品質の再査読ではない（REQ-0136-030 との整合）。強制ゲート（保存拒否条件: 重複所有、配置不一致）は SPEC 宣言形式の定義完了後に有効化し、宣言未完了の既存 SPEC は警告モードで経過観察する（REQ-0136-035）。段階適用（宣言形式定義 → 警告モード棚卸し → 重複解消 → 新規/変更 SPEC 強制 → 全件強制）とし、bootstrap 問題を回避する。

## 代替案

- REQ-0103（Artifact責任分界、90行）への APPEND による新規概念集約: SPLIT シグナル +2 に達し肥大化を加速させるため採用しない。代わりに新規概念を既存の正規所有者 REQ へ分配する（文書統治→REQ-0101、req-define 責務→REQ-0102、検査・診断→REQ-0108、正規所有単位→REQ-0119、req-define/spec-save 接続→REQ-0136、文書粒度→REQ-0155、SPEC 体系化→REQ-0156）。
- ADR-0103 を supersede する方式: ADR-0103 の意味（文書種別責務境界）は維持すべきため採用しない。relates-to 関係とし、新規概念を追加するのみとする。
- 分類根拠の厳格スキーマ化: 既存の採用済み成果物、RU、req_draft を一斉に無効にする後方互換性問題を生むため採用しない。ADR-0124（soft-contract）を維持し、欠落時は unknown 既定値で警告する。

## 結果、影響

REQ-0101、REQ-0102、REQ-0108、REQ-0119、REQ-0136、REQ-0155、REQ-0156 に新規要件行を追加する。SPEC 群（document-model.md、document-type-responsibilities.md、artifact-responsibilities.md、artifact-contracts.md、req-health-metrics.md、spec-health-metrics.md、req-define.md、spec-save.md、backlog-review.md、learning-promote.md、intake-promote.md、responsibility-boundary-purification.md）へ意味分類と正規所有モデルの詳細を配置する。既存 REQ/SPEC を新基準で横断再評価する基準は responsibility-boundary-purification.md へ定義し、実施は case-open/case-run 工程で行う。

## 関連する決定

- ADR-0103（文書種別責務境界）: relates-to。本 ADR は ADR-0103 の枠組みを維持しつつ、REQ 候補の受理基準（4妥当性基準）と SPEC 論理区分（5区分）を追加する。
- ADR-0123（SPEC lifecycle と spec-save の導入）: relates-to。本 ADR は spec-save の検証観点を「配置一貫性検証」として拡張し、SPEC lifecycle は維持する。
- ADR-0107（コマンド・スキル・テンプレート・スクリプト責任分界）: relates-to。本 ADR は「責務ごと」の単位を「安定した関心キー」として精緻化し、責任分界の原則は維持する。
- ADR-0124（req_draft soft-contract 原則）: 分類根拠の soft-contract 運用（unknown 既定値、警告）の基盤として維持する。
