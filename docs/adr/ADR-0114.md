# ADR-0114: case-run 実行責務の外部実行バックエンド委譲

## 背景

AgentDevFlow の `case-run` は、これまで Issue 解決、実装、検証、PR 作成を単一コマンド内で完結する構造を持っていた。
しかし、実装実行の規模、複雑性が増大する中で、以下の課題が顕在化した。

1. **case-run の肥大化**: case-run が worktree 管理、実装実行、乖離検出、PR 作成、キャプチャ収集を全て抱え込み、200行上限を維持することが困難。
2. **実行品質の属人化**: 実装実行の判断が case-run の内部手順に埋め込まれ、外部の強力な実行エンジンを活用できない。
3. **責務境界の不明確さ**: ワークフロー状態管理（Issue/PR/worktree）と実装実行が同一コマンドに混在し、委譲境界が定義されていない。

OMO（OpenCode Managed Orchestration）等の外部実行バックエンドは、計画駆動な実装実行を得意とするが、agentdev 側のワークフロー状態管理と明確に分離する必要がある。

## 決定

case-run は実装実行を外部実行バックエンドへ委譲し、自身は Issue/PR/worktree 状態のオーケストレーションに専念する。
委譲境界はドライバーサブエージェントが担う。

### 1. 委譲モデル

- **case-run 本体**: 1 Issue 単位でドライバーサブエージェントを起動し、ドライバー結果を処理する。複数 Issue / Epic 順序制御、worktree 前提確認、ドライバー起動、ドライバー結果処理を責務とする。
- **ドライバーサブエージェント**: 外部実行バックエンドのアダプタプロトコルを実装する独立スキル（`agentdev-case-run-execution-adapter`）が定義する。ADR/REQ/SPEC/docs/Issue 本文/リポジトリコンテキストを再確認し、回答可能なブロッカーを自律的に差し戻す。
- **外部実行バックエンド**: 実装実行、検証、PR 本文作成を行う。agentdev 側からは不透明な外部成果物として扱う。

### 2. ドライバー結果契約

ドライバーは以下の4状態のいずれかを返す（ADR-0136 により3状態から4状態へ拡張）:

- **completed-pr**: 実装完了、PR作成済み。PR 番号を伴う。case-run の成功成果は PR 作成である。
- **blocked**: 回答可能なブロッカーに遭遇。詳細本文は Issue コメントに SSoT として記録される。ドライバーが回答可能なブロッカーを自律的に外部実行バックエンドへ差し戻せる。
- **failed**: リポジトリコンテキストで回答不能なブロッカー。実装試行後の領域レベル失敗。Issue コメントに構造化して記録される。
- **delegation-unavailable**: 実行インフラ起動不能。`failed` とは異なる回復アクションを要する独立状態（ADR-0136、REQ-0162-003/004）。

### 3. 完了条件チェックボックス責務の分離

完了条件チェックボックスの評価、更新は case-close 責務とし、PR 作成後に別コンテキストで行う。

### 4. 検出事項、キャプチャの配置

検出事項、キャプチャ候補が存在する場合、PR 本文に記述される。
case-close は引き続き PR 本文をキャプチャ入力源として扱い、Issue 本文の完了条件を再読込して最終完了判定する。

### 5. オーケストレーションの保持

複数 Issue / Epic Issue のオーケストレーションは agentdev 側に残る。
外部実行バックエンド側の計画成果物（plan artifact）は agentdev から不透明な外部成果物として扱う。

## 代替案

1. **case-run 内部で実装実行を維持**: case-run の肥大化、実行品質の限界が解消されないため不採用。
2. **ドライバーなしで外部バックエンドを直接呼ぶ**: アダプタプロトコルが不在で委譲境界が不明確になるため不採用。
3. **完了条件チェックボックスをドライバーが更新する**: PR 作成後の最終完了判定と競合し、二重更新のリスクがあるため不採用。

## 結果、影響

- case-run 本体はオーケストレーションに専念し、150行目標を維持しやすくなる。
- `agentdev-case-run-execution-adapter` スキル（ドライバーアダプタプロトコル定義）が新設される。
- ドライバー結果の3状態契約により、blocked / failed の処理フローが明確化する。
- 完了条件チェックボックスの評価責務が case-close に集約される。
- 外部実行バックエンド側の計画成果物は agentdev の管理対象外となる。
- case-auto は case-run の新境界に追従し、QG を再実装せず構成コマンドの QG を継承する。
- 外部実行基盤利用時の momus による実行計画確認は case-run の実装開始前確認として扱う（REQ-0139-006）。momus は QG-3/QG-4 のいずれの代替でもない。
- AgentDevFlow は外部実行手段の中間成果物の内部構造を永続状態として扱わない（REQ-0139-007）。

## 関連する決定

- ADR-0112（サブエージェント委譲の一般化）（本判断は ADR-0112の一般原則の特定適用である）。ドライバーサブエージェントは委譲時最小契約（inputs, side_effect_boundary, output_contract, capture_handoff）に従う。
- ADR-0109（Epic Issue 本文を実行順序 SSoT）（オーケストレーションは Epic Issue を SSoT として扱い続ける）。
- ADR-0128（case-run の実行モデル: 実行担当サブエージェント委譲）（本 ADR の実行モデルを外部実行バックエンド CLI 起動から実行担当サブエージェント委譲へ移行。result 契約4状態、worktree 隔離、Findings/Capture 配置は維持）。
- ADR-0136（配布物ハーネス境界の浄化）（result 契約を3状態から4状態へ拡張し、`delegation-unavailable` を追加。harness 固有詳細を配布物から除去）。
- ADR-0137（case-auto における case-run インライン実行）（case-auto 経由時の委譲起点を折りたたみ、多重委譲を回避。本 ADR の委譲モデルと直交する実行形態の正規化）。

## 承認記録

- 承認日: 2026-06-16（proposed → accepted）
- 承認根拠: OU-08（Issue #836）にて、委譲モデル、ドライバー結果3状態契約、完了条件チェックボックス責務分離、検出事項配置、オーケストレーション保持を実装により検証。`agentdev-case-run-execution-adapter` スキルを新設しドライバーアダプタプロトコルを定義、case-run 本体をオーケストレーションに純化、case-close に完了条件チェックボックス評価責務を集約した。
