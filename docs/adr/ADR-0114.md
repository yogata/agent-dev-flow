# ADR-0114: case-run 実行責務の外部実行バックエンド委譲

## 背景

AgentDevFlow の `case-run` は、これまで Issue 解決・実装・検証・PR 作成を単一コマンド内で完結する構造を持っていた。しかし、実装実行の規模・複雑性が増大する中で、以下の課題が顕在化した。

1. **case-run の肥大化**: case-run が worktree 管理・実装実行・乖離検出・PR 作成・capture 収集を全て抱え込み、200行上限を維持することが困難。
2. **実行品質の属人化**: 実装実行の判断が case-run の内部手順に埋め込まれ、外部の強力な実行エンジンを活用できない。
3. **責務境界の不明確さ**: workflow state 管理（Issue/PR/worktree）と実装実行が同一コマンドに混在し、委譲境界が定義されていない。

OMO（OpenCode Managed Orchestration）等の外部実行バックエンドは、plan-driven な実装実行を得意とするが、agentdev 側の workflow state 管理と明確に分離する必要がある。

## 決定

case-run は実装実行を外部実行バックエンドへ委譲し、自身は Issue/PR/worktree 状態の orchestration に専念する。委譲境界は driver subagent が担う。

### 1. 委譲モデル

- **case-run 本体**: 1 Issue 単位で driver subagent を起動し、driver result を処理する。複数 Issue / Epic 順序制御・worktree 前提確認・driver 起動・driver 結果処理を責務とする。
- **driver subagent**: 外部実行バックエンドの adapter protocol を実装する独立 skill（`agentdev-execution-backend`）が定義する。ADR/REQ/SPEC/docs/Issue 本文/repository context を再確認し、回答可能な blocker を自律的に差し戻す。
- **外部実行バックエンド**: 実装実行・検証・PR 本文作成を行う。agentdev 側からは不透明な外部成果物として扱う。

### 2. driver result 契約

driver は以下の3状態のいずれかを返す:

- **completed(pr)**: 実装完了・PR作成済み。PR番号を伴う。case-run の成功成果は PR 作成である。
- **blocked**: 回答可能な blocker に遭遇。詳細本文は Issue コメントに SSoT として記録される。driver が回答可能な blocker を自律的に外部実行バックエンドへ差し戻せる。
- **failed**: repository context で回答不能な blocker。Issue コメントに構造化して記録される。

### 3. 完了条件チェックボックス責務の分離

Issue 本文の完了条件チェックボックスの評価・更新は、PR 作成後に case-close が別コンテキストで行う責務とする。case-run・driver subagent・外部実行バックエンドは完了条件チェックボックスを更新しない。

### 4. Findings / Capture の配置

Findings / Capture 候補が存在する場合、PR 本文に記述される。case-close は引き続き PR 本文を capture 入力源として扱い、Issue 本文の完了条件を再読込して最終完了判定する。

### 5. orchestration の保持

複数 Issue / Epic Issue の orchestration は agentdev 側に残る。外部実行バックエンド側の plan artifact は agentdev から不透明な外部成果物として扱う。

## 代替案

1. **case-run 内部で実装実行を維持**: case-run の肥大化・実行品質の限界が解消されないため不採用。
2. **driver なしで外部バックエンドを直接呼ぶ**: adapter protocol が不在で委譲境界が不明確になるため不採用。
3. **完了条件チェックボックスを driver が更新する**: PR 作成後の最終完了判定と競合し、二重更新のリスクがあるため不採用。

## 結果・影響

- case-run 本体は orchestration に専念し、150行目標を維持しやすくなる。
- `agentdev-execution-backend` skill（driver adapter protocol 定義）が新設される。
- driver result の3状態契約により、blocked / failed の処理フローが明確化する。
- 完了条件チェックボックスの評価責務が case-close に集約される。
- 外部実行バックエンド側の plan artifact は agentdev の管理対象外となる。
- case-auto は case-run の新境界に追従し、QG を再実装せず構成コマンドの QG を継承する。

## 関連する決定

- ADR-0112（サブエージェント委譲の一般化）— 本判断は ADR-0112の一般原則の特定適用である。driver subagent は委譲時最小契約（inputs, side_effect_boundary, output_contract, capture_handoff）に従う。
- ADR-0109（Epic Issue 本文を実行順序 SSoT）— orchestration は Epic Issue を SSoT として扱い続ける。
