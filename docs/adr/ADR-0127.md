# ADR-0127: case-auto 構成工程の委譲によるスケーラビリティ確立

## 背景

AgentDevFlow の `case-auto` は、req-save → spec-save → case-open → case-run → case-close を順次自走する最大自走モードである（REQ-0114）。
ADR-0114 により case-run の実装実行は外部実行バックエンドへ委譲され、コンテキスト分離が実現されている。

しかし、残る4工程（req-save / spec-save / case-open / case-close）は case-auto と同一の sisyphus コンテキストでインライン実行される設計だった。
この設計は以下の課題を生んでいた。

1. **コンテキスト枯渇**: case-auto が各工程のコマンド定義と参照スキル群を自コンテキストで読込、実行するため、最大ケース（複数OU、Epic flow）でコンテキストが枯渇する。
2. **親コンテキスト非累積要件との矛盾**: 既に「Epic 全体の実装詳細を親コンテキストに累積させず、永続状態から進行、再開できること」が要件化されていたが、インライン実行モデルはこの要件に構造的に反していた。
3. **スケーラビリティの不均一**: case-run のみが外部実行バックエンドでスケールし、他4工程はスケールしない状態だった。

## 決定

case-auto は構成工程（req-save / spec-save / case-open / case-close）を委譲起動し、自身は薄いオーケストレータ（入力解決、工程分岐、状態引き継ぎ、停止判定、完了報告、OU/子Issueループ制御）に専念する。
case-run は引き続き外部実行バックエンド委譲モデル（ADR-0114）を使用する。

### 1. 委譲モデル

- **case-auto 本体**: 入力解決、工程分岐、工程間状態引き継ぎ、停止条件検出、完了報告、Epic/OUループ制御のみを担う。工程の内部ロジックを実行しない。
- **req-save / spec-save / case-open / case-close 委譲**: 各コマンド定義を authoritative source として読み込み実行する。case-auto から受け取った入力（draft path、OU ID、Issue番号、PR番号）を処理し、結果（保存済みファイル、Issue/PR番号、pass/warn/fail）を返す。
- **case-run**: ADR-0137 により case-auto 内でインライン実行される（委譲起点の折りたたみ）。実行担当サブエージェントへの委譲は case-auto から直接行う。本 ADR は case-run を構成工程委譲の対象から除外し、実行形態は ADR-0128、ADR-0137、ADR-0138 に従う。

### 2. 工程別委譲契約

各工程の委譲起動は ADR-0112 §5（委譲時最小契約）に従う:

| 工程 | inputs | output_contract | side_effect_boundary |
|------|--------|-----------------|---------------------|
| req-save | draft path, OU ID | 保存済みREQ/ADRリスト, draft status=saved, pass/warn/fail | `docs/requirements/**`, `docs/adr/**`, `docs/README.md`, `.agentdev/drafts/**`, git commit/push |
| spec-save | draft path | 保存済みSPECリスト, SPEC消費済みフラグ, pass/warn/fail | `docs/specs/**`, `.agentdev/drafts/**`, git commit/push |
| case-open | draft path, OU ID | Issue番号(Epic含む), pass/warn/fail | GitHub Issue作成, `.agentdev/drafts/**` 削除, `.agentdev/backlog/req-units/**` 削除, git commit/push |
| case-run | Issue番号, worktree root | completed-pr/blocked/failed/delegation-unavailable | worktree内ファイル編集, GitHub PR作成（外部実行バックエンド、ADR-0114） |
| case-close | Issue番号, PR番号 | マージ結果, Capture保存結果, 削除済みブランチ, pass/warn/fail | GitHub PRマージ/Issue close, `.agentdev/learning/**`, `.agentdev/intake/**`, git commit/push, branch/worktree削除 |

### 3. case-auto の保持責務（オーケストレーション）

case-auto は委譲後も以下を保持する（工程内部ロジックではないため）:

- 入力解決（draft path特定、OU ID解決、Issue番号/URL分岐）
- 工程分岐（artifact_actions に基づく動的判定）
- 工程間状態引き継ぎ（Issue番号、PR番号、RU パス、capture対象情報）
- 停止条件検出
- 完了報告（タイミング情報含む）
- Wave/子Issue選択、case-run インライン実行制御（ADR-0125、ADR-0137 による委譲起点折りたたみ、ADR-0138 Phase 2 並列）
- OU逐次処理ループ
- クリーンアップ検証ゲート（draft/RU 残存確認）

Epic Issue 本文の単一書き手は case-close が担う（ADR-0128 と整合）。case-auto は Epic Issue 本文の更新責務を持たない。

### 4. コンテキスト分離効果とインライン実行

case-auto は構成工程（req-save / spec-save / case-open / case-close）を独立コンテキストへ委譲することでコンテキスト枯渇を回避する。case-run は ADR-0137 により case-auto 内でインライン実行し、実行担当サブエージェントへの委譲を case-auto から直接行う（委譲起点の折りたたみ、多重委譲回避）。

各工程が独立コンテキストで実行されるため:
- case-auto 本体は5コマンド定義を読み込まない（工程別委譲契約と結果処理のみ保持）
- Epic flow の子Issueループで case-close を起動するたびにコンテキストがリセットされる
- OU逐次処理ループで各工程を起動するたびにコンテキストがリセットされる
- case-run インライン実行時は Phase 制御（ADR-0138）と bg task 状態管理で安全性を確保する
- 結果として最大ケース（複数OU、Epic flow）でも case-auto 本体のコンテキストが飽和しない

委譲起動が失敗（ツール不在、ハードリジェクト）、または結果が delegation-unavailable、もしくは blocked/failed でかつその理由が委譲 chain の破綮の場合、当該工程をインライン実行へフォールバックする。genuine blocker（実装上の問題、スコープ外操作等）はフォールバック対象外とし停止条件として扱う。

## 代替案

1. **インライン実行の維持（通常時）**: コンテキスト枯渇が構造的に解消されないため不採用（維持）。ただし委譲失敗時のフォールバックとしてのインライン実行は例外的に許可する（harness のコンテキスト管理でコンテキスト管理）。親コンテキスト非累積要件もフォールバック時は例外扱いとする。
2. **全工程を外部実行バックエンド化**: 最大スケーラビリティだが、req-save/spec-save/case-open/case-close 用のアダプタスキルを4つ新設する実装コストが過大。
委譲で同等のコンテキスト分離効果が得られるため不採用。
将来的に case-close 等の重い工程から段階的に外部実行バックエンド化する拡張ポイントは本 ADR と両立する。
3. **工程を統合して1つの委譲で実行**: req-save〜case-close を1つの委譲起動にまとめる案。
これでは委譲先内でコンテキスト枯渇が再発するため不採用。

## 結果、影響

- case-auto 本体が薄いオーケストレータに純化し、コマンド定義サイズの削減と保守性向上が見込める。
- 親コンテキスト非累積要件が構造的に満たされる。
- 各工程の委譲起動が独立コンテキストで実行されるため、コンテキスト枯渇が解消される。
- case-auto は工程の完了結果（Issue/PR番号、pass/warn/fail）のみを受領し、次工程への入力とする。工程内部の調査過程、中間ログは各委譲先のコンテキストで完結し case-auto に累積しない。
- 各工程コマンド（req-save / spec-save / case-open / case-close）の責務、定義は変更しない（ADR-0114 と同じく委譲先の権威を維持）。
- ADR-0112（委譲一般化）の適用事例として位置づく。委譲時最小契約（inputs/side_effect_boundary/output_contract）に従う。
- SPEC `docs/specs/workflows/workflow-contracts.md`（横断ワークフロー契約）の Implementation Pattern 表に case-auto（manager-orchestrator）を追加し、汎用サブエージェント委譲契約の適用対象に加える。
- 導入後、case-auto の実行でコンテキスト枯渇指摘が解消されることを運用で確認する。
- フォールバック例外の追加により、委譲 chain 破綾時（ハーネス制限、ツール不在等）でも case-auto が処理を継続できる。コンテキスト枯渇リスクは harness のコンテキスト管理で管理し、通常時の委譲モデルは維持される。
