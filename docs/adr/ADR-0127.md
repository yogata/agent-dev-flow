# ADR-0127: case-auto 構成工程の task() 委譲によるスケーラビリティ確立

## 背景

AgentDevFlow の `case-auto` は、req-save → spec-save → case-open → case-run → case-close を順次自走する最大自走モードである（REQ-0114）。ADR-0114 により case-run の実装実行は CLI subprocess（外部実行バックエンド）へ委譲され、コンテキスト分離が実現されている。

しかし、残る4工程（req-save / spec-save / case-open / case-close）は case-auto と同一の sisyphus コンテキストでインライン実行される設計だった（REQ-0114-006「既存コマンド定義を読み込み順次実行」）。この設計は以下の課題を生んでいた。

1. **コンテキスト枯渇**: case-auto が5コマンド定義（約594行）と各コマンドが参照するスキル群（各100〜数百行）を全て自コンテキストで読込・実行するため、最大ケース（複数OU・Epic flow）でコンテキストが枯渇する。直近の case-auto 実行でこの指摘が複数回あった。
2. **REQ-0114-073 との矛盾**: 既に「Epic 全体の実装詳細を親コンテキストに累積させず、永続状態から進行・再開できること」が要件化されていたが、インライン実行モデルはこの要件に構造的に反していた。
3. **スケーラビリティの不均一**: case-run のみが CLI subprocess でスケールし、他4工程はスケールしない状態だった。

## 決定

case-auto は構成工程（req-save / spec-save / case-open / case-close）を task() で起動し、自身は薄いオーケストレータ（入力解決・工程分岐・状態引き継ぎ・停止判定・完了報告・OU/子Issueループ制御）に専念する。case-run は引き続き CLI subprocess 委譲モデル（ADR-0114）を使用する。

### 1. 委譲モデル

- **case-auto 本体**: 入力解決・工程分岐・工程間状態引き継ぎ・停止条件検出・完了報告・Epic/OUループ制御のみを担う。工程の内部ロジックを実行しない。
- **req-save / spec-save / case-open / case-close task()**: 各コマンド定義を authoritative source として読み込み実行する。case-auto から受け取った入力（draft path・OU ID・Issue番号・PR番号）を処理し、結果（保存済みファイル・Issue/PR番号・pass/warn/fail）を返す。
- **case-run**: ADR-0114 に基づき CLI subprocess（外部実行バックエンド）へ委譲。本 ADR は case-run の委譲モデルを変更しない。

### 2. 工程別委譲契約

各工程の task() 起動は ADR-0112 §5（委譲時最小契約）に従う:

| 工程 | inputs | output_contract | side_effect_boundary |
|------|--------|-----------------|---------------------|
| req-save | draft path, OU ID | 保存済みREQ/ADRリスト, draft status=saved, pass/warn/fail | `docs/requirements/**`, `docs/adr/**`, `docs/README.md`, `.agentdev/drafts/**`, git commit/push |
| spec-save | draft path | 保存済みSPECリスト, SPEC消費済みフラグ, pass/warn/fail | `docs/specs/**`, `.agentdev/drafts/**`, git commit/push |
| case-open | draft path, OU ID | Issue番号(Epic含む), pass/warn/fail | GitHub Issue作成, `.agentdev/drafts/**` 削除, `.agentdev/backlog/req-units/**` 削除, git commit/push |
| case-run | Issue番号, worktree root | completed(pr)/blocked/failed | worktree内ファイル編集, GitHub PR作成（CLI subprocess・ADR-0114） |
| case-close | Issue番号, PR番号 | マージ結果, Capture保存結果, 削除済みブランチ, pass/warn/fail | GitHub PRマージ/Issue close, `.agentdev/learning/**`, `.agentdev/intake/**`, git commit/push, branch/worktree削除 |

### 3. case-auto の保持責務（オーケストレーション）

case-auto は task() 委譲後も以下を保持する（工程内部ロジックではないため）:

- 入力解決（draft path特定・OU ID解決・Issue番号/URL分岐）
- 工程分岐（artifact_actions に基づく動的判定）
- 工程間状態引き継ぎ（Issue番号・PR番号・RU パス・capture対象情報）
- 停止条件検出（REQ-0114-016）
- 完了報告（タイミング情報含む）
- Epic Issue ステータス追跡テーブル更新（単一書き手制約・REQ-0114-081）
- Wave/子Issue選択・case-run 並行委譲制御（ADR-0125）
- OU逐次処理ループ（REQ-0114-065〜067）
- クリーンアップ検証ゲート（draft/RU 残存確認・REQ-0114-060〜063）

### 4. コンテキスト分離効果

各工程が task() で独立コンテキストを実行するため:
- case-auto 本体は5コマンド定義を読み込まない（工程別委譲契約と結果処理のみ保持）
- Epic flow の子Issueループで case-close を起動するたびにコンテキストがリセットされる
- OU逐次処理ループで各工程を起動するたびにコンテキストがリセットされる
- 結果として最大ケース（複数OU・Epic flow）でも case-auto 本体のコンテキストが飽和しない

## 代替案

1. **インライン実行の維持（現状）**: コンテキスト枯渇が構造的に解消されないため不採用。REQ-0114-073 にも反する。
2. **全工程を CLI subprocess 化**: 最大スケーラビリティだが、req-save/spec-save/case-open/case-close 用のアダプタスキルを4つ新設する実装コストが過大。task() で同等のコンテキスト分離効果が得られるため不採用。将来的に case-close 等の重い工程から段階的に subprocess 化する拡張ポイントは本 ADR と両立する。
3. **工程を統合して1つの task() で実行**: req-save〜case-close を1つの task() にまとめる案。これでは task() 内でコンテキスト枯渇が再発するため不採用。

## 結果・影響

- case-auto 本体が薄いオーケストレータに純化し、コマンド定義サイズの削減と保守性向上が見込める。
- REQ-0114-073（親コンテキスト非累積）が構造的に満たされる。
- 各工程の task() が独立コンテキストで実行されるため、コンテキスト枯渇が解消される。
- case-auto は工程の完了結果（Issue/PR番号・pass/warn/fail）のみを受領し、次工程への入力とする。工程内部の調査過程・中間ログは各 task() のコンテキストで完結し case-auto に累積しない。
- 各工程コマンド（req-save / spec-save / case-open / case-close）の責務・定義は変更しない（ADR-0114 と同じく委譲先の権威を維持）。
- ADR-0112（委譲一般化）の適用事例として位置づく。委譲時最小契約（inputs/side_effect_boundary/output_contract）に従う。
- SPEC `docs/specs/workflow-contracts.md` の Implementation Pattern 表に case-auto（manager-orchestrator）を追加し、汎用サブエージェント委譲契約の適用対象に加える。
- 導入後、case-auto の実行でコンテキスト枯渇指摘が解消されることを運用で確認する。

## 関連する決定

- ADR-0112（サブエージェント委譲の一般化と委譲時最小契約）（本判断は ADR-0112 の一般原則の case-auto への特定適用である）。各工程の task() は委譲時最小契約（inputs, side_effect_boundary, output_contract）に従う。
- ADR-0114（case-run 実行責務の外部実行バックエンド委譲）（本判断は case-run の CLI subprocess 委譲モデルを変更しない）。case-auto の残り4工程に対して task() 委譲を適用する点で補完関係にある。
- ADR-0125（case-auto Wave 内並列子Issue実行モデル）（本判断は Wave/子Issueオーケストレーション（case-auto 保持責務）を変更しない）。子Issue単位の case-run委譲・case-close起動は本判断の task() モデルに自然に統合される。
- REQ-0114（case-auto 最大自走モード）（REQ-0114-006/007 を本委譲モデルに合わせて更新し、REQ-0114-073（親コンテキスト非累積）との整合を明確にする）。
