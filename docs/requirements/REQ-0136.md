## 目的

AgentDevFlow の REQ/SPEC 責務分離を徹底するため、ワークフローを req-define → req-save → spec-save（新設）→ case-* に修正し、req-define 強化、spec-save 新設、case-* の SPEC 確定フローにより REQ/SPEC 境界を明確化する。

## 変更履歴

- 2026-06-19: 3層ゲート自動化（REQ-0136-001, 002, 004, 005, 006, 007, 008）および AgentDevFlow 本体リポジトリ特有の運用自動化（REQ-0136-017, 019）を取り下げ。GitHub Actions 依存および `repo-agentdev-integrity` skill 内の配布対象外実装に依存する仕組みは対象外とし、AgentDevFlow を適用する他プロジェクトでも使用可能な汎用仕組み（コマンド、スキル、テンプレート層）のみを対象とする。要件行 ID は欠番扱い（REQ-ID 安定 ID 規約）。
- 2026-06-19: REQ-0136-009 を廃止し REQ-0138-018 に統合。SPEC候補セクション approach は既に REQ-0138-009/REQ-0136-025 で artifact_actions に移行済み。「REQ 必須セクション維持」を REQ-0138 に移管。

## 要件

| ID | 要件 |
|---|---|
| REQ-0136-003 | REQ 健全性メトリクス（要件行数、関心対象数、成果物種別数）の定量閾値を SPEC として定義すること |
| REQ-0136-010 | `req-define` はSPEC保存対象をdraft-dataの`artifact_actions`へ分離し、各actionに既存SPECパスまたは構造化`target_spec`、operation、target_area、保存本文を保持すること |
| REQ-0136-011 | req-define が既存REQ の肥大化シグナル（要件行数、関心対象数、成果物種別数）を計測し、SPLIT 予兆を早期検知してユーザーに提案すること |
| REQ-0136-012 | `/agentdev/spec-save` はdraft-dataのSPEC対象`artifact_actions`をSPECへ保存し、SPECライフサイクルの `draft`、`accepted`、`superseded` 契約と整合すること |
| REQ-0136-013 | `operation_units` の操作種別（`operation_type`）に `spec-create` / `spec-update` が追加され、`target_spec` フィールドにより SPEC 操作も OU として管理されること。既存の操作種別（`create`/`append`/`update`）は維持すること |
| REQ-0136-014 | `case-auto` は `req-save`、`spec-save`、`case-open`、`case-run`、`case-close` の順で実行し、各保存コマンドの実行可否を対象`artifact_actions`の有無で判断すること |
| REQ-0136-015 | case-run で実装時に発見された SPEC 詳細が PR 本文の `## SPEC確定候補` セクションに記録され、case-close で SPEC 確定チェックが強化されること |
| REQ-0136-016 | inspect-promote が高確信度の検出事項（SPEC分離基準違反の確信度が高い検出事項等）を自動で採用し、HITL（人の判断を挟む）を経ずに intake/backlog パイプラインへ流入させること。自動採用の実行ログを記録し、誤検知の取り消し手順を整備すること |
| REQ-0136-018 | case-auto は今回の変更に伴い、関連ドキュメント（ガイド、SKILL.md、README、DOC-MAP）を更新すること |
| REQ-0136-020 | spec-save は draft 全体を読み、artifact_actions のうち artifact: spec の action を処理すること（REQ-0136-012 から更新） |
| REQ-0136-021 | spec-save の実行可否は work_type ではなく SPEC 対象 artifact_actions の有無で判断すること。空の場合は no-op として完了できること（REQ-0136-012/014 から更新） |
| REQ-0136-022 | spec-save は全 work_type（feature/bugfix/maintenance/docs_chore）で実行できること（REQ-0136-014 から更新） |
| REQ-0136-023 | spec-save は SPEC 対象 artifact_actions を OU 単位で部分処理せず、draft 全体の SPEC 保存対象を保存すること（REQ-0136-013 から更新） |
| REQ-0136-024 | spec-save は SPEC `status` を `accepted` にしないこと。`draft` から `accepted` への昇格は case-close の責務とすること |
| REQ-0136-025 | req-define は draft-meta.spec-candidates と SPEC候補セクションを正とする旧出力をやめ、draft-data の artifact_actions のうち artifact: spec を正とすること（REQ-0136-010 から更新。REQ-0136-009 は REQ-0138-018 に統合済み） |
| REQ-0136-026 | Issue の完了条件は Update Notes セクション参照ではなく、REQ-0101 拡張要件（達成記録先標準）に基づく正規記録先（SPEC / guide / 当該 REQ 要件行）参照で記述すること。3層ゲート達成状況、false positive 対応、再発finding対応の完了条件は、各々の正規記録先（該当 guide、該当 SPEC false positive 表、当該 REQ 要件行）に記録されたことを検証する記述とすること |
| REQ-0136-027 | spec-save は target_area が指定された update または spec-update 操作の場合、対象 SPEC ファイル内で target_area に一致する見出しから次の同レベル（または上位レベル）見出しの直前までを「セクション」として特定し、content で置換すること。見出し階層の解釈規則、複数マッチ時の挙動（最初のマッチ採用 + warn 出力）、未検出時の挙動（当該 action のスキップ + follow-up 報告）の詳細は SPEC（`docs/specs/commands/spec-save.md`）に配置すること |
| REQ-0136-028 | spec-save は target_area 未指定の draft、または operation が create/spec-create の場合は従来の「追記」動作を維持し、後方互換を保つこと。target_area が指定された場合のみ「置換」動作を適用すること |
| REQ-0136-029 | req-save/spec-save の決定的処理（REQ番号採番、ADR番号採番、要件行ID採番、frontmatter id↔ファイル名整合性確認、README/DOC-MAP/mapping-table エントリ存在確認、変更範囲検証、target_area 見出し検索）は各 script 種別の正規所有者 skill 配下の決定的スクリプトで実行すること。REQ 番号採番、要件行 ID 採番、REQ 固有検証は agentdev-req-file-manager 配下、ADR 番号採番、ADR 固有検証は agentdev-adr-file-manager 配下、SPEC 固有処理（target_area 見出し検索含む）は agentdev-spec-file-manager 配下、文書種別横断の共通検証と共有 lib は agentdev-artifact-validation 配下に配置する（AG-003、AG-009、AG-019、RU-20260722-01 合意で所有権を責務別へ変更）。command はこれらを LLM 推論で実行せず、所有 skill の公開操作契約経由で Script へ委譲することを原則とする（REQ-0136-020/027/028 の延長、REQ-0103-159 と整合） |
| REQ-0136-030 | spec-save の品質ゲートは「適用結果の整合性検証」として位置づけること。検証対象は target_area 置換結果の整合性、SPEC status の整合性（新規作成時 `status: draft` 付与）、インデックスの整合性（`docs/specs/README.md` エントリと新規 SPEC の一致）、変更範囲の妥当性とすること。内容の品質は req-define の QG-1 の責務とし、spec-save は再検証しないこと（REQ-0136-020〜028 の延長） |
| REQ-0136-031 | 全現行 REQ の要件行は command 定義または SPEC の Step 番号を直接参照せず、機能名、フェーズ名で参照すること。Step 番号は SPEC 詳細に属し、REQ 要件行に混入しないこと。検出の詳細シグナル、exemption 条件は SPEC（integrity-rule-catalog.md IR-044）に配置すること |
| REQ-0136-032 | `agentdev-spec-file-manager` 新設に伴い、SPEC 操作（SPEC 作成、更新、配置先判断、target_area による更新判断、SPEC 固有整合性確認、SPEC 固有 script 呼出契約）の正規所有者を agentdev-spec-file-manager とすること（AG-007、RU-20260722-01 合意）。req-save と spec-save の操作対象境界を明確にし、SPEC 操作は spec-save が、REQ/ADR 操作は req-save が担うこと。SPEC 内容推論、accepted 昇格判断（case-close 責務、ADR-0123 / REQ-0136-024 準拠）、ユーザー承認、commit、push は agentdev-spec-file-manager の対象外とすること。共通検証 script の重複実装は行わず、agentdev-artifact-validation の公開検証契約へ委譲すること |

## 適用範囲

- **対象**: AgentDevFlow 本体（リポジトリ全体）
 - `req-define.md`, `req-save.md`, `case-auto.md`, `case-run.md`, `case-close.md`, `inspect-promote.md`
 - `spec-save.md`（新設）
 - `doc_requirement.md` テンプレート
 - 各種 SPEC（REQ健全性メトリクス、文書パターン、ワークフロー契約 等）
 - 各種ガイド、SKILL.md、README、DOC-MAP
 - 新規 ADR-0123「SPEC lifecycle と spec-save の導入」
- **対象外**:
 - AgentDevFlow を導入する導入先リポジトリ側の独自運用
 - 3層ゲート自動化（Delta Guard / Impact Guard / Full Audit）、GitHub Actions による定期実行、`repo-agentdev-integrity` skill 内の整合性チェック実装。これらは AgentDevFlow 本体リポジトリ専用であり配布対象外（ADR-0106）
