## 目的

AgentDevFlow リポジトリの REQ/SPEC/ADR を継続的に適正運用する仕組みを、3層（事前予防・実装時検証・事後監視）すべてで自動化する。あわせてワークフローを req-define → req-save → spec-save（新設）→ case-* に修正し、REQ/SPEC 責務分離を徹底する。

## 要件

| ID | 要件 |
|---|---|
| REQ-0136-001 | check_integrity.ts は `--gate` / `--paths` / `--reqs` 引数を持ち、3層ゲート切り替えと対象ファイル絞り込みを支援すること。既存の引数なし呼び出しは後方互換性を維持すること |
| REQ-0136-002 | integrity-rule-catalog.md と req-impact-map.md をパースし、TypeScript から機械利用可能なデータ構造に変換するパーサが提供されること |
| REQ-0136-003 | REQ 健全性メトリクス（要件行数・関心対象数・artifact種別数）の定量閾値を SPEC として定義すること |
| REQ-0136-004 | `.githooks/pre-commit` が変更ファイル種別に応じた Delta Guard を自動起動し、strict 違反時は commit を block すること |
| REQ-0136-005 | `.githooks/pre-push` が変更された REQ/ADR の影響範囲（直接・間接関連REQ + 直接関連SPEC）を含む Impact Guard を自動起動し、strict 違反時は push を block すること |
| REQ-0136-006 | check_integrity.ts が IR-044（REQ/SPEC 境界違反検出）を実装し、9つの SPEC分離基準違反シグナルを検出すること。安定契約例外（REQ-0101-069）は検出対象外とすること |
| REQ-0136-007 | Full Audit が月1回の定期実行 + トリガーイベント（integrity-rule-catalog 変更時・ADR 追加時・REQ retire 時）で自動実行されること |
| REQ-0136-008 | git hooks の setup スクリプト（Windows/Unix 両対応）が提供され、チーム成员が容易に hook を有効化・無効化できること |
| REQ-0136-009 | doc_requirement.md テンプレートに SPEC候補セクション（補助セクション）が追加され、REQ 必須セクション（目的/要件/適用範囲）は維持されること |
| REQ-0136-010 | req-define が分類判定結果をもとに SPEC 候補をドラフト内の SPEC候補セクションに自動分離し、各候補に想定配置先 SPEC をメタデータとして記録すること（対象ステップの詳細は command reference 参照） |
| REQ-0136-011 | req-define が既存REQ の肥大化シグナル（要件行数・関心対象数・artifact種別数）を計測し、SPLIT 予兆を早期検知してユーザーに提案すること |
| REQ-0136-012 | `/agentdev/spec-save` コマンドが新設され、req-define で分離された SPEC 候補を SPEC ファイルに保存・確定すること。SPEC status（draft / accepted）を管理すること |
| REQ-0136-013 | operation_units の operation type に `spec-create` / `spec-update` が追加され、`target_spec` フィールドにより SPEC 操作も OU として管理されること。既存 operation type（create/append/update）は維持すること |
| REQ-0136-014 | case-auto のワークフローが req-save → spec-save → case-open → case-run → case-close の順で実行すること。draft-meta の spec-candidates が空の場合は spec-save をスキップすること。旧形式 draft は後方互換性を維持すること |
| REQ-0136-015 | case-run で実装時に発見された SPEC 詳細が PR 本文の `## SPEC確定候補` セクションに記録され、case-close で SPEC 確定チェックが強化されること |
| REQ-0136-016 | inspect-promote が高確信度 finding（SPEC分離基準違反の high-specificity signal 等）を自動 promote し、HITL を経ずに intake/backlog パイプラインへ流入させること。自動 promote 実行ログを記録し、誤検知の revoke 手順を整備すること |
| REQ-0136-017 | 既存REQ の SPEC 混入が IR-044 導入後に洗い出され、ユーザー承認を得た上で SPEC ファイルへ移行されること |
| REQ-0136-018 | 今回の変更に伴い、関連ドキュメント（ガイド・SKILL.md・README・DOC-MAP）が更新されること |
| REQ-0136-019 | REQ-0108 の3層ゲート達成状況が frontmatter `updated` フィールドで追跡されること |
| REQ-0136-020 | spec-save は draft 全体を読み、artifact_actions のうち artifact: spec の action を処理すること（REQ-0136-012 から更新） |
| REQ-0136-021 | spec-save の実行可否は work_type ではなく SPEC 対象 artifact_actions の有無で判断すること。空の場合は no-op として完了できること（REQ-0136-012/014 から更新） |
| REQ-0136-022 | spec-save は全 work_type（feature/bugfix/maintenance/docs_chore）で実行できること（REQ-0136-014 から更新） |
| REQ-0136-023 | spec-save は SPEC 対象 artifact_actions を OU 単位で部分処理せず、draft 全体の SPEC 保存対象を保存すること（REQ-0136-013 から更新） |
| REQ-0136-024 | spec-save は SPEC status を accepted にしないこと。draft から accepted への昇格は case-close の責務とすること |
| REQ-0136-025 | req-define は draft-meta.spec-candidates と SPEC候補セクションを正とする旧出力をやめ、draft-data の artifact_actions のうち artifact: spec を正とすること（REQ-0136-009/010 から更新） |

## 適用範囲

- **対象**: AgentDevFlow 本体（リポジトリ全体）
  - `check_integrity.ts`, `cli_utils.ts`
  - `.githooks/`, setup スクリプト
  - `req-define.md`, `req-save.md`, `case-auto.md`, `case-run.md`, `case-close.md`, `inspect-promote.md`
  - `spec-save.md`（新設）
  - `doc_requirement.md` テンプレート
  - 各種 SPEC（`req-health-metrics.md`, `patterns.md`, `workflow-contracts.md` 等）
  - 各種ガイド・SKILL.md・README・DOC-MAP
  - 新規 ADR-0123「SPEC lifecycle と spec-save の導入」
- **対象外**: AgentDevFlow を導入する consumer リポジトリ側の独自運用（consumer 側は `.githooks/setup` を参考にする）
