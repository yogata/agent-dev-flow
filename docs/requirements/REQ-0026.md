## 目的

intake 配下のディレクトリ構造を active queue と archive に分離し、item の lifecycle を明確にする。

現行の intake workflow では、`accepted/` と `promoted/` に同じ intake 題材が併存し、利用者から見てどの段階にあるか分かりにくい。`rejected/` が top-level directory として増大し続ける。`promoted/` の artifact に `route` / `status` frontmatter を持たせる設計は、状態管理がディレクトリと frontmatter に分散する原因となっている。また `archive/` 済みファイルを後続処理の active input として扱う設計は、終端領域の安全性を損なう。

本要件は以下を達成する:

1. active queue（inbox / accepted / promoted）と archive（rejected / promoted）を分離する
2. ディレクトリ配置を一次状態管理とし、frontmatter による状態管理を廃止する
3. 取り込み済み artifact を削除し、active queue に未処理のみを残す
4. archive を終端・履歴領域とし、後続処理の input として扱わない

## 要件

### ディレクトリ構造

| ID | 要件 |
|---|---|
| REQ-0026-001 | `.agentdev/intake/rejected/`（top-level）は使用しない。却下された intake item は `.agentdev/intake/archive/rejected/` に移動しなければならない（SHALL + MUST NOT） |
| REQ-0026-002 | `.agentdev/intake/accepted/` は、採用済みかつ `intake-promote` 未実行の raw intake item のみを置く。`intake-promote` 済み item を `.agentdev/intake/accepted/` に残してはならない（SHALL + MUST NOT） |
| REQ-0026-003 | `.agentdev/intake/promoted/` は raw intake item ではなく、後続 command 用の派生 artifact を置く。route は frontmatter ではなくサブディレクトリ（`promoted/req-define/` または `promoted/intake-open/`）で表さなければならない（SHALL + MUST NOT） |

### intake-review の振り分け

| ID | 要件 |
|---|---|
| REQ-0026-004 | `intake-review` は採用 item を `inbox/{item}.md → accepted/{item}.md` に移動し、保留 item を `inbox/{item}.md` に残し、却下 item を `inbox/{item}.md → archive/rejected/{item}.md` に移動する。`intake-review` は promoted artifact を作成してはならない（SHALL + MUST NOT） |

### intake-promote の処理

| ID | 要件 |
|---|---|
| REQ-0026-005 | `intake-promote` は `accepted/{item}.md` を読み、後続 route に応じて派生 artifact を `promoted/req-define/{item}.md` または `promoted/intake-open/{item}.md` に作成しなければならない。派生 artifact 作成に成功した場合、元 raw intake item を `accepted/{item}.md → archive/promoted/{item}.md` に移動しなければならない（SHALL） |

### 取り込み後の artifact 削除

| ID | 要件 |
|---|---|
| REQ-0026-006 | `intake-open` が GitHub Issue 作成と VERIFY に成功した場合、対象 artifact（`promoted/intake-open/{item}.md`）を削除しなければならない。失敗した場合は削除してはならない（SHALL + MUST NOT） |
| REQ-0026-007 | `req-define` が `promoted/req-define/` の artifact を取り込みし要件定義が完了した場合、対象 artifact を imported として処理しなければならない。imported 処理（削除）は後続ルート（`req-save` または `case-open`）の成功後に実行する。`req-define` 完了直後には削除しない（SHALL） |

### 状態管理方針

| ID | 要件 |
|---|---|
| REQ-0026-008 | 状態判定の一次情報はディレクトリ配置とする。`route` / `status` frontmatter による active queue 管理は行ってはならない。許容する frontmatter は archive 配下の履歴補助情報（`archived_at`、`archive_reason`、`promoted_artifact` 等）のみとする（SHALL + MUST NOT） |
| REQ-0026-009 | `.agentdev/intake/archive/` 配下の item は後続処理の active input として扱ってはならない。`archive/promoted/` の item を `intake-promote` 対象にしてはならない。`archive/rejected/` の item を `intake-review` 対象にしてはならない（MUST NOT） |
| REQ-0026-010 | 以下の状態を維持する: `inbox/` は未レビューのみ、`accepted/` は promote 待ちのみ、`promoted/req-define/` は req-define 待ちのみ、`promoted/intake-open/` は intake-open 待ちのみ、`archive/` は終端履歴のみ（SHALL） |

### 既存REQ への影響

| ID | 要件 |
|---|---|
| REQ-0026-011 | 本要件は REQ-0017-036（「intake item の状態表現のためにディレクトリ移動を必須にしてはならない」）を intake lifecycle domain において narrow する。新設計は frontmatter 状態管理を廃止しディレクトリ配置に一本化するものであり、過剰な複雑化を防ぐという REQ-0017-036 の本来の意図と合致する（SHALL） |

## command 変更方針

### intake-review

変更後:

```text
inbox → accepted
inbox → archive/rejected
inbox に保留
```

削除:

```text
inbox → rejected
```

### intake-promote

変更後:

```text
accepted → promoted/req-define
accepted → promoted/intake-open
accepted → archive/promoted（派生 artifact 作成成功後）
```

削除:

```text
accepted に元 item を残す
promoted artifact に route/status frontmatter を必須化する
```

### intake-open

変更後:

```text
promoted/intake-open を読む
Issue 作成成功後に対象 artifact を削除
```

削除:

```text
status: issued への更新
```

### req-define

変更後:

```text
promoted/req-define を明示入力ファイルとして読む
要件定義完了後、後続ルート（req-save / case-open）成功時に imported 処理（削除）
```
