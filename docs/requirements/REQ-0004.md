## 目的

要件（REQ）ファイルとアーキテクチャ決定記録（ADR）の管理規則を定義する。REQは `REQ-{NNNN}.md` を永続正本とし、観点別体系化には `views/*.md` を非正本ビューとして使用する。ADRは `ADR-{NNNN}.md` を判断履歴の正本とし、本文を area / topic / view 別ファイルへ再配置しない。両者の整合性を維持するための規約である。area-based current document 構造への移行方針は撤回し、per-file 構造を永続標準とする。

## 要件

### 操作分類の判定軸

| ID | 要件 |
|---|---|
| REQ-0004-028 | req-file-manager スキルの既存REQ照合方法論に「操作分類の判定軸」セクションを新設し、関連REQ特定後の CREATE/APPEND/UPDATE 選択において以下の5つの判定軸を評価するものとする（SHALL）: 対象コンポーネント（重み:高 — 同じUI部品・画面・機能単位への変更か）、対象スコープ（重み:高 — 同じ利用者操作・業務範囲・表示範囲への変更か）、目的の連続性（重み:高 — 既存REQの目的に対する追加・補強・調整か）、要件行追加可能性（重み:高 — 既存REQの要件テーブルへ自然に新規行として追加できるか）、独立目的の有無（重み:高 — 既存REQとは独立した目的を持つ新規要件か） |
| REQ-0004-029 | REQファイルの title / tags / 適用範囲には、将来の APPEND/UPDATE 判定（REQ-0004-028の判定軸）で参照できる対象コンポーネント・対象画面・対象操作・対象スコープを明示するものとする。frontmatterへの新規フィールド追加は行わず、既存セクションの記述規則として定義する（SHALL） |

### REQ正本構造とviews運用

| ID | 要件 |

| REQ-0004-032 | `docs/requirements/views/*.md` は、正本REQを観点別に体系化した非正本文書として扱うこと（SHALL） |

| REQ-0004-034 | `views/*.md` と `REQ-{NNNN}.md` が矛盾する場合、`REQ-{NNNN}.md` を正とすること（SHALL） |

| REQ-0004-036 | `docs/requirements/areas/` は作成しないこと（SHALL） |

| REQ-0004-039 | `docs/requirements/README.md` には、`REQ-{NNNN}.md` が正本であり、`views/*.md` が非正本の観点別ビューであることを明記すること（SHALL） |
| REQ-0004-040 | `views/*.md` の冒頭にも、当該ファイルが非正本ビューであり、正本は `REQ-{NNNN}.md` であることを明記すること（SHALL） |
| REQ-0004-041 | `INDEX.md` は補助インデックスとする。generated artifact とするか手動管理とするかは本要件では未確定とすること（SHALL） |

### views更新契機と整合性維持

| ID | 要件 |
|---|---|
| REQ-0004-042 | `REQ-{NNNN}.md` の CREATE / APPEND / UPDATE 時には、関連する `docs/requirements/views/*.md` への影響有無を確認すること。影響がある場合は、同一変更内で `views/*.md` を更新すること（SHALL） |
| REQ-0004-043 | `views/*.md` への影響がない場合、更新は不要とする。ただし、完了報告で「views更新なし」と明示すること（SHALL） |
| REQ-0004-044 | `views/*.md` の更新が必要だが当該作業の範囲を超える場合、REQ保存は中止せず、follow-up として明示すること（SHALL） |
| REQ-0004-045 | `views/*.md` と `REQ-{NNNN}.md` の矛盾を検出した場合、`REQ-{NNNN}.md` を正とし、`views/*.md` を修正対象とすること（SHALL） |
| REQ-0004-046 | 既存の `req-save` は、引き続き `docs/requirements/REQ-{NNNN}.md` を保存先とすること（SHALL） |

### ADR作成基準と参照ルール

| ID | 要件 |
|---|---|
| REQ-0004-057 | ADRは、アーキテクチャ上の重要性、長期的影響、逆転困難性、技術スタック、主要方式、責務分界、主要構造を決める場合に作成対象とすること（SHALL） |
| REQ-0004-058 | 単純なバグ修正、既存設計に従う通常実装、軽微なリファクタリング、UI微調整はADR対象外とすること（SHALL） |
| REQ-0004-059 | ADR-0004 は削除せず、area-based SSoT 方針を撤回した判断履歴として残すこと（SHALL） |
| REQ-0004-060 | ADR-0004 は後継ADRにより置き換え、`superseded-by:[ADR-XXXX]` または `deprecated` に遷移させること（SHALL） |
| REQ-0004-061 | REQ → ADR の参照は許可すること。要件が依存する判断を示すために使用する（MAY） |
| REQ-0004-062 | ADR → ADR の参照は許可すること。関連判断・後継判断を示すために使用する（MAY） |
| REQ-0004-063 | Issue → ADR の参照は許可すること。実装対象が従う判断を示すために使用する（MAY） |
| REQ-0004-064 | ADR → Issue の逆参照は原則禁止すること。ADRを実装履歴に引きずらせないため（SHALL） |
| REQ-0004-065 | ADR本文からREQへの逆参照は必須にしないこと（SHALL） |
| REQ-0004-066 | `docs/adr/README.md` の Related REQ / Decision Map は分類ビューであり、ADR本文のSSoTではないこと（SHALL） |
| REQ-0004-067 | `docs/adr/README.md` の Related REQ / Decision Map は、REQ本文に記載された Related ADRs などの正本情報から構成すること（SHALL） |

### views更新契機一覧

| 契機 | `views/` 更新 |
|------|-------------|
| REQ新規作成 | 関連する view がある場合は追記する |
| REQへのAPPEND | view に影響する場合は更新する |
| REQのUPDATE | view の記述と矛盾する場合は更新する |
| REQ削除・deprecated・superseded | view からの扱いを更新する |
| view未作成の領域 | 更新不要。必要なら follow-up にする |
