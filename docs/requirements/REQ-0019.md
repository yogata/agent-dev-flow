## 目的

AgentDevFlow の intake workflow と learning pipeline の責任境界を明確にし、両者が独立した領域として正しく協調するようにする。また、通常の case lifecycle に、非ブロッキングな発見を本筋に混ぜずに退避する導線を組み込む。

現在、README / specs / command-map / REQ 間で intake が learning pipeline の前段であるかのような説明が残っており、intake item の定義に learning item が含まれ、intake と learning の分割判断ルールが SSoT 化されていない。さらに case-run / case-close に、本筋外の発見を intake / learning 候補として退避する導線が弱い。

本要件は以下を達成する:

1. intake と learning の定義・判断質問・分割ルール（split rule）を SSoT として文書化する
2. intake が learning pipeline の前段であるという誤解を排除し、後続ルートの説明を全ファイルで一致させる
3. case-run に本筋外の発見を intake / learning 候補として退避する方針を追加する
4. case-close に post-run capture（learning capture + intake capture）を追加する
5. intake / learning / integrity の各 command の責務を split rule と整合させる
6. `docs/tips/` と `.agentdev/learning/` の位置づけを現状と矛盾なく説明する

## 要件

### Intake / Learning 定義と境界

| ID | 要件 |
|---|---|
| REQ-0019-001 | AgentDevFlow は `intake` を、今回の作業本筋では扱わないが後で対応要否を判断すべき具体的な作業候補・不整合・規約違反・未回収課題を保存する workflow と定義すること（SHALL） |
| REQ-0019-002 | AgentDevFlow は `learning` を、作業中の失敗・回避・修正・判断ミス・手順漏れから次回以降の再発防止に使う知見を保存・分類・昇華する pipeline と定義すること（SHALL） |
| REQ-0019-003 | intake を learning pipeline の前段として扱ってはならない。`intake-promote` は learning pipeline の入力を生成しないこと（MUST NOT） |
| REQ-0019-004 | `intake-promote` は review 済み intake item を `req-define` 用または `intake-open` 用の入力 artifact に整形すること。learning pipeline への変換は行わないこと（SHALL + MUST NOT） |
| REQ-0019-005 | 同じ観測から具体的な修正対象と再発防止知見の両方が得られる場合、intake item と learning item を別々に作成すること（SHALL） |
| REQ-0019-006 | 具体的に直す対象があるものは intake、次回同じ失敗を防ぐ知見は learning、両方ある場合は分割する、という split rule を SSoT として文書化すること（SHALL） |

### Split Rule SSoT

| ID | 要件 |
|---|---|
| REQ-0019-007 | split rule の SSoT を `.opencode/skills/agentdev-workflow-lifecycle/reference/capture-boundaries.md` に配置すること（SHALL） |
| REQ-0019-008 | capture-boundaries.md は以下を含むこと（SHALL）: (1) intake / learning の定義 (2) 判断質問（「後で対応すべきか？」「次回同じ失敗をどう防ぐか？」） (3) split rule (4) 代表例と非対象例のテーブル (5) 同じ観測から intake item と learning item を分ける具体例 (6) 各 command が参照すべきルール |

### case-run: 非ブロッキング発見の退避

| ID | 要件 |
|---|---|
| REQ-0019-009 | `case-run` は、作業中に本筋外の不整合・規約違反・小改善候補を見つけた場合、現在の完了条件を勝手に拡大して修正してはならないこと（SHOULD / MUST NOT） |
| REQ-0019-010 | `case-run` は、本筋外の発見を intake 候補として記録し、case-close または後続処理が拾える導線を持つこと（SHOULD） |
| REQ-0019-011 | `case-run` は、作業中の失敗・回避・判断ミス・手順漏れを learning 候補として区別できるようにすること（SHOULD） |
| REQ-0019-012 | `case-run` は intake 候補と learning 候補を混ぜた単一成果物にしないこと（SHALL） |

### case-close: post-run capture

| ID | 要件 |
|---|---|
| REQ-0019-013 | `case-close` は、完了処理中に発見した本筋外の変更候補のうち、具体的な修正対象が特定できるものだけを自律的に intake item として保存すること（SHALL） |
| REQ-0019-014 | `case-close` は、曖昧な変更候補、修正対象が特定できない候補、単なる違和感・改善案を自律保存せず、完了報告に候補として提示すること（SHALL） |
| REQ-0019-015 | `case-close` は、intake item の保存先を `.agentdev/intake/inbox/` に限定すること（SHALL） |
| REQ-0019-016 | `case-close` は、intake item の採用可否判断、GitHub Issue 作成、後続対応の優先度判断を行わないこと（SHALL） |
| REQ-0019-017 | `case-close` は、今回の Issue / PR の完了条件に含まれる未対応事項を intake item に逃がして完了扱いしないこと（SHALL） |
| REQ-0019-018 | `case-close` は、既存の learning capture に加えて post-run capture を行い、未対応の積み残し候補を intake、再発防止知見を learning に分けること（SHALL） |
| REQ-0019-019 | `case-close` は、intake item と learning item を作成・通知する場合、件数・保存先・次ステップを別々に表示すること（SHALL） |
| REQ-0019-020 | `case-close` の既存の learning capture の原則（ユーザーに学びを問わない、承認なしで蓄積し通知する）は維持すること（SHALL） |

### Integrity Check の位置づけ

| ID | 要件 |
|---|---|
| REQ-0019-021 | `integrity-check` の finding は原則 intake 対象であること。finding の発生原因や予防策がある場合のみ learning 対象にもなりうることを明記すること（SHALL） |
| REQ-0019-022 | `integrity-check` は、検出結果を learning item に直接変換する責務を持たないこと（MUST NOT） |

### Intake Command 群の責務境界

| ID | 要件 |
|---|---|
| REQ-0019-023 | `intake-capture` / `intake-review` / `intake-promote` / `intake-open` は learning item の保存・分類・昇華を担当しないこと（MUST NOT） |
| REQ-0019-024 | `intake-capture` は「未分類の作業候補・不整合・規約違反・未回収課題」を主語とし、作業知見だけの内容は対象外とすること（SHALL） |
| REQ-0019-025 | `intake-review` は review 観点に「これは intake ではなく learning に分けるべきものではないか」を含めること。ただし learning item を作成する責務を持たないこと（SHALL） |
| REQ-0019-026 | `intake-open` は intake item 群から learning item を作成しないこと（MUST NOT） |

### Learning Command / Skill の責務境界

| ID | 要件 |
|---|---|
| REQ-0019-027 | `agentdev-learning-capture` は intake item を作成しないこと。具体的な修正対象が残る場合は split rule に従って intake workflow に委ねること（SHALL） |
| REQ-0019-028 | `agentdev-learning-capture` は、learning が「改善提案そのもの」ではなく「改善提案へ昇華されうる再発防止知見」であることを明記すること（SHALL） |

### 説明整合性

| ID | 要件 |
|---|---|
| REQ-0019-029 | README / specs / command-map / command files / REQ 間で、intake と learning の責任境界および後続ルートの説明が一致していること（SHALL） |
| REQ-0019-030 | intake workflow の基本フローは `積み残し候補発生 → intake-capture / intake-from-github / integrity-check → intake-review → req-define または intake-promote → intake-open` とすること。`intake-promote → learning pipeline` という説明を全箇所から削除すること（SHALL） |
| REQ-0019-031 | learning pipeline の基本フローは `学び発生 → agentdev-learning-capture → learning-refine → learning-promote → Requirement Source staging stub → req-define` とすること（SHALL） |

### Storage 整合性

| ID | 要件 |
|---|---|
| REQ-0019-032 | `docs/tips/` と `.agentdev/learning/` の位置づけについて、当面の learning pipeline の canonical storage は既存互換のため `docs/tips/` とすること。`.agentdev/learning/` への移行は別 Requirement / 別 Case とすることを明記すること（SHALL） |
