> **履歴として残す理由**: Command薄化・Skill切り出しの判断基準の設計経緯、1 command = 1 orchestration skill を原則としない判断の記録

## 目的

Command / Skill / Template / Script の責任分界を明確化し、learning pipeline の昇華結果（staging stub）を req-define の汎用入力ファイルとして扱えるようにする。

現在、一部の command に詳細な状態機械・schema・分類表・テンプレート選定ルール・共通安全手順が入り込んでおり、Command / Skill / Template / Script の責任分界が曖昧になっている。また、learning-promote の出力を req-define に渡す前提がある一方で、req-define 側の Input / Guardrails が自然文・Issue URL・エラーログに限定されており、要件化前 source file を扱う責務が抽象化されていない。

今回の改定では、Command 責任分界の再整理と、learning pipeline の要件ソース化を一体で扱う。

### artifact-boundaries.md 項目リスト詳細

各アーティファクトの責務を以下の項目リストで定義する:

**Command — 置くもの**:
- ユーザー向け入口
- Input / Output
- 高レベル Steps（5〜12個）
- 使用する Skill（frontmatter load_skills + Steps 内参照）
- 成果物の読み書き対象
- command 固有 Guardrails

**Command — 置かないもの**:
- 大きな状態機械
- 詳細な判定表
- schema
- template
- script
- 共通安全手順
- 複数 command で使う workflow protocol

**Skill — 置くもの**:
- 再利用可能な判断基準
- domain knowledge
- workflow protocol
- safety protocol
- テンプレート選定ルール
- owner artifact の構造定義

**Template — 置くもの**:
- 出力構造
- 見出し
- 必須/任意マーカー
- プレースホルダー

**Script — 置くもの**:
- 採番
- validation
- INDEX 生成
- Markdown 更新
- 整合性チェック

**Orchestration skill 作成判断基準**:
- 大きな状態機械を持つ
- 再開ポイント検出が必要
- CI loop を含む
- Wave scheduling を含む
- サブエージェント protocol を含む

上記のいずれかを満たす場合にのみ orchestration skill 化を認める。
