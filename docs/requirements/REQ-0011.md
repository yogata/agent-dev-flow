## 目的

issue-create, issue-work, issue-close コマンドが GitHub Issue/PR に書き込んだ内容の品質を、書き込み直後に自動検証する。文字化け（エンコーディング問題）およびフォーマット崩れ（Markdown構造の破壊・テンプレート準拠性）を検出し、問題があれば自動リトライすることで、Issue/PRの内容品質を担保する。

## 要件

| ID | 要件 |
|---|---|
| REQ-0011-001 | 各コマンド（issue-create, issue-work, issue-close）の完了報告前に、書き込んだ Issue/PR の本文を `gh issue view` / `gh pr view` の `--json body` で読み戻し、元テキストと比較する検証ステップを追加すること（SHALL） |
| REQ-0011-002 | 検証では以下の3点を確認すること（SHALL）: (a) 文字化けパターンの検出 (b) Markdown構造の整合性確認 (c) テンプレート必須セクションの存在確認 |
| REQ-0011-003 | 文字化けパターンの検出では、元テキストと読み戻しテキストを比較し、日本語文字の差異・制御文字の混入・非Unicode文字を検出すること（SHALL） |
| REQ-0011-004 | Markdown構造の整合性確認では、テーブル行のカラム数、チェックボックス記法（`- [ ]` / `- [x]`）、コードブロック（`` ``` `` ペア）、リスト構造のインデントが維持されていることを確認すること（SHALL） |
| REQ-0011-005 | テンプレート必須セクションの存在確認では、該当テンプレートの `【必須】` マーカーが付与されたセクション見出しが全て読み戻しテキストに存在することを確認すること（SHALL） |
| REQ-0011-006 | 各コマンドで書き込み操作が複数ある場合（Issue作成＋コメント追加等）、各書き込みごとに検証すること（SHALL） |
| REQ-0011-007 | 問題を検出した場合、差分の種類に応じて3段階で対応すること（SHALL）: (a) 同一内容リトライ（最大2回＝計3回）で解決する可能性がある差分は再送する (b) 書き込み内容自体に問題がある差分は内容を再生成する（再生成後は(a)に戻さない） (c) (a)(b)で解決しない場合はユーザーに報告して停止する（SHALL） |
| REQ-0011-008 | 検証結果（成功/失敗/リトライ回数/検出した問題の詳細）を完了報告に含めること（SHOULD） |
| REQ-0011-009 | 読み戻しには `agentdev-gh-cli` スキルの安全な読み取り手順（一時ファイル経由で Read tool 使用）を使用すること（SHALL） |
| REQ-0011-010 | 検証ロジックは `agentdev-gh-cli` スキル内に `VERIFY操作` セクションとして定義し、issue-create / issue-work / issue-close の各コマンドから参照可能にすること（SHALL） |

## 適用範囲

- **対象**: issue-create, issue-work, issue-close, issue-update, issue-backlog コマンドにおける Issue/PR への書き込み操作（Issue作成、PR作成、コメント追加、Issue本文更新）
  - issue-update: 通常検証（本文更新・コメント追加の各書き込み操作を検証）
  - issue-backlog: Epic本文・子Issue本文=通常検証、抽出コメント=best-effort（警告）
- **対象外**: issue-req, issue-save-req, issue-next コマンド。gh CLI以外の書き込み操作。レビュー・マージ等の非テキスト操作
