## 目的

AgentDevFlow の ADR ライフサイクル、文書種別責務境界、runtime独立性、workflow 分類、状態モデルを正規化し、docs 永続文書と runtime 配布物の責務混線を解消する。

## 要件

| ID | 要件 |
|---|---|
| REQ-0112-001 | ADR status 形式を正規化すること: `status: superseded` + frontmatter に `superseded_by: ADR-XXXX`（SHALL）。旧形式 `superseded-by:[ADR-XXXX]` は使用しない |
| REQ-0112-002 | ADR テンプレートから `## 実装後の記録` セクションを削除すること（SHALL） |
| REQ-0112-003 | `req-save` は合意済み ADR を `accepted` として作成すること（SHALL）。新規 ADR の初期 status は `proposed` とし、壁打ち合意後に `accepted` に遷移させる |
| REQ-0112-004 | `case-run` は同一ブランチ・同一 PR 内で ADR 変更を含めること（SHALL）。ADR 変更のみの独立 PR は原則として作成しない |
| REQ-0112-005 | `case-close` は ADR 差分の最終確認ゲートとして機能し、新規判断の追加は行わないこと（SHALL） |
| REQ-0112-006 | `accepted` status の ADR のみを現行根拠として使用すること（SHALL）。`proposed` / `superseded` / `deprecated` の ADR を現行要件判断の根拠として引用しない |
| REQ-0112-007 | docs 永続文書（REQ/ADR/SPEC/guides）から特定 RU-ID を根拠参照として引用しないこと（SHALL）。RU は `case-open` で削除される一時成果物であり、docs の永続根拠に不適である |
| REQ-0112-008 | REQ ファイルの Requirement Source セクションに RU パスを記録しないこと（SHALL）。req-save は RU パスを docs 永続文書に保存しない |

| REQ-0112-016 | runtime command が `docs/specs/system.md` / `docs/specs/patterns.md` に固定依存している構造を削除すること（SHALL）。runtime command は自己完結し、repo 内部 SPEC に依存しない |
| REQ-0112-017 | `case-run` は 150 行を目標とし、200 行を超える状態を維持しないこと（SHALL）。詳細判断は skill/reference に切り出す |

| REQ-0112-030 | command-map を状態遷移エンジン化しないこと（SHALL） |
| REQ-0112-031 | ADR文書はADR-0017責務境界に適合すること（SHALL）。技術判断を含まない文書はADRとして扱わない |
| REQ-0112-032 | 技術判断を含むADRは本文を圧縮し、REQ/SPEC詳細を移管すること（SHALL） |
| REQ-0112-033 | 技術判断を含まないADRはdeprecated化し、現行根拠として使用しないこと（SHALL）。本文に誤分類理由・移管状況を記載する |
| REQ-0112-034 | 既存deprecated/supersededのADRは原則再編集しないこと（SHALL）。docs/adr/README.mdの矛盾のみ最小限修正を推奨する（SHOULD） |
| REQ-0112-035 | docs/adr/README.mdの各ビューはADR本文のstatusと矛盾しないこと（SHALL） |
| REQ-0112-036 | deprecated化したADRの分離内容は既存REQ/SPECへ移管すること（SHALL）。判定基準: 振る舞い・制約・状態はREQ、schema・path・lifecycleはSPEC |
| REQ-0112-037 | 移管後にREQ/SPEC側で仕様情報が欠落しないこと（SHALL）。移管元・先の照合確認を必須とする |
| REQ-0112-038 | req-defineはADR候補を提示する前にREQ/SPEC相当判定を行うこと（SHALL）。除外基準: 仕様変更のみ・command動作仕様・workflow定義・状態遷移・命名規約・directory規約・運用ルール・template変更・入出力形式・非技術的合意 |
| REQ-0112-039 | REQ/SPEC相当と判定された場合、ADR候補から除外しREQ/SPEC更新候補として整理すること（SHALL）。分類: 振る舞い→REQ、schema/lifecycle→SPEC、方針→guide |
| REQ-0112-040 | req-saveはADR保存直前に妥当性を再検証すること（SHALL）。技術判断不在の場合は保存を停止し理由を報告する |
| REQ-0112-041 | adr-guidelinesにADR禁止条件を明示すること（SHALL）。仕様変更のみ・command仕様・artifact contract変更はADR化禁止 |
| REQ-0112-042 | adr-file-managerはADR要否の意味判定を主責務とすることを禁止すること（SHALL）。adr-guidelinesの判定完了を前提として明記する |
| REQ-0112-043 | repo-agentdev-integrityはADR意味妥当性のstrict判定を禁止すること（SHALL）。誤分類兆候のwarning/observation検出を推奨する（SHOULD）。兆候: 技術判断不在・REQ/SPEC相当内容の混入・ADR-0017適合外・文書種別不一致 |
| REQ-0112-044 | ADR体系の全面改定・責務再分類・圧縮を目的とするユーザー承認済みの req-save 実行では、既存 deprecated/superseded ADR および誤分類 ADR の最小限を超える編集を許可してもよい（MAY）。ただし、編集目的、対象ADR、変更種別、移管先REQ/SPEC/guide、現行根拠として残すADRをドラフトまたは完了報告に明示すること（SHALL） |
| REQ-0112-045 | ADR全面改定時でも、accepted ADR の決定内容を別判断へ意味変更してはならない（SHALL NOT）。既存判断を覆す場合は新規ADRを作成し、旧ADRを superseded または deprecated として扱うこと（SHALL） |
| REQ-0112-046 | ADR全面改定後、docs/adr/README.md は ADR本文の status と矛盾せず、accepted ADR のみを現行根拠として読める分類ビューを提供すること（SHALL） |
| REQ-0112-047 | ADR全面改定後の現行ADR baseline は `ADR-0101` 以降の番号帯を使用すること（SHALL）。`ADR-0001` から `ADR-0099` は再編前ADRの履歴番号帯として扱うこと（SHALL） |
| REQ-0112-048 | 再編前ADRは、現行ADR baseline 作成後に `docs/adr/retired/ADR-00XX.md` へ移動すること（SHALL）。旧ADRを現行根拠として引用してはならず、履歴参照する場合は retired path を明示すること（SHALL） |
| REQ-0112-049 | ADR全面改定時、旧ADRの単純リネームではなく、必要な判断を `ADR-01XX` の新規ADRへ引き継ぎ、旧ADRは retired ADR として保持すること（SHALL） |
| REQ-0112-050 | integrity-check は current ADR collection（`docs/adr/ADR-01XX.md`）と retired ADR collection（`docs/adr/retired/ADR-00XX.md`）を区別して検査すること（SHALL）。retired ADR への履歴参照は現行根拠引用 warning と区別すること（SHOULD） |

## 適用範囲

- **対象**: ADR ライフサイクル標準化、RU 参照排除、work_type 移行、command 薄型化、integrity 反転、docs 整合、状態モデル明確化、ADR全面改定時例外規定、ADR-01XX baseline、旧ADR retired移動、current/retired ADR collection区別
- 主対象 REQ: REQ-0101/0103/0104/0105/0108/0109
- **対象外**: 個別 command の詳細実装手順、テストコードの具体的記述、各 SPEC ファイルの本文内容
- 新規ADRの具体的本文作成
- 実装コードの変更
- Issue/PRの作成
- retired REQの大規模再編集
