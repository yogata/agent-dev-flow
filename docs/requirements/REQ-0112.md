## 目的

AgentDevFlow の ADR ライフサイクル、文書種別責務境界、runtime独立性、workflow 分類、状態モデルを正規化し、docs 永続文書と runtime 配布物の責務混線を解消する。

## 要件

| ID | 要件 |
|---|---|
| REQ-0112-001 | ADR status 形式を正規化すること: `status: superseded` + frontmatter に `superseded_by: ADR-XXXX`。旧形式 `superseded-by:[ADR-XXXX]` は使用しない |
| REQ-0112-002 | ADR テンプレートから `## 実装後の記録` セクションを削除すること |
| REQ-0112-003 | `req-save` は合意済み ADR を `accepted` として作成すること。新規 ADR の初期 status は `proposed` とし、壁打ち合意後に `accepted` に遷移させる。ADR status の有効値は `proposed` / `accepted` / `superseded` / `deprecated` の4種類とする |
| REQ-0112-004 | `case-run` は同一ブランチ・同一 PR 内で ADR 変更を含めること。ADR 変更のみの独立 PR は原則として作成しない |
| REQ-0112-005 | `case-close` は ADR 差分の最終確認ゲートとして機能し、新規判断の追加は行わないこと |
| REQ-0112-006 | `accepted` status の ADR のみを現行根拠として使用すること。`proposed` / `superseded` / `deprecated` の ADR を現行要件判断の根拠として引用しない |
| REQ-0112-007 | docs 永続文書（REQ/ADR/SPEC/guides）から特定 RU-ID を根拠参照として引用しないこと。RU は `case-open` で削除される一時成果物であり、docs の永続根拠に不適である |
| REQ-0112-008 | REQ ファイルの Requirement Source セクションに RU パスを記録しないこと。req-save は RU パスを docs 永続文書に保存しない |
| REQ-0112-009 | `case-open` は RU 削除対象を docs 永続文書の Requirement Source ではなく、一時メタ情報から取得すること |
| REQ-0112-010 | `ru-sources` frontmatter フィールドを使用しないこと |
| REQ-0112-011 | work_type を `bugfix` / `feature` / `maintenance` / `docs_chore` の 4 値に固定すること |
| REQ-0112-012 | scale は `feature` のみ `standard` / `large` を持つこと。他の work_type は scale を持たない |
| REQ-0112-013 | workflow_route を保存せず、work_type + scale から都度導出すること。派生ルール: feature+standard → req_backed_case, feature+large → epic_case, その他 → direct_case |
| REQ-0112-014 | draft-meta に以下を保存しないこと: pattern, workflow_route, branch_type, labels, implementation_pattern, secondary_pattern, load_skills |
| REQ-0112-015 | Pattern A/B/C/D の分類を現行分類から退場させること。work_type + scale + workflow_route の3層分類に置き換える |
| REQ-0112-016 | runtime command が `docs/specs/system.md` / `docs/specs/patterns.md` に固定依存している構造を削除すること。runtime command は自己完結し、repo 内部 SPEC に依存しない |
| REQ-0112-017 | `case-run` は 150 行を目標とし、200 行を超える状態を維持しないこと。詳細判断は skill/reference に切り出す |
| REQ-0112-018 | `learning-*` / `intake-*` command を入口 command に縮小すること。詳細処理は skill/script に委譲する |
| REQ-0112-019 | `docs-check` の frontmatter 検査において、implementation_pattern / secondary_pattern / load_skills の存在を禁止すること |
| REQ-0112-020 | command authoring standards も implementation_pattern / secondary_pattern / load_skills の frontmatter 記述を禁止すること |
| REQ-0112-021 | `docs/specs/` を repo-internal 設計文書として明記すること。runtime 配布物の依存先ではない |
| REQ-0112-022 | `docs/guides/` を人間向け navigation 層に位置づけること。規範的権限を持たない |
| REQ-0112-023 | 6 マイクロフェーズ（requirement/analyzed/created/in_progress/review/done）は状態管理モデルではなく説明用ラベルとすること。AgentDevFlow は全体横断の状態遷移モデルを持たない |
| REQ-0112-024 | `.agentdev/` を domain state として明確化し、配布物ではないことを明記すること |
| REQ-0112-025 | `.sisyphus/` を gitignore 対象とすること |
| REQ-0112-026 | `.sisyphus/drafts/` を一時領域として明確化すること |
| REQ-0112-027 | REQ / SPEC に workflow status を追加しないこと |
| REQ-0112-028 | intake promoted に route / status を追加しないこと |
| REQ-0112-029 | Issue / PR の状態を docs に複製しないこと |
| REQ-0112-030 | command-map を状態遷移エンジン化しないこと |
| REQ-0112-031 | ADR文書はADR-0017責務境界に適合すること。技術判断を含まない文書はADRとして扱わない |
| REQ-0112-032 | 技術判断を含むADRは本文を圧縮し、REQ/SPEC詳細を移管すること |
| REQ-0112-033 | 技術判断を含まないADRはdeprecated化し、現行根拠として使用しないこと。本文に誤分類理由・移管状況を記載する |
| REQ-0112-034 | 既存deprecated/supersededのADRは原則再編集しないこと。docs/adr/README.mdの矛盾のみ最小限修正とする |
| REQ-0112-035 | docs/adr/README.mdの各ビューはADR本文のstatusと矛盾しないこと |
| REQ-0112-036 | deprecated化したADRの分離内容は既存REQ/SPECへ移管すること。判定基準: 振る舞い・制約・状態はREQ、schema・path・lifecycleはSPEC |
| REQ-0112-037 | 移管後にREQ/SPEC側で仕様情報が欠落しないこと。移管元・先の照合確認を必須とする |
| REQ-0112-038 | req-defineはADR候補を提示する前にREQ/SPEC相当判定を行うこと。除外基準: 仕様変更のみ・command動作仕様・workflow定義・状態遷移・命名規約・directory規約・運用ルール・template変更・入出力形式・非技術的合意 |
| REQ-0112-039 | REQ/SPEC相当と判定された場合、ADR候補から除外しREQ/SPEC更新候補として整理すること。分類: 振る舞い→REQ、schema/lifecycle→SPEC、方針→guide |
| REQ-0112-040 | req-saveはADR保存直前に妥当性を再検証すること。技術判断不在の場合は保存を停止し理由を報告する |
| REQ-0112-041 | adr-guidelinesにADR禁止条件を明示すること。仕様変更のみ・command仕様・artifact contract変更はADR化禁止 |
| REQ-0112-042 | adr-file-managerはADR要否の意味判定を主責務とすることを禁止すること。adr-guidelinesの判定完了を前提として明記する |
| REQ-0112-043 | repo-agentdev-integrityはADR意味妥当性のstrict判定を禁止すること。誤分類兆候のwarning/observation検出を行うこと。兆候: 技術判断不在・REQ/SPEC相当内容の混入・ADR-0017適合外・文書種別不一致 |
| REQ-0112-044 | ADR体系の全面改定・責務再分類・圧縮を目的とするユーザー承認済みの req-save 実行では、既存 deprecated/superseded ADR および誤分類 ADR の最小限を超える編集を許可してよい。ただし、編集目的、対象ADR、変更種別、移管先REQ/SPEC/guide、現行根拠として残すADRをドラフトまたは完了報告に明示すること |
| REQ-0112-045 | ADR全面改定時でも、accepted ADR の決定内容を別判断へ意味変更してはならない。既存判断を覆す場合は新規ADRを作成し、旧ADRを superseded または deprecated として扱うこと |
| REQ-0112-046 | ADR全面改定後、docs/adr/README.md は ADR本文の status と矛盾せず、accepted ADR のみを現行根拠として読める分類ビューを提供すること |
| REQ-0112-047 | ADR全面改定後の現行ADR baseline は `ADR-0101` 以降の番号帯を使用すること。`ADR-0001` から `ADR-0099` は再編前ADRの履歴番号帯として扱うこと |
| REQ-0112-048 | 再編前ADRは、現行ADR baseline 作成後に `docs/adr/retired/ADR-00XX.md` へ移動すること。旧ADRを現行根拠として引用してはならず、履歴参照する場合は retired path を明示すること |
| REQ-0112-049 | ADR全面改定時、旧ADRの単純リネームではなく、必要な判断を `ADR-01XX` の新規ADRへ引き継ぎ、旧ADRは retired ADR として保持すること |
| REQ-0112-050 | docs-check は current ADR collection（`docs/adr/ADR-01XX.md`）と retired ADR collection（`docs/adr/retired/ADR-00XX.md`）を区別して検査すること。retired ADR への履歴参照は現行根拠引用 warning と区別すること |
| REQ-0112-051 | 廃止ADR（ADR-0017→0103, ADR-0018→0104, ADR-0019→0105, ADR-0020→0106）を現在の一次参照源としている箇所を後継ADRに更新すること。歴史的参照としては元のADR番号を保持可能 |
| REQ-0112-052 | 廃止ADR参照の更新対象ファイル（8ファイル: REQ-0101, REQ-0103, REQ-0108, REQ-0112, AGENTS.md, DOC-MAP.md, system.md, document-model.md）を網羅的に修正すること |
| REQ-0112-053 | retired ADR / superseded ADR / deprecated ADR への履歴参照は、migration table / history section / superseded-by chain（frontmatter `superseded_by` 参照）に限定すること。現行要件判断の根拠としてこれら非accepted ADRを引用してはならず、常に最新の accepted ADR を一次参照源とすること |
| REQ-0112-054 | `agentdev-learning-pipeline` は ADR 候補分類の前に禁止条件フィルタリングゲートを実行すること。`agentdev-adr-guidelines` の除外基準（技術判断不在・仕様変更のみ・command仕様・運用ルール等）を適用し、該当項目を ADR 候補分類前に除外し、除外理由を evaluation-report に記録すること |
| REQ-0112-055 | `agentdev-req-define` は ADR 判定の根拠をドラフトに記録すること。記録内容は以下を含むこと: 判断結果（ADR必要/不要）、適用基準（ADR閾値に達した理由 または 除外基準に該当した理由）、該当/非該当の根拠となった事実 |
| REQ-0112-056 | `agentdev-req-save` は `adr-revision-mode: full-reclassification` 指定時に既存 ADR の全面的な再分類処理をサポートすること。REQ-0112-044 の例外規定に従い、移管先情報（REQ/SPEC/guide）と変更内容（対象ADR・変更種別）をドラフトに明示すること |
| REQ-0112-057 | `docs/specs/workflow-contracts.md` の ADR path 参照は baseline（`docs/adr/ADR-01XX.md`）と retired（`docs/adr/retired/ADR-00XX.md`）の区別を反映すること。REQ-0112-050 の current/retired collection 区別に整合する glob パターンまたは path 表現を使用すること |
| REQ-0112-058 | `docs/specs/design-principles.md` は ADR 見逃し防止の false negative 基準を記載すること。ADR化すべき判断が見逃されるリスクの兆候（REQ/SPEC境界の曖昧な内容、複数モジュールにまたがる判断、長期間有効な技術選定）と防止策を含むこと |
| REQ-0112-059 | `docs/specs/integrity-contracts.md` の docs-check 承認フロー表現は REQ-0108-225（実行＝保存承認、追加のユーザー承認を要求しない）に整合すること。別途ユーザー承認を示唆する表現（"on approval" 等）を使用しないこと |

## 適用範囲

- **対象**: ADR ライフサイクル標準化、RU 参照排除、work_type 移行、command 薄型化、integrity 反転、docs 整合、状態モデル明確化、ADR全面改定時例外規定、ADR-01XX baseline、旧ADR retired移動、current/retired ADR collection区別、廃止ADR参照の後継ADR更新、ADR status有効値列挙、retired ADR参照境界の明確化、ADR baseline/retired 導入後の SPEC・Command・Skill 追従漏れ解消（learning-pipeline フィルタリングゲート、req-define ADR判定根拠記録、req-save full-reclassification モード、workflow-contracts ADR path参照、design-principles false negative基準、integrity-contracts 承認フロー表現）
- 主対象 REQ: REQ-0101/0103/0104/0105/0108/0109
- **対象外**: 個別 command の詳細実装手順、テストコードの具体的記述、各 SPEC ファイルの本文内容
- 新規ADRの具体的本文作成
- 実装コードの変更
- Issue/PRの作成
- retired REQの大規模再編集
