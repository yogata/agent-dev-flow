## 目的

AgentDevFlow の ADR ライフサイクル、文書種別責務境界、runtime独立性、workflow 分類、状態モデルは一貫した基盤を持ち、docs 永続文書と runtime 配布物の責務は分離されている。

## 要件

| ID | 要件 |
|---|---|
| REQ-0112-001 | ADR status 形式は `status: superseded` + frontmatter に `superseded_by: ADR-XXXX` を使用すること |
| REQ-0112-002 | ADR テンプレートは `## 実装後の記録` セクションを対象外とすること |
| REQ-0112-003 | `req-save` は合意済み ADR を `accepted` として作成すること。新規 ADR の初期 status は `proposed` であり、壁打ち合意後に `accepted` に遷移する。ADR status の有効値は `proposed` / `accepted` / `superseded` / `deprecated` の4種類であること |
| REQ-0112-004 | `case-run` は同一ブランチ・同一 PR 内で ADR 変更を含めること。ADR 変更のみの独立 PR は原則として作成しない |
| REQ-0112-005 | `case-close` は ADR 差分の最終確認ゲートとして機能すること |
| REQ-0112-006 | `accepted` status の ADR のみを現行根拠として使用すること |
| REQ-0112-007 | docs 永続文書（REQ/ADR/SPEC/guides）の根拠参照は RU-ID を除外すること。RU は `case-open` で削除される一時成果物であり、docs の永続根拠に不適である |
| REQ-0112-009 | `case-open` は RU 削除対象を一時メタ情報から取得すること |
| REQ-0112-010 | `ru-sources` frontmatter フィールドに代わる永続根拠管理方式を使用すること |
| REQ-0112-011 | work_type を `bugfix` / `feature` / `maintenance` / `docs_chore` の 4 値に固定すること |
| REQ-0112-012 | `feature` のみが scale を持ち、値は `standard` / `large` であること |
| REQ-0112-013 | workflow_route は派生値として work_type + scale から都度導出すること。派生ルール: feature+standard → req_backed_case, feature+large → epic_case, その他 → direct_case |
| REQ-0112-015 | Pattern A/B/C/D は廃止済みとし、work_type + scale + workflow_route の3層分類を使用すること |

| REQ-0112-018 | `learning-*` / `intake-*` command は入口 command であり、詳細処理は skill/script にあること |
| REQ-0112-019 | `docs-check` の frontmatter 検査は `description` / `agent` のみを許容すること |
| REQ-0112-020 | command authoring standards の frontmatter も `description` / `agent` のみを許容すること |
| REQ-0112-021 | `docs/specs/` を repo-internal 設計文書として明記すること。runtime 配布物の依存先ではない |
| REQ-0112-022 | `docs/guides/` を人間向け navigation 層に位置づけ、導線・案内・手順の記述に限定すること |
| REQ-0112-023 | 6 マイクロフェーズ（requirement/analyzed/created/in_progress/review/done）は説明用ラベルとすること |
| REQ-0112-024 | `.agentdev/` は domain state であり、配布物ではないこと |
| REQ-0112-025 | `.sisyphus/` を gitignore 対象とすること |
| REQ-0112-026 | `.agentdev/drafts/` は一時領域であること |
| REQ-0112-027 | REQ / SPEC は workflow status の記述を対象外とすること |
| REQ-0112-028 | intake promoted は route / status の記述を対象外とすること |
| REQ-0112-029 | Issue / PR の状態管理は Issue / PR に一元化すること |
| REQ-0112-030 | command-map は静的参照表として維持すること |
| REQ-0112-031 | ADR文書はADR-0103責務境界に適合し、技術判断を含む文書のみを ADR として扱うこと |
| REQ-0112-032 | 技術判断を含むADRは本文が圧縮されており、REQ/SPEC詳細は REQ/SPEC 側に配置されていること |
| REQ-0112-033 | 技術判断を含まないADRは `deprecated` として履歴参照のみに限定すること。本文に誤分類理由・配置状況が記載されていること |
| REQ-0112-034 | 既存 deprecated/superseded の ADR は原則として再編集対象外であり、docs/adr/README.md の矛盾のみが最小限修正の対象であること |
| REQ-0112-035 | docs/adr/README.md の各ビューは ADR 本文の status と矛盾しないこと |
| REQ-0112-036 | deprecated とした ADR の分離内容（振る舞い・制約・状態は REQ、schema・path・lifecycle は SPEC）は、対応する既存 REQ/SPEC に配置されていること |
| REQ-0112-037 | REQ/SPEC に配置された分離内容に仕様情報の欠落がないこと。配置元・先の照合が必須であること |
| REQ-0112-038 | req-define は ADR 候補を提示する前に REQ/SPEC 相当判定を行うこと。除外基準: 仕様変更のみ・command動作仕様・workflow定義・状態遷移・命名規約・directory規約・運用ルール・template変更・入出力形式・非技術的合意 |
| REQ-0112-039 | REQ/SPEC 相当と判定された項目は ADR 候補に含まれず、REQ/SPEC 更新候補として分類されていること（振る舞い→REQ、schema/lifecycle→SPEC、方針→guide） |
| REQ-0112-040 | req-save は ADR 保存直前に妥当性を再検証すること。技術判断不在の場合は保存を停止し理由を報告する |
| REQ-0112-041 | adr-guidelines に ADR 禁止条件を明示すること。仕様変更のみ・command仕様・artifact contract変更はADR化禁止 |
| REQ-0112-042 | adr-file-manager は ADR 要否の意味判定を主責務とせず、adr-guidelines の判定完了を前提とすることを明記すること |
| REQ-0112-043 | repo-agentdev-integrity は ADR 意味妥当性の strict 判定を行わず、誤分類兆候の warning / observation 検出を行うこと。兆候: 技術判断不在・REQ/SPEC相当内容の混入・ADR-0103適合外・文書種別不一致 |
| REQ-0112-044 | ADR体系の全面改定・責務再分類・圧縮を目的とするユーザー承認済みの req-save 実行では、既存 deprecated/superseded ADR および誤分類 ADR の最小限を超える編集が許容される。ただし、編集目的、対象ADR、変更種別、配置先REQ/SPEC/guide、現行根拠として残すADRをドラフトまたは完了報告に明示すること |
| REQ-0112-045 | ADR全面改定時でも、accepted ADR の決定内容は安定して維持すること。既存判断を覆す場合は新規 ADR が作成され、旧 ADR は superseded または deprecated として扱われること |
| REQ-0112-046 | ADR全面改定後、docs/adr/README.md は ADR 本文の status と矛盾せず、accepted ADR のみを現行根拠として読める分類ビューを提供すること |
| REQ-0112-047 | ADR全面改定後の現行ADR baseline は `ADR-0101` 以降の番号帯を使用すること。`ADR-0001` 〜 `ADR-0023` は再編前ADRの履歴番号帯として扱うこと |
| REQ-0112-048 | 再編前ADRは、現行ADR baseline 作成後に `docs/adr/retired/ADR-00XX.md` に配置されていること。旧ADRを現行根拠として引用せず、履歴参照する場合は retired path を明示すること |
| REQ-0112-049 | ADR全面改定時、旧ADRの単純リネームではなく、必要な判断は `ADR-01XX` の新規 ADR に引き継がれ、旧 ADR は retired ADR として保持されていること |
| REQ-0112-050 | docs-check は current ADR collection（`docs/adr/ADR-01XX.md`）と retired ADR collection（`docs/adr/retired/ADR-00XX.md`）を区別して検査すること。retired ADR への履歴参照は現行根拠引用 warning と区別すること |
| REQ-0112-053 | retired ADR / superseded ADR / deprecated ADR への履歴参照は、migration table / history section / superseded-by chain（frontmatter `superseded_by` 参照）に限定すること。現行要件判断の根拠は常に最新の accepted ADR を一次参照源とすること |
| REQ-0112-054 | `agentdev-learning-pipeline` は ADR 候補分類の前に禁止条件フィルタリングゲートを実行すること。`agentdev-adr-guidelines` の除外基準（技術判断不在・仕様変更のみ・command仕様・運用ルール等）を適用し、該当項目を ADR 候補分類前に除外し、除外理由を evaluation-report に記録すること |
| REQ-0112-055 | `agentdev-req-define` は ADR 判定の根拠をドラフトに記録すること。記録内容は以下を含むこと: 判断結果（ADR必要/不要）、適用基準（ADR閾値に達した理由 または 除外基準に該当した理由）、該当/非該当の根拠となった事実 |
| REQ-0112-056 | `agentdev-req-save` は `adr-revision-mode: full-reclassification` 指定時に既存 ADR の全面的な再分類処理をサポートすること。REQ-0112-044 の例外規定に従い、配置先情報（REQ/SPEC/guide）と変更内容（対象ADR・変更種別）をドラフトに明示すること |
| REQ-0112-057 | `docs/specs/workflow-contracts.md` の ADR path 参照は baseline（`docs/adr/ADR-01XX.md`）と retired（`docs/adr/retired/ADR-00XX.md`）の区別を反映すること。REQ-0112-050 の current/retired collection 区別に整合する glob パターンまたは path 表現を使用すること |
| REQ-0112-058 | `docs/specs/design-principles.md` は ADR 見逃し防止の false negative 基準を記載すること。ADR化すべき判断が見逃されるリスクの兆候（REQ/SPEC境界の曖昧な内容、複数モジュールにまたがる判断、長期間有効な技術選定）と防止策を含むこと |
| REQ-0112-059 | `docs/specs/integrity-contracts.md` の docs-check 承認フロー表現は REQ-0108-225（実行＝保存承認）に整合すること |
| REQ-0112-060 | REQ-0112 が ADR lifecycle / ADR運用品質維持に相当し、runtime独立性・work_type・command薄型化・state model が他REQ / SPEC へ切り出されていること |

## 適用範囲

- **対象**: ADR ライフサイクル、RU 参照の不在、work_type 4値分類、command の薄型構造、integrity 検査の方向、docs 整合、状態モデル、ADR全面改定時例外規定、ADR-01XX baseline、旧ADR の retired 配置、current/retired ADR collection 区別、廃止ADR参照の後継ADR参照、ADR status有効値列挙、retired ADR参照境界、ADR baseline/retired 導入後の SPEC・Command・Skill の追従状態（learning-pipeline フィルタリングゲート、req-define ADR判定根拠記録、req-save full-reclassification モード、workflow-contracts ADR path参照、design-principles false negative基準、integrity-contracts 承認フロー表現）
- 主対象 REQ: REQ-0101/0103/0104/0105/0108/0109
- **対象外**: 個別 command の詳細実装手順、テストコードの具体的記述、各 SPEC ファイルの本文内容
- 新規ADRの具体的本文作成
- 実装コードの変更
- Issue/PRの作成
- retired REQの大規模再編集
