## 目的

現行REQ群（REQ-0001〜REQ-0040）には、現行仕様・旧仕様・既存成果物への反映作業・実装分解・個別ファイル修正指示が混在しており、実装者が旧仕様をactive要件として読むリスクがある。REQ群を現行AgentDevFlowの仕様基準として再構成し、旧REQを履歴として保持しつつ、新基準REQ群を作成することで、要件の読み手保護と後続工程（req-define / req-save / case-open / case-run）の入力品質を担保する。

## 要件

| ID | 要件 |
|---|---|
| REQ-0041-001 | AgentDevFlow は、既存REQ群を削除せず、履歴REQとして保持しなければならない（MUST） |
| REQ-0041-002 | AgentDevFlow は、現行仕様に従う新しい基準REQ群を作成しなければならない（MUST） |
| REQ-0041-003 | 新しい基準REQ群の要件行には、変更後に満たすべき振る舞い・制約・状態のみを記述しなければならない（MUST） |
| REQ-0041-004 | 新しい基準REQ群の要件行に、既存成果物への更新・削除・移動・名称変更・廃止・置換・参照修正・インデックス修正・整合性修正そのものを記述してはならない（MUST NOT） |
| REQ-0041-005 | 既存成果物への反映作業は、対象REQ/ADR/SPEC等への UPDATE、APPEND、関連ドキュメント更新候補、または後続Caseの変更対象として分離しなければならない（MUST） |
| REQ-0041-006 | 旧REQは `retained` / `partially superseded` / `superseded` のいずれかに分類しなければならない（MUST） |
| REQ-0041-007 | `partially superseded` の旧REQには、後継REQ、対象範囲、現行仕様として読んではならない範囲を明記しなければならない（MUST） |
| REQ-0041-008 | 旧REQと新基準REQの対応表を作成し、旧REQごとに継承・置換・廃止・未継承の範囲を明記しなければならない（MUST） |
| REQ-0041-009 | 新基準REQは、旧REQの個別ファイル修正指示、対象ファイル一覧、実装分解、line番号依存の修正指示を要件行として継承してはならない（MUST NOT） |
| REQ-0041-010 | 新基準REQは、現行command、現行domain state、現行workflow、現行artifact lifecycle を基準に記述しなければならない（MUST） |
| REQ-0041-011 | 旧command名・旧directory名・旧workflow名は、履歴説明または旧REQ対応表に限定して使用しなければならない（MUST） |
| REQ-0041-012 | README と DOC-MAP は、新基準REQ群を文書探索の入口として示し、旧REQは履歴・参照用であることを区別しなければならない（MUST） |
| REQ-0041-013 | `req-define` は、新規REQ作成前に、要件行候補が変更後仕様か既存成果物への反映作業かを分類しなければならない（MUST） |
| REQ-0041-014 | `req-define` は、既存成果物への反映作業のみを表す候補を、新規REQの独立要件行として扱ってはならない（MUST NOT） |
| REQ-0041-015 | `req-save` は、CREATE対象REQの保存前に、既存成果物への反映作業だけを表す要件行が残っていないか検査しなければならない（MUST） |
| REQ-0041-016 | `req-save` は、反映作業のみの要件行を検出した場合、保存を停止し、該当行・判定理由・移送先を報告しなければならない（MUST） |
| REQ-0041-017 | REQ-IDは `REQ-{NNNN}-{MMM}` の数値形式に統一し、英字suffixを新規に使用してはならない（MUST NOT） |
| REQ-0041-018 | 欠番がある場合は、削除済み・移管済み・superseded済み等の理由を記録しなければならない（MUST） |

## 旧REQ分類定義

| 分類 | 意味 |
|------|------|
| `retained` | 旧REQのID・本文をそのまま現行基準として維持する |
| `partially superseded` | 旧REQの一部だけを後継REQへ移し、残りは履歴または限定的基準として残す |
| `superseded` | 旧REQを現行基準として読まず、後継REQまたは履歴参照に置き換える |

## Superseded表記ルール

`superseded` または `partially superseded` の旧REQは、本文冒頭に次を置く。

| 項目 | 内容 |
|------|------|
| Status | `superseded` または `partially superseded` |
| Superseded by | 後継REQ |
| 現行仕様として読む範囲 | なし、または限定範囲 |
| 履歴として残す理由 | 過去判断、Issue/PR追跡、設計経緯 |
| 注意 | active要件として実装判断に使わない範囲 |

## 対応表必須項目

旧REQごとに、次を記録する。

| 項目 | 内容 |
|------|------|
| 旧REQ ID | 例: REQ-0016 |
| 現行分類 | retained / partially superseded / superseded |
| 後継REQ | 新基準REQ ID |
| 継承する内容 | 現行仕様として残す内容 |
| 継承しない内容 | 旧workflow、旧command、個別反映作業など |
| 反映作業の移送先 | UPDATE / APPEND / 関連ドキュメント更新候補 / follow-up |
| 備考 | 履歴として残す理由 |

## 新基準REQ群構成

新基準REQ群は、旧番号を無理に踏襲せず、現行仕様の責務境界で再構成する。

| REQ | 内容 |
|-----|------|
| REQ-0042: REQ/ADR/SPEC/DOC-MAP基準構造 | 要件・判断・現在仕様・探索入口の境界 |
| REQ-0043: req-define / req-save / REQ分類ゲート | CREATE/APPEND/UPDATE、反映作業混入防止、保存前検査 |
| REQ-0044: Command / Skill / Template / Script責任分界 | 各artifactに置くもの・置かないもの |
| REQ-0045: AgentDevFlow command protocol | req/case/intake/learning/integrity の現行入出力 |
| REQ-0046: intake / learning / req-backlog / RU lifecycle | promoted artifact、RU、Requirement Source の lifecycle |
| REQ-0047: case-run / case-close / post-run capture | 実行、完了、Findings回収、intake/learning分離 |
| REQ-0048: reporting / GitHub body / link / writing quality | 完了報告、本文品質、リンク正規化、AI-slop抑止 |
| REQ-0049: integrity / validation / tests | 静的検査、validator、テスト体系 |
