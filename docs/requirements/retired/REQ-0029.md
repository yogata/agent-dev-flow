## 目的

intake-open コマンドは現在、引数なし実行時に promoted artifact が1件なら自動検出、2件以上なら候補一覧を表示して停止する仕様である。
しかし、intake-promote が複数の artifact を同時に生成することがあり、その都度手動でファイル名を指定して再実行する必要がある。
また、git pull/commit/push が artifact ごとに実行され、冗長な git 操作が発生する。

本要件は、引数なし実行時に全 promoted artifact をファイル名昇順で一括処理するバッチモードを導入し、オペレーション効率を向上させる。
同時に、continue-on-error による安全な継続処理、一括 git 永続化、batch 向け完了報告フォーマットを導入する。
引数ありの場合も内部フローを共通化し、単一処理とバッチ処理の二重定義を排除する。

## 要件

### 対象集合の決定

| ID | 要件 |
|---|---|
| REQ-0029-001 | intake-open は引数なしの場合、対象 artifact が 0件のときエラー停止すること（SHALL） |
| REQ-0029-002 | 引数なし + 対象 artifact が 1件以上の場合、`.agentdev/intake/promoted/intake-open/*.md` をファイル名昇順で全件取得し、順次処理すること（SHALL） |
| REQ-0029-003 | 引数ありの場合、指定された artifact 1件のみを処理対象集合とすること。処理フローは対象集合の件数にかかわらず batch flow に共通化すること（SHALL） |
| REQ-0029-013 | 現行 Step 1 の「引数なし + promoted artifact が2件以上 → 候補一覧表示して停止」の動作を廃止すること（MUST NOT） |

### artifact ごとの処理

| ID | 要件 |
|---|---|
| REQ-0029-004 | 各 artifact ごとに、Issue 構成判定（Single Issue flow / Epic flow）、Issue 作成、VERIFY 操作を順次実行すること（SHALL） |
| REQ-0029-005 | Issue 作成 + VERIFY が成功した artifact のみを `.agentdev/intake/promoted/intake-open/` から削除すること（SHALL） |
| REQ-0029-006 | artifact 単位の処理失敗時は、当該 artifact を残したうえで後続 artifact の処理を継続すること（continue-on-error）（SHALL） |
| REQ-0029-014 | failed artifact はファイル内容、ファイル名を変更せず `.agentdev/intake/promoted/intake-open/` に残すこと。artifact に `status: failed` 等の frontmatter を追加してはならない（MUST NOT）。artifact を `.failed` 等にリネームしてはならない（MUST NOT） |
| REQ-0029-015 | Issue 化に必要な情報が不足する artifact は failed として扱い、ファイルを変更せず残すこと（SHALL） |
| REQ-0029-016 | intake-open は promoted artifact の内容を実現するための質問を行わないこと（MUST NOT）。候補抽出、解消チェック、採否判断、review は引き続き行わないこと（REQ-0017-026b 準拠） |
| REQ-0029-017 | 再実行時は、残存 artifact を通常の処理対象として再試行すること。failed は永続状態ではなく前回実行結果の分類であること（SHALL） |

### git 操作

| ID | 要件 |
|---|---|
| REQ-0029-007 | git pull --ff-only を全 artifact 処理の前に1回だけ実行すること（SHALL） |
| REQ-0029-008 | git pull --ff-only が失敗した場合は、artifact 処理を開始せず全体停止すること（SHALL） |
| REQ-0029-009 | 全 artifact 処理後に、`.agentdev/intake/` 配下の変更を1 commit / 1 push すること（SHALL） |
| REQ-0029-010 | git push が失敗した場合は完了扱いにせず、構造化エラーとして報告すること（SHALL） |
| REQ-0029-018 | 変更がない場合は commit/push をスキップすること（SHALL） |

### 完了報告

| ID | 要件 |
|---|---|
| REQ-0029-011 | 完了報告には以下を含めること: 成功件数、失敗件数、削除済み artifact 一覧、残存 artifact 一覧、作成 Issue 番号一覧、停止理由（ある場合）、git 永続化結果（SHALL） |
| REQ-0029-012 | 完了報告フォーマットは `agentdev-workflow-reporting` の完了報告フォーマット（completion-reports.md）に従うこと（SHALL） |
| REQ-0029-019 | `completion-reports.md` にバッチ処理向けの intake-open 完了報告形式を追加すること。全件成功（✅）、部分成功（⚠️）、全件失敗（❌）の3形式を定義すること（SHALL） |
| REQ-0029-020 | 全 artifact が成功した場合、`✅ intake-open 完了` として報告すること（SHALL） |
| REQ-0029-021 | 1件以上の artifact が Issue 化成功済みで一部失敗がある場合、`⚠️ intake-open 部分完了` として報告すること（SHALL） |
| REQ-0029-022 | 全 artifact が失敗した場合、`❌ intake-open 未完了` として報告すること（SHALL） |
| REQ-0029-023 | `/agentdev/case-run` は作成済み Issue が存在する場合のみ案内し、その Issue 番号に限定して案内すること（SHALL） |
| REQ-0029-024 | Issue 番号が未確定の場合、`{N}` または `#{N}` を出力してはならない（MUST NOT） |
