## 目的

<!-- 【必須】 -->

Epic Issue を子 Issue の一覧だけでなく「実行順序の SSoT（Single Source of Truth）」として扱えるようにする。
`/agentdev/case-run {epic_issue}` を実行した場合、case-run は Epic Orchestrator として動作し、Epic 本文に明記された Wave 順序に従って子 Issue を subagent に渡して実行する。

現状の課題:
1. case-open の Epic 出力は推奨実行順序を GitHub Issue 本文の実行 SSoT として保持しない
2. case-run は Epic Issue を入力として受け取り、Epic 本文から子 Issue の実行順序を読み取る定義がない
3. ユーザーが手動で `/agentdev/case-run {child1}` → `/agentdev/case-run {child2} {child3}` のように順番を維持する必要がある
4. 子 Issue の実装コンテキストがメインセッションに混ざる

## 要件

<!-- 【必須】 -->

### Epic テンプレート、case-open

| ID | 要件 |
|---|---|
| REQ-0020-001 | case-open は Epic flow で Epic Issue 本文に実行順序を明記する。実行順序は人間向け推奨ではなく、case-run が読み取る実行 SSoT とする（SHALL） |
| REQ-0020-002 | 実行順序は `## 実行順序` セクション（`<!-- 【必須】 -->` マーカー付き）内の Wave テーブルで表現する。Wave 間は直列実行、同一 Wave 内は並列実行可能とする（SHALL） |
| REQ-0020-003 | Wave テーブルは `Wave / Issue / 実行方法 / 前提` の4列形式とする。前提列には依存先 Wave 番号を記載する。前提のない Wave は `—` とする（SHALL） |
| REQ-0020-004 | 前提 Issue がある Issue は、必ず前提 Issue の Wave より後続の Wave に配置する（SHALL） |
| REQ-0020-005 | `## 実行順序` セクションは `issue_desc_epic.md` テンプレートの `## 分解（Decomposition）` とは独立したセクションとする。既存分解テーブルとステータス追跡テーブルの互換性を維持する（SHALL） |
| REQ-0020-006 | case-open 完了報告は `次のステップ: /agentdev/case-run {epic_N}` を提示する。子 Issue 番号のスペース区切り列挙形式に代えて Epic Issue を指定したコマンドを提示する。補足として Wave 概要を表示してよい（SHALL + MAY） |

### case-run Epic Orchestrator モード

| ID | 要件 |
|---|---|
| REQ-0020-007 | case-run は入力 Issue が `epic` ラベルを持つ場合、Epic Orchestrator モードとして動作する。`epic` ラベルがない場合でも、本文に `## 実行順序` セクションと Wave 列を持つ Markdown テーブルが存在する場合は Epic Orchestrator モードとして動作する。どちらにも該当しない場合は通常の単一 Issue として実行する（SHALL） |
| REQ-0020-008 | Epic Orchestrator モードでは、Epic 本文の `## 実行順序` から Wave テーブルを読み取り、子 Issue を抽出する（SHALL） |
| REQ-0020-009 | Wave テーブルに記載された順序を崩してはならない。依存関係分析で安全側の直列化が必要と判断した場合は Wave をさらに分割してよい。ただし前提 Wave より前に後続 Wave の Issue を実行してはならない（MUST NOT + MAY） |
| REQ-0020-010 | 各子 Issue の実行は subagent に渡す。subagent は case-run 相当の処理（準備フェーズ + 実装フェーズ + 提出フェーズ）を実行する。specs 更定は行わない（SHALL） |

### メインセッション責務

| ID | 要件 |
|---|---|
| REQ-0020-011 | Epic Orchestrator モードのメインセッションは以下の Orchestrator 責務に限定し、子 Issue 個別の実装詳細を抱え込まない: (1) Epic 本文の読み取り (2) Wave 順序の解釈 (3) 子 Issue の subagent 起動 (4) Wave 完了待機 (5) 子 Issue の成功、失敗集約 (6) Epic ステータス更新 (7) specs 更新が必要な場合の直列処理 (8) 集約完了報告（SHALL） |
| REQ-0020-012 | specs 更新は親エージェントのみが実行する。subagent は specs 更新を行わない。この既存方針を維持する（SHALL） |
| REQ-0020-013 | Epic Orchestrator は各 Wave 開始時に対象子 Issue の Epic ステータスを一括更新（`☐ 未着手` → `🔄 進行中`）する。同一 Wave 内の複数 Issue については親エージェントが一括更新し、subagent 同士による Epic 本文更新競合を回避する（SHALL） |

### Wave 失敗時の後続制御

| ID | 要件 |
|---|---|
| REQ-0020-014 | Wave 内の一部 Issue が失敗した場合、同一 Wave 内の他 Issue は継続する。失敗 Issue は兄弟 Issue の実行をブロックしない（SHALL） |
| REQ-0020-015 | Wave 完了後、失敗 Issue と後続 Wave の前提関係を確認し、後続 Wave の実行可否を判定する: (1) 後続 Wave が失敗 Issue に依存しない場合: 継続 (2) 後続 Wave が失敗 Issue に依存する場合: 該当 Wave をスキップ (3) 依存有無が判定不能な場合: 安全側に倒して該当 Wave をスキップ（SHALL） |
| REQ-0020-016 | 全 Issue が失敗した場合、後続 Wave を実行せず集約失敗報告で停止する（SHALL） |

### 再開ポイント

| ID | 要件 |
|---|---|
| REQ-0020-017 | Epic Orchestrator 再実行時は Wave 1 から全 Wave を再評価する。各子 Issue は既存 case-run のべき等性、再開判定に従いスキップ、再開、新規実行する（SHALL） |
| REQ-0020-018 | Epic 本文のステータス追跡テーブルは再開判断の補助情報として参照してよい。ただし主判定は各子 Issue の実成果物（worktree、コミット、PR、Issue 状態）と既存 case-run の再開判定とする（MAY） |

### 制約、互換性

| ID | 要件 |
|---|---|
| REQ-0020-019 | 子 Issue 番号直接指定の既存動作（`/agentdev/case-run {child1} {child2}`）は維持する。Epic Orchestrator モードは追加モードであり、既存多重 Issue モードを置き換えない（SHALL） |
| REQ-0020-020 | 1 Wave 内の同時実行子 Issue 上限は5件とする。Epic 全体の子 Issue 数には5件制限を適用しない（Epic は1件の呼び出しであり、ユーザーが複数 Issue を直接指定しているのではないため）（SHALL） |
| REQ-0020-021 | subagent は既存の多重 Issue モードの責務分界を維持し、親エージェント専管の処理（specs 更新、Epic ステータス更新等）を行わない（SHALL） |

## 適用範囲

<!-- 【必須】 -->

- **対象**:
 - Epic Issue 本文への `## 実行順序` セクション（Wave テーブル）追加
 - `issue_desc_epic.md` テンプレート更新
 - case-open の Epic flow における Wave テーブル生成、完了報告変更
 - case-run の Epic Orchestrator モード追加（検出、Wave 解析、subagent 起動、障害伝播、再開）
 - `agentdev-workflow-orchestration` スキルの Epic Orchestrator モード定義追加
 - `agentdev-epic-tracker` スキルの Wave レベル一括更新対応
 - `agentdev-workflow-reporting` の Epic Orchestrator 完了報告フォーマット追加
 - `completion-reports.md` の case-open Epic 作成時フォーマット更新

- **対象外**:
 - 子 Issue の実装方針の詳細化
 - 新しい実装計画の作成
 - GitHub Issue 以外のローカル Issue 管理方式の導入
 - Epic 以外の通常 Issue に対する挙動変更
 - subagent を使わない完全同期実行への変更
 - 既存の単一 Issue 実行モードの変更
 - 既存の多重 Issue モード（子 Issue 番号直接指定）の変更
 - `## 分解（Decomposition）` テーブルと `## ステータス追跡` テーブルの形式変更
 - case-open の Standard flow の変更
