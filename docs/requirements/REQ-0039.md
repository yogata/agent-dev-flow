## 目的

AgentDevFlow において `/agentdev/req-backlog` コマンドと Requirement Unit（RU）を導入し、intake・learning パイプラインで生成された promoted artifact を要件化前の中間成果物として一元管理する。RU は req-define への構造化入力として機能し、promoted artifact から要件定義への橋渡しを担う。

## 要件

### A. req-backlog コマンド新設

| ID | 要件 |
|---|---|
| REQ-0039-001 | `req-backlog` コマンドは `.agentdev/intake/promoted/*.md` 及び `.agentdev/learning/promoted/*.md` の全 promoted artifact を読み込み、Requirement Unit（RU）を `.agentdev/backlog/req-units/RU-*.md` として生成しなければならない（MUST） |
| REQ-0039-002 | RU-*.md は frontmatter（source_type, generated_by, generated_at, status, sources）＋ Sources セクション＋ Source Summary セクション＋統合理由セクション＋要件化の方向セクションの構造化フォーマットに従わなければならない（MUST） |
| REQ-0039-003 | `req-backlog` は promoted artifact の単純コピー（パススルー）を生成してはならない（MUST NOT）。全 RU は分析・統合された構造化内容でなければならない |
| REQ-0039-004 | RU の粒度は N:1（複数 artifact → 1 RU 統合）及び 1:N（1 artifact → 複数 RU 分割）を許可しなければならない（MUST）。同一趣旨の artifact は 1 RU に統合し、1 artifact 内の複数論点は複数 RU に分割する |
| REQ-0039-005 | promoted artifact 間に矛盾が検出された場合、`req-backlog` は矛盾する artifact を RU 化せず、ユーザーに確認を求めなければならない（MUST）。矛盾しない artifact は通常通り RU 化を継続する |
| REQ-0039-006 | `req-backlog` は RU 生成に成功した promoted artifact を削除しなければならない（MUST）。失敗・矛盾の artifact は `promoted/` に残置する |
| REQ-0039-007 | promoted artifact が 0 件の場合、`req-backlog` は正常終了しなければならない（MUST）。エラー扱いとしない |
| REQ-0039-008 | 全件失敗の場合でも、`req-backlog` は構造化された実行報告を出力して終了しなければならない（MUST） |
| REQ-0039-009 | `req-backlog` は intake/learning の raw item（inbox.md, archive.md, entries/）を更新してはならない（MUST NOT）。raw item の更新は各パイプラインのコマンド（intake-promote, learning-promote）の責務である |
| REQ-0039-010 | `req-backlog` は partial success をサポートしなければならない（MUST）。成功分は RU 化 + 元 artifact 削除、失敗分は `promoted/` に残置して報告に含める |

### B. RU lifecycle（削除管理）

| ID | 要件 |
|---|---|
| REQ-0039-015 | `req-save` は RU の内容が REQ ファイルに永続化された後、該当 RU ファイル（`.agentdev/backlog/req-units/RU-*.md`）を削除しなければならない（MUST） |
| REQ-0039-016 | `case-open` は RU の内容が GitHub Issue に永続化された後、該当 RU ファイルを削除しなければならない（MUST） |
| REQ-0039-017 | RU 削除のトリガーは Requirement Source の永続化完了（REQ ファイル保存成功 または GitHub Issue 作成成功）に限定しなければならない（MUST）。永続化が未完了の場合は RU を残置する |

### C. req-define 入力制約

| ID | 要件 |
|---|---|
| REQ-0039-018 | `req-define` は `.agentdev/backlog/req-units/RU-*.md` を Requirement Source として受け入れなければならない（MUST） |
| REQ-0039-019 | `req-define` は `.agentdev/intake/promoted/*.md` 及び `.agentdev/learning/promoted/*.md` を直接読み込んではならない（MUST NOT）。promoted artifact は req-backlog による RU 化を経由しなければならない |
| REQ-0039-020 | `req-define` はユーザーの直接入力及び一般 source file を引き続き Requirement Source として受け入れなければならない（MUST）。RU 経由のみに制限しない |

### D. req-backlog 対象指定と RU 生成基準

| ID | 要件 |
|---|---|
| REQ-0039-032 | `req-backlog` は引数なしで実行された場合、`.agentdev/intake/promoted/*.md` 及び `.agentdev/learning/promoted/*.md` の全 promoted artifact を対象とする。引数ありで実行された場合、指定されたファイルパスのみを対象とする（SHALL） |
| REQ-0039-033 | 同一の関心領域・変更意図を持つ複数の promoted artifact は 1 つの RU に統合しなければならない（MUST）。統合時は RU の Sources セクションに全元ファイルを列挙し、統合理由セクションに統合の根拠を記録する |
| REQ-0039-034 | 1 つの promoted artifact 内に複数の独立した論点が含まれる場合、論点ごとに別々の RU に分割しなければならない（MUST）。分割時は各 RU の Sources セクションに元ファイルを記録し、要件化の方向セクションに分割の根拠を記録する |
| REQ-0039-035 | promoted artifact 間に矛盾が検出された場合（REQ-0039-005）、矛盾する artifact ペア・矛盾の具体的内容・各 artifact の主張を構造化してユーザーに提示しなければならない（MUST） |

### E. reference ファイル責務分離

| ID | 要件 |
|---|---|
| REQ-0039-037 | `artifact-lifecycle.md` は成果物のライフサイクルに関する宣言的ルール（状態遷移・削除条件・禁止事項）を定義し、コマンド定義ファイルの手順（Step 番号・ファイルパス・エラーメッセージ）を記述してはならない（SHALL + MUST NOT）。両者が同じ事象を扱う場合、`artifact-lifecycle.md` は「何をすべきか」を、コマンド定義は「どのように実行するか」を定義する |

## 適用範囲

### 対象

- **新設**: `/agentdev/req-backlog` コマンド, RU-*.md lifecycle, `artifact-lifecycle.md`
- **変更**: `req-define`（RU 入力対応）, `req-save`（RU 削除トリガー）, `case-open`（RU 削除トリガー）

- intake-open コマンド削除（実装済み）
- learning-promote 出力先変更（REQ-0007 UPDATE で対応済み）
- 既存ドキュメント整合性更新（実装済み）
- `case-run`, `case-close` コマンド（RU 削除トリガー以外の変更なし）
- `learning-refine` コマンド（原則変更なし）
- `intake-review`, `intake-capture` コマンド（原則変更なし）
- intake archive フラット化（REQ-0026 UPDATE で別途対応）
- learning archive 再構成（REQ-0027 UPDATE で別途対応）
