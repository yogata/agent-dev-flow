## 目的

作業候補、学び、要件化前の構造化入力を `.agentdev/` domain state として管理し、短期Issue化候補と要件化候補を混同しない。

## 適用範囲

- **対象**: intake、learning、backlog-review、backlog-save、backlog-review ↔ backlog-save の受け渡し（`.sisyphus/drafts/backlog-review-*.md`）、RU lifecycle、session-sourced RU、`.agentdev/` domain state、`intake-review` の後続ルート提示・判定表・結果レポート・完了報告、`.agentdev/intake/accepted/` の状態意味、通常 intake と REQ再構成intake の後続ルート分離
- **対象外**: `.sisyphus/`（`backlog-review-*.md` を除く）、case実行、REQ本文の再構成手順、`intake-promote` の整形ロジック、`backlog-review` の RU 統合・分割判定の詳細ルール、`req-define` の RU 読み込み仕様、`accepted/` 配下のサブ状態追加、個別 intake item の内容

## Design Notes

### backlog-review draft の status フィールドについて（INC-0019 対応注記）

REQ-0105-041 ~ 044 が参照する `status: reviewed` / `status: saved` は、backlog-review draft（`.sisyphus/drafts/`）の軽量ドキュメントライフサイクル注記である。これらは REQ-0108-123 が禁止するワークフロー状態管理モデル（6 マイクロフェーズによる進行管理）ではなく、backlog-review → backlog-save 間の受け渡し成立条件を示す 2 値フラグである。実際の draft 生成・状態遷移の実装は backlog-review / backlog-save コマンドが担う。
