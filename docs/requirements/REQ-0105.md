## 目的

作業候補、学び、要件化前の構造化入力を `.agentdev/` domain state として管理し、短期Issue化候補と要件化候補を混同しない。

## 要件

| ID | 要件 |
|---|---|
| REQ-0105-001 | intake pipeline の domain state は `.agentdev/intake/` に配置すること（SHALL） |
| REQ-0105-002 | `intake-capture` はユーザー入力から明示できる内容のみを intake item として整理すること（SHALL） |
| REQ-0105-003 | `intake-review` は intake item を採用・却下・保留に分類し、状態を配置で表現すること（SHALL） |
| REQ-0105-004 | `intake-promote` は採用済み intake item を promoted artifact として `.agentdev/intake/promoted/` に配置すること（SHALL） |
| REQ-0105-005 | learning pipeline の domain state は `.agentdev/learning/` に配置すること（SHALL） |
| REQ-0105-006 | learning は再発防止・抽象化・反映価値のある知見を蓄積すること（SHALL） |
| REQ-0105-007 | 同一観測から intake と learning の両方が発生する場合、成果物を分離すること（SHALL） |
| REQ-0105-008 | `req-backlog` は intake / learning の promoted artifact を分析・統合し、RU（Requirement Unit）を `.agentdev/backlog/req-units/` に生成すること（SHALL） |
| REQ-0105-009 | RU は promoted artifact の単純コピーではなく、要件化に向けた構造化内容とすること（SHALL） |
| REQ-0105-010 | `req-define` は RU を Requirement Source として受け入れること（SHALL） |
| REQ-0105-011 | `req-define` は promoted artifact を直接読み込まず、RU または明示入力ファイルを経由すること（SHALL） |
| REQ-0105-012 | RU は case-open の Issue作成 + VERIFY 成功後にのみ削除すること（SHALL） |
| REQ-0105-013 | REQ再構成intakeは通常の短期Issue化候補と区別し、`req-backlog` 向け短期候補として処理しないこと（SHALL） |
| REQ-0105-014 | req-save は RU ファイルを削除せず、REQファイルの Requirement Source に RU パスを記録すること（SHALL） |
| REQ-0105-015 | case-open は Issue作成 + VERIFY 成功後に RU を削除する唯一のコマンドであること（SHALL） |
| REQ-0105-016 | case-open は Issue作成失敗または VERIFY 失敗時に RU を残置すること（SHALL） |
| REQ-0105-017 | チャット内でユーザーと合意形成済みの内容は、session-sourced RU として .agentdev/backlog/req-units/ に保存できること（SHALL） |
| REQ-0105-018 | session-sourced RU の frontmatter は source_type: chat, generated_by: session, sources[].type: chat を許容すること（SHALL） |
| REQ-0105-019 | session-sourced RU の sources[].path は session:YYYY-MM-DD-{topic} 形式とし、チャット内合意の識別子として扱うこと（SHALL） |
| REQ-0105-020 | session-sourced RU の本文は後続工程（req-define）で必要な情報を自足していること（SHALL） |
| REQ-0105-021 | session-sourced RU は req-define に渡せる粒度まで整理済みの内容に限定すること（SHALL） |
| REQ-0105-022 | 未合意・未整理・採否未確定の内容は session-sourced RU 化しないこと（SHALL） |
| REQ-0105-023 | `case-open` は RU ファイル削除を commit/push した場合、main 作業ディレクトリの HEAD と `origin/main` が一致し、`git status --porcelain` に削除済み RU ファイルが残っていないことを確認すること（SHALL） |
| REQ-0105-024 | `case-open` は RU ファイル削除 commit/push 後の同期確認に失敗した場合、完了扱いにせず、対象ファイル・現在の HEAD・`origin/main` を表示して停止すること（SHALL） |
| REQ-0105-025 | `intake-review` は、通常 intake item の判定が `採用` の場合、後続ルートとして `/agentdev/intake-promote` のみを提示すること（SHALL） |
| REQ-0105-026 | `intake-review` は、通常 intake item の採用後続として `/agentdev/req-define` を提示してはならない（SHALL） |
| REQ-0105-027 | `.agentdev/intake/accepted/` は、採用済み通常 intake item の promote 待ち状態として扱うこと（SHALL） |
| REQ-0105-028 | 通常 intake 起点の item が `/agentdev/req-define` に渡されるのは、`/agentdev/intake-promote` と `/agentdev/req-backlog` を経て RU 化された後であること（SHALL） |
| REQ-0105-029 | `intake-review` の判定表・結果レポート・完了報告において、通常 intake の採用後続を `/agentdev/intake-promote` に統一すること（SHALL） |
| REQ-0105-030 | REQ再構成intake の後続ルートは通常 intake の `/agentdev/intake-promote` ルートと混同せず、REQ-0109 に基づく独立した扱いを維持すること（SHALL） |

## 適用範囲

- **対象**: intake、learning、req-backlog、RU lifecycle、session-sourced RU、`.agentdev/` domain state、`intake-review` の後続ルート提示・判定表・結果レポート・完了報告、`.agentdev/intake/accepted/` の状態意味、通常 intake と REQ再構成intake の後続ルート分離
- **対象外**: `.sisyphus/`、case実行、REQ本文の再構成手順、`intake-promote` の整形ロジック、`req-backlog` の RU 統合・分割ルール、`req-define` の RU 読み込み仕様、`accepted/` 配下のサブ状態追加、個別 intake item の内容
