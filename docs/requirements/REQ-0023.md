## 目的

learning-promote が生成する staging stub（`.agentdev/learning/elevation-staging/*.md`）は、req-define → case-open → case-close のライフサイクルを通じて消費される。現在、staging stub がどの Issue で消費されたかの追跡機能と、消費後のアーカイブ機能が存在しない。

本要件は、staging stub の消費追跡（req-define / case-open / case-close 間でのパス伝播）と、消費後のアーカイブ（case-close での移動・消費記録追記）を実現する。

## 要件

### req-define: Requirement Source セクション

| ID | 要件 |
|---|---|
| REQ-0023-001 | req-define は、入力に `.agentdev/learning/elevation-staging/*.md` が含まれる場合、生成する REQ 文書の `## 適用範囲` セクションの後に `## Requirement Source` セクションを追加しなければならない（SHALL） |
| REQ-0023-002 | `## Requirement Source` セクションには、入力となった staging stub のファイルパスをリスト形式で記録しなければならない（SHALL） |
| REQ-0023-003 | `## Requirement Source` セクションの機構は、learning staging stub 以外の入力ソースにも適用可能な汎用形式とするべきである（SHOULD） |

### case-open: Issue 本文転記

| ID | 要件 |
|---|---|
| REQ-0023-004 | case-open は、REQ 文書に `## Requirement Source` セクションが存在する場合、その内容を Issue 本文の対応セクションに転記しなければならない（SHALL） |

### case-close: consumed 判定

| ID | 要件 |
|---|---|
| REQ-0023-005 | case-close は、Issue 本文に staging stub パスが記録されている場合、`agentdev-learning-pipeline` skill を load しなければならない（SHALL） |
| REQ-0023-006 | case-close は、Issue 本文から staging stub のファイルパスを抽出し、当該ファイルが `.agentdev/learning/elevation-staging/` に存在することを確認しなければならない（SHALL） |
| REQ-0023-007 | staging stub の consumed 判定は、以下の 4 条件を全て満たす場合にのみ consumed としなければならない（SHALL）: (1) Issue に stub パスが記録されている、(2) 対応 PR がマージ済み、(3) Issue が completed としてクローズされている、(4) case-close が正常完了している |
| REQ-0023-008 | consumed 判定において、staging stub の内容と Issue/PR の内容の意味的一致検証を実行してはならない（MUST NOT） |
| REQ-0023-009 | staging stub パスが Issue 本文に記録されていない場合、archive 処理を実行してはならない（MUST NOT） |
| REQ-0023-010 | case-close は、consumed 判定の結果を完了報告に含めなければならない（SHALL） |

### staging stub archive

| ID | 要件 |
|---|---|
| REQ-0023-011 | consumed と判定された staging stub を `.agentdev/learning/elevation-staging/archive/` に移動しなければならない（SHALL） |
| REQ-0023-012 | 移動前に、stub 本文の末尾に `## 消費記録` セクションを追記しなければならない（SHALL） |
| REQ-0023-013 | `## 消費記録` セクションには以下を記録しなければならない（SHALL）: 消費元 Issue 番号、マージ済み PR 番号、消費日（YYYY-MM-DD）、処理（case-close） |
| REQ-0023-014 | archive 先に同名ファイルが既存の場合、上書きしてはならない（MUST NOT） |
| REQ-0023-015 | consumed と判定された stub のみを archive 対象としなければならない（SHALL） |
| REQ-0023-016 | 不採用の stub は archive 対象外とし、自動処理してはならない（MUST NOT） |
| REQ-0023-017 | archive 処理は case-close の Step 9c の前に実行しなければならない（SHALL） |
| REQ-0023-018 | archive 処理の結果は Step 9c の `.agentdev` commit に含めなければならない（SHALL） |

### 消費記録形式

`## 消費記録` セクションの記述形式:

```
## 消費記録

| 項目 | 値 |
|------|-----|
| Issue | #N |
| PR | #N |
| 消費日 | YYYY-MM-DD |
| 処理 | case-close |
```

## Requirement Source

- `.agentdev/learning/elevation-staging/*.md`（learning-promote が生成した staging stub）

## 適用範囲

- **対象**:
  - `.opencode/commands/agentdev/req-define.md` — Requirement Source セクション追加
  - `.opencode/commands/agentdev/case-open.md` — Issue 本文への Requirement Source 転記
  - `.opencode/commands/agentdev/case-close.md` — consumed 判定・archive 実行・learning-pipeline skill load
  - `.opencode/skills/agentdev-learning-pipeline/SKILL.md` — staging stub archive ルール定義
  - `.agentdev/learning/elevation-staging/` — archive 対象ディレクトリ

- **対象外**:
  - case_id 等の新規識別子の導入
  - elevation-staging 専用分岐（req-define への専用ロジック追加）
  - archive.md への追記
  - inbox.md / archive.md の lifecycle との混同
  - staging stub と Issue/PR の意味的一致検証
  - 不採用 stub の自動処理（ユーザー手動対応）
