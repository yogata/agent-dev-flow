## 目的

要件定義と保存の入口は、既存REQ照合・操作分類・反映作業混入防止を含む一貫したプロセスである。

## 要件

| ID | 要件 |
|---|---|
| REQ-0102-001 | `req-define` はユーザーとの対話、明示入力ファイル、RU を Requirement Source として受け入れること |
| REQ-0102-002 | `req-define` は要件展開前に active REQ を照合し、関連する関心対象と対象REQを推論すること |
| REQ-0102-003 | `req-define` は CREATE を選択する前に APPEND / UPDATE の可否を評価すること |
| REQ-0102-004 | 同一関心対象の active REQ が存在する場合、要件追加は APPEND / UPDATE を既定候補とすること |
| REQ-0102-005 | CREATE を選択する場合、既存 active REQ に吸収できない理由を記録すること |
| REQ-0102-006 | active REQ は現在満たすべき状態・振る舞い・制約のみを保持すること。要件行候補は変更後に満たすべき振る舞い・制約・状態として記述すること |
| REQ-0102-007 | 移行・変更・再定義・削除・改名・移管・除去・更新・移動・名称変更・参照修正などの作業手段を要件行として扱わないこと |
| REQ-0102-008 | `req-define` は要件行候補を「変更後仕様」と「反映作業」に分類すること |
| REQ-0102-009 | `req-save` は保存前に反映作業のみの要件行が残っていないか検査すること |
| REQ-0102-010 | `req-save` は反映作業のみの要件行を検出した場合、保存を停止し、該当行・理由・移送先を報告すること |
| REQ-0102-011 | `req-define` は関連ドキュメント更新候補を抽出し、対象パス・判定・必要対応を出力すること |
| REQ-0102-012 | `req-save` は REQ/ADR 保存後に README / DOC-MAP / mapping-table との整合性を確認すること |
| REQ-0102-013 | 未決分岐がある場合、`req-define` は分岐・判断軸・推奨案・ユーザー確認事項を明示すること |
| REQ-0102-014 | `req-define` は複数RUを同時に入力として受け入れること。各RUを独立したRequirement Sourceとして扱うこと |
| REQ-0102-015 | 複数RU入力時、`req-define` は以下の基準で統合・分割を判断すること：(a) 同一主題・同一対象REQ・同一変更理由に収まるRU群 → 1つのREQ操作に統合 (b) 対象REQ・変更理由・受け入れ条件・対象外が分かれるRU群 → 複数REQ操作として分割 |
| REQ-0102-016 | 複数REQ操作がある場合、`req-define` は各操作の対象REQ・操作種別・要件行候補・依存関係を operation unit 構造として operation_units セクションに明示すること |
| REQ-0102-017 | `req-define` は large / Epic規模と判定した場合、`scale: large` と `decomposition`（分割方針）を記録すること |
| REQ-0102-018 | REQ間または子Issue間に順序制約がある場合、`req-define` は wave候補または依存関係を記録すること |
| REQ-0102-019 | `req-define` はIssue階層そのものを確定しないこと。Issue階層の決定は `case-open` の責務である |
| REQ-0102-020 | `req-save` は operation unit 単位で保存結果を draft の OU result に保持すること: (a) 保存したREQ doc一覧 (b) 各OU操作と保存先REQ docの対応 (c) source RUとOU操作の対応 (d) case-open が入力として扱える保存結果 |
| REQ-0102-021 | `req-save` はREQ doc保存時にIssueを作成しないこと。Issue作成は `case-open` の責務である |
| REQ-0102-022 | `req-define` は実装フェーズの活動（ソースコード編集、テスト実行、ビルドコマンド実行、ソースファイルの作成・変更）を実行しないこと。`req-define` の出力は要件定義成果物（要件行候補・対象REQ・操作種別・依存関係・wave候補）に限定すること |
| REQ-0102-023 | `req-define` の要件行候補に実装指示（具体的なファイル編集内容・コード変更箇所の行番号指定・ステップバイステップの実装手順・テストコマンド・実装コード断片）を含めないこと。要件行は要件定義基準（REQ-0102-006 に従い変更後に満たすべき振る舞い・制約・状態として記述）に適合すること |
| REQ-0102-024 | REQ 要件行は満たす必要がある検証可能な要件に限定し、推奨・任意・将来候補を要件行として扱わない |
| REQ-0102-025 | REQ 要件行に 規範語（必達/推奨・任意相当）の使用を必須としない |
| REQ-0102-026 | 推奨 相当の内容は必達要件として再定義できる場合のみ REQ 要件行に残す |
| REQ-0102-027 | 必達要件として再定義できない 推奨 相当の内容は、注記・SPEC・guide・backlog・report のいずれか適切な文書へ移す |
| REQ-0102-028 | 任意 相当の内容は原則として REQ 要件行から除外し、許容仕様・設計余地・将来候補として適切な文書へ移す |
| REQ-0102-029 | 主語・適用範囲・既存 active REQ から一意に導ける再説明は、active REQ の要件行に含まれないこと |
| REQ-0102-030 | 既存要件の言い換えに過ぎず、新しい判定基準・例外・対象外・検査境界・優先関係を含まない文は、要件行に含まれないこと |
| REQ-0102-031 | 既存制約との関係説明（括弧内補足の「整合する」「追従する」等）が判定境界を増やさない場合、当該説明は要件行に含まれないこと |
| REQ-0102-032 | 対象外条件・例外条件・検査対象範囲・finding条件・出力粒度・優先関係を実際に変える文は、要件行に残ること |
| REQ-0102-033 | draft に `operation_units` セクションを定義し、1 件以上の operation unit（OU）を保持すること。OU は Issue 単位ではなく、要件整理結果の構成単位である |
| REQ-0102-034 | 各 OU は `ou_id`, `source_ru`, `target_req`, `operation`（CREATE / APPEND / UPDATE）, `scale`, `depends_on`, `recommended_order`, `issue_policy` の各フィールドを持つこと |
| REQ-0102-035 | 各 OU は `result` セクションを持ち、req-save / case-open が処理結果を書き戻す先とすること |
| REQ-0102-036 | draft に `execution_groups` セクションを定義し、Epic 候補グループを記録すること。各 execution_group は `id`, `type`, `purpose`, `included_ou`, `rationale` を持つこと |
| REQ-0102-037 | req-define は複数 RU 入力時の統合/分離結果（REQ-0102-015）を出力する際、結果を OU 構造として `operation_units` セクションに構造化すること |
| REQ-0102-038 | req-define は OU 群を分析し、Epic 候補グループを `execution_groups` セクションに記録すること。記録は提案であり、Issue 発行は行わない |
| REQ-0102-039 | req-save は OU ID 指定時に指定 OU だけを処理対象とし、他 OU は処理しないこと |
| REQ-0102-040 | req-save は OU ID 指定なしで draft に OU が 1 件の場合、その OU を自動選択して処理すること |
| REQ-0102-041 | req-save は OU ID 指定なしで draft に OU が 2 件以上の場合、OU 一覧を表示して停止すること |
| REQ-0102-042 | req-save は OU の保存結果を draft の当該 OU `result` に書き戻すこと |
| REQ-0102-043 | req-save は REQ-0108 への APPEND 操作時に、追加対象要件行に関連する integrity-rule-catalog.md の catalog entry 有無を確認すること |
| REQ-0102-044 | catalog entry が未記載の場合、req-save はユーザーに catalog entry 追記を促すこと。req-save 自身は docs/specs/ 配下を直接編集しない |
| REQ-0102-045 | `req-define` は全 work_type（feature / bugfix / maintenance / docs_chore）で `.agentdev/drafts/req-draft-{topic-slug}.md` を作成すること |
| REQ-0102-046 | draft-meta の work_type が後続コマンドの消費パターンを決定すること（feature: req-save が消費、bugfix/maintenance/docs_chore: req-save をスキップして case-open が消費） |
| REQ-0102-047 | bugfix / maintenance / docs_chore の draft も case-open および case-auto の入力として検出・処理されること |
| REQ-0102-048 | draft 削除タイミングが全 work_type で共通であること（case-open の Issue 作成 + VERIFY 成功後） |
| REQ-0102-049 | 作業手段は case / Issue / 受け入れ条件 / 作業記録で扱い、active REQ の要件行に混入しないこと |
| REQ-0102-050 | 要件分析・REQ作成支援 skill は状態要件と反映作業の分離基準を判断基準として持つこと |

## 適用範囲

- **対象**: `req-define`、`req-save`、Requirement Source、分類ゲート、関連ドキュメント更新候補
- **対象外**: case 実行、intake / learning lifecycle、コード実装
