## 目的

要件定義と保存の入口は、既存REQ照合、操作分類、反映作業混入防止を含む一貫したプロセスである。本 REQ は req-define / req-save プロセスの phases 横断基本契約を定める。構造化 req_draft 契約（operation_units、artifact_actions、test_strategy スキーマ）は REQ-0138、文書品質ゲート（QG-1 適用）は REQ-0140 へ分割済（AG-004、Phase1 CR-004、2026-07-19 SPLIT）。

## 要件

| ID | 要件 |
|---|---|
| REQ-0102-001 | `req-define` は当該セッション履歴、現在コンテキスト、明示入力ファイル、RU、ユーザーとの対話を受け入れること |
| REQ-0102-002 | `req-define` は要件展開前に、リポジトリ上の REQ/ADR 実ファイル列挙に基づき現行REQ を照合すること。文書記載の番号レンジと実ファイル一覧の乖離を発見した場合は、文書修正または実ファイル確認により解消すること |
| REQ-0102-003 | `req-define` は CREATE を選択する前に APPEND / UPDATE の可否を評価すること |
| REQ-0102-004 | 同一関心対象の現行REQ が存在する場合、要件追加は APPEND / UPDATE を既定候補とすること |
| REQ-0102-005 | CREATE を選択する場合、既存REQ に吸収できない理由を記録すること |
| REQ-0102-006 | 現行REQ は現在満たすべき状態、振る舞い、制約のみを保持すること。要件行候補は変更後に満たすべき振る舞い、制約、状態として記述すること |
| REQ-0102-007 | 要件行は変更後に満たすべき振る舞い、制約、状態に限定し、移行、変更、再定義、削除、改名、移管、除去、更新、移動、名称変更、参照修正などの作業手段は対象外とすること |
| REQ-0102-008 | `req-define` は要件行候補を「変更後仕様」と「反映作業」に分類すること |
| REQ-0102-009 | `req-save` は保存前に反映作業のみの要件行が残っていないか検査すること |
| REQ-0102-010 | `req-save` は反映作業のみの要件行を検出した場合、保存を停止し、該当行、理由、移送先を報告すること |
| REQ-0102-012 | `req-save` は REQ/ADR 保存後に README / DOC-MAP / mapping-table との整合性を確認すること |
| REQ-0102-014 | `req-define` は複数RUを同時に入力として受け入れること。各RUを独立した入力として扱うこと |
| REQ-0102-015 | 複数RU入力時、`req-define` は以下の基準で統合、分割を判断すること：(a) 同一主題、同一対象REQ、同一変更理由に収まるRU群 → 1つのREQ操作に統合 (b) 対象REQ、変更理由、受け入れ条件、対象外が分かれるRU群 → 複数REQ操作として分割 |
| REQ-0102-017 | `req-define` は large / Epic規模と判定した場合、`scale: large` と `decomposition`（分割方針）を記録すること |
| REQ-0102-018 | REQ間または子Issue間に順序制約がある場合、`req-define` は wave候補または依存関係を記録すること |
| REQ-0102-019 | `req-define` はIssue階層そのものを確定しないこと。Issue階層の決定は `case-open` の責務である |
| REQ-0102-021 | `req-save` はREQ doc保存時にIssueを作成しないこと。Issue作成は `case-open` の責務である |
| REQ-0102-022 | `req-define` は実装フェーズの活動（ソースコード編集、テスト実行、ビルドコマンド実行、ソースファイルの作成、変更）を実行しないこと。`req-define` の出力は要件定義成果物（要件行候補、対象REQ、操作種別、依存関係、wave候補）に限定すること |
| REQ-0102-023 | `req-define` の要件行候補に実装指示（具体的なファイル編集内容、コード変更箇所の行番号指定、ステップバイステップの実装手順、テストコマンド、実装コード断片）を含めないこと。要件行は要件定義基準（REQ-0102-006 に従い変更後に満たすべき振る舞い、制約、状態として記述）に適合すること |
| REQ-0102-024 | REQ 要件行は満たす必要がある検証可能な要件に限定し、推奨、任意、将来候補を要件行として扱わない |
| REQ-0102-025 | REQ 要件行における規範語（必達/推奨、任意相当）の使用は任意とすること |
| REQ-0102-026 | 推奨 相当の内容は必達要件として再定義できる場合のみ REQ 要件行に残す |
| REQ-0102-027 | 必達要件として再定義できない 推奨 相当の内容は、注記、SPEC、guide、backlog、report のいずれか適切な文書へ移す |
| REQ-0102-028 | 任意 相当の内容は原則として許容仕様、設計余地、将来候補として適切な文書に配置し、REQ 要件行は対象外とすること |
| REQ-0102-029 | 主語、適用範囲、既存REQ から一意に導ける再説明は、現行REQ の要件行の対象外とすること |
| REQ-0102-030 | 既存要件の言い換えに過ぎず、新しい判定基準、例外、対象外、検査境界、優先関係を含まない文は、要件行の対象外とすること |
| REQ-0102-031 | 既存制約との関係説明（括弧内補足の「整合する」「追従する」等）が判定境界を増やさない場合、当該説明は要件行に含まれないこと |
| REQ-0102-032 | 対象外条件、例外条件、検査対象範囲、検出事項条件、出力粒度、優先関係を実際に変える文は、要件行に残ること |
| REQ-0102-038 | Epic / Wave / Issue 構成の生成は `case-open` の責務であり、`req-define` は execution_groups セクションを出力しないこと |
| REQ-0102-039 | req-save は OU ID 指定時に指定 OU だけを処理対象とし、他 OU は処理しないこと |
| REQ-0102-040 | req-save は OU ID 指定なしで draft に OU が 1 件の場合、その OU を自動選択して処理すること |
| REQ-0102-041 | req-save は OU ID 指定なしで draft に OU が 2 件以上の場合、OU 一覧を表示して停止すること |
| REQ-0102-042 | req-save は OU の保存結果を draft の当該 OU `result` に書き戻すこと |
| REQ-0102-043 | req-save は REQ-0108 への APPEND 操作時に、追加対象要件行に関連する integrity-rule-catalog.md の catalog entry 有無を確認すること |
| REQ-0102-044 | catalog entry が未記載の場合、req-save はユーザーに catalog entry 追記を促すこと。docs/specs/ 配下の編集はユーザー作業に限定すること |
| REQ-0102-045 | `req-define` は全 work_type（feature / bugfix / maintenance / docs_chore）で draft を作成すること |
| REQ-0102-046 | draft-meta の work_type が後続コマンドの消費パターンを決定すること（feature: req-save が消費、bugfix/maintenance/docs_chore: req-save をスキップして case-open が消費） |
| REQ-0102-047 | bugfix / maintenance / docs_chore の draft も case-open および case-auto の入力として検出、処理されること |
| REQ-0102-048 | draft 削除タイミングが全 work_type で共通であること（case-open の Issue 作成 + VERIFY 成功後） |
| REQ-0102-049 | 作業手段は case / Issue / 受け入れ条件 / 作業記録で扱い、現行REQ の要件行に混入しないこと |
| REQ-0102-050 | 要件分析、REQ作成支援 skill は状態要件と反映作業の分離基準を判断基準として持つこと |
| REQ-0102-051 | req-define と req-save のREQ責務が明確に区画化されていること。req-define は要件展開、既存REQ照合、操作分類を担い、req-save はREQ/ADR保存、整合性確認を担うこと |
| REQ-0102-053 | REQ 要件行は、外部から見える責務、command 間契約、ドメイン状態の位置づけ、公開入口、安全境界、恒久的な文書体系制約を表す内容を含めること |
| REQ-0102-054 | REQ 要件行は、当該行が存在しない場合に対象 REQ が何を満たすべきか不明になる内容に限定すること |
| REQ-0102-055 | req-define および req-save は、要件行候補が REQ に記述すべき外部契約、状態要件であるか、SPEC、ルールカタログ、command reference 等に配置すべき詳細、内部パラメータであるかを、保存前に判定すること |
| REQ-0102-056 | req-define は、ドラフト内の実装詳細セクション（修正候補リスト、検出事項カタログ、影響ファイル一覧等）から実装スコープを推定し、要件の複雑さに関わらず実装スコープが大きい場合は scale: large に昇格すること。昇格の閾値は別途定義する |
| REQ-0102-057 | req-define は、ドラフトに実装詳細（個別ファイルの編集指示、修正候補リスト、検出事項カタログ等）が含まれる場合、当該内容を要件定義部分（要件行、acceptance criteria、適用範囲）とは分離されたセクションに配置し、要件定義部分の判読性を確保すること。実装詳細がドラフト全体の過半を占める場合は、実装詳細の要約をユーザーに提案すること |
| REQ-0102-058 | `req-define` は引数なし単体実行時、RU 自動検出に進む前に当該セッション履歴、現在コンテキストを要件ソース候補として評価すること |
| REQ-0102-059 | 当該セッション履歴、現在コンテキストから有効な要件ソースを構成できない場合に限り、RU 自動検出に進むこと |
| REQ-0102-060 | 当該セッション履歴、現在コンテキストが部分的に不足し補足質問で解消可能な場合、RU 自動検出に進まず壁打ち対話を継続すること |
| REQ-0102-061 | RU 自動検出で 2 件以上の RU が存在する場合、候補一覧を提示し自動選択しないこと |
| REQ-0102-062 | 当該セッション履歴、現在コンテキストおよび RU のいずれからも有効な入力を構成できない場合、壁打ち対話を開始すること |
| REQ-0102-067 | req-save は SPEC を編集せず、SPEC 保存は spec-save の責務とすること |
| REQ-0102-068 | req-define は ADR 判断が必要な要件について、ADR 判断確定前にアーキテクチャ確認ステップを実行すること |
| REQ-0102-069 | req-define は ADR 番号を推測指定せず `new:{topic-slug}` 形式を使用すること。req-save は ADR ファイル保存時に ADR 採番ルールで確定した番号を振り、draft 内の全 ADR 参照を当該確定番号で置換すること |
| REQ-0102-070 | req-define は要件doc保存後、要件doc確認前に auto_gate を確認し、auto_ready:false または未解決 item（unresolved_questions/conflicts/out_of_repo_operations/stop_reasons）が残る場合、当該 stop_reasons を解消する方策を壁打ちフェーズで合意すること |
| REQ-0102-071 | ユーザーが明示的に「auto_ready:false のまま標準フローで手動実行する」と選択した場合のみ false を許容すること。当該選択は conflict_resolutions に記録すること |
| REQ-0102-072 | req-define は RU の frontmatter、本文から対象領域キーワードを抽出し、glob/grep で関連 REQ/ADR/SPEC を事前特定してから explore 委譲の調査スコープを絞り込むこと。ただし REQ-0102-002 が要求する実ファイル完全列挙は維持し、絞り込みは explore 委譲の調査優先対象リストにのみ適用すること（ハードフィルタではなくヒントとして扱う） |
| REQ-0102-073 | req-define はアーキテクチャ助言サブエージェントへの相談入力を `agentdev-architecture-advisory` skill が定める標準テンプレートで構築し、助言サブエージェントの出力を 4 ラベル構造（確定事項/推定事項/ユーザー確認事項/ブロッカー）で要求すること。ラベル構造は soft-contract（ADR-0124）とし厳格スキーマ検証を導入しないこと。最終的なラベル分類は親エージェントが行うこと |
| REQ-0102-083 | `req-define` は壁打ち開始時、入力（RU、セッションコンテキスト、明示入力ファイル）の構造を入力データの性質に応じた分析フレームで先行して整理し、個別論点の深掘り前に全体構造をユーザーに提示すること |
| REQ-0102-084 | `req-define` は二項選択を求められた質問に対し、件数と根拠でいずれかを明示して回答すること。「混在」「要確認」単独の回答を出力しないこと。両選択肢に該当する場合は両方の件数と根拠を明示すること |
| REQ-0102-085 | `req-define` は修正の要否を検討する際、実装面（ソースコード、スクリプト、スキル定義ファイル等）と SPEC 面（docs/specs/ 配下の文書）の両面を分析し、各面の修正対象と修正内容を明示すること。片面のみの分析で修正要否を断定しないこと |

## 適用範囲

- **対象**: `req-define`、`req-save`、分類ゲート
- **対象外**: case 実行、intake / learning lifecycle、コード実装
