## 目的

要件定義と保存の入口を、既存REQ照合・操作分類・反映作業混入防止を含む一貫したプロセスとして定義する。

## 要件

| ID | 要件 |
|---|---|
| REQ-0102-001 | `req-define` はユーザーとの対話、明示入力ファイル、RU を Requirement Source として受け入れること（SHALL） |
| REQ-0102-002 | `req-define` は要件展開前に active REQ を照合し、関連する関心対象と対象REQを推論すること（SHALL） |
| REQ-0102-003 | `req-define` は CREATE を選択する前に APPEND / UPDATE の可否を評価すること（SHALL） |
| REQ-0102-004 | 同一関心対象の active REQ が存在する場合、要件追加は APPEND / UPDATE を既定候補とすること（SHALL） |
| REQ-0102-005 | CREATE を選択する場合、既存 active REQ に吸収できない理由を記録すること（SHALL） |
| REQ-0102-006 | 要件行候補は、変更後に満たすべき振る舞い・制約・状態として記述すること（SHALL） |
| REQ-0102-007 | 既存成果物への更新・削除・移動・名称変更・参照修正そのものを要件行として扱わないこと（SHALL） |
| REQ-0102-008 | `req-define` は要件行候補を「変更後仕様」と「反映作業」に分類すること（SHALL） |
| REQ-0102-009 | `req-save` は保存前に反映作業のみの要件行が残っていないか検査すること（SHALL） |
| REQ-0102-010 | `req-save` は反映作業のみの要件行を検出した場合、保存を停止し、該当行・理由・移送先を報告すること（SHALL） |
| REQ-0102-011 | `req-define` は関連ドキュメント更新候補を抽出し、対象パス・判定・必要対応を出力すること（SHOULD） |
| REQ-0102-012 | `req-save` は REQ/ADR 保存後に README / DOC-MAP / mapping-table との整合性を確認すること（SHALL） |
| REQ-0102-013 | 未決分岐がある場合、`req-define` は分岐・判断軸・推奨案・ユーザー確認事項を明示すること（SHALL） |
| REQ-0102-014 | `req-define` は複数RUを同時に入力として受け入れること（SHALL）。各RUを独立したRequirement Sourceとして扱うこと |
| REQ-0102-015 | 複数RU入力時、`req-define` は以下の基準で統合・分割を判断すること（SHALL）：(a) 同一主題・同一対象REQ・同一変更理由に収まるRU群 → 1つのREQ操作に統合 (b) 対象REQ・変更理由・受け入れ条件・対象外が分かれるRU群 → 複数REQ操作として分割 |
| REQ-0102-016 | 複数REQ操作がある場合、`req-define` は各操作の対象REQ・操作種別・要件行候補・依存関係を後続コマンドが参照可能な形で明示すること（SHALL） |
| REQ-0102-017 | `req-define` は large / Epic規模と判定した場合、`scale: large` と `decomposition`（分割方針）を記録すること（SHALL） |
| REQ-0102-018 | REQ間または子Issue間に順序制約がある場合、`req-define` は wave候補または依存関係を記録すること（SHALL） |
| REQ-0102-019 | `req-define` はIssue階層そのものを確定しないこと（SHALL）。Issue階層の決定は `case-open` の責務である |
| REQ-0102-020 | `req-save` は複数REQ操作を保存した場合、以下を後続コマンドが参照できる形で保持すること（SHALL）：(a) 保存したREQ doc一覧 (b) 各REQ操作と保存先REQ docの対応 (c) source RUとREQ操作の対応 (d) case-open が入力として扱える保存結果 |
| REQ-0102-021 | `req-save` はREQ doc保存時にIssueを作成しないこと（SHALL）。Issue作成は `case-open` の責務である |
| REQ-0102-022 | `req-define` は実装フェーズの活動（ソースコード編集、テスト実行、ビルドコマンド実行、ソースファイルの作成・変更）を実行しないこと（MUST NOT）。`req-define` の出力は要件定義成果物（要件行候補・対象REQ・操作種別・依存関係・wave候補）に限定すること（SHALL） |
| REQ-0102-023 | `req-define` の要件行候補に実装指示（具体的なファイル編集内容・コード変更箇所の行番号指定・ステップバイステップの実装手順・テストコマンド・実装コード断片）を含めないこと（SHALL）。要件行は要件定義基準（REQ-0102-006 に従い変更後に満たすべき振る舞い・制約・状態として記述）に適合すること（SHALL） |
| REQ-0102-024 | REQ 要件行は満たす必要がある検証可能な要件に限定し、推奨・任意・将来候補を要件行として扱わない |
| REQ-0102-025 | REQ 要件行に RFC 2119 語（SHALL/SHOULD/MAY）の使用を必須としない |
| REQ-0102-026 | SHOULD 相当の内容は必達要件として再定義できる場合のみ REQ 要件行に残す |
| REQ-0102-027 | 必達要件として再定義できない SHOULD 相当の内容は、注記・SPEC・guide・backlog・report のいずれか適切な文書へ移す |
| REQ-0102-028 | MAY 相当の内容は原則として REQ 要件行から除外し、許容仕様・設計余地・将来候補として適切な文書へ移す |

## 適用範囲

- **対象**: `req-define`、`req-save`、Requirement Source、分類ゲート、関連ドキュメント更新候補
- **対象外**: case 実行、intake / learning lifecycle、コード実装
