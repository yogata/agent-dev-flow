## 目的

`req-define` の段階で、要件変更に伴って更新が必要になり得る既存仕様・REQ・ADR・README を限定的に検出し、`case-open` / `case-run` / `case-close` へ作業対象として引き渡す。

### 背景

staff-schedule の Issue #815 で、REQ-0030 は `staffs.is_active` / `inactive_detected_at` の導入を要求していたが、`docs/specs/system.md` に「is_active フラグは使用しない」という旧仕様が残存し、`case-close` の G09 で停止した。`case-close` は、変更後仕様と矛盾するドキュメント更新漏れを完了不可とする責務を既に持つため、この停止は正当である。

現行 `req-define` は既存REQ照合を行うが、`docs/specs/**` などの関連仕様まで限定探索する責務は明示されていない。また G05 は `docs/` 配下の広範探索を禁止している。

## 要件

### req-define: 変更種別判定・キーワード抽出・限定探索

| ID | 要件 |
|---|---|
| REQ-0034-001 | `req-define` は、ユーザー入力・既存REQ照合結果・要件展開結果から、DBカラム、状態フラグ、判定条件、API仕様、画面表示、既存REQ/ADR/仕様の意味変更に該当する変更種別を自律的に判定すること（SHALL） |
| REQ-0034-002 | `req-define` は、変更種別に該当すると判定した場合、関連キーワードを抽出すること（SHALL） |
| REQ-0034-003 | `req-define` は、抽出キーワードを用いて、関連ドキュメント更新候補を read-only で限定探索すること（SHALL） |
| REQ-0034-004 | 探索対象は `docs/specs/**`、`docs/requirements/REQ-*.md`、`docs/requirements/views/*.md`、`docs/adr/**`、`docs/README.md`、`docs/requirements/README.md` に限定すること（SHALL） |
| REQ-0034-005 | 上記探索は `req-define` の Step 4b「関連ドキュメント更新候補抽出」でのみ許可し、`docs/` 配下の無限定探索は禁止すること（SHALL） |
| REQ-0034-006 | `req-define` は、検出した候補ごとに、対象パス、検出語句、判定、必要対応を出力すること（SHALL） |
| REQ-0034-007 | 判定は、少なくとも `直接矛盾`、`更新候補`、`影響なし` に分類すること（SHALL） |
| REQ-0034-008 | 既存REQ・ADR・仕様の記述を更新対象として扱えば新要件が成立する場合、`req-define` は停止せず、関連ドキュメント更新候補として後続工程へ引き渡すこと（SHOULD） |
| REQ-0034-009 | 既存REQ・ADR・仕様のどれを正とするかの判断が必要で、単なる追随更新では解消できない場合、`req-define` は要件成立判断としてユーザー判断を求めること（SHALL） |
| REQ-0034-010 | `req-define` は、関連ドキュメント更新候補をドラフトまたはセッション内要件docに保持すること（SHALL） |

### case-open: Issue 本文転記と完了条件反映

|---|---|
| REQ-0034-011 | `case-open` は、`req-define` が出力した関連ドキュメント更新候補を Issue 本文へ転記すること（SHALL） |
| REQ-0034-012 | `case-open` は、`直接矛盾` に分類された候補を Issue の完了条件へ反映すること（SHALL） |

### case-run: 実装計画への組み込み

| ID | 要件 |
|---|---|
| REQ-0034-013 | `case-run` は、Issue本文に関連ドキュメント更新候補が存在する場合、それを実装計画および関連ドキュメント更新確認の対象に含めること（SHALL） |

## 適用範囲

- **対象**:
  - `/agentdev/req-define` — Step 4b 追加、G05 例外拡張
  - `/agentdev/case-open` — Issue 本文への関連ドキュメント更新候補転記、直接矛盾の完了条件反映
  - `/agentdev/case-run` — 関連ドキュメント更新候補の実装計画組み込み
  - `agentdev-req-file-manager` — 要件docへの関連ドキュメント更新候補セクション追加
  - 必要に応じて `agentdev-spec-compliance`
- **対象外**:
  - `req-save` による `docs/specs/**` 編集
  - case-close G09 の緩和
  - 自動修正の実装
  - コード実装計画
  - `docs/` 全体の網羅探索
  - ユーザーに関連ファイル列挙を求める変更
  - case-close の停止条件の緩和

## Step 4b: 関連ドキュメント更新候補抽出

現行 `req-define` は Step 3 で既存REQ照合、Step 4 で要件展開、Step 5 でADR閾値判断を行う。追加位置は **Step 4 と Step 5 の間** とする。

| 処理 | 内容 |
|---|---|
| 入力 | Step 4 で展開済みの要件 |
| 判定 | DBカラム、状態フラグ、判定条件、API仕様、画面表示、既存REQ/ADR/仕様の意味変更に該当するかを自律判定 |
| キーワード抽出 | 要件本文、既存REQ照合結果、対象コンポーネント、API名、DBカラム名、状態名、表示文言から抽出 |
| 探索 | G05例外として許可された範囲を read-only で限定探索 |
| 分類 | `直接矛盾` / `更新候補` / `影響なし` |
| 出力 | 対象パス、検出語句、判定、必要対応 |
| 後続連携 | Step 6 の要件doc生成へ反映 |

## G05 例外拡張

現行 G05 は維持する。ただし、Step 4b に限り例外を拡張する。

| 区分 | 扱い |
|---|---|
| `docs/` 配下の広範探索 | 禁止 |
| 明示入力ファイル | 許可 |
| `docs/requirements/**` の read-only 参照 | 許可 |
| Step 4b の抽出キーワードベース限定探索 | 許可 |
| 無関係な docs 再構成・網羅探索 | 禁止 |

## 判定基準

| 判定 | 定義 | 後続扱い |
|---|---|---|
| 直接矛盾 | 既存記述が新要件と明示的に反対の内容を持つ | Issue本文へ転記し、完了条件へ反映 |
| 更新候補 | 新要件の影響を受ける可能性が高いが、直接矛盾までは確認できない | Issue本文へ転記し、case-runで確認 |
| 影響なし | 検索ヒットしたが、更新不要と判断できる | 出力してもよいが、完了条件には入れない |

## 停止判断

| 状況 | req-define の扱い |
|---|---|
| 旧仕様を更新すれば新要件が成立する | 停止しない。関連ドキュメント更新候補として後続工程へ渡す |
| 既存REQ・ADR・仕様のどれを正とするか判断が必要 | ユーザー判断を求める |
| ADRの決定と新要件が根本衝突する | ADRの改訂・supersede 判断が必要なためユーザー判断を求める |
| 単なるREADMEリンク不足 | 直接矛盾ではないため、完了条件化しない |

## 出力形式

`req-define` は、関連ドキュメント更新候補を以下の形式で出力する。

```markdown
## 関連ドキュメント更新候補

| パス | 検出語句 | 判定 | 必要対応 |
|---|---|---|---|
| docs/specs/system.md | is_active フラグは使用しない | 直接矛盾 | REQ-0030に合わせてアクティブ判定基準を更新する |
```

## case-close の扱い

関連ドキュメント更新候補が未解消で、変更後仕様と矛盾する場合の完了不可判定は、REQ-0015-006 および case-close G09 により既に担保されている。本要件では再定義しない。

## req-save の扱い

`req-save` の編集スコープは広げない。現行 `req-save` は `docs/requirements/**`、`docs/adr/**`、`docs/README.md`、`.sisyphus/drafts/**` を編集対象としており、`docs/specs/system.md` は対象外である。仕様書更新は `case-run` の作業対象として扱う。
