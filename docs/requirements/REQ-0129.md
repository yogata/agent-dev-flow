## 目的

intake / learning / diagnostics の採用済み成果物を分析、統合し、ユーザー承認を経て RU（Requirement Unit）を `.agentdev/backlog/req-units/` に直接生成する。
採用済み成果物間の矛盾検出と depends_on による依存順序制御を含む。

## 要件

| ID | 要件 |
|---|---|
| REQ-0129-001 | `backlog-review` は intake / learning の採用済み成果物を分析、統合し、ユーザー承認を経て RU を backlog ドメイン状態に生成すること。backlog-save を経由せず、backlog-review 内で直接 RU を生成する |
| REQ-0129-002 | RU は採用済み成果物の単純コピーではなく、要件化に向けた構造化内容とすること |
| REQ-0129-003 | REQ再構成intakeは通常の短期Issue化候補と区別し、専用経路で処理すること（`backlog-review` 向け短期候補は対象外） |
| REQ-0129-004 | `backlog-review` は intake / learning / inspect の promoted 成果物を読み込み、統合、分割の可能性を分析し、判定をユーザーに提示して承認を得ること |
| REQ-0129-005 | `backlog-review` は採用済み成果物間の矛盾を検出し、解決をユーザーに委ねること |
| REQ-0129-006 | RU 化に成功した採用済み成果物のみが `promoted/` から削除されること。RU 化に失敗した採用済み成果物は `promoted/` に残存すること。矛盾成果物は RU 化されず `promoted/` に残存すること |
| REQ-0129-007 | `backlog-review` 実行後、ドメイン状態配下の変更は git commit / push 済みであること |
| REQ-0129-008 | RU frontmatter に `depends_on` を記述できること。`depends_on` の値はRU-IDに限定すること。採用済み成果物 path を依存先として扱わないこと |
| REQ-0129-009 | `backlog-review` は、同時処理するRU群に `depends_on` が含まれる場合、依存先RUを先に処理すること |
| REQ-0129-011 | 未解決依存、循環依存、または同時処理対象外の依存がある場合、`backlog-review` は当該RUを保留し理由を提示すること |
| REQ-0129-012 | backlog-review は上流の恒久契約候補判定を独立して再評価し、恒久契約として不適格な成果物を RU 化しないこと。恒久契約適格性基準は document-model.md SPEC が定義し、backlog-review はその参照を保持すること |

## 適用範囲

- **対象**: backlog-review、RU 直接生成（backlog-save 経由なし）、採用済み成果物読込、統合、矛盾検出、RU の構造化内容、depends_on（RU-ID限定、依存順序、循環依存検証）、REQ再構成intake のルーティング分離、backlog-review 後の git 永続化
- **対象外**: intake パイプライン（REQ-0127）、learning-promote（REQ-0128）、RU ライフサイクル、RU 削除タイミング（REQ-0105）、`backlog-review` の RU 統合、分割判定の詳細ルール、req-define の RU 読み込み仕様（REQ-0102）
