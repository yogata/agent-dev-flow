## 目的

AgentDevFlow の標準ワークフロー（req-define → req-save → case-open → case-run → case-close）を維持しつつ、ユーザーが明示した場合に req-define 完了後の後続工程を最大限自走させる入口を提供する。

repo に閉じる変更に限り、エージェントが自律的に実装、検証、PR作成、merge、close まで進める。
DB マイグレーション実行、デプロイ、cloud apply 等、repo 外の実体を変更する操作は自走対象外とする。

> **REQ-0114-096/097 適用範囲追記**
>
> REQ-0114 の適用範囲（対象）に以下を追加すること:
> - case-auto と case-run の実行方式記述責務境界（case-auto は外部契約のみ、case-run が内部実行方式の SSoT）
> - Epic Wave 実行の外部契約参照（最大5件並列実行）への記述限定

> **REQ-0114-088 破壊的 UPDATE ノート（CR-001）**
>
> - **変更前**: 「case-open は複数の独立 OU（depends_on 空、L0 相当）を検出した場合、自動的に Epic Issue 化し Wave 1 に全 OU を配置すること」
> - **変更後**: 「case-open は OU 群の依存グラフの連結成分（必須依存のみをエッジとする）を Epic 候補の出発点とし、依存強度、Epic サイズ、機能的一貫性の3軸判断で複数 Standard Issue / 複数 Epic Issue / 混在を自律生成すること。単独根（1 OU だけの連結成分）は Standard flow とする」
> - **後方互換性**: 従来の depends_on 空（技術的独立のみ）運用では、全 OU が別 Standard Issue として分散される。REQ-0114-087〜093 のうち、独立 OU 自動 Epic 化（REQ-0114-088）のみ本 UPDATE の対象であり、他は維持する。既存の単一 Epic 運用（旧 REQ-0114-088 前提のガイド、テンプレート）は新モデルに追随して更新する。

> **REQ-0114-016(8) 停止条件改訂ノート（REQ-0151-006）**
>
> - **変更前**: 停止条件 (8)「merge 競合 / リモート hash 不一致」により、コンフリクト発生時に case-auto は即停止していた。
> - **変更後**: 停止条件 (8) は「case-auto がアクセス可能な文脈を総動員しても解消不能なコンフリクト」に段階化される（REQ-0151-006、ADR-0132 参照）。機械的競合（rebase で自動解決可能）は case-close（Level 1）が解消するため停止条件から除外する。Level 2 の再委譲を上限回数（2回）試行しても解消しない場合のみ停止する。
> - **後方互換性**: コンフリクト発生時の即停止は廃止。3レベルコンフリクト解消モデル（Level 1: case-close rebase、Level 2: case-auto 文脈付き再委譲、Level 3: case-auto オーケストレーション級判断）を順次試行した後にのみ停止する。

- **対象**:
  - `/agentdev/case-auto` コマンド定義の構成
  - 入力解決（3段階優先順位）
  - 実行モデル（原本参照によるオーケストレーション）

  - ドキュメント構成（README / command reference / guides）

  - Epic Issue 本文の単一書き手制約
  - 完了報告のタイミング情報（開始時刻、終了時刻、所要時間）
  - 構成工程の task() 委譲モデル（req-save / spec-save / case-open / case-close、ADR-0127）
