## 目的

REQ体系の再構成を、履歴を失わず、active基準を曖昧にせず、段階的に実行できる運用として定義する。

## 要件

| ID | 要件 |
|---|---|
| REQ-0109-001 | REQ再構成では、現行判断に使う active REQ と履歴参照用 retired REQ を明確に分離すること（SHALL） |
| REQ-0109-002 | 旧REQを廃止する場合、削除せず `docs/requirements/retired/` に移動すること（SHALL） |
| REQ-0109-003 | retired REQ の内容を現行要件として再利用する場合、active REQ に要件行として移行すること（SHALL） |
| REQ-0109-004 | active REQ に移行しない旧REQは `retired-no-successor` または `historical-only` として理由を記録すること（SHALL） |
| REQ-0109-005 | 移行表は旧REQごとに移行判定、移行先、非移行理由、注意点を記録すること（SHALL） |
| REQ-0109-006 | REQ再構成候補は通常intakeと区別し、REQ再構成レビューの入力として扱うこと（SHALL） |
| REQ-0109-007 | REQ再構成候補の raw intake item は `.agentdev/intake/inbox/req-restructure/` に保存すること（SHALL） |
| REQ-0109-008 | REQ再構成intakeは保存時点でREQの分割・統合・移動・廃止を要求するものではなく、レビュー入力材料として扱うこと（SHALL） |
| REQ-0109-009 | REQ再構成レビューでは SPLIT / MERGE / MOVE / DUPLICATE / RETIRE / DRIFT を検知観点として扱うこと（SHOULD） |
| REQ-0109-010 | active REQ の数を増やす場合、関心対象モデルを更新し、README / DOC-MAP / mapping-table と整合させること（SHALL） |
| REQ-0109-011 | retired REQ 内のリンク切れは履歴保持を優先し、必要な場合のみ注記で補足すること（MAY） |
| REQ-0109-012 | REQ再構成レビューは `/agentdev/req-restructure-review` コマンドで実行できること（SHALL） |
| REQ-0109-013 | `/agentdev/req-restructure-review` は診断専用とし、ファイル変更、Issue作成、PR作成、worktree作成、retired移動、mapping-table更新を行わないこと（SHALL） |
| REQ-0109-014 | `/agentdev/req-restructure-review` は `docs/requirements/`、`docs/requirements/retired/`、`docs/requirements/mapping-table.md`、`docs/DOC-MAP.md`、`docs/README.md`、`.opencode/`、`docs/guides/`、`docs/specs/` を確認対象とすること（SHALL） |
| REQ-0109-015 | `/agentdev/req-restructure-review` は SPLIT / MERGE / MOVE / DUPLICATE / RETIRE / DRIFT の観点でREQ体系を診断すること（SHALL） |
| REQ-0109-016 | `/agentdev/req-restructure-review` は再構成に伴うREQ参照ID、第一参照導線、active / retired / 世代境界の整合性を確認すること（SHALL） |
| REQ-0109-017 | `/agentdev/req-restructure-review` は出力を最小限の診断結果、問題候補、推奨アクション、必要な場合の req-define 入力案に限定すること（SHALL） |
| REQ-0109-018 | `/agentdev/req-restructure-review` は旧REQ全件の移行先一覧、全修正ファイル候補、mapping-table完成案、Issue本文完成版、PR本文完成版を出力しないこと（SHALL） |
| REQ-0109-019 | `/agentdev/req-restructure-review` は未処理の intake / learning / RU がある場合、その存在と診断への影響可能性だけを出力すること（SHALL） |
| REQ-0109-020 | `/agentdev/req-restructure-review` は intake item の accept / reject、learning の refine / promote、RU生成、promoted artifact の直接処理を行わないこと（SHALL） |
| REQ-0109-021 | active set の世代交代を伴う大規模再構成の場合のみ、次世代REQは100の位をインクリメントして採番すること。通常の APPEND / UPDATE では既存 active REQ を維持すること（SHOULD） |

## 適用範囲

- **対象**: REQ再構成、retired archive、migration table、REQ再構成intake、REQ再構成レビューコマンド、100番台採番運用
- **対象外**: 個別旧REQ本文の履歴改変、`.sisyphus/` 運用
