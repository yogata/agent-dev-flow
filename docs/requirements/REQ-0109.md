## 目的

REQ体系は履歴を保持し、active基準を曖昧にせず、段階的な構成変更を許容する整合性を維持する。

## 要件

| ID | 要件 |
|---|---|
| REQ-0109-001 | REQ体系において、現行判断に使う active REQ と履歴参照用 retired REQ は明確に区分されていること |
| REQ-0109-002 | active判断から外れた旧REQは `docs/requirements/retired/` に配置され、ファイルの削除は行われないこと |
| REQ-0109-003 | retired REQ の内容が現行要件として採用される場合、当該内容は active REQ の要件行として存在すること |
| REQ-0109-004 | active REQ に後継がない旧REQは `retired-no-successor` または `historical-only` として、その理由が移行表に記録されていること |
| REQ-0109-005 | 移行表は旧REQごとに移行判定、移行先、非移行理由、注意点を保持すること |
| REQ-0109-006 | REQ再構成候補は通常intakeとは区別され、REQ再構成レビューの入力として扱われること |
| REQ-0109-007 | REQ再構成候補の raw intake item は `.agentdev/intake/inbox/req-restructure/` に配置されること |
| REQ-0109-008 | REQ再構成intakeは、保存時点ではREQの分割・統合・移動・廃止を要求せず、レビュー入力材料として位置づけられること |
| REQ-0109-009 | REQ再構成レビューの検知観点は SPLIT / MERGE / MOVE / DUPLICATE / RETIRE / DRIFT であること |
| REQ-0109-010 | active REQ の構成変更に伴い、関心対象モデル・README・DOC-MAP・mapping-table は整合状態を維持すること |
| REQ-0109-011 | retired REQ 内のリンク切れに対しては履歴保持が優先され、注記は必要な場合のみ付与されること |
| REQ-0109-012 | REQ再構成レビューは `/agentdev/inspect-docs` コマンドで実行できること |
| REQ-0109-013 | `/agentdev/inspect-docs` は診断専用とし、ファイル変更、Issue作成、PR作成、worktree作成、retired移動、mapping-table更新を行わないこと |
| REQ-0109-014 | `/agentdev/inspect-docs` は `docs/requirements/`、`docs/requirements/retired/`、`docs/requirements/mapping-table.md`、`docs/DOC-MAP.md`、`docs/README.md`、`.opencode/`（runtime projection; sourceは`src/opencode/`）、`docs/guides/`、`docs/specs/` を確認対象とすること |
| REQ-0109-015 | `/agentdev/inspect-docs` は SPLIT / MERGE / MOVE / DUPLICATE / RETIRE / DRIFT の観点でREQ体系を診断すること |
| REQ-0109-016 | `/agentdev/inspect-docs` は再構成に伴うREQ参照ID、第一参照導線、active / retired / 世代境界の整合性を確認すること |
| REQ-0109-017 | `/agentdev/inspect-docs` は出力を最小限の診断結果、問題候補、推奨アクション、必要な場合の req-define 入力案に限定すること |
| REQ-0109-018 | `/agentdev/inspect-docs` は旧REQ全件の移行先一覧、全修正ファイル候補、mapping-table完成案、Issue本文完成版、PR本文完成版を出力しないこと |
| REQ-0109-019 | `/agentdev/inspect-docs` は未処理の intake / learning / RU がある場合、その存在と診断への影響可能性だけを出力すること |
| REQ-0109-020 | `/agentdev/inspect-docs` は intake item の accept / reject、learning の refine / promote、RU生成、promoted artifact の直接処理を行わないこと |
| REQ-0109-021 | active set の世代交代を伴う大規模な構成変更の場合のみ、次世代REQは100の位をインクリメントした番号を採番すること。通常の APPEND / UPDATE では既存 active REQ の番号を維持すること |
| REQ-0109-022 | inspect-docs は SPLIT / MERGE / MOVE / DUPLICATE / RETIRE / DRIFT の各観点について具体的な検出シグナルを持つこと |
| REQ-0109-023 | SPLIT の検出シグナルは 1つのREQに複数の関心対象、複数 artifact 種別、複数 command family、複数 lifecycle の混在とすること |
| REQ-0109-024 | MERGE の検出シグナルは複数REQが同じ目的、同じ対象 artifact、同じ command、同じ責務を扱っていることとすること |
| REQ-0109-025 | MOVE の検出シグナルは REQ行が変更後仕様ではなく反映作業そのものになっていることとすること |
| REQ-0109-026 | DUPLICATE の検出シグナルは active REQ 間または REQ と spec / guide / command の間で同じ 必達要件 相当の責務が重複していることとすること |
| REQ-0109-027 | RETIRE の検出シグナルは active REQ が現行案内から参照されず関心対象も既存 active REQ に吸収済みであることとすること |
| REQ-0109-028 | DRIFT の検出シグナルは REQ が要求する対象と実体ファイルまたは command 定義が矛盾していることとすること |
| REQ-0109-029 | 1シグナルのみの場合は観察メモに留め、問題候補 finding としては出さないこと |
| REQ-0109-030 | 2シグナル以上が揃った場合に問題候補として出すこと |
| REQ-0109-031 | 3シグナル以上、または active / retired 判断に影響する場合は高優先度候補として出すこと |
| REQ-0109-032 | 機械的なリンク切れ・ID不整合の一覧化を主目的にせず、再構成判断に影響する場合だけ根拠として使うこと |
| REQ-0109-033 | 未処理 intake / learning / RU / requirements-review finding は存在と影響可能性の報告に留め、処理・accept / reject / promote / RU生成は行わないこと |
| REQ-0109-034 | 問題候補を出す場合は観点、対象、根拠、シグナル数、確信度、推奨アクション、req-define入力案を含めること |
| REQ-0109-035 | 確信度は high / medium / low とすること |
| REQ-0109-036 | 推奨アクションは SPLIT / MERGE / MOVE / RETIRE / UPDATE / APPEND / no-action のいずれかとすること |
| REQ-0109-037 | 入力対象として `.agentdev/intake/inbox/req-restructure/`、`.agentdev/drafts/requirements-review-finding-*.md`、`.agentdev/backlog/req-units/RU-*.md` の存在確認を行うこと |
| REQ-0109-038 | REQ-0122（規範語ルールおよび記載）は retired 状態であり、`docs/requirements/retired/REQ-0122.md` に配置されている。REQ-0122 の条項は目的達成により他REQへの吸収を伴わず退役済みである。理由: RFC2119 規範語の全面除外が PR #743（commit 90064c2）で達成され、REQ-0122 自身の自己矛盾（除外対象の列挙に RFC2119 規範語 SHALL/MUST/SHOULD/MAY 等を使用していた）を解消するため |

## 適用範囲

- **対象**: REQ再構成、retired archive、migration table、REQ再構成intake、REQ再構成レビューコマンド、100番台採番ルール、inspect-docs の検出シグナル定義（SPLIT / MERGE / MOVE / DUPLICATE / RETIRE / DRIFT）、finding 化基準（シグナル数閾値、機械的検査との分離、未処理 artifact の扱い）、出力スキーマ（観点、対象、根拠、シグナル数、確信度、推奨アクション、req-define入力案）、入力対象（req-restructure intake、requirements-review finding、RU）

- **対象外**: 個別旧REQ本文の履歴変更、`.sisyphus/` の扱い、検出シグナルの具体的な閾値パラメータ、出力スキーマの詳細な型定義・正規表現、未処理 artifact の処理・accept / reject / promote
