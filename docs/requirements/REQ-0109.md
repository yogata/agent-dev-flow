## 目的

REQ体系は履歴を保持し、現行基準を曖昧にせず、段階的な構成変更を許容する整合性を維持する。

## 要件

| ID | 要件 |
|---|---|
| REQ-0109-001 | REQ体系において、現行判断に使う現行REQ と履歴参照用の廃止REQ は明確に区分されていること |
| REQ-0109-002 | 現行判断から外れた旧REQは `docs/requirements/retired/` に配置され、ファイルの削除は行われないこと |
| REQ-0109-003 | 廃止REQ の内容が現行要件として採用される場合、当該内容は現行REQ の要件行として存在すること |
| REQ-0109-004 | 現行REQ に後継がない旧REQは `retired-no-successor` または `historical-only` として、その理由が移行表に記録されていること |
| REQ-0109-005 | 移行表は旧REQごとに移行判定、移行先、非移行理由、注意点を保持すること |
| REQ-0109-006 | REQ再構成候補は通常intakeとは区別され、REQ再構成レビューの入力として扱われること |
| REQ-0109-007 | REQ再構成候補の raw intake item は `.agentdev/intake/inbox/req-restructure/` に配置されること |
| REQ-0109-008 | REQ再構成intakeは、保存時点ではREQの分割・統合・移動・廃止を要求せず、レビュー入力材料として位置づけられること |
| REQ-0109-009 | REQ再構成レビューの検知観点は SPLIT / MERGE / MOVE / DUPLICATE / RETIRE / DRIFT であること |
| REQ-0109-010 | 現行REQ の構成変更に伴い、関心対象モデル・README・DOC-MAP・mapping-table は整合状態を維持すること |
| REQ-0109-011 | 廃止REQ 内のリンク切れに対しては履歴保持が優先され、注記は必要な場合のみ付与されること |
| REQ-0109-012 | REQ再構成レビューは `/agentdev/inspect-docs` コマンドで実行できること |
| REQ-0109-013 | `/agentdev/inspect-docs` は診断専用とし、操作スコープを読み取りおよび診断結果の出力に限定すること |
| REQ-0109-014 | `/agentdev/inspect-docs` は `docs/requirements/`、`docs/requirements/retired/`、`docs/requirements/mapping-table.md`、`docs/DOC-MAP.md`、`docs/README.md`、`.opencode/`（実行時の配置先; 原本は `src/opencode/`）、`docs/guides/`、`docs/specs/` を確認対象とすること |
| REQ-0109-015 | `/agentdev/inspect-docs` は SPLIT / MERGE / MOVE / DUPLICATE / RETIRE / DRIFT の観点でREQ体系を診断すること |
| REQ-0109-016 | `/agentdev/inspect-docs` は再構成に伴うREQ参照ID、第一参照導線、現行・廃止・世代境界の整合性を確認すること |
| REQ-0109-017 | `/agentdev/inspect-docs` は出力を最小限の診断結果、問題候補、推奨アクション、必要な場合の req-define 入力案に限定すること |
| REQ-0109-018 | `/agentdev/inspect-docs` の出力は再構成判断に影響する根拠に限定し、旧REQ全件の移行先一覧、全修正ファイル候補、mapping-table完成案、Issue本文完成版、PR本文完成版は出力スコープから除外すること |
| REQ-0109-019 | `/agentdev/inspect-docs` は未処理の intake / learning / RU がある場合、その存在と診断への影響可能性だけを出力すること |
| REQ-0109-020 | `/agentdev/inspect-docs` は intake item の accept / reject、learning の refine / promote、RU生成、採用済み成果物の処理を後続コマンド（intake-promote / learning-promote / backlog-review / inspect-promote）に委ねること |
| REQ-0109-021 | 現行セットの世代交代を伴う大規模な構成変更の場合のみ、次世代REQは100の位をインクリメントした番号を採番すること。通常の APPEND / UPDATE では既存REQ の番号を維持すること |
| REQ-0109-022 | inspect-docs は SPLIT / MERGE / MOVE / DUPLICATE / RETIRE / DRIFT の各観点について具体的な検出シグナルを持つこと |
| REQ-0109-037 | 入力対象として `.agentdev/intake/inbox/req-restructure/`、`.agentdev/drafts/requirements-review-finding-*.md`、`.agentdev/backlog/req-units/RU-*.md` の存在確認を行うこと |
| REQ-0109-039 | SPLIT / MERGE / MOVE / DUPLICATE / RETIRE / DRIFT が inspect-docs の診断観点に加え、REQ運用品質維持の恒常的基準として参照可能であること |
| REQ-0109-047 | inspect-docs は、REQ/SPEC 境界違反・REQ 粒度過小・SPEC 詳細混入・外部契約と内部パラメータの誤分類を診断観点として扱えること。REQ-0101-067〜069 の境界基準・SPEC 移管基準・安定契約例外判定を参照し、現行REQ の再分類候補抽出時の判断基準として機能すること |

## 適用範囲

- **対象**: REQ再構成、廃止REQアーカイブ、移行表、REQ再構成intake、REQ再構成レビューコマンド、100番台採番ルール、inspect-docs の診断観点（SPLIT / MERGE / MOVE / DUPLICATE / RETIRE / DRIFT）、出力責務、入力対象（req-restructure intake、requirements-review finding、RU）

- **対象外**: 個別旧REQ本文の履歴変更、`.sisyphus/` の扱い、検出シグナル定義、confidence enum、recommended action enum、signal threshold、出力スキーマの詳細な型定義・正規表現、未処理 artifact の処理・accept / reject / promote、作業履歴
