## 目的

AgentDevFlow の要件・判断・仕様・参照入口を、読み手が迷わない基準構造として維持する。active REQ は現在満たすべき要件を表し、retired REQ は履歴参照に限定する。

## 要件

| ID | 要件 |
|---|---|
| REQ-0101-001 | active REQ は `docs/requirements/REQ-{NNNN}.md` 直下に配置すること |
| REQ-0101-002 | retired REQ は `docs/requirements/retired/REQ-{NNNN}.md` に配置し、現行要件判断に使用しないこと |
| REQ-0101-003 | REQ ID は4桁ゼロ埋めの安定IDとし、active / retired を問わず再利用しないこと |
| REQ-0101-004 | REQ の要件行IDは `REQ-{NNNN}-{MMM}` 形式で一意に識別すること |
| REQ-0101-005 | REQ frontmatter は `id`、`title`、`created`、`updated` を含めること |
| REQ-0101-006 | active REQ は関心対象の総体を表し、追加要件は原則として既存 active REQ への APPEND / UPDATE で扱うこと |
| REQ-0101-007 | 新規 active REQ を作成する場合、既存 active REQ に吸収できない独立した関心対象である理由を記録すること |
| REQ-0101-008 | ADR は将来の設計・運用・文書システムを制約する決定の記録として `docs/adr/ADR-{NNNN}.md` に配置すること |
| REQ-0101-009 | SPEC は現在仕様の基準として `docs/specs/*.md` に配置し、REQ/ADR の判断内容を代替しないこと |
| REQ-0101-010 | `docs/DOC-MAP.md` は文書探索入口であり、REQ/ADR/SPEC の内容を代替しないこと |
| REQ-0101-011 | 文書間に矛盾がある場合、active REQ を優先すること |
| REQ-0101-012 | `docs/requirements/README.md` は active REQ の第一参照一覧、retired REQ の履歴入口、移行表への導線を提供すること |
| REQ-0101-013 | 旧REQから新REQへの移行有無は `docs/requirements/mapping-table.md` に記録すること |
| REQ-0101-014 | guides は人間向けナビゲーション層であり、REQ/ADR/SPEC/DOC-MAP の内容を代替しないこと。docs/guides/ は README.md（入口）と 10 ガイド（quickstart, command-selection, req-case-flow, intake-learning-backlog-flow, diagnostics-and-maintenance, artifacts-and-state, project-docs-and-specs, consumer-project-setup, troubleshooting, glossary）で構成する |
| REQ-0101-015 | `.sisyphus/` 配下の成果物は現行REQ体系の管理対象に含めないこと |
| REQ-0101-016 | retired REQ の参照は current primary reference（現行要件判断に使われる参照）と historical reference（移行表・履歴説明・置換関係の参照）を区別して扱うこと。`mapping-table.md` 内の migration 用 retired REQ 参照は historical reference とし、現行要件判断に使用されている導線の検査対象外とすること |
| REQ-0101-017 | 利用者が現在の入力状態から次に実行すべきコマンドを選べる入口表を README またはコマンド選択ガイドに設けること |
| REQ-0101-018 | コマンド系統の説明において「本流」「支流」「合流」のような比喩的上下関係を構造名として使用しないこと |
| REQ-0101-020 | root README は identity、入口表、参照先リンク、最小限のクイックスタートに限定し、guide や command README と重複する概念説明やフロー説明を含めないこと |
| REQ-0101-021 | 対象文書に、現行要件のように読める旧 REQ 参照、非推奨 command を現行 producer とする説明、canonical ではない path 参照は含まれないこと |
| REQ-0101-022 | `.agentdev/README.md` は domain state の早見表とし、各 path について状態、producer、consumer / next command、retention / removal condition を記載すること |
| REQ-0101-023 | `.agentdev/README.md` は `.agentdev/` を永続 domain state、`.sisyphus/` を runtime workspace と明記し、working draft が canonical domain state ではないことを示すこと。ADR status 正規化形式（`status` + `superseded_by`）の記載基準を含める |
| REQ-0101-024 | 旧 `artifact-model.md` の内容は新ガイド（artifacts-and-state 等）に位置し、docs/guides/ は README.md（ガイド入口）と 10 ガイドファイルで構成される人間向けナビゲーション層であること |
| REQ-0101-025 | 旧 `domain-state-lifecycle.md` の内容は新ガイド（artifacts-and-state, diagnostics-and-maintenance 等）に位置し、旧 3 ガイド（workflow-overview, artifact-model, domain-state-lifecycle）は docs/guides/ に存在しないこと |
| REQ-0101-026 | `.opencode/commands/agentdev/README.md` は概念説明の重複を避け、各コマンドの command、primary input、primary output、next command / next action を一覧化するリファレンスとして構成すること |
| REQ-0101-027 | ガイドは規範文書ではなく人間向けナビゲーション層であり、ガイド内に 必達要件 規範表現、Pattern 詳細、REQ番号注記を含めないこと |
| REQ-0101-028 | ADR status 形式は `status: superseded` + frontmatter に `superseded_by: ADR-XXXX` とし、旧形式 `superseded-by:[ADR-XXXX]` を使用しないこと |
| REQ-0101-029 | docs 永続文書の Requirement Source セクションに RU-ID（RU-XXXX 等）を記載しないこと。RU は一時成果物であり docs の永続根拠に不適である |
| REQ-0101-030 | `.sisyphus/` は `.gitignore` に含まれ、runtime workspace として追跡対象外であること |
| REQ-0101-031 | `.agentdev/` は永続 domain state であり、配布物ではないこと。runtime command は `.agentdev/` を配布対象としないこと |
| REQ-0101-040 | active REQ 本文に見え消し記述（取り消し線）を残さない |
| REQ-0101-041 | 廃止・更新・移行の履歴を残す必要がある場合、本文ではなく Update Notes 等の履歴管理箇所に記載する |
| REQ-0101-042 | 現行判断に必要な制約・例外・境界は、見え消しではなく現行文として記述する |
| REQ-0101-043 | accepted ADRは、将来の構造・責務境界・運用原則などの「あるべき状態」を意思決定として記述すること |
| REQ-0101-044 | accepted ADRに、削除・廃止・移行・統合・再構築・完全削除そのものを主題にしたADRが存在しないこと |
| REQ-0101-045 | 過去の判断を現行基盤から外す場合、新規ADRではなく既存ADRのstatus遷移（retire/supersede）で処理すること |
| REQ-0101-046 | 新規ADRは、旧判断を置き換える新しい「あるべき状態」の意思決定が存在する場合のみ作成すること |
| REQ-0101-047 | req-defineのADR作成可否ゲートが、作業手段を主題にしたADR候補を拒否すること |
| REQ-0101-048 | agentdev-adr-guidelinesがADR作成可・不可の明示的条件を定義すること |
| REQ-0101-049 | agentdev-adr-file-managerが、単なる廃止・削除・移行を新規ADRではなくretire/supersedeで扱う基準を定義すること |
| REQ-0101-050 | docs/specsの文書種別責務境界が「ADRはwhy/decision、SPECはcurrent architecture/what is、REQは満たすべき要件」という役割分担を反映し、矛盾しないこと |
| REQ-0101-051 | ADR作成前に既存accepted ADRとの内容重複を確認し、重複時は新規ADR作成ではなく既存ADRのUPDATEを優先すること |
| REQ-0101-052 | active REQ の各ファイルが、文書統治REQ、ワークフロー全体REQ、command-level REQ、artifact/runtime/skill責務REQ、validation/inspection REQ、ADR lifecycle REQ のいずれかの分類に属し、関心対象の総体として説明できること |
| REQ-0101-053 | 新規REQ作成基準が、既存active REQに吸収できない独立関心対象に限定されていること |
| REQ-0101-054 | command-level REQ が、公開commandの入力・出力・副作用境界・停止条件・他commandとの接続を持つ単位として定義されていること |
| REQ-0101-055 | SPECに置くべき schema / lifecycle / command topology / rule catalog / fixture detail を active REQ から切り出す基準が定義されていること |
| REQ-0101-056 | retire候補判定基準が、バグ修正由来・移行完了状態・他REQ吸収済み・作業手段主題の4類型で文書化されていること |
| REQ-0101-057 | 文書分類ポリシーの恒久内容が REQ-0101 の関心対象に含まれ、独立した分類ポリシーREQの維持有無が明示的に判断されていること |
| REQ-0101-058 | 文書分類定義の一次所有先が REQ-0101 + `docs/specs/document-model.md` に統一されていること。独立した分類ポリシーREQを維持しないこと |
| REQ-0101-059 | active docs が現行仕様だけを説明し、廃止済み機能・旧構造・検討経緯を現行仕様本文として保持しないこと |
| REQ-0101-060 | docs/specs/* に過去前提・移行経緯・将来予定・未実装予定・検討過程が混入していないこと |
| REQ-0101-061 | 英語用語が文脈上の実体に基づく自然な日本語に書き直され、内部処理段階名が上位docs本文に露出していないこと |
| REQ-0101-062 | docs/README.md、docs/DOC-MAP.md、guides が入口・索引として整理され、REQ/ADR/SPEC本文を重複して保持していないこと |
| REQ-0101-063 | active docs 内の HTMLコメント（provenance marker を含む）は retired REQ ID を単独で参照しないこと。retired REQ に関連する内容は後継 active REQ へ置換するか、本文中に履歴であることを明記して後継REQへのポインタを伴うこと |
| REQ-0101-064 | REQ/ADR/SPEC の各要件行・Decision 本文・SPEC Rule は、主たる文意を肯定文で記述すること。主たる文意とは、当該記述が単独で表現する「満たすべき状態・振る舞い・制約」を指す |
| REQ-0101-065 | 否定文（「〜しない」「〜を禁止する」「〜を含めない」「〜てはならない」等）は、肯定文で記述された主たる文意に対する境界条件・例外・補足として併記する場合に限り使用すること |
| REQ-0101-066 | 既存の REQ/ADR/SPEC 記述で主たる文意が否定文のみで構成されているものは、対応する肯定文へ書き換えること |

## 適用範囲

- **対象**: REQ/retired REQ/ADR/SPEC/DOC-MAP/guides の基準境界、REQ移行表、ID規約、README / command-selection / command README / artifact-model / domain-state-lifecycle / DOC-MAP の責務境界と内容基準、`.agentdev/README.md` の早見表形式、現行文書における旧 REQ 参照・obsolete path 参照の扱い、ADR記述対象境界（意思決定のみ・作業手段除外）、REQ/ADR/SPEC の主たる文意の肯定文記述原則・境界条件としての否定文の許容範囲
- **対象外**: 各コマンドの実行手順、コード実装、`.sisyphus/` 運用、各ドキュメントの具体的な本文内容、新規ドキュメントの追加
