## 目的

ユーザー入力、クローズ済み Issue/PR からの作業候補を `.agentdev/intake/` に蓄積し、採用判定と HITL（人の判断を挟む）確定を経て採用済み成果物として要件化パイプラインに供給する。
intake / learning capture の責務境界の一次定義と、他 command の capture 非関与を含む。

## 要件

| ID | 要件 |
|---|---|
| REQ-0127-001 | intake pipeline は専用のドメイン状態を持つこと |
| REQ-0127-002 | `intake-capture` はユーザー入力から明示できる内容のみを intake 項目（item）として整理すること |
| REQ-0127-003 | `intake-promote` は inbox の intake 項目を直接読み込み、採用判定、HITL 確定を経て promoted に配置または削除すること |
| REQ-0127-004 | `intake-promote` は inbox の intake 項目を入力とし、accepted を経由せず直接処理すること |
| REQ-0127-005 | `intake-promote` は内部フェーズとしてレビュー（review）相当の採用、却下、保留判定を実行すること |
| REQ-0127-006 | `intake-promote` は HITL 確定ステップを持ち、採用済み成果物はユーザーの明示的な承認を得た後に生成すること |
| REQ-0127-007 | `intake-promote` の完了報告は採用、却下、保留の判定結果と後続経路を提示すること |
| REQ-0127-008 | intake pipeline の成果物配置先は inbox または promoted に限定すること（accepted は対象外）。intake コマンドは既存ファイルを inbox または promoted に配置すること |
| REQ-0127-009 | 通常 intake 起点の項目が `/agentdev/req-define` に渡されるのは、`/agentdev/intake-promote`、`/agentdev/backlog-review` を経て RU 化された後であること |
| REQ-0127-010 | REQ再構成intake の後続経路は通常 intake の `/agentdev/intake-promote` 経路と混同せず、REQ-0109 に基づく独立した扱いを維持すること。`intake-promote` は内部で REQ再構成intake を分離処理する |
| REQ-0127-011 | 同一観測から intake と learning の両方が発生する場合、成果物を分離すること |
| REQ-0127-012 | 単一の観測から intake と learning の両方が得られる場合、別々の成果物に分離すること（intake/learning を混ぜた単一成果物は対象外） |
| REQ-0127-013 | システムは intake / learning capture の責務境界を一次参照として定義すること |
| REQ-0127-014 | PR本文テンプレートの capture セクション名を `## Findings / Capture候補` とすること |
| REQ-0127-015 | PR本文テンプレートの `## Findings / Capture候補` セクションに `### intake` と `### learning` 小見出しを含めること |
| REQ-0127-016 | capture 候補分類の判断質問を capture 境界定義に含めること |
| REQ-0127-017 | intake / learning の代表例と非対象例を capture 境界定義に含めること |
| REQ-0127-018 | Post-Run Capture 参照先が capture 境界定義を含むこと |
| REQ-0127-019 | req-save Allowed Changes に capture 非関与（REQ再構成intake除く）が反映されていること |
| REQ-0127-020 | `req-save` の capture 生成対象は REQ再構成intake のみとすること（通常の intake / learning capture は対象外） |
| REQ-0127-021 | `case-auto` は構成コマンド（case-run / case-close）の capture 責務境界に従うこと（case-auto 固有の capture 振る舞いは対象外） |
| REQ-0127-022 | `intake-promote` において reject 判定された intake 項目は、HITL による分類確定後に削除すること。`archive/` 配下への移動は行わないこと |

## 適用範囲

- **対象**: intake-capture、intake-from-github、intake-promote（内部レビューフェーズ含む）、`.agentdev/intake/` ドメイン状態、`.agentdev/intake/accepted/` 不使用、intake / learning capture 責務境界、PR テンプレート capture セクション、capture 非関与 command の制約（req-save、case-auto）、REQ再構成intake のルーティング分離
- **対象外**: learning-promote の詳細処理（REQ-0128）、backlog-review の RU 生成（REQ-0129）、RU ライフサイクル（REQ-0105）、case-run / case-close / case-open の capture 振る舞い（REQ-0130 / 0131 / 0132）、`intake-promote` の整形ロジック、個別 intake 項目の内容、capture 境界の具体的な intake/learning 分類アルゴリズム
