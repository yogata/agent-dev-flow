## 目的

Command / Skill / Template / Script の責務境界を明確にし、重複記述・責務混線・参照不整合を防止する。

## 要件

| ID | 要件 |
|---|---|
| REQ-0103-001 | command はユーザー向け入口、入出力、ガードレール、高レベル手順を定義すること（SHALL） |
| REQ-0103-002 | command は共通判断基準や大きな状態機械を直接所有せず、skill を参照すること（SHALL） |
| REQ-0103-003 | skill は再利用可能な判断基準、共通知識、宣言的ルールを定義すること（SHALL） |
| REQ-0103-004 | skill は command 固有の Step 番号、局所的なファイルパス、エラーメッセージを一次情報として所有しないこと（SHALL） |
| REQ-0103-005 | template は Issue/PR description やコメントの本文構造、必須/任意セクション、プレースホルダーを定義すること（SHALL） |
| REQ-0103-006 | script はガードレール・検査・補助処理の決定的でテスト可能な実行ロジックを提供すること（SHALL） |
| REQ-0103-007 | command 定義ファイルは `.opencode/commands/agentdev/` に配置すること（SHALL） |
| REQ-0103-008 | skill 定義ファイルは `.opencode/skills/` に配置すること（SHALL） |
| REQ-0103-009 | AgentDevFlow の canonical namespace は `agentdev` とし、public command prefix は `/agentdev/*` とすること（SHALL） |
| REQ-0103-010 | domain state directory は `.agentdev/` とすること（SHALL） |
| REQ-0103-011 | skill naming は小文字・数字・ハイフンのみの kebab-case とすること（SHALL） |
| REQ-0103-012 | skill の frontmatter は `name` と `description` を基本構造とし、description に使用条件を明示すること（SHALL） |
| REQ-0103-013 | Skill は Progressive Disclosure を適用し、詳細は必要に応じて参照ファイルへ分離すること（SHOULD）。参照ファイルは Agent Skills 仕様に基づき `references/` ディレクトリに配置すること（SHALL）。`reference/`（単数形）は canonical ではなく obsolete structure として扱うこと |
| REQ-0103-014 | ガードレールスクリプトは該当 skill の `scripts/` 配下に配置し、command から参照されること（SHALL） |
| REQ-0103-015 | 各 agentdev command は、command の内部構造を分類する primary implementation pattern を1つ定義すること（SHALL）。primary pattern は command の主たる実行パターンを示す |
| REQ-0103-016 | implementation pattern は command の内部構造に基づく分類軸であり、REQ-0104 で定義される Pattern A/B/C/D（Issue 種別分類）とは直交する概念であること（SHALL）。Pattern A/B/C/D は Issue 種別とワークフロー分岐を示し、implementation pattern は command の責務境界と許容副作用を示す |
| REQ-0103-017 | 各 implementation pattern は主責務、許可される副作用、禁止される副作用、参照すべき skill の性質、完了報告で示すべき情報を定義すること（SHALL）。定義は lifecycle / command-map 系 reference file に集約する |
| REQ-0103-018 | command は、複数の pattern の特性を併せ持つ場合に限り、secondary implementation pattern を定義してもよい（MAY） |
| REQ-0103-019 | implementation pattern taxonomy（pattern 一覧・定義・command ↔ pattern 対応表）は lifecycle / command-map 系 reference file に集約すること（SHALL）。command 定義ファイルには primary / secondary pattern の参照のみを最小限（frontmatter または短いセクション）で持たせ、詳細説明は reference 側に置く |
| REQ-0103-020 | command の load_skills は command の implementation pattern の skill 要件と整合すること（SHOULD）。整合性の検査は integrity-check で行う |
| REQ-0103-021 | agentdev command は公開入口、Input / Output、Guardrails、高レベル Steps、固有制約、参照先に限定し、詳細な判断表・状態機械・共通手順・長い失敗時手順を直接保持しないこと（SHALL） |
| REQ-0103-022 | agentdev command は原則 100 行以内とし、150 行を実運用上限、200 行超を例外状態として扱うこと（SHOULD） |
| REQ-0103-023 | command が 150 行を超える場合、切り出し候補と残置理由を review 可能な形で記録すること（SHOULD） |
| REQ-0103-024 | command が 200 行を超える場合、orchestration skill / reference / script への切り出し、または明示的な例外理由なしに維持してはならないこと（SHALL） |
| REQ-0103-025 | command の Steps は原則 5〜12 個に保ち、細かなサブ手順は skill reference または script に移すこと（SHOULD） |
| REQ-0103-026 | deprecated command は互換入口・移行先・最小限の警告に限定し、詳細処理を保持しないこと（SHALL） |
| REQ-0103-027 | command の load_skills は当該 command の主責務を実行するために即時必要な skill に限定すること（SHALL） |
| REQ-0103-028 | 必要時にのみ参照すれば足りる skill は load_skills ではなく manual reference として扱うこと（SHOULD） |
| REQ-0103-029 | command は implementation pattern ごとの load_skills 目安上限を持つこと（SHOULD） |
| REQ-0103-030 | 初期目安上限は capture-only は 2、wall-session は 4、file-pipeline は 5、read-only-diagnostic は 3、manager-orchestrator は 8 とすること（SHOULD） |
| REQ-0103-031 | load_skills が pattern 目安上限を超える場合、超過理由と必要性を明示すること（SHOULD） |
| REQ-0103-032 | agentdev-no-ai-slop-writing は自然言語成果物の最終生成・レビュー段階で参照することを基本とし、すべての command に常時 load してはならないこと（SHOULD） |
| REQ-0103-033 | agentdev-workflow-reporting は完了報告 variant の解決と出力段階に必要な範囲で参照し、すべての variant 詳細を command 実行時に常時読ませないこと（SHOULD） |
| REQ-0103-034 | command が skill を load する場合、その skill の USE FOR / DO NOT USE FOR と command の implementation pattern が矛盾しないこと（SHALL） |
| REQ-0103-035 | SKILL.md は progressive disclosure の入口として、目的、使用条件、禁止条件、最重要ルール、参照先を中心に構成すること（SHOULD） |
| REQ-0103-036 | 長い例、詳細な判定表、スキーマ、分類表、失敗時手順、command 固有の分岐は SKILL.md 本体ではなく references/ または script に移すこと（SHOULD） |
| REQ-0103-037 | SKILL.md が 200 行を超える場合、分割候補として integrity-check または authoring review で報告すること（SHOULD） |
| REQ-0103-038 | SKILL.md が command 固有の Step 番号、局所ファイルパス、局所エラーメッセージを一次情報として保持しないこと（SHALL） |
| REQ-0103-039 | Skill reference は references/ を canonical とし、reference/ 表記や旧リンクを active guidance に残さないこと（SHALL） |
| REQ-0103-040 | 複数 command で使う共通処理（Git 同期、push、pull 後 hash 照合、push 失敗時の構造化エラー等）は共通 skill reference または script に集約すること（SHALL） |
| REQ-0103-041 | 共通処理の集約先は既存 skill への reference / script 追加を優先し、新規 skill 作成は既存 skill で責務を表現できない場合に限定すること（SHOULD） |
| REQ-0103-042 | command 側には共通処理の詳細本文ではなく、使用する共通処理名、入出力、許可パス、停止条件のみを保持すること（SHALL） |
| REQ-0103-043 | 共通処理の変更時に参照元 command を一括更新せずに挙動を変えられる構造にすること（SHOULD） |

## 適用範囲

- **対象**: command / skill / template / script の責務、配置、命名、namespace、implementation pattern 分類、pattern 別責務境界と skill 要件の参照整合性、Command 薄型化基準（行数、Steps 数、deprecated shim）、load_skills prompt budget（即時必要制限、pattern 別目安上限、常時 load 禁止）、Skill 入口カード化（SKILL.md 構成、progressive disclosure、references/ canonical）、共通処理の中央集約（Git 同期等の共通化、集約先優先順位）
- **対象外**: 個別commandの詳細手順、テストケース本文、`.sisyphus/` 運用、個別 implementation pattern の詳細定義（lifecycle reference 側に集約）、個別 command の具体的な縮小目標行数、個別 SKILL.md の具体的な分割方法、共通処理の具体的な配置先ファイルパス
