## 目的

AgentDevFlow の開発ワークフロー、コマンド間データフロー、Pattern分類、SSoT遷移を定義する。

## 要件

| ID | 要件 |
|---|---|
| REQ-0104-001 | ワークフローは壁打ち、構造的実行、レビュー完了の3つのマクロフェーズで構成すること（SHALL） |
| REQ-0104-002 | 各フェーズは入力SSoTと出力SSoTを明示すること（SHALL） |
| REQ-0104-003 | フェーズ境界を跨ぐ場合、次フェーズが参照する成果物へ情報を永続化すること（SHALL） |
| REQ-0104-004 | ワークフローパターンは Pattern A（bug）、Pattern B（feature/enhancement）、Pattern C（refactor/maintenance）、Pattern D（docs/chore）を基本とすること（SHALL） |
| REQ-0104-005 | Pattern A は原則として新規REQ作成を要求せず、Issue本文で要件を管理すること（SHALL） |
| REQ-0104-006 | Pattern B は新しい振る舞いを導入するため、必要に応じて active REQ と SPEC を更新すること（SHALL） |
| REQ-0104-007 | Epic規模の作業は親Issueと子Issueに分割し、親Issueを実行順序のSSoTとして扱うこと（SHALL） |
| REQ-0104-008 | `case-open` は要件doc、active REQ、SPEC、ADRを読み取り、GitHub Issueを出力すること（SHALL） |
| REQ-0104-009 | `case-run` は GitHub Issue と SPEC を読み取り、実装済みブランチとPRを出力すること（SHALL） |
| REQ-0104-010 | `case-close` は PR と Issue を入力とし、完了検証、merge、Issue close、後処理を実行すること（SHALL） |
| REQ-0104-011 | ワークフロー実行中のエラー回復はデータ損失を伴わないこと（SHALL） |
| REQ-0104-012 | label と template metadata が矛盾する場合、コマンドは停止して確認を求めること（SHALL） |
| REQ-0104-013 | `.sisyphus/` を現行ワークフローのSSoTまたは必須成果物として扱わないこと（SHALL） |

## 適用範囲

- **対象**: ワークフロー3フェーズ、Pattern分類、SSoT、case-open / case-run / case-close の基本契約、Epic flow
- **対象外**: `.sisyphus/` プラン基盤、個別command内部の詳細手順
