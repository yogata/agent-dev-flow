## 目的

AgentDevFlow の開発ワークフロー、コマンド間データフロー、work_type 分類、SSoT遷移を定義する。

## 要件

| ID | 要件 |
|---|---|
| REQ-0104-001 | ワークフローは壁打ち、構造的実行、レビュー完了の3つのマクロフェーズで構成すること |
| REQ-0104-002 | 各フェーズは入力SSoTと出力SSoTを明示すること |
| REQ-0104-003 | フェーズ境界を跨ぐ場合、次フェーズが参照する成果物へ情報を永続化すること |
| REQ-0104-004 | ワークフローは work_type（bugfix / feature / maintenance / docs_chore）と scale（feature のみ standard / large）で分類し、workflow_route（req_backed_case / epic_case / direct_case）を都度導出すること。workflow_route を保存・永続化しない |
| REQ-0104-005 | work_type が bugfix / maintenance / docs_chore の場合、新規REQ作成を要求せず Issue 本文で要件を管理すること |
| REQ-0104-006 | work_type が feature の場合、新しい振る舞いを導入する際、必要に応じて active REQ と SPEC が現行状態に保たれること |
| REQ-0104-007 | scale が large の feature（workflow_route: epic_case）は親 Issue と子 Issue に分割し、親 Issue を実行順序の SSoT として扱うこと |
| REQ-0104-008 | `case-open` は要件doc、active REQ、SPEC、ADR を読み取り、GitHub Issue を出力すること。Issue labels は work_type から単純マッピングして付与する |
| REQ-0104-009 | `case-run` は GitHub Issue と SPEC を読み取り、実装済みブランチとPRを出力すること。draft metadata は work_type, scale（feature のみ）, req_operation, target_req, adr_required のみを格納し、workflow_route / branch_type / labels / pattern は格納しない（これらは派生値または dev メタデータである） |
| REQ-0104-010 | `case-close` は PR と Issue を入力とし、完了検証、merge、Issue close、後処理を実行すること |
| REQ-0104-011 | ワークフロー実行中のエラー回復はデータ損失を伴わないこと |
| REQ-0104-012 | work_type 由来の label と template metadata が矛盾する場合、コマンドは停止して確認を求めること |
| REQ-0104-013 | `.sisyphus/` を現行ワークフローの runtime workspace として扱い、SSoT・必須成果物は対象外とすること |
| REQ-0104-014 | work_type は bugfix（バグ修正・軽微変更）、feature（機能追加）、maintenance（リファクタリング・保守作業）、docs_chore（ドキュメント・雑務）の 4 種とすること |
| REQ-0104-015 | workflow_route 派生ルール: feature+standard → req_backed_case、feature+large → epic_case、bugfix/maintenance/docs_chore → direct_case とすること |
| REQ-0104-016 | draft-meta における pattern, workflow_route, branch_type, labels は保存対象外とすること（派生値または dev メタデータのため） |
| REQ-0104-017 | AgentDevFlow workflow の実装対象は現在のプロジェクトであること |
| REQ-0104-018 | AgentDevFlow 適用プロジェクトでは AgentDevFlow 本体・配布 command・配布 skill・配布 template・配布 script を upstream handoff 対象として扱うこと |
| REQ-0104-019 | AgentDevFlow 適用プロジェクトで AgentDevFlow 本体の不具合・改善点を発見した場合、現在プロジェクトでは適用せず、upstream handoff 用 RU として整理すること |
| REQ-0104-020 | agent-dev-flow repository では AgentDevFlow 本体が現在プロジェクトの成果物であるため、通常の req/case workflow で改修対象にできること |
| REQ-0104-021 | upstream handoff metadata として `agentdev_handoff: true` を RU frontmatter に記録できること。`agentdev_handoff: true` は AgentDevFlow 本体向け handoff marker であること。`handoff_target` および `apply_in_current_project` に代わり `agentdev_handoff: true` を使用すること |
| REQ-0104-022 | `req-define` は AgentDevFlow 本体改善要求を upstream handoff 用 RU 入力として整理すること。現在プロジェクトの通常要件docは対象外とする。出力には `agentdev_handoff: true` を含めること |
| REQ-0104-023 | `backlog-review` は AgentDevFlow 本体改善 artifact に upstream handoff marker として `agentdev_handoff: true` を付与すること |
| REQ-0104-024 | `backlog-review` は `agentdev_handoff: true` を RU frontmatter に転記すること。RU 本文に現在プロジェクトでは実装しない upstream handoff 用 RU であることを記載すること |
| REQ-0104-025 | `case-open` は consumer repository において、要件docまたは RU に `agentdev_handoff: true` がある場合、停止して agent-dev-flow repository への手動取り込み対象として報告すること。Issue 作成は対象外とすること |
| REQ-0104-026 | `case-run` は consumer repository において、Issue 本文、要件doc、Requirement Source に `agentdev_handoff: true` がある場合、停止して agent-dev-flow repository への手動取り込み対象として報告すること。実装の開始は対象外とすること |
| REQ-0104-027 | 各 command は upstream handoff の共通方針を `agentdev-workflow-lifecycle/references/upstream-handoff.md` に集約し、command 側の長文の重複実装を references への参照のみに限定すること |
| REQ-0104-028 | `case-open` は単一 OU の場合、当該 OU 対応 Issue を作成すること |
| REQ-0104-029 | `case-open` は execution_groups が提案されている場合、execution_groups の検証（REQ-0104-039）を経て Epic Issue を作成すること |
| REQ-0104-030 | Epic配下には operation unit 単位で子Issueを作成すること |
| REQ-0104-031 | 必要な場合のみ、子Issue配下に孫Issueを作成すること |
| REQ-0104-032 | wave候補または依存関係がある場合、Epic本文または子Issue本文に反映すること |
| REQ-0104-033 | 子Issue構成は OU 単位で作成し、OU単位のトレーサビリティを維持すること。wave単位のみの子Issue構成は対象外とする |
| REQ-0104-034 | bugfix 実行時も、影響を受ける docs（REQ/ADR/SPEC/guide）の更新を完了条件に含めること。REQ作成を不要とする bugfix の軽量経路であっても、既存ドキュメントとの整合性維持は免除されない |
| REQ-0104-035 | case-open は OU ID 指定時に指定 OU の req-save result を読み、その OU だけを Issue 化すること |
| REQ-0104-036 | case-open は OU ID 指定なしで draft に OU が 1 件の場合、その OU を自動選択して処理すること |
| REQ-0104-037 | case-open は OU ID 指定なしで draft に OU が 2 件以上の場合、OU 一覧を表示して停止すること |
| REQ-0104-038 | case-open は作成した Issue / Epic 番号を draft の当該 OU `result` に書き戻すこと |
| REQ-0104-039 | case-open は draft の `execution_groups` を読み、Epic 候補の根拠を検証すること。根拠が明示されていれば Epic / child Issue を作成し、根拠不足・矛盾時は Epic 化せず停止または単体 Issue 化すること |
| REQ-0104-040 | case-open は `execution_groups` の提案に基づいて Epic Issue を作成し、独自推論による作成は対象外とすること |
| REQ-0104-041 | case-open は複数 OU が存在する場合でも `execution_groups` の検証を経て Epic Issue を作成すること |
| REQ-0104-042 | case-open の Issue 化単位は REQ doc 単位ではなく OU 単位とすること |
| REQ-0104-043 | `agentdev_handoff` は通常 RU / 要件doc の必須 frontmatter ではないこと。通常 RU には `agentdev_handoff` を記載せず、`agentdev_handoff: false` も記載しないこと |
| REQ-0104-044 | agent-dev-flow 本体 repository では `agentdev_handoff: true` を停止条件として扱わず、取り込み元の履歴 metadata として扱い、通常の req/case workflow 入力として処理すること |
| REQ-0104-045 | case-open の共通終了処理（コメント追加・ドラフト削除・RU削除・完了報告）が Standard flow と Epic flow の全フロー共通処理として独立セクションに分離されていること |
| REQ-0104-046 | case-open の Epic flow セクション完了後に共通終了処理を実行する旨の明示的ルーティングがあること |
| REQ-0104-047 | case-open の Step 18（ドラフト削除）・18-1（RU削除）に Standard / Epic 全フロー共通である旨の注記があること |
| REQ-0104-048 | 全公開commandが、主フロー・最大自走入口・補助フロー・検出フロー・repo-local検査のいずれかに分類されていること |
| REQ-0104-049 | case-auto が標準フローを置換しない追加入口として位置づけられていること |
| REQ-0104-050 | REQ-0123 の題名・内容・関心対象が一致し、workflow-lifecycle 責務分界またはskill配置責務として読めること |
| REQ-0104-051 | workflow-contracts.md と REQ 側の説明に矛盾がないこと |
| REQ-0104-052 | REQ が保証・制約・責務境界を持ち、詳細なI/O表やphase説明が SPEC 側に置かれていること |

## 適用範囲

- **対象**: ワークフロー3フェーズ、work_type + scale 分類、workflow_route 派生、SSoT、case-open / case-run / case-close の基本契約、case-open 共通終了処理の全フロー共通化、Epic flow（epic_case）、upstream handoff workflow protocol、`agentdev_handoff: true` metadata convention、req-define / backlog-review / case-open / case-run の handoff 停止条件、`agentdev-workflow-lifecycle/references/upstream-handoff.md` の参照（共通方針集約先）
- **対象外**: `.sisyphus/` プラン基盤、個別command内部の詳細手順、新規 guard skill の作成、repo_scope / self-hosting 判定 script、git diff ベースの AgentDevFlow managed artifact 検査、全 command への長文ルール複製、適用プロジェクトでの AgentDevFlow 本体改修の自動適用、`/repo/intake-*` または `/repo/learning-*` の新設
