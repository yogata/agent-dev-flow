## 目的

AgentDevFlow の開発ワークフロー、コマンド間データフロー、work_type 分類、SSoT遷移を定義する。

## 要件

| ID | 要件 |
|---|---|
| REQ-0104-001 | ワークフローは壁打ち、構造的実行、レビュー完了の3つのマクロフェーズで構成すること（SHALL） |
| REQ-0104-002 | 各フェーズは入力SSoTと出力SSoTを明示すること（SHALL） |
| REQ-0104-003 | フェーズ境界を跨ぐ場合、次フェーズが参照する成果物へ情報を永続化すること（SHALL） |
| REQ-0104-004 | ワークフローは work_type（bugfix / feature / maintenance / docs_chore）と scale（feature のみ standard / large）で分類し、workflow_route（req_backed_case / epic_case / direct_case）を都度導出すること（SHALL）。workflow_route を保存・永続化しない |
| REQ-0104-005 | work_type が bugfix / maintenance / docs_chore の場合、新規REQ作成を要求せず Issue 本文で要件を管理すること（SHALL） |
| REQ-0104-006 | work_type が feature の場合、新しい振る舞いを導入するため必要に応じて active REQ と SPEC を更新すること（SHALL） |
| REQ-0104-007 | scale が large の feature（workflow_route: epic_case）は親 Issue と子 Issue に分割し、親 Issue を実行順序の SSoT として扱うこと（SHALL） |
| REQ-0104-008 | `case-open` は要件doc、active REQ、SPEC、ADR を読み取り、GitHub Issue を出力すること（SHALL）。Issue labels は work_type から単純マッピングして付与する |
| REQ-0104-009 | `case-run` は GitHub Issue と SPEC を読み取り、実装済みブランチとPRを出力すること（SHALL）。draft metadata は work_type, scale（feature のみ）, req_operation, target_req, adr_required のみを格納し、workflow_route / branch_type / labels / implementation_pattern / secondary_pattern / load_skills / pattern は格納しない（これらは派生値または dev メタデータである） |
| REQ-0104-010 | `case-close` は PR と Issue を入力とし、完了検証、merge、Issue close、後処理を実行すること（SHALL） |
| REQ-0104-011 | ワークフロー実行中のエラー回復はデータ損失を伴わないこと（SHALL） |
| REQ-0104-012 | work_type 由来の label と template metadata が矛盾する場合、コマンドは停止して確認を求めること（SHALL） |
| REQ-0104-013 | `.sisyphus/` を現行ワークフローのSSoTまたは必須成果物として扱わないこと（SHALL） |
| REQ-0104-014 | work_type は bugfix（バグ修正・軽微変更）、feature（機能追加）、maintenance（リファクタリング・保守作業）、docs_chore（ドキュメント・雑務）の 4 種とすること（SHALL） |
| REQ-0104-015 | workflow_route 派生ルール: feature+standard → req_backed_case、feature+large → epic_case、bugfix/maintenance/docs_chore → direct_case とすること（SHALL） |
| REQ-0104-016 | draft-meta に implementation_pattern, secondary_pattern, load_skills, pattern, workflow_route, branch_type, labels を保存しないこと（SHALL） |
| REQ-0104-017 | AgentDevFlow workflow の実装対象は現在のプロジェクトであること（SHALL） |
| REQ-0104-018 | AgentDevFlow 適用プロジェクトでは AgentDevFlow 本体・配布 command・配布 skill・配布 template・配布 script を直接改修対象にしないこと（SHALL） |
| REQ-0104-019 | AgentDevFlow 適用プロジェクトで AgentDevFlow 本体の不具合・改善点を発見した場合、現在プロジェクトでは適用せず、upstream handoff 用 RU として整理すること（SHALL） |
| REQ-0104-020 | agent-dev-flow repository では AgentDevFlow 本体が現在プロジェクトの成果物であるため、通常の req/case workflow で改修対象にできること（SHALL） |
| REQ-0104-021 | upstream handoff metadata として `handoff_target` と `apply_in_current_project` を RU frontmatter に記録できること（SHALL）。基本値は `handoff_target: agent-dev-flow`, `apply_in_current_project: false` とすること |
| REQ-0104-022 | `req-define` は AgentDevFlow 本体改善要求を現在プロジェクトの通常要件docとして定義せず、upstream handoff 用 RU 入力として整理すること（SHALL）。出力には `handoff_target: agent-dev-flow` と `apply_in_current_project: false` を含めること（SHALL） |
| REQ-0104-023 | `backlog-review` は AgentDevFlow 本体改善 artifact に upstream handoff metadata（`handoff_target: agent-dev-flow`, `apply_in_current_project: false`）を付与すること（SHALL） |
| REQ-0104-024 | `backlog-review` は upstream handoff metadata を RU frontmatter に転記すること（SHALL）。RU 本文に現在プロジェクトでは実装しない upstream handoff 用 RU であることを記載すること（SHALL） |
| REQ-0104-025 | `case-open` は要件docまたは RU に `apply_in_current_project: false` がある場合、Issue を作成せず停止すること（SHALL）。agent-dev-flow repository への手動取り込み対象として報告すること（SHALL） |
| REQ-0104-026 | `case-run` は Issue 本文、要件doc、Requirement Source に `apply_in_current_project: false` がある場合、実装を開始せず停止すること（SHALL）。agent-dev-flow repository への手動取り込み対象として報告すること（SHALL） |
| REQ-0104-027 | 各 command は upstream handoff の共通方針を `agentdev-workflow-lifecycle/references/upstream-handoff.md` に集約し、command 側に長文の重複実装を持たないこと（SHALL） |
| REQ-0104-028 | `case-open` は単一REQ docの場合、既存どおりREQ doc対応Issueを作成すること（SHALL） |
| REQ-0104-029 | `case-open` は複数REQ docまたは `scale: large` の場合、Epic Issueを作成すること（SHALL） |
| REQ-0104-030 | Epic配下にはREQ doc単位で子Issueを作成すること（SHALL） |
| REQ-0104-031 | 必要な場合のみ、子Issue配下に孫Issueを作成すること |
| REQ-0104-032 | wave候補または依存関係がある場合、Epic本文または子Issue本文に反映すること（SHALL） |
| REQ-0104-033 | REQ doc単位のトレーサビリティを崩してwave単位だけの子Issue構成にはしないこと（SHALL） |
| REQ-0104-034 | bugfix 実行時も、影響を受ける docs（REQ/ADR/SPEC/guide）の更新を完了条件に含めること（SHALL）。REQ作成を不要とする bugfix の軽量経路であっても、既存ドキュメントとの整合性維持は免除されない |

## 適用範囲

- **対象**: ワークフロー3フェーズ、work_type + scale 分類、workflow_route 派生、SSoT、case-open / case-run / case-close の基本契約、Epic flow（epic_case）、upstream handoff workflow protocol、`apply_in_current_project: false` metadata convention、req-define / backlog-review / case-open / case-run の handoff 停止条件、`agentdev-workflow-lifecycle/references/upstream-handoff.md` の追加・参照
- **対象外**: `.sisyphus/` プラン基盤、個別command内部の詳細手順、新規 guard skill の作成、repo_scope / self-hosting 判定 script、git diff ベースの AgentDevFlow managed artifact 検査、全 command への長文ルール複製、適用プロジェクトでの AgentDevFlow 本体改修の自動適用、`/repo/intake-*` または `/repo/learning-*` の新設
