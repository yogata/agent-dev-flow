## 目的

AgentDevFlow の開発ワークフロー、コマンド間データフロー、work_type 分類、SSoT遷移を定義する。

## 要件

| ID | 要件 |
|---|---|
| REQ-0104-001 | ワークフローは壁打ち、構造的実行、レビュー完了の3つのマクロフェーズで構成すること（SHALL） |
| REQ-0104-002 | 各フェーズは入力SSoTと出力SSoTを明示すること（SHALL） |
| REQ-0104-003 | フェーズ境界を跨ぐ場合、次フェーズが参照する成果物へ情報を永続化すること（SHALL） |
| REQ-0104-004 | ワークフローは work_type（bugfix / feature / maintenance / docs_chore）と scale（feature のみ standard / large）で分類し、workflow_route（req_backed_case / epic_case / direct_case）を都度導出すること（SHALL）。workflow_route を保存・永続化しない |
| REQ-0104-005 | work_type が bugfix / maintenance / docs_chore の場合、新規REQ作成を要求せず Issue 本文で要件を管理すること（SHALL） |
| REQ-0104-006 | work_type が feature の場合、新しい振る舞いを導入するため必要に応じて active REQ と SPEC を更新すること（SHALL） |
| REQ-0104-007 | scale が large の feature（workflow_route: epic_case）は親 Issue と子 Issue に分割し、親 Issue を実行順序の SSoT として扱うこと（SHALL） |
| REQ-0104-008 | `case-open` は要件doc、active REQ、SPEC、ADR を読み取り、GitHub Issue を出力すること（SHALL）。Issue labels は work_type から単純マッピングして付与する |
| REQ-0104-009 | `case-run` は GitHub Issue と SPEC を読み取り、実装済みブランチとPRを出力すること（SHALL）。draft metadata は work_type, scale（feature のみ）, req_operation, target_req, adr_required のみを格納し、workflow_route / branch_type / labels / implementation_pattern / secondary_pattern / load_skills / pattern は格納しない（これらは派生値または dev メタデータである） |
| REQ-0104-010 | `case-close` は PR と Issue を入力とし、完了検証、merge、Issue close、後処理を実行すること（SHALL） |
| REQ-0104-011 | ワークフロー実行中のエラー回復はデータ損失を伴わないこと（SHALL） |
| REQ-0104-012 | work_type 由来の label と template metadata が矛盾する場合、コマンドは停止して確認を求めること（SHALL） |
| REQ-0104-013 | `.sisyphus/` を現行ワークフローのSSoTまたは必須成果物として扱わないこと（SHALL） |
| REQ-0104-014 | work_type は bugfix（バグ修正・軽微変更）、feature（機能追加）、maintenance（リファクタリング・保守作業）、docs_chore（ドキュメント・雑務）の 4 種とすること（SHALL） |
| REQ-0104-015 | workflow_route 派生ルール: feature+standard → req_backed_case、feature+large → epic_case、bugfix/maintenance/docs_chore → direct_case とすること（SHALL） |
| REQ-0104-016 | draft-meta に implementation_pattern, secondary_pattern, load_skills, pattern, workflow_route, branch_type, labels を保存しないこと（SHALL） |

## 適用範囲

- **対象**: ワークフロー3フェーズ、work_type + scale 分類、workflow_route 派生、SSoT、case-open / case-run / case-close の基本契約、Epic flow（epic_case）
- **対象外**: `.sisyphus/` プラン基盤、個別command内部の詳細手順
