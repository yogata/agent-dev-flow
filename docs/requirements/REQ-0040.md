> **新基準REQに移行した範囲**: case-run への Step 追加・検証制約（REQ-0040-002〜004, 006）は REQ-F候補に移行。workflow-orchestration のサブエージェント指示（REQ-0040-006, 012）は REQ-D候補に移行。テンプレート定義の更新（REQ-0040-001）は REQ-G候補に移行
> **後継REQ**: REQ-F候補、REQ-G候補、REQ-D候補
> **履歴として残す理由**: PR 本文を intake/learning の永続チャネルとする設計経緯

## 目的

case-run で発見した本筋外 Finding（intake 候補・learning 候補）を、PR 本文の「## Findings / Intake候補」セクションに永続化する。case-close の Step 9b が PR 本文から回収し、Epic 時は子 Issue PR 群を横断回収する。並列実行時の `.agentdev/intake/inbox/` コンフリクトを回避しつつ、本筋外発見を確実に後続パイプラインに渡す。

本要件は REQ-0019（intake / learning 責任境界）の退避・回収方針（009-020）を実装する永続チャネルメカニズムを追加する。REQ-0019 は「何を退避・回収するか」、本要件は「PR 本文を用いてどのように永続化・回収するか」。

## 適用範囲

### 対象

- pr_desc.md（PR本文テンプレート）
- workflow-templates/SKILL.md（テンプレート定義）
- case-run.md（Step 10.5追加、Step 11b制約追加）
- case-close.md（Step 9b PR回収、Epic横断回収）
- workflow-orchestration/SKILL.md（サブエージェント指示、親エージェント集約）
- capture-boundaries.md（PR本文永続チャネル定義）
- completion-reports.md（報告フォーマット追加）

### 対象外

- intake自動Issue化（intake-review → intake-promote → req-backlog の既存フローで対応）
- intake候補の採用・却下判断（intake-reviewの責務）
- PR本文＝intake item本体（case-closeがPR本文から抽出してintake item形式に変換）
- subagent一時ログの取得元特定（本要件はPR本文セクションへの記録が対象）
- 複数PRから直接inbox変更（case-closeが単一PRまたはEpic単位で回収）
