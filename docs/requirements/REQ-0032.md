## 目的

case-close コマンドの完了ゲートにおいて、Issue本文に未チェックのチェックボックスが残っている場合の挙動を定義する。5条件プロトコルに基づく根拠付き判定を行い、達成可能な項目は `[x]` に更新、達成不可項目は自律解離判定（検証適用不能 / 代替検証実施）を経て解消を試みる。自律解決也不能な項目のみユーザーアクションを求める構造化エラーで停止する。

## 達成判定方針

case-close の未チェック項目達成判定は、エージェントの自律判断で行う。

ただし、case-close は根拠なしにチェックボックスを `[x]` に更新してはならない。
各項目について、以下を満たす場合のみ達成済みと判定する。

1. チェックボックス文言が要求している完了状態を特定できること
2. PR本文、PR差分、コミット内容、CI結果、Issue/PRコメント、実装済みファイルのいずれかに対応する根拠があること
3. 根拠がチェックボックス文言と直接対応していること
4. 反証または未対応を示す情報がないこと
5. 判定理由を完了コメントまたは内部処理ログ相当の出力に記録できること

上記を満たさない場合は自律解決判定へ進む。

### 自律解決判定フロー

unchecked 項目検出
→ 5条件プロトコルで達成判定
→ 達成可能なら `[x]` 更新
→ 達成不可なら自律解決判定
   → **検証適用不能**: 変更対象と検証種別の対応関係から、要求された検証が当該 PR diff の変更に対して直接証拠にならないことを説明できる場合 → `[x]` 更新（適用不能理由を記録）
   → **代替検証実施**: エージェントが代替手段で検証を実施し、直接証拠を得られる場合 → 代替検証実施 → `[x]` 更新
   → **自律解決不能**: 上記いずれでもない場合 → `[ ]` のまま G08 停止

### 自律解決の判定基準

自律解決可否の判定では、少なくとも以下を項目単位で確認する。

1. **PR diff の変更対象を分類**:
   - docs
   - command definition
   - skill definition
   - script/code
   - test
   - config/CI
   - generated artifact

2. **チェックボックスが要求する検証種別を分類**:
   - E2E
   - unit/integration test
   - static inspection
   - document consistency
   - diff/file existence
   - CI result
   - manual/business judgment

3. **変更対象と検証種別の対応関係を説明**

4. **検証が論理的に適用不能な場合、その理由を記録**

5. **代替検証が可能な場合、代替検証の実施内容・証拠・判定結果を記録**

### 判定例

| 未チェック項目 | 自律解決タイプ | 判定根拠 |
|---|---|---|
| E2E テスト: case-run 実行で関連ドキュメント更新候補が Step 10 対象に含まれること | 検証適用不能 | PR diff は command/REQ 文書変更のみ。実行可能コードの変更なし。E2E は本 PR の変更対象に対する直接証拠にならない。代替として REQ-command 整合性確認を実施 |
| テストが通っていること | — (5条件で達成) | CI結果、ローカルテスト結果 |
| 7つの views ファイルが作成されていること | — (5条件で達成) | PR差分、実ファイル一覧 |
| ビジネスロジックの正確性確認 | 自律解決不能 | ビジネス判断が必要 |

### 適用不能判定の具体例

```
例:
- PR diff が docs/requirements/REQ-0032.md と .opencode/commands/agentdev/case-close.md のみ
- 変更内容は case-close の達成判定手順の文書定義であり、実行可能コードは変更していない
- unchecked 項目が「E2E テスト実行」を要求
- この変更に対する主検証は、E2E ではなく、REQ と command 定義の整合性確認、G08 手順の静的確認、完了ゲート文言の確認
- よって E2E は本 PR の変更対象に対して適用不能
```

**注意**: 「不要」とは書かない。書くべきなのは「このチェックボックスが要求する検証は、この PR diff の変更対象に対して直接証拠にならない」である。

## 記録要件

### 自律解決時の記録先

| 状態 | 記録先 |
|---|---|
| 自律解決して close 継続 | Issue 完了コメントに必須記録 |
| 自律解決した件数の要約 | case-close 完了報告に任意記録 |
| 自律解決不能で停止 | 構造化エラーに必須記録 |
| Issue本文チェックボックス更新 | `[x]` 更新時、文言に理由を含める場合あり（REQ-0032-021） |

### 自律解決時の完了コメント記録内容

`case-close` は、自律解決経路により unchecked 項目を解消した場合、Issue の完了コメントに以下を項目単位で記録する:

- 対象チェックボックス文言
- 5条件プロトコルで達成不可となった理由
- 自律解決タイプ（検証適用不能 / 代替検証実施）
- 判断根拠
- 実施した代替検証または適用不能判定の内容
- 最終判定
- Issue本文上の更新結果

### 自律解決不能時の構造化エラー内容

`case-close` は、自律解決できない unchecked 項目が残る場合、G08 の構造化エラーに以下を含めて停止する:

- 対象チェックボックス文言
- 5条件プロトコルで達成不可となった理由
- 自律解決不能と判定した理由
- 必要なユーザーアクション

## 期待する処理順序

1. Issue本文を取得する
2. 完了条件・テスト戦略セクションからチェックボックスを抽出する
3. 未チェック項目がある場合、即停止せず、項目ごとに達成根拠を探索する
4. 達成根拠がある項目を `[x]` に更新する
5. 達成根拠がない項目は自律解決判定に進む
6. 自律解決判定で解消できた項目を `[x]` に更新する（適用不能理由を含む）
7. 自律解決也不能な項目は `[ ]` のまま残す
8. Issue本文を更新する
9. Issue本文を再取得する
10. 未チェック項目が残っていないことを確認する
11. 未チェック項目が残っている場合は構造化エラーで停止する
12. 未チェック項目が残っていない場合のみ、PR merge / Issue close に進む

## 構造化エラー要件

達成判定・自律解決判定後も未チェック項目が残る場合、以下の情報を表示して停止する。

```text
## 完了ゲートエラー: 未達チェックボックス検出

**Issue**: #{N}
**未達項目数**: {count}件
**未達項目一覧**:
{未達項目のリスト}

**停止理由**:
達成根拠を確認できず、自律解決も不能な未チェック項目が残っています。

**自律解決判定結果**:
{項目単位の判定詳細: 5条件未充足理由・自律解決不能理由}

**ユーザーアクション**:
未達項目を実装・検証するか、不要な項目であればIssue本文から削除してください。
```

## 適用範囲

- **対象**:
  - `case-close.md` の Step 2 の達成判定に自律解決経路を追加
  - G08 完了ゲートの構造化エラーメッセージの更新
  - 自律解決判定プロトコルの追加
  - 完了コメント記録要件の追加
  - 達成判定方針・処理順序の更新
- **対象外**:
  - case-close が根拠なしに未チェック項目を完了扱いすること
  - 未達項目を intake item や learning item に逃がして Issue を完了扱いすること
  - case-run のチェックボックス更新責務の削除
  - Issue本文の完了条件・テスト戦略そのものの勝手な削除
  - 達成判定のキーワードベース固定ルール化
  - 自律解決判定の緩やかな「エージェントの技術的判断に委ねる」アプローチ（変更対象分類 × 検証種別分類の明示的分類が必須）
  - 「不要」という表現による適用不能判定（「直接証拠にならない」理由の明示が必須）
