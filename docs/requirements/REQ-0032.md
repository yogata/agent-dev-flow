## 目的

case-close コマンドの完了ゲートにおいて、Issue本文に未チェックのチェックボックスが残っている場合の挙動を再定義する。現行では Step 2 が未チェック項目を検出すると即時エラー停止し、Step 5（達成済み項目の `[x]` 反映）に到達できない。これを解消し、達成判定プロトコルに基づく根拠付き判定 → 未達項目のみ停止というフローに変更する。

## 要件

### 達成判定フロー（Step 再構成）

| ID | 要件 |
|---|---|
| REQ-0032-001 | `case-close` は、Issue本文の完了条件・テスト戦略に未チェック項目がある場合、即時停止してはならない（SHALL） |
| REQ-0032-002 | `case-close` は、未チェック項目ごとに達成根拠を探索し、達成済みと判断できる項目を `[x]` に更新すること（SHALL） |
| REQ-0032-003 | `case-close.md` の Step 2 は、未チェック検出時に即停止する記述から、未チェック項目の達成判定へ進む記述に変更すること（SHALL） |
| REQ-0032-004 | `case-close.md` の Step 5 は、到達不能な後続処理ではなく、完了ゲート前の達成反映処理として再配置または再定義すること（SHALL） |

## 達成判定方針

case-close の未チェック項目達成判定は、エージェントの自律判断で行う。

ただし、case-close は根拠なしにチェックボックスを `[x]` に更新してはならない。
各項目について、以下を満たす場合のみ達成済みと判定する。

1. チェックボックス文言が要求している完了状態を特定できること
2. PR本文、PR差分、コミット内容、CI結果、Issue/PRコメント、実装済みファイルのいずれかに対応する根拠があること
3. 根拠がチェックボックス文言と直接対応していること
4. 反証または未対応を示す情報がないこと
5. 判定理由を完了コメントまたは内部処理ログ相当の出力に記録できること

上記を満たさない場合は `[ ]` のまま残し、達成判定後の完了ゲートで停止する。

### 判定例

| 未チェック項目 | `[x]` にしてよい根拠 | CIだけで足りるか |
|---|---|---|
| テストが通っていること | CI結果、ローカルテスト結果 | 足りる |
| 7つの views ファイルが作成されていること | PR差分、実ファイル一覧 | 足りない |
| README に views セクションがあること | PR差分、実ファイル内容 | 足りない |
| ADR本文を移動していないこと | PR差分でADR本文移動がないこと | 足りない |
| Accepted済みADRのDecision本文に意味変更がないこと | PR差分で該当箇所変更なし、または変更内容確認 | 足りない |

## 期待する処理順序

1. Issue本文を取得する
2. 完了条件・テスト戦略セクションからチェックボックスを抽出する
3. 未チェック項目がある場合、即停止せず、項目ごとに達成根拠を探索する
4. 達成根拠がある項目を `[x]` に更新する
5. 達成根拠がない項目は `[ ]` のまま残す
6. Issue本文を更新する
7. Issue本文を再取得する
8. 未チェック項目が残っていないことを確認する
9. 未チェック項目が残っている場合は構造化エラーで停止する
10. 未チェック項目が残っていない場合のみ、PR merge / Issue close に進む

## 構造化エラー要件

達成判定後も未チェック項目が残る場合、以下の情報を表示して停止する。

```text
## 完了ゲートエラー: 未達チェックボックス検出

**Issue**: #{N}
**未達項目数**: {count}件
**未達項目一覧**:
{未達項目のリスト}

**停止理由**:
達成根拠を確認できない未チェック項目が残っています。

**ユーザーアクション**:
未達項目を実装・検証するか、不要な項目であればIssue本文から削除してください。
```

## 適用範囲

- **対象**:
  - `case-close.md` の Step 2（前提確認）と Step 5（テスト戦略チェックボックス更新）の再構成
  - G08（完了ゲート品質ゲート）の再定義
  - 達成判定プロトコルの追加
  - 構造化エラーメッセージの更新
- **対象外**:
  - case-close が根拠なしに未チェック項目を完了扱いすること
  - 未達項目を intake item や learning item に逃がして Issue を完了扱いすること
  - case-run のチェックボックス更新責務の削除
  - Issue本文の完了条件・テスト戦略そのものの勝手な削除
  - 達成判定のキーワードベース固定ルール化
